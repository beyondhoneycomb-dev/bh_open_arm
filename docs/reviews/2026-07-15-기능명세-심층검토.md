# OpenArm v2.0 기능명세 심층 검토 보고서

**검토 방식**: 교차-계열 삼각검증. (1) Claude 병렬 워크플로우(15문서 추출 → 교차정합성 매핑, 16 에이전트), (2) Codex(gpt-5.6-sol, xhigh, blind, 자체 웹검증), (3) 표적 웹검증. 두 축은 서로를 못 본 채 독립 검토 후 대조.
**범위**: 기능 명세의 기술적 타당성·내부 정합성·완결성·근거 건전성. **기획(일정·비용·시장)은 배제.**
**날짜**: 2026-07-15.

> 두 계열(Claude/Codex)이 **독립적으로 같은 결론**에 도달한 항목은 최고 신뢰도(★수렴)로 표기. 한쪽만 발견한 항목은 상호 보완.

## 종합 판정

명세는 **예외적으로 근거가 탄탄하나(3차 반증검증), 구현 착수 전 반드시 닫아야 할 구조적 BLOCKER가 남아 있다.** 대부분 "조용한 실패"이며 세 뿌리로 수렴한다:
1. **주파수/CAN 버스 예산** — 16 vs 32 프레임/사이클이 미판별인데 문서마다 반대 결론을 확정으로 씀 → 전 성능 NFR이 흔들림.
2. **영점/좌표계 SoT + 정지 의미론** — connect()/set_zero 계약이 문서마다 상충 + 홀드/E-Stop 의미가 안전표준과 어긋남 + 무브레이크 낙하 경로.
3. **버전·하드웨어 실존 종속** — LeRobot "0.6.1"(유령 버전) + Pibiger 어댑터(실존 의심) 위에 명세 절반이 얹힘.

## BLOCKER

### B1. LeRobot `v0.6.1`은 존재하지 않는 버전 ★수렴(4중: Claude/Codex/WebSearch/sim-research)
- **위치**: 00 §2·§3.1, 14 §2.11/FR-OPS-089, 16 §3.1, 09 §6, 그리고 "v0.6.1 스냅샷" 전 문서.
- **문제**: 공개 최신 = **0.6.0**(2026-07-06), 다음 = 0.7.0. PyPI·GitHub 태그에 0.6.1 없음. `main`의 pyproject만 0.6.1로 자칭. → pip/태그 재현 설치·라인번호 인용·공급망 고정 불성립.
- **Codex 해소**: 0.6.1의 근거로 든 `lerobot-rollout`·RealSense depth는 **이미 0.6.0에 존재** → 기능은 릴리스됨, 버전 라벨만 틀림.
- **근거**: github.com/huggingface/lerobot/releases/tag/v0.6.0, .../blob/v0.6.0/pyproject.toml, PyPI.
- **권고**: 정본을 **v0.6.0 또는 commit SHA + wheel hash**로 교체하고 소스 인용·회귀테스트를 SHA 기준으로 재생성.

### B2. 모터 영점 ↔ URDF 좌표계 단일 진실원(SoT) 부재 [Codex 심화 + Claude C-02 수렴]
- **위치**: 00 §3.2, 02 FR-CON-061/063, 03 FR-MOT-045, 14 §2.5/FR-OPS-058, 16 M-1/M-20; 상태표 04/11/12.
- **문제**: (a) 한 문서는 0xFE 영점이 전원사이클마다 휘발이라 하고 다른 곳은 NV 영속 전제 → connect()/set_zero 자동송신을 **01/04/11/12 상태표는 전제**, **02/03/12/14는 오버라이드 제거를 필수**로 규정 = 같은 이벤트 정반대 계약(상태표대로면 세션마다 영점 파괴). (b) 디스크 캘리브에 **모터영점 오프셋·motor↔URDF 프레임 변환이 없음** → 눈으로 rest 자세 재현만으론 비대칭 URDF 리밋(j2 −10~190°, j6 ±45°)·충돌검사가 동일 좌표계라는 보장 없음.
- **권고**: 관절별 `{motor_zero_raw, sign, scale, urdf_zero_offset, gripper_endpoints}` + 버전·체크섬 캘리브 계약. 영점 지그/기준면·허용오차·readback·전원재인가 시험 필수화. 모든 connect()/상태표의 영점 동작 단일화.

### B3. `STOP_HOLD`를 IEC 60204-1 Cat-2 E-Stop / safety-rated로 오정의 [Codex 고유]
- **위치**: 03 FR-MOT-043/044, 05 FR-TEL-086, 11 FR-INF-045, 12 §2.10/FR-SAF-053~056, 14 FR-OPS-038.
- **문제**: Cat-2는 전력 유지 controlled stop이지 비상정지 범주가 아니다(E-Stop은 Cat 0/1). 또 Python/CAN 소프트 홀드는 인증 safety-rated 시스템이 아님(12 Q13 스스로 인정) — FR-SAF-056과 충돌. `disable_all()` 토크OFF를 soft stop이라 부르는 문구도 powered hold와 정반대.
- **근거**: ISO 10218-1, IEC 60204-1 정지 카테고리.
- **권고**: operational hold / protective stop / e-stop Cat 0·1 분리. 원인별 {감속·enable·최종토크·낙하·해제권한} 단일 stop matrix. 인증 하드웨어 없으면 "safety-rated" 표현 삭제.

### B4. 프로세스 사망·CAN bus-off에서 소프트 홀드는 물리적으로 불가능 ★수렴(Codex 직접모순 + Claude 완결성갭)
- **위치**: 12 NFR-SAF-001/Q17, 13 FR-GUI-093, 14 FR-OPS-024.
- **문제**: FR-GUI-093은 백엔드/프로세스 사망 시 STOP_HOLD 요구, FR-OPS-024는 같은 상황을 "CAN 스트림 중단→RID9→토크소실→낙하"로 규정 = 직접 모순. SIGKILL/OOM/deadlock 후엔 명령 송신 불가, bus-off에선 hold 명령 전송 자체 불가. 무브레이크라 낙하 직결.
- **권고**: 외부 안전컨트롤러/독립 CAN takeover 프로토콜이 없는 한 "process death/bus-off = 토크소실·낙하"로 명시. 기계적 지지·낙하영역 격리·전원차단 회로를 실패안전에 포함.

### B5. 관측·action을 같은 24/48차원 스키마로 묶어 비실행 채널을 학습 ★수렴(Codex + Claude action_torque_semantics)
- **위치**: 05 FR-TEL-010/013, 07 §2.3.3, 08 FR-DAT-012, 10 FR-TRN-074, 11 action 계약.
- **문제**: `use_velocity_and_torque=True`가 observation뿐 아니라 **action feature에도** pos/vel/torque 추가. 그러나 `send_action()`은 `.pos`만 읽고 vel/torque=0 하드코딩. VR/KER action은 vel/torque=0인데 07/08은 action.torque를 "리더 측정값"으로 기록하라 규정 → 손실함수가 실행 안 되는 채널을 최적화, 정책 출력차원·VLA 입력제한·실기 의미가 어긋남.
- **권고**: `observation.state=24/48`과 `action=position 8/16` 분리. 데이터셋 action = 안전게이트 후 실제 전송된 위치명령. vel/torque action 지원하려면 bus 전송~학습~추론 전 경로를 실행가능 계약으로 별도 정의.

### B6. `save_episode()`가 REVIEW보다 먼저 호출 → 폐기·재녹화가 원자적이지 않음 [Codex 고유]
- **위치**: 07 녹화 상태머신 T16/T19/T20/T22.
- **문제**: T16에서 save_episode 성공 후 REVIEW로 이동하나, LeRobot save_episode는 episode/parquet/meta 기록 + 내부버퍼 clear. 이후 COMMIT/DISCARD/RERECORD가 이미 영속화된 파일·인덱스·stats를 되돌려야 하는데 롤백 계약 부재 + 일부 전이 중복저장 위험.
- **권고**: REVIEW를 save 전에 하거나 quarantine 데이터셋에 저장 후 승인 시 원자적 promote. 폐기 시 parquet/video/meta/index/stats 전량 복구 트랜잭션 + "episode당 save 1회" 불변식.

### B7. CAN 프레임/사이클 16 vs 32 → 주파수 예산 전체가 2배 어긋남 ★수렴(Claude C-01 BLOCKER + Codex 1kHz HIGH)
- **위치**: 01 §2.6/NFR-SYS-002, 03 §2.14, 04 §2.5, 14 §2.7, 15 §2.1/NFR-PRF-003/M-25, 12 NFR-SAF-001.
- **문제**: 01/03/04/14는 16프레임(패턴A)→1000Hz≈83%·500Hz=42%를 '달성가능'으로, 15는 damiao.py:676 `_batch_refresh`가 매 사이클 REFRESH 라운드를 얹어 **32프레임(패턴B)→CAN-FD 상한 625Hz·500Hz=80%·1000Hz 불가**로 판정(소스 검증됨). 16 vs 32는 15가 [미확인](candump M-25)으로 남긴 미결인데 여러 문서가 반대 결론을 확정으로 씀. 12의 1kHz 안전루프는 이 위에서 성립 불가.
- **권고**: candump로 사이클 프레임 실측(M-25). 고주파(≥300Hz)는 패턴A(get_observation 우회, robot.bus 직접) 계약 명시 + classic-CAN 폴백 ≤350Hz 클램프를 01/03/04에 반영. 1kHz 안전루프 요구를 실측 가능 값으로 조정.

### B8. 원격 추론 action의 최종 안전게이트 미결 ★수렴(Codex + Claude 실현가능성리스크)
- **위치**: 11 NFR-INF-008/FR-INF-072/Q5, 16 D-11, 12 FR-SAF-074.
- **문제**: 모든 action이 동일 safety 파이프라인을 통과해야 한다면서 강제 지점(wrapper vs processor)을 미결로 남김. 공식 async robot client는 수신 action을 `robot.send_action()`으로 직접 전달 → rollout processor만으론 우회 가능. 그런데 16은 원격추론을 확정으로 취급.
- **권고**: 모든 입력원 바로 아래 **우회 불가 단일 action gateway**(단위·영점·리밋·freshness·workspace/collision·slew/jerk·stop-state 검사 후에만 CAN 호출). 12 FR-SAF-074(send_action 단일 집행점)와 일치.
