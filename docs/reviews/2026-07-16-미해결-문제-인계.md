## 1. 🔴 최우선 — 착수를 물리적으로 막는 것

### 1.1 `02c`(Wave 4A~5) 36개 WP가 기계판독 불가 — **BOOT 전면 차단**

**두 칸이 동시에 깨져 있다.**

| 칸 | 상태 | 수 |
|---|---|---|
| **워크플로우 형상** | 산문 서술. `SHAPE-*` 토큰 없음 | 36 중 **34** (2개는 이번에 채움) |
| **실행 클래스** | **한 칸에 2개** 선언 (`AI-offline` + `AI-on-HW` 등) | **13** |

**왜 차단인가.** `WP-BOOT-02`(`02a` §−2)가 WP 매니페스트 필수 필드를 `{wp_id, exec_class, workflow, owns[], consumes[], produces[], gates[], normalization_hash, contract_refs[]}`로 못 박고 **"`exec_class`·`workflow` 값 공간은 §0.2·§0.3 어휘로 닫힌다 — 제3의 어휘 발명 = 거부"**라 선언한다.

- `exec_class`는 **스칼라**다(`owns[]`·`gates[]`와 달리 `[]` 없음) → **"`AI-offline` + `AI-on-HW`"는 넣을 자리가 없다.**
- `workflow`도 스칼라 + 닫힌 어휘 → 산문은 못 넣는다.

→ 매니페스트를 못 만든다 → `WP-BOOT-02` 수용 ①("전 WP가 매니페스트 보유 — 누락 0건") 실패 → **`BOOT` 실패 → 어떤 WP도 착수 불가.**

**`01`은 Wave 4 형상을 아예 안 적는다** → 코퍼스 어디에도 이 36개의 토큰이 없다.

#### 1.1b 🔴 담당(`exec_class`) 2개 선언 — **13개. 이게 진짜 결정이다**

**전수(`02c` 기준):**

| 조합 | 수 | 성격 |
|---|---|---|
| `AI-offline` + `AI-on-HW` | **6** | 계산은 로봇 불필요 / 측정은 **리그 1대 배타** → 묶으면 계산까지 리그를 기다린다 |
| `AI-offline` + `Human-judgment` | **3** | 자동 부분은 안 막히는데 사람 판정이 붙어 **전체가 블록**된다 |
| `Human-assisted-HW` + `AI-on-HW` | **2** | 둘 다 리그. 사람 가용성이 DAG에 들어온다 |
| `Human-judgment` + `Human-assisted-HW` | **1** | |
| `Human-assisted-HW` + `Human-judgment` | **1** | |

`02b`(표 형식)에도 **`AI-offline` → `AI-on-HW` 4건**, **`AI-on-HW` → `Human-judgment` 1건**이 있다. **화살표(`→`, 순차)와 플러스(`+`, 병존)가 구분 없이 쓰인다** — 새 세션은 이 둘이 같은 뜻인지부터 정하라.

**결정해야 할 질문 (사용자 판정):**

> **한 WP에 담당 2개를 허용할 것인가?**

| 선택 | 대가 |
|---|---|
| **A. 허용** | 매니페스트 스키마를 바꿔야 한다(`exec_class`를 배열로). 그러면 `00` §4의 근거 — *"`AI-on-HW`는 블로킹 없이 fan-out되고 `Human-assisted-HW`는 멈춘다. **이 둘을 뭉치면 계획의 절반이 근거 없이 직렬화된다**"* — 가 **한 WP 안에서 그대로 재발**한다. `CI-14`(`exec_class ∈ {Human-*} ↔ workflow=SHAPE-HG`)도 재정의 필요 |
| **B. 쪼갠다** (권고) | 13개 → 26개. **WP 총계 177 → 190쯤.** `177`이 **8개 파일 30군데**에 박혀 있어 전수 스윕 필요. `00` §2.5가 경고한 *"ID 공간을 다시 연다"* 비용을 정면으로 지불한다 |

> 🔴 **내가 처음에 "`WP-4B-04` 하나만 쪼개자"고 한 것은 철회했다.** 13개가 같은 모양이므로 하나만 쪼개면 ID 공간을 열어놓고 일관성만 잃는다. **전수로 하거나, 안 하거나 둘 중 하나다.**

**B를 고를 때의 근거 자료** — `00` §4가 클래스를 가른 이유는 *"사람이 필요한가"의 이진 질문이 아니라 **사람이 무엇을 하는가**"*이고, 판정 기준은 **"사람의 가용성이 DAG에 들어오는가"**다. `AI-offline`+`Human-judgment`를 한 WP로 묶으면 자동 부분까지 사람을 기다린다 — 이게 §4가 막으려던 바로 그 직렬화다.

### 1.2 런처 부재 — **BOOT을 다 지어도 못 돌린다** (Codex 발견 · 내가 직접 검증함)

`BOOT`(`WP-BOOT-01`~`05`)이 짓는 것: 레지스트리 · 매니페스트/인덱스 · CI 검사기 · 상태 저장소 + 취소 · 계약 동결 잠금.

**없는 것: Claude 워크플로우를 실제로 스폰·취소하는 어댑터.**

| 후보 | 실제로 소유하는 것 | 런처인가 |
|---|---|---|
| `WP-BOOT-04` | `registry/state/**`, `ops/cancel/**` | ❌ 상태 기록 + 취소 **규약**이지 스폰이 아니다 |
| `02c:150` "런처" | `lerobot-train` **subprocess** 런처 | ❌ 학습 잡용 |

`05` §0.1이 *"fan-out 발행"*을 자기 절차로 선언하지만 **그 발행을 실행할 물건을 소유한 WP가 없다.** 마찬가지로 `registry/check.py`는 **로컬 검사기**이지 CI 결선이 아니다.

→ **`BOOT`은 시뮬레이션을 자기검사할 뿐 실제 소비자(Claude 워크플로우)를 제어하지 않는다.**

**결정**: 런처·CI 결선·병합 집행을 소유하는 WP를 `BOOT` 대역에 발급할 것인가(`WP-BOOT-06`?). 발급하면 ID 공간을 또 연다.

## 5. 새 세션 권장 순서

1. **전수 조사부터.** §0의 표를 보라 — 이 문서의 "N개" 숫자를 믿지 말고 직접 세라. 최소 3종: ① `02c`·`02b` 담당 칸 다중 선언 전수 ② `177` 박힌 곳 전수(8파일 30군데) ③ `01`↔카탈로그 형상 재대조(현재 39/39지만 새 WP 발급 시 깨진다).
2. **§1.1b 결정**(담당 2개 허용 vs 쪼개기) — **이게 나머지 전부를 지배한다.** 쪼개면 §2.2(발급처)를 먼저 닫아야 하고, `177`이 전부 바뀐다.
3. **§1.2 런처** 발급 여부.
