# 실행 부트스트랩 사용 가이드

이 문서는 **`registry/`와 `ops/`에 있는 도구를 어떻게 쓰는가**를 다룬다.
계획이 무엇을 요구하는지는 `docs/plan/`이 갖고, 여기는 그것을 돌리는 방법만 있다.

---

## 0. 이게 무엇인가 — 한 문단

`docs/plan/`은 177개 작업 패키지를 정의하면서 "CI가 검증한다", "후손이 stale 된다",
"계약이 동결된다"고 171번 말한다. **그런데 그 CI도, stale 전파도, 동결 잠금도 코드로
존재한 적이 없었다.** 규칙만 있고 실행체가 없으면 그것은 규율이 아니라 희망이다.

`registry/`와 `ops/`가 그 실행체다. 이 부트스트랩이 착지하기 전에는 어떤 작업 패키지도
착수할 수 없다(`00` §3.5 — 미달 시 `FAIL_BLOCKING`).

---

## 1. 설치

Python 3.11 이상과 [uv](https://docs.astral.sh/uv/)가 필요하다.

```bash
cd bh_open_arm
uv venv --python 3.12
uv pip install -e ".[dev]"
```

이후 명령은 전부 `uv run`을 앞에 붙이거나 `.venv/bin/python`을 직접 쓴다.
아래 예시는 `uv run` 기준이다.

```bash
uv run python -c "import registry; print('ok')"
```

## 2. 5분 안에 해보기 — 세 가지 명령

### 2.1 레지스트리가 계획과 일치하는지 확인

```bash
uv run python -m registry.ingest.cli --check
```

출력:

```
records=1215 packages=177/177 schema_errors=0
```

읽는 법:

| 값 | 뜻 | 이 값이 아니면 |
|---|---|---|
| `records` | 레지스트리 레코드 수 = 명세 요구사항 1,207 + 계획 축 8 | 명세에 요구사항이 추가·삭제됐다 |
| `packages` | 등재된 작업 패키지 / 카탈로그가 발급한 수 | **177/177이 아니면 대역 수용 게이트 미달**이다 |
| `schema_errors` | 스키마 위반 | 0이 아니면 레코드가 계약을 깼다 |

`--check`는 **읽기만 한다.** 파일을 쓰지 않는다.

### 2.2 레지스트리 다시 만들기

계획 문서(`docs/plan/02a`~`02d`, `06`)나 명세(`docs/spec/`)를 고쳤다면:

```bash
uv run python -m registry.ingest.cli
```

두 파일이 갱신된다.

- `registry/traceability.yaml` — 레지스트리 본체
- `registry/build/reconciliation.md` — **산문↔레지스트리 대조 리포트**

> 🔴 **방향에 주의.** 파종은 **부트스트랩 때 한 번만** 산문에서 레지스트리로 흐른다.
> 착지 후에는 **레지스트리가 정본이고 산문이 뷰다**(`05` §0.1). 다시 돌리는 목적은
> "계획을 고쳤으니 레지스트리를 따라오게 한다"가 아니라 **"둘이 아직 일치하는지 본다"**이다.
> 절차와 레지스트리가 갈리면 레지스트리가 이긴다.

### 2.3 대조 리포트 읽기

```bash
less registry/build/reconciliation.md
```

이 리포트가 이 부트스트랩에서 **가장 중요한 문서**다. 무엇을 만들었는지보다
**산문이 말하지 않은 것이 무엇인지**를 담기 때문이다. §3이 사람이 고쳐야 할 목록이다.

## 3. 알아야 할 개념 넷

### 3.1 레코드는 요구사항 단위다 (WP 단위가 아니다)

레지스트리 한 줄 = 요구사항 하나다. 한 작업 패키지가 요구사항 여럿을 구현하면
**같은 `wp` 값을 가진 레코드가 여럿** 생긴다.

```yaml
- req: FR-CON-061        # 이 줄의 키
  wp: WP-1-02            # 같은 값이 다른 레코드에도 나온다
```

그래서 `CI-14c`가 존재한다 — WP에 속한 필드(`gate`, `owns`, `downstream` …)는
같은 `wp`의 모든 레코드에서 **완전히 같아야** 한다. 하나만 어긋나면 stale 전파가
계산하는 후손 집합이 조용히 달라진다.

### 3.2 `DEFERRED`는 실패가 아니라 정직함이다

현재 831개 요구사항이 `wp: DEFERRED`다. 이건 버그가 아니다.

**왜 이렇게 되는가**: 카탈로그는 요구사항을 *인용*한다(다대다). `NFR-SAF-007`은
8개 WP가 인용한다 — 8개가 다 그걸 지켜야 하니까. 그런데 레지스트리가 필요한 건
*소유*(다대일)다. 그 소유 함수는 **코퍼스 어디에도 전수로 존재하지 않는다.**
`06` §6이 영역 규칙과 대표 예시 47건만 두고, 전수는 레지스트리가 갖는다고 명시한다 —
즉 지금 만들고 있는 바로 그 물건이 갖는다고 말한다.

**왜 발명하지 않는가**: 831건에 그럴듯한 소유자를 채우면 레지스트리는 검증을 통과하고,
초록불이 되고, **어떤 검사기도 볼 수 없는 방식으로 틀린다.** 계획 자신이 이걸
가장 나쁜 결과로 지목한다 — *"아무것도 잡지 못한 채 green인 검사기는 지켜진다는 증거를
위조한다"*(`02a` §−2.3).

스키마는 이 상태를 위해 `DEFERRED`를 마련해 뒀고, `CI-04`(게이트 필수)와
`CI-07`(정규화 해시 필수)이 둘 다 이 값을 면제한다.

**어떻게 줄이는가**: `06` §6 표에 소유 WP를 명시하면 된다. 한 줄 추가할 때마다
`doc06-section6` 규칙이 잡아가고 `DEFERRED`가 하나 줄어든다.

### 3.3 모든 레코드는 자기 출처를 안다

```yaml
  provenance:
    wp_rule: ambiguous-citation
    wp_candidates: [WP-0A-01, WP-1-03, WP-2A-01]
```

`wp_rule`이 이 배정을 **어느 규칙이 결정했는지** 말한다. 여섯 가지가 있고
대조 리포트 §2에 건수별로 정리돼 있다. `ambiguous-citation`이면 `wp_candidates`가
후보를 남기므로, 사람이 고를 때 다시 찾을 필요가 없다.

### 3.4 게이트는 두 종류이고 섞으면 안 된다

| | `PG-*` 측정 게이트 | `CG-*` 수용 검사 |
|---|---|---|
| 무엇 | 통과 여부가 **하류 착수를 바꾸는** 물리 판정점 | 아티팩트에서 결정론적으로 계산되는 검사 |
| 상태 | **5상태 전부** (`PASS`/`RETRY_WITH_VARIANT`/`DEGRADED_ACCEPTED`/`FAIL_BLOCKING`/`SUPERSEDED`) | **`PASS`/`FAIL` 이진** |
| 후손 | 있다 (stale 전파) | **없다** |
| ID 발급 | `03-측정-게이트.md` | 카탈로그 수용 항목에서 **위치로 도출** |

`CG-*`에 `DEGRADED_ACCEPTED`를 달면 `CI-05e`가 빌드를 실패시킨다. 이유는 표기 문제가
아니다 — `CG-*`는 그 WP가 정의했으므로 **심판과 선수가 같고**, 5상태를 주면
WP가 자기 수용선을 자기가 낮추고 통과를 선언할 수 있게 된다.

**`CG-*` ID는 손으로 짓지 않는다.** `CG-<대역>-<번호><영문자>`에서 영문자는
수용 칸의 **몇 번째 ①②③ 항목인가**다(`06` §2.4a). 그래서 수용 항목을 **재정렬하면
ID가 바뀐다** — 항목을 지우거나 앞에 끼우면 `CI-04c`가 잡는다.

## 4. 자주 하는 작업

### 4.1 계획 문서에 WP를 추가했다

```bash
uv run python -m registry.ingest.cli        # 다시 파종
uv run python -m pytest tests/boot01 -q     # 177 카운트가 맞는지
```

`packages=178/178`처럼 양쪽이 같이 늘면 정상이다. **한쪽만 늘면** 카탈로그 표 형식이
파서가 아는 셋 중 어디에도 안 맞는다는 뜻이다.

파서가 아는 형식:

| 문서 | 형식 |
|---|---|
| `02a` | 가로 9열, WP당 한 행 (Wave 1만 `순서` 열이 앞에 하나 더) |
| `02b`·`02d` | A표(6열) + B표(5열)를 WP ID로 조인 |
| `02c` | WP당 세로 `\| 항목 \| 내용 \|` 카드, ID는 제목에 |

### 4.2 요구사항을 명세에 추가했다

요구사항은 **선언 표**에 있어야 인식된다:

```markdown
| ID | 요구사항 | 우선 | 근거 | 비고 |
|---|---|---|---|---|
| FR-CAM-099 | 시스템은 … | M | … | [신규구현] |
```

산문에 ID를 쓰는 것만으로는 **선언이 아니다.** 이건 의도적이다 — 명세 `00` §5.1은
`NFR-NFR-001`을 이름 규칙의 반례로 언급하는데, 산문을 정규식으로 긁으면 그런 **예시 68개가
진짜 요구사항으로 잡힌다.**

`우선` 칸은 `M`/`S`/`C` 중 하나여야 한다. `비고`의 `[…]` 표기는
`확정`/`미확인`/`결정필요`/`신규구현` 중 하나여야 한다. 어휘 밖 값은 보수적으로
`미확인`으로 매핑되고 원본이 `provenance`에 남으며 대조 리포트 §3.3에 열거된다.

### 4.3 특정 WP의 레코드만 보기

```bash
uv run python - <<'PY'
import yaml
doc = yaml.safe_load(open("registry/traceability.yaml"))
for e in doc["entries"]:
    if e["wp"] == "WP-1-03":
        print(e["req"], "|", e["gate"])
PY
```

### 4.4 `DEFERRED`가 왜 그렇게 됐는지 보기

```bash
uv run python - <<'PY'
import collections, yaml
doc = yaml.safe_load(open("registry/traceability.yaml"))
rules = collections.Counter(e["provenance"]["wp_rule"] for e in doc["entries"])
print(rules)
# 후보가 여럿이라 미배정된 것 중 앞 10건
for e in doc["entries"]:
    if e["provenance"]["wp_rule"] == "ambiguous-citation":
        print(e["req"], "->", e["provenance"]["wp_candidates"])
PY
```

## 5. 걸리면 이렇게 읽는다

| 증상 | 원인 | 대응 |
|---|---|---|
| `packages=176/177` | 카탈로그 표 형식이 파서가 아는 셋 밖 | §4.1 형식표와 대조. 열 이름·`WP` 열 위치 확인 |
| `schema_errors` > 0 | 레코드가 `06` §2.2 계약 위반 | 출력의 `entries/<n>/<필드>` 경로를 보고 해당 레코드 확인 |
| 요구사항이 등재 안 됨 | 선언 표 밖에 있음 | §4.2. `ID`+`우선` 또는 `ID`+`요구사항` 열이 있는 표여야 한다 |
| 표 행이 열 밀림 | 코드 스팬 안에 이스케이프 안 된 `\|` | `\\\|`로 바꾼다. 위치는 대조 리포트 §3.1 |
| `CG-*` ID가 어긋남 | 수용 항목을 재정렬·삽입·삭제했다 | ID는 항목 **위치**에서 나온다. 항목 순서를 되돌리거나 참조를 갱신 |

## 6. 이 도구가 하지 않는 것

경계를 분명히 해 두는 편이 낫다.

- **요구사항의 소유 WP를 추측하지 않는다.** 확정할 수 없으면 `DEFERRED`다(§3.2).
- **게이트 임계 수치를 정하지 않는다.** `00` §7.1 "측정 전 목표 금지"가 전역 규칙이고,
  `CI-11`이 증거 해시를 참조하지 않는 임계 상수를 빌드 실패시킨다.
- **계획 문서를 고치지 않는다.** 결함은 대조 리포트 §3에 위치와 함께 **보고만** 한다.
- **하드웨어를 만지지 않는다.** BOOT 대역 5개는 전부 `AI-offline`이다 — 로봇도 사람도
  필요 없다.
- **`registry/build/**`를 손으로 고치면 안 된다.** 생성물이며 pre-commit 훅이 거부한다.
