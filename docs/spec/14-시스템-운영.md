# 14. 시스템·운영·실패 복구

> 영역 코드: **OPS** · 대상: OpenArm v2.0 (enactic) 기반 피지컬 AI 토탈 플랫폼
> 이 문서의 요구사항 ID는 전부 `FR-OPS-xxx` / `NFR-OPS-xxx`다.

## 1. 범위

**다룬다**: 워크스페이스·프로파일·세션의 영속화와 임포트/익스포트, 설정 계층과 오버라이드 우선순위, 구조화 로그·에러 코드·이벤트 타임라인·진단 번들, **실패 모드 전수표(감지/표시/자동복구/수동복구)**, **전원 상실 후 복구 시퀀스**, 헬스체크·하트비트·서브시스템 상태, 배포(udev/systemd/Docker/버전 고정), 실험 추적·계보, 제어권 락과 권한.

**다루지 않는다**: 텔레옵 상태머신 전이 자체(→ 텔레옵 문서), IK 알고리즘·솔버 선택(→ 기구학 문서), 데이터셋 스키마 상세와 학습 하이퍼파라미터(→ 데이터/학습 문서), GUI 화면 레이아웃(→ GUI 문서). 여기서는 그 축들이 **실패했을 때 시스템이 무엇을 해야 하는가**만 다룬다.

**전제 (v2 런타임)**: CAN 배타 소유자는 **LeRobot Python API를 임베드한 단일 GUI 백엔드 프로세스**다. 그 외 어떤 프로세스도 SocketCAN을 열지 않는다. **LeRobot CLI를 서브프로세스로 스폰하지 않는다**(FR-OPS-085). `dora` 데몬도 `controller_manager`도 이 경로에 **없다**.

## 2. 확정된 사실 기반

### 2.1 런타임 토폴로지 — 무엇이 무엇을 소유하는가

```
GUI 백엔드 프로세스 (단일, LeRobot Python API 임베드)
  ├─ CanGuard              ← flock(can0..can3) + ip link 검증 + 침입자 감시
  ├─ OpenArmFollower / BiOpenArmFollower   ← can0/can1 소유. 세션당 connect() 1회
  │    └─ DamiaoMotorsBus (python-can, socketcan)   ×2
  ├─ Teleoperator (우리가 만듦)
  │    ├─ VRTeleoperator   ← UDP :5006 수신 → 포즈 → openarm_control IK → 관절각 deg
  │    └─ KERTeleoperator  ← USB (pyusb, VID 0x303A / PID 0x4002). CAN 0채널
  ├─ SafetyLayer (우리가 만듦)   ← 속도/가속/저크 클램프, 소프트 스톱, 워치독
  ├─ LeRobotDataset        ← 기록 (정본 포맷)
  ├─ Policy                ← 추론 (lerobot-rollout 등가 루프)
  └─ EvalHarness (우리가 만듦)   ← 성공/실패 라벨링 + 성공률 집계
```

**한 프로세스, 한 robot 객체.** 양팔도 같은 프로세스 안에 버스 객체 2개다(`bi_openarm_follower.py:93-94`, `utils/bimanual.py:47-63` — **순차 동기 호출**, 별도 스레드도 asyncio도 없다).

**GUI = 웹 단일 [확정, D-2 종결].** 헤드리스 **FastAPI 백엔드**가 LeRobot·CAN·안전계층·`robot` 객체를 인프로세스로 소유(connect 1회, `flock`, 모드=상태전이)하고, **웹 SPA** 프론트가 브라우저에서 3D(Three.js + urdf-loader)와 카메라 프리뷰를 그린다. **브라우저↔백엔드 = 단일 WebSocket**(control/telemetry/camera 멀티플렉싱) **+ HTTP REST**(비실시간 CRUD). **gRPC-web은 브라우저 양방향 스트리밍 제약으로 배제** → WS 채택. gRPC는 **백엔드↔원격 추론(async PolicyServer :8080) 홉 하나에만.** ⇒ 이전 판의 "GUI 전송 계층 `[결정필요]`"(WebRTC/Foxglove/rosbridge 다채널 분리)와 PyQt6 데스크톱은 **둘 다 배제한다.** 참조 구현(`bh_indy7_LeRobot/src/bh_indy7_lerobot_gui`)은 **백엔드 로직 패턴**(안전가드·캘리브레이션 SoT·이중추론)만 이식하고 UI는 웹으로 대체한다. 상세 → `01` §2.16.

**🔴 프로세스를 늘리면 안 되는 이유가 두 개다:**
1. **LeRobot CLI는 각각 독립 프로세스다** (`pyproject.toml [project.scripts]`: `lerobot-teleoperate` / `lerobot-record` / `lerobot-rollout` / `lerobot-calibrate` / `lerobot-replay` / `lerobot-setup-can`). 각 스크립트가 시작 시 `robot.connect()`, 종료 시 `disconnect()`를 한다 → **모드마다 CLI를 스폰하면 CAN 중재 문제가 그대로 살아있고 §2.6의 영점 파괴 사고까지 따라온다.**
2. **커널이 배타성을 강제하지 않는다** (§2.3).

### 2.3 CAN 이중 점유 — "조용한 실패"의 정확한 메커니즘

Linux 커널 공식 문서 (`docs.kernel.org/networking/can.html`) 원문:
> - "**Multiple sockets can be opened at the same time**, on different or the same protocol module and they can listen/send frames on different or the same CAN IDs."
> - "To meet multi user needs **the local loopback is enabled by default**."
> - bind 시점 기본 필터: "**The filters are set to exactly one filter receiving everything.**"

`openarm_can`의 실제 소켓 코드는 `CAN_RAW_LOOPBACK`, `CAN_RAW_RECV_OWN_MSGS`, `CAN_RAW_FILTER`를 **어느 것도 설정하지 않는다** → 커널 기본값 그대로.

**따라서 두 프로세스 A, B가 can0을 동시에 bind하면:**
1. `bind()`가 **둘 다 성공**한다. 예외도 errno도 없다.
2. 루프백 기본 ON → A는 B가 보낸 명령 프레임(0x01–0x08)까지 수신한다. `openarm_can`은 recv id(0x11–0x18)로 디스패치하므로 조용히 버려진다 → **에러 없음**.
3. 모터가 B의 질의에 응답한 프레임(0x11–0x18)을 **A의 `recv_all()`이 먼저 소비**할 수 있다. A는 자기 refresh의 응답으로 착각해 상태 캐시를 갱신한다 → **A는 "정상"으로 보인다**. 동시에 B는 프레임을 뺏겨 타임아웃 → **낡은 상태로 제어한다**.
4. 둘 다 MIT 명령을 쓴다. 게인이 다르면(§2.7) 모터는 마지막 셋포인트를 따르며 두 제어기 사이에서 **진동**한다.
5. 이 모든 과정에서 예외/로그/리턴코드가 **하나도 발생하지 않는다**.

**우리 경로의 정본 CAN 클라이언트는 하나다.** 그러나 **침입 가능한 클라이언트는 여전히 여럿이다:**

| 클라이언트 | 스택 | 우리 경로에서의 지위 | 출처 |
|---|---|---|---|
| **LeRobot `DamiaoMotorsBus`** | **python-can `socketcan` 백엔드** (`openarm_can` **미링크**, 완전 독립 구현) | ✅ **정본. GUI 백엔드 프로세스가 소유** | `lerobot/motors/damiao/damiao.py` |
| `openarm-can-cli` | 자체 SocketCAN RAW | ⚠️ **침입자.** 캘리브레이션·파라미터 조회에 손이 가기 쉽다 | `openarm_can/setup/cli` |
| `cansend` / `candump` | iproute2 / can-utils | ⚠️ `cansend`는 **침입자**. `candump`는 읽기만 하므로 물리적으로 안전(단 그건 커널이 아니라 candump의 선의) | can-utils |
| `openarm_teleop` (C++, **500 Hz** 바이래터럴) | 자체 SocketCAN RAW | ⚠️ **잠재 침입자.** M-21에서 Python 500 Hz가 실패하면 이걸 붙여야 하고, 그 순간 **배타 소유권 이양 프로토콜이 필요해진다** | `openarm_teleop` |
| `openarm_driver` (Python→pybind) | `openarm_can` 경유 | ⚠️ **[미확인]** — 내부에서 CAN을 여는지 확인 필요(`16` M-24). **연다면 우리 경로에서 배제한다** | `openarm_driver/driver.py` |
| 두 번째 LeRobot 인스턴스 | python-can | ⚠️ **침입자.** 사용자가 터미널에서 `lerobot-teleoperate`를 치면 즉시 발생한다 | — |

**CAN ID 맵이 전 스택 동일**(send `0x01`–`0x08` / recv `0x11`–`0x18`)하다. 그래서 서로의 모터를 **완벽하게 침범할 수 있다.**

⚠️ **`openarm_teleop`의 채널 관례가 LeRobot과 정반대다** — `openarm_bilateral_control.cpp:154-155`: `leader_can_interface = "can0"`, `follower_can_interface = "can2"`. LeRobot 관례는 `can0` = follower-left다. **둘을 한 리그에서 섞어 쓸 거면 반드시 CLI 인자로 명시하라.**

✅ **런타임 CAN 해제는 가능하다** — `damiao.py:250-269` `disconnect()`가 `canbus.shutdown()` + `canbus = None`을 한다. **그러나 재연결이 영점을 파괴하므로**(§2.6) **모드 전환에 이를 쓰면 안 된다.** 해제는 세션 종료 시에만.

### 2.5 캘리브레이션 영속 · 전원 상실과 영점

**🔴 캘리브레이션 영속의 정본 = 디스크 JSON SoT (참조레포 패턴, m1 확정).** 참조 구현(`bh_indy7_LeRobot`)이 검증한 **이원화 모델**을 채택한다:

| 데이터 | SoT | 방식 |
|---|---|---|
| 관절 부호(sign) · 그리퍼 open/close **rad** 엔드포인트 · captured 플래그 · 타임스탬프 | **디스크 JSON** | pydantic 모델 → `save_calibration_atomic`(mkstemp → write → flush → `os.fsync` → `os.replace`, POSIX 원자적). 경로는 LeRobot 레이아웃 미러 `~/.cache/huggingface/lerobot/calibration/<kind>/<id>.json` |
| 관절 zero(영점) | **모터 NV**(`set_zero_position` = `CAN_CMD_SET_ZERO`) | 디스크 오프셋으로 표현 불가 — DM 모터의 firmware 연산이다 |

- **디스크가 SoT인 이유**: 관절 zero를 제외한 모든 캘리브레이션(부호·그리퍼 엔드포인트·플래그)은 디스크 JSON에 영속되고 부팅 시 live backend에 **재적용**된다. 손상 시 modal 복구(파일 보존 vs `.broken-<ts>` 이동). 이것이 M-1의 정답 구조다.
- 🔴 **레지스터 flash store(0xAA `存储参数`)에는 의존하지 않는다** — firmware별 신뢰성이 확인되지 않았다(§5 Q1). 우리 SoT는 **디스크**이지 모터 flash가 아니다.
- **connect()는 자동 set_zero를 하지 않는다(오버라이드)** — 상세 §2.6. 영점은 **명시적 operator 플로우**로 확정한다: `connect_readonly()`(torque-OFF 버스 오픈, 손으로 방향·영점 검증) → 손으로 rest(URDF-zero) 자세 정렬 → `set_zero`(disable → per-motor zero → verify readback) → 그리퍼 open/close 물리 끝단에서 `capture_gripper_endpoint`(현재 native rad 캡처, 로드셀 없음) → 디스크 영속.
- **set_zero는 그리퍼 캡처를 무효화한다** — 그리퍼 모터도 재-zero되므로 캡처된 절대 rad가 더 이상 물리 open/close를 가리키지 않는다 → captured 플래그를 리셋하고 **재캡처를 요구**한다.

> ⚠️ **참조레포는 OpenArm이 아니다**(Seeed reBot B601-DM + Indy7). 위는 **패턴만** 이식한 것이다 — reBot 그리퍼 수치(−270° 등)·관절수(6)·모터 ID(0x01–0x07)는 쓰지 않는다.

---

**✅ 확정: 출력축 단회전 절대 엔코더가 있으므로 전원 재인가만으로 관절 절대각(물리 자세)이 복원된다.**

1차 출처(Damiao 공식 매뉴얼 p.1 电机特点):
> **双编码器，输出轴单圈绝对位置，不惧掉电输出轴绝对位置丢失。**
> (영문판: "Dual encoders with single-turn absolute position **on the output shaft** — ensures no position loss even after power failure.")

레지스터가 이를 증명한다: **RID 80 `p_m`** = 电机当前位置(로터측), **RID 81 `xout`** = **输出轴位置(출력축)**. 두 엔코더가 각각 별도 레지스터로 노출된다. 사양: 14-bit, 엔코더 2개, 자기식 단회전, 감속비 40:1.

⚠️ **단회전**이므로 복원 범위는 출력축 1회전 이내다. OpenArm 관절 리밋은 전부 그 안에 들어간다.

**⚠️ 단서 — 영점 오프셋(m_off)의 flash 영속성은 [미확인]이나, 우리 설계는 이에 의존하지 않는다.** SoT는 디스크 JSON이고 영점은 bring-up 시 operator `set_zero`로 재확립한다(위). 아래 레지스터 사실은 진단·교차검증용이며, **0xAA 강제 저장은 정본 경로가 아니다(m1).**

| 명령 | CAN ID | 데이터 | 비고 |
|---|---|---|---|
| 파라미터 읽기 | `0x7FF` | `[ID_L][ID_H] 0x33 [RID] 00 00 00 00` | |
| 파라미터 쓰기 | `0x7FF` | `[ID_L][ID_H] 0x55 [RID] [val 4B LE]` | **휘발성** |
| **파라미터 저장(存储参数)** | `0x7FF` | `[ID_L][ID_H] 0xAA 0x01` | ① **disable 상태에서만 유효** ② 전 파라미터 일괄 ③ 1회 최대 **30 ms** ④ **flash 수명 약 10,000회** |
| enable / disable / **영점 설정** | `send_can_id` | `FF…FF FC` / `FF…FF FD` / **`FF…FF FE`** | |

- `m_off`(RID 54, 영점 오프셋)는 **RO**다 → `0x55` 쓰기 경로에 없다 → 매뉴얼의 "쓰기는 휘발" 문장의 적용 대상이 **아니다**. 0xFE는 별개의 특수 명령이다.
- **OpenArm은 0xAA를 어디에서도 보내지 않는다.** `set_zero` CLI에 `--save` 플래그가 없고, `openarm_can`의 `openarm.hpp`에 `save_param`/`store_param` 계열 API가 **존재하지 않는다**.
- OpenArm 저수준 시퀀스: **disable(0xFD) → set_zero(0xFE) → disable(0xFD)**, 각 단계 사이 100 ms (`zero_position_commands.cpp`).
- 정황은 전부 "영속된다" 쪽(공식 문서가 영점을 1회성 셋업 Step 4로 배치 / i2rt가 동일 프레임을 `save_zero_position()`으로 명명 / Damiao SDK가 0xFE를 保存零点으로 표기)이나, **어느 것도 1차 명문이 아니다.** → §5 Q1의 실기 검증으로 확정할 것.

**영점 캘리브레이션의 실체 = 기계적 하드스톱 범핑** (`openarm-can-zero-position-calibration`, 413줄 Python):
- `bump_to_limit()`이 각 관절을 0.2°씩 기계 스토퍼까지 밀어붙이고 `|dq| < dq_th && |τ| > tau_th`로 접촉을 판정한다.
- 접촉 판정 임계: **J1 `dq<0.0125 rad/s && |τ|>5.0 Nm`**, **J2~J7 `dq<0.1 && |τ|>2.0`**, **그리퍼 `dq<0.3 && |τ|>0.3`**.
- 🔴 **결함**: 이상 홈으로 이동하는 `move_to_precise_home()`이 `pass` **스텁**이고 호출조차 주석 처리돼 있다. 산출된 `arm_goal_abs`는 **버려지고**, 영점은 "bump 후 복귀 시퀀스가 남긴 자세"에 찍힌다.

### 2.6 🔴 `connect()`가 영점을 파괴한다 — 이 문서에서 가장 중요한 사실

```python
# robots/openarm_follower/openarm_follower.py:137-156
self.bus.connect()
if not self.is_calibrated and calibrate:
    self.calibrate()
for cam in self.cameras.values():
    cam.connect()
self.configure()
if self.is_calibrated:
    self.bus.set_zero_position()     # ← 🔴 여기
self.bus.enable_torque()
```

```python
# motors/damiao/damiao.py:335-340
def set_zero_position(self, motors: str | list[str] | None = None) -> None:
    """Set current position as zero for selected motors."""
    target_motors = self._get_motors_list(motors)
    for motor in target_motors:
        self._send_simple_command(motor, CAN_CMD_SET_ZERO)
```

**`CAN_CMD_SET_ZERO`는 현재 물리 자세를 영점으로 확정하는 모터 명령이다.** `connect()`의 독스트링도 이를 인정한다(`openarm_follower.py:133-134`): *"We assume that at connection time, the arms are in a safe rest position."*

⇒ **모드 전환하려고 disconnect/reconnect 하면, 팔이 rest 자세가 아닌 순간 영점이 그 자세로 덮어써진다.** 조인트 리밋과 데이터셋 좌표계가 통째로 어긋난다. **예외도, 경고도, 로그도 없다.**

**따라서 명세가 못 박는 것:**

| 규칙 | 요구사항 |
|---|---|
| **세션당 `connect()`는 정확히 1회.** IDLE → CONNECTED 전이에서만. | FR-OPS-083 |
| **모드 전환은 활성 제어 루프 교체다.** 버스는 건드리지 않는다. | FR-OPS-083 |
| **`disconnect()`는 세션 종료에서만.** | FR-OPS-083 |
| **LeRobot CLI를 모드별로 스폰하지 않는다** (각 CLI가 시작 시 connect, 종료 시 disconnect를 한다). | FR-OPS-085 |
| GUI에 "재연결" 버튼을 두려면 **"팔을 rest 자세로 두라"는 모달 확인**을 반드시 건다. | FR-OPS-083 |

⚠️ **실증이 필요하다** — 소스는 명백하지만 실기에서 관측 좌표계가 실제로 얼마나 어긋나는지는 [미측정]이다 → `16` **M-20**.

**모드 상태머신 (v2)**

```
IDLE ──connect()──> CONNECTED ──┬── TELEOP        (VR/KER 어댑터 → 안전필터 → send_action)
   (세션당 1회)                 ├── TELEOP+RECORD (동일 루프 + LeRobotDataset 라이터)
                                ├── INFERENCE     (정책 → 안전필터 → send_action)
                                ├── MANUAL        (조그/freedrive)
                                ├── CALIBRATE     (토크 off, 사람이 팔을 잡음)
                                └── BILATERAL     ← 특수. M-21 결과에 따라 존재 여부가 갈린다
                                                    (Python 500 Hz 실패 시: LeRobot 버스를 정말로 닫고
                                                     flock을 별도 RT 프로세스에 넘긴다 → 복귀 시 재연결
                                                     = 영점 재설정 → rest 확인 모달 필수)
```

- **robot 객체는 세션당 정확히 하나.** 모드 전환 = **활성 컨트롤 루프 교체**.
- `send_action()` 호출권을 **단일 뮤텍스**로 보호한다. 전이 시 이전 루프의 종료를 join으로 확인한 뒤 새 루프를 시작해 **겹침 0을 보장**한다. LeRobot에 중재자는 없다.
- 배타성의 단위는 "버스 소유"가 아니라 **"누가 `send_action()`을 부를 권리를 갖는가"**다.

🔴 **런타임이 LeRobot이라는 사실이 이 문제를 뾰족하게 만든다.** LeRobot의 기본 게인은 **240 계열 = v1 유래**다. 근거는 커밋 이력이다 — `openarm_teleop`의 `follower.yaml`은 2025-07-23 이후 무수정인데 `openarm_description`에 v2.0 자산이 처음 등장한 것은 **2026-05-18**(PR #52)이다. **아무것도 안 하면 v1 게인으로 v2 로봇을 돌리게 된다.** → 게인 프로파일을 명시적으로 선택·표시하고, v1 유래에 **"비정본" 라벨**을 붙여야 한다(FR-OPS-011).

**관절 리밋도 다중 충돌이다.**

| 출처 | 값 | 단위/기준 |
|---|---|---|
| URDF `joint_limits.yaml` (v2.0) | joint1 `[-1.3963, 3.4907]`, joint4 `[0, 2.4435]` | **rad**, URDF 프레임 |
| **MJCF `jnt_range`** (`openarm_mujoco` v2) | right joint_1 **−80°…+200°** | **rad**, MJCF 프레임. 🔴 **`mink.ConfigurationLimit`이 보는 유일한 리밋** |
| **LeRobot `config_openarm_follower.py`** | joint_1 `(-75, +75)`, joint_2 R`(-9, +90)`/L`(-90, +9)`, joint_4 `(0, 135)` | **deg**, 캘리브레이션(호밍 오프셋) 적용 후 **모터 프레임**. 같은 파일 주석: *"the default joint limit values are small for safety"* → **의도적 보수 기본값** |
| `openarm_driver/configs/openarm_cell.yaml` | `right_arm: [[-1.3,2.0],[-0.174533,2.0],…]` | rad, 좌우 미러링 |

🔴 **IK와 LeRobot의 리밋이 어긋난다.** MJCF right joint_1 = −80°…+200°인데 LeRobot 소프트 리밋은 −75°…+75°다 → **IK가 LeRobot이 클램프할 해를 낸다.** 게다가 `openarm_control/kinematics.py:220-231`이 `NoSolutionFound` 시 **`limits=[]`로 재시도**한다 → **리밋을 통째로 버린 해**를 낼 수 있다.

⇒ **두 방어선이 모두 필요하다**: ① `ArmSetup` 생성 후 `model.jnt_range`를 LeRobot 리밋으로 **덮어쓴 뒤** `Kinematics(...)`를 만든다(**순서 중요**), ② **어댑터 레벨 클램프가 1차 방어선**이어야 한다.

⚠️ `side`를 지정하지 않으면 LeRobot은 전 축 **±5°로 잠긴다**(`config_openarm_follower.py:107-120`). GUI는 side를 강제해야 한다(FR-OPS-014).
⇒ **어느 것을 클램프 정본으로 쓸지** → `16` D-3.

**안전 체커 계층 — 실제 존재하는 유일한 방어선** (`openarm_driver/safety.py`)

| 체커 | 위반 시 | `force_stop` | 결과 |
|---|---|---|---|
| `JointPosChecker` | `np.clip(positions, low, high)` | **False** | **소프트 클램프** — 잘라서 계속 진행 |
| `JointDeltaPosChecker` | `abs(positions[i] − last_command[i]) > delta_limits[i]` | **True** | **`raise RuntimeError`** — **노드 크래시** |

⚠️ **`joint_delta_position_limits`의 단위가 코드와 주석이 다르다.** YAML 주석은 `# (rad/s)`인데 `check()`는 **dt로 나누지 않는다** → 실제로는 **"이벤트당 절대 델타 [rad]"**다. 500 Hz 틱에서 `1.8 rad/step`은 **900 rad/s**에 해당 → 속도 리밋으로서는 무력하고, 사실상 "1회 명령 점프 가드"로만 동작한다. **우리는 (a) 진짜 속도 리밋(`|Δq|/dt`)과 (b) 점프 가드를 분리 정의해야 한다.**

**그리퍼 파지력**: `gripper_posforce_limits: [50.0, 1.0]` 리터럴은 실재하나, API는 **per-unit**(`torque_pu ∈ [0,1]`, `gripper_component.cpp:41`에서 clamp)이고 드라이버는 `torque_pu = limits[1] / 4.5`를 넘기며 바로 위에 `# TODO: Now We should multiply 10 to convert Nm to pu?`(driver.py:206-210)가 달려 있다. 실제 전달값은 **0.222 pu**. 또 50 rad/s는 DM4310의 V_MAX=30 rad/s를 **초과**한다. ⇒ **"파지력 상한 1.0 Nm"은 근거가 없다.** 실측 캘리브레이션으로 확정해야 한다 → §5 Q3.

**제어 주파수의 상한**: `openarm_can` 공식 문서 경고 — **8모터 구성에서 1000 Hz 초과 제어 사이클은 CAN 연결을 불안정하게 만든다.** 선언된 주파수: `openarm_teleop` **1000 Hz** 전역(`#define FREQUENCY 1000.0`) / 바이래터럴 스레드 **500 Hz**(`openarm_bilateral_control.cpp:42,72,103`), 카메라 **30 fps**.

🔴 **LeRobot 제어 루프의 실측 사이클 타임은 [미측정]이다.** 그리고 LeRobot 루프의 한 사이클은 `get_observation()`(refresh 라운드) + `send_action()`(MIT 라운드)이므로 **32 프레임/사이클(패턴 B)** → CAN-FD 이론 상한 **625 Hz**다(`15` §2.1). **500 Hz는 버스 부하 80%.** 즉 Python 속도 이전에 **버스부터 빠듯하다.** → `15` NFR-PRF-054, `16` M-21 / M-25.

**recv 타임아웃**: 최소 100 µs(수동 sleep 필요), 표준 **500 µs**, 느린 연산 2000 µs. python-can socketcan 경로에서 우리가 이 값을 어떻게 제어하는지는 `[미확인]`.

### 2.8 설정 계층 — 무엇이 어디에 사는가 [신규구현]

LeRobot의 설정은 **전부 CLI 인자(draccus)에만 존재**하고 영속화 계층이 **없으므로** 신규 정의한다. 포맷 선택 근거: 프로파일·게인·리밋은 사람이 읽고 diff해야 하므로 YAML, 세션·계보는 다중 필드 조인 질의가 필요하므로 SQLite(YAML로 불가), 시계열은 append-only 복구성이 필요하므로 MCAP이다.

**워크스페이스 레이아웃** (루트 = `$OPENARM_WORKSPACE`, 기본 `/var/lib/openarm`):

```
$OPENARM_WORKSPACE/
├── profiles/<profile_id>/
│   ├── profile.yaml            # 로봇 프로파일 (하드웨어 불변 사항 + 참조)
│   ├── gains/<gain_set>.yaml   # 게인 프로파일 (복수 공존, §2.7)
│   ├── limits/<limit_set>.yaml # 리밋 프로파일 (복수 공존)
│   ├── cameras.yaml            # 카메라 역할↔udev 심링크↔해상도/FPS (뎁스=RealSense, RGB=OpenCV 혼합)
│   ├── calibration/<cal_id>.yaml   # 영점/그리퍼/카메라 내외부/VR↔베이스 정합
│   ├── zero_golden.yaml        # 모터별 m_off(RID 54) 골든값 (§4.2 무결성 검사용)
│   └── mjcf/                   # openarm_control IK용 MJCF (jnt_range를 LeRobot 리밋으로 덮어쓴 사본)
├── tasks/<task_id>.yaml        # 태스크 정의 (프로파일과 독립)
├── sessions/
│   ├── index.sqlite            # 세션·계보·감사 로그 (질의 대상)
│   └── <session_id>/
│       ├── snapshot/           # 불변 설정 스냅샷 (§2.11)
│       ├── events.jsonl        # 이벤트 타임라인
│       └── *.mcap              # 시계열 기록 (관절·명령·진단·영상 메타)
├── datasets/<repo_id>/         # 🔴 LeRobotDataset (정본). 로컬 우선, push_to_hub는 명시적 opt-in
├── runs/<run_id>/              # 학습 실행 (config·체크포인트·로그)
├── evals/<eval_id>/            # 실기 평가 (롤아웃·성공라벨·성공률+CI) [신규구현]
├── logs/*.jsonl                # 구조화 로그
├── diagnostics/*.tar.gz        # 진단 번들
└── state/                      # 크래시 링버퍼, 로컬 상태
/run/openarm/<iface>.lock       # CAN 배타 락 (flock, tmpfs)
```

⚠️ **`datasets/`가 `sessions/<id>/episodes/`를 대체했다.** LeRobot 포맷이 처음부터 정본이므로 별도 1차 저장소가 없다. 세션 ↔ 데이터셋 연결은 `index.sqlite`의 계보 테이블이 담당한다.

**설정 우선순위 (낮음 → 높음)**

| # | 계층 | 파일/출처 | 오버라이드 가능 | 비고 |
|---|---|---|---|---|
| 0 | **안전 엔벨로프** | `profiles/<id>/limits/_envelope.yaml` | **불가** (관리자 + 감사 로그로만 변경) | 절대 상한. 아래 어떤 계층도 이보다 넓힐 수 없다 |
| 1 | 패키지 기본값 | upstream 하드코딩 | — | `dm_motor_constants.hpp` P_MAX/V_MAX/T_MAX |

**불변식**: 계층 2~6의 어떤 값도 계층 0의 안전 엔벨로프를 넘길 수 없다. 넘기는 설정은 **로드 시점에 거부**되어야 하며 조용히 클램프해서는 안 된다.

### 2.9 로깅·기록 포맷

| 대상 | 포맷 | 근거 |
|---|---|---|
| 구조화 로그 | **JSONL** (1줄=1이벤트) | grep/jq 가능, append-only, 부분 손상 내성 |
| 이벤트 타임라인 | `sessions/<id>/events.jsonl` | 위와 동일 |
| 시계열(관절·명령·진단·영상 메타) | **MCAP** (`mcap` Python 라이브러리 직접 사용 — rosbag2 불필요) | "MCAP has an **append-only design**, meaning that even if recording is interrupted, MCAP can guarantee that **only the last recorded message** has any danger of being corrupted." |
| 세션·계보·감사 | **SQLite** (`sessions/index.sqlite`) | 다중 필드 조인 질의 필요 |
| **데이터셋** | 🔴 **LeRobotDataset (정본, 변환 없음)** | `openarm_dataset` pip(0.3.0)에는 **라이터가 없다** — src 트리 확인(dataset/camera/convert/merge/repair/validate/sampler/lerobot_v21/lerobot_v30 — **writer 모듈 부재**). 유일한 라이터는 `dora-openarm-dataset-recorder`(dora 노드)다. → **1차 저장 = LeRobot 포맷.** `openarm_dataset`은 **읽기/변환 전용 도구**로만 쓴다 |
| 프로파일 임포트/익스포트 | **단일 `.tar.gz`** = `profile.yaml` + 서브프로파일 + `mjcf/` + `calibration/` + `manifest.json`(스키마 버전·체크섬·git commit·생성 시각) | — |

⚠️ **MCAP은 시스템 시계열용이지 데이터셋용이 아니다.** 학습 데이터는 LeRobotDataset(parquet + 비디오)이고, MCAP은 **진단·이벤트·CAN 트레이스**를 담는다. 둘을 혼동하지 마라. (구 명세가 rosbag2/Humble을 전제로 MCAP을 골랐던 근거는 **소멸**했으나, append-only 무결성 보장이라는 선택 이유 자체는 유효하다 → 유지한다. rosbag2 없이 `mcap` Python 라이브러리를 직접 쓴다.)

**시각화는 인프로세스다** — 3D 뷰어는 `pyqtgraph.opengl`로 GUI 창 안에 임베드된다(외부 Foxglove/Rerun 서버·라이브 WS 전송 없음, D-2 종결). MCAP은 **오프라인 진단 파일 포맷**일 뿐 라이브 전송 의존이 아니다(필요 시 Foxglove Studio로 사후 열람 가능하나 런타임 의존 없음).

**데이터셋 무결화**: LeRobot v3 문서 — "you **must** call `dataset.finalize()` … otherwise the parquet files will be corrupt." 녹화 종료 경로가 어떻게 끝나든 finalize가 반드시 실행되도록 구조적으로 보장해야 한다.

**🔴 `observation.state` 인덱스 매핑 — 반드시 레지스트리로 관리한다**

`observation.effort`는 **존재하지 않는다**(LeRobot 전체 트리 grep 0건). 토크는 `feature_utils.py:68-89`에서 **`observation.state` 벡터 안으로 평탄화**된다:

| 구성 | 차원 | 내용 |
|---|---|---|
| 단완, `use_velocity_and_torque=False` | **8** | `joint_1.pos … gripper.pos` (deg) |
| 단완, `use_velocity_and_torque=True` | **24** | 8모터 × (`.pos` deg, `.vel` deg/s, `.torque` **Nm**) |
| 양완, `use_velocity_and_torque=True` | **48** | 위 × 2 (`left_` / `right_` 프리픽스) |

`names`에는 `joint_1.torque` 같은 **문자열만** 남는다. ⚠️ **한 벡터에 deg와 Nm이 섞인다**(`damiao.py:568` — 토크는 변환 없이 Nm) → **정규화 통계(mean/std) 산정 시 반드시 고려해야 한다.** 시스템은 인덱스 ↔ 채널 매핑을 레지스트리로 관리하고 스냅샷에 기록해야 한다(FR-OPS-087).

### 2.10 에러 코드 체계 [신규구현]

OpenArm에 에러 코드 체계가 **전무하므로** 신규 정의한다. 형식: `OA-<도메인>-<3자리>`.

| 프리픽스 | 도메인 | 예 |
|---|---|---|
| `OA-CAN-xxx` | CAN 버스 | `001` 버스오프 · `002` 프레임 스테일 · `003` FD 미활성 · **`004` 이중 점유 감지** · `005` 비트레이트 불일치 |
| `OA-MOT-xxx` | 모터 (**Damiao ERR에 1:1 매핑**, §2.4) | `000` 미인에이블 · `008` 과전압 · `009` 저전압 · `00A` 과전류 · `00B` MOS과온 · `00C` 코일과온 · `00D` 통신두절 · `00E` 과부하 |
| `OA-CTL-xxx` | 제어 루프 | `001` 오버런 · `002` 조인트 리밋 클램프 · `003` 명령 점프 가드 발동 · `004` 명령 신선도 초과 |
| `OA-CAM-xxx` | 카메라 | `001` disconnect · `002` 프레임 드롭 · `003` age 초과 |
| `OA-TEL-xxx` | 텔레옵/VR | `001` 하트비트 소실 · `002` 추적 INVALID · `003` 추적 STALE · `004` 왕복 지연 임계 초과 |
| `OA-IK-xxx` | 기구학 | `001` 솔버 무해(`solve() → None`) · `002` EE 잔차 초과 · `003` **무제약 폴백(`limits=[]`) 발동** · `004` 개루프 발산 · `005` rad↔deg 변환 검증 실패 |
| `OA-SYS-xxx` | 시스템 | `001` 디스크 풀 · `002` OOM · `003` RT 권한 없음 · **`004` GUI 백엔드(CAN 소유 프로세스) 사망** · `005` 제어 루프 스레드 사망 · `006` 포트 충돌 |
| `OA-INF-xxx` | 추론 | `001` 정책 서버 무응답 · `002` 액션 큐 고갈 · `003` 추론 폭주(리밋 위반) · `004` 미지원 정책 |
| `OA-DAT-xxx` | 데이터셋/학습 | `001` 에피소드 손상(finalize 누락) · `002` 체크포인트-데이터셋 불일치 · `003` 학습 프로세스 사망 · **`004` 토크 피처 불일치(팔로워/리더 `use_velocity_and_torque` 비대칭)** · **`005` `push_to_hub` 의도치 않은 활성화** |
| `OA-ZRO-xxx` | 영점/캘리브레이션 | `001` 영점 무결성 실패 · `002` m_off 골든값 불일치 · `003` 캘리브레이션 만료 · 🔴 **`004` `connect()` 재호출로 인한 영점 파괴 의심** |

각 코드는 다음 필드를 갖는다:
`{code, severity, message_ko, message_en, hardware_id, subsystem, recovery_hint, doc_url, first_seen_t, count}`

`severity`는 4단계로 한다: **`OK=0 / WARN=1 / ERROR=2 / STALE=3`**. (ROS 2 `diagnostic_msgs/DiagnosticStatus`의 정의를 **의미론만 차용**한다 — ROS 2는 우리 런타임에 없으므로 메시지 타입 의존은 없다.)

**RT 권한** (`/etc/security/limits.conf`):
```
@realtime soft rtprio 99
@realtime soft priority 99
@realtime soft memlock unlimited
@realtime hard rtprio 99
@realtime hard priority 99
@realtime hard memlock unlimited
```
ros2_control 공식 문서(런타임은 안 쓰지만 판단 근거로 유효): "the normal linux kernel is optimized for computational throughput and therefore is **not well suited for hardware control**" → PREEMPT_RT 권장. **PREEMPT_RT 필수 여부는 지터 실측으로 판정한다**(`15` NFR-PRF-040).

**🔴 CAN 인터페이스 구성 — 코드가 대신할 수 없는 필수 단계**

python-can socketcan 백엔드는 `bitrate` / `data_bitrate`를 **무시한다**(`SocketcanBus.__init__` 시그니처에 없음, `socketcan.py:694-703`). 소켓에 작용하는 것은 `fd`뿐(`:765-769` `setsockopt(SOL_CAN_RAW, CAN_RAW_FD_FRAMES, 1)`). **따라서 `ip link`로 FD를 켜는 단계는 선택이 아니라 필수이고, GUI는 기동 시 이를 검증해야 한다.**

```bash
# 방법 A — LeRobot 내장 (권장)
lerobot-setup-can --mode=setup --interfaces=can0,can1,can2,can3
#   내부적으로: ip link set canX down
#              ip link set canX type can bitrate 1000000 dbitrate 5000000 fd on
#              ip link set canX up
lerobot-setup-can --mode=test  --interfaces=can0,can1,can2,can3

# 방법 B — 직접 (restart-ms / txqueuelen은 LeRobot도 enactic 스크립트도 안 건드린다)
sudo ip link set can0 type can bitrate 1000000 dbitrate 5000000 fd on restart-ms 100
sudo ip link set can0 txqueuelen 1000 up      # 기본 10은 8모터 고주파에 얕다

# 검증 (기동 시 필수 — FR-OPS-084)
ip -details link show can0    # fd on / bitrate 1000000 / dbitrate 5000000 확인
```

⚠️ **enactic의 `openarm-can-configure-socketcan-4-arms`는 쓰지 않는다** — `FD_MODE` 기본값이 **`false`**라 `-fd`를 안 주면 CAN 2.0으로 올라가는데, LeRobot은 `use_can_fd=True`가 기본이라 **어긋난다.** LeRobot이 동등물을 내장한다(`lerobot_setup_can.py:96-121`).

커널 문서: "An automatic bus-off recovery can be enabled by setting the `restart-ms` to a non-zero value."
커널 문서: "The system generates *error message frames* … Applications request these via **`CAN_RAW_ERR_FILTER`**." → **에러 프레임은 애플리케이션이 명시적으로 구독해야 받는다.**

**udev — 고정 이름 (전부 우리가 작성)**

동일 VID/PID 어댑터 2개를 꽂으면 `can0`~`can3` 할당은 **부팅 시 비결정적이다**(USB 열거 순서 경합) → **udev 고정은 선택이 아니다.**

```udev
# CAN 어댑터 — 채널별 구분은 dev_id로 (멀티채널 장치는 netdev->dev_id = channel)
# ⚠️ NAME= 을 can0~can3 으로 주지 마라 — 커널이 이미 canN을 할당 중이라 이름을 두고 경합한다.
#    "can"으로 시작하지 않는 이름을 써서 네임스페이스를 분리한다.
SUBSYSTEM=="net", ACTION=="add", ATTR{type}=="280", \
  ATTRS{idVendor}=="XXXX", ATTRS{idProduct}=="YYYY", ATTRS{serial}=="AAAA", \
  ATTR{dev_id}=="0x0", NAME="oa_fl"          # follower left
SUBSYSTEM=="net", ACTION=="add", ATTR{type}=="280", \
  ATTRS{idVendor}=="XXXX", ATTRS{idProduct}=="YYYY", ATTRS{serial}=="AAAA", \
  ATTR{dev_id}=="0x1", NAME="oa_fr"          # follower right

# 카메라: 역할 기반 심링크 (ATTR{index}=="0" 필수 — UVC 1대가 /dev/videoN을 2개 이상 생성)
SUBSYSTEM=="video4linux", ATTRS{serial}=="SN123", ATTR{index}=="0", SYMLINK+="openarm/cam_wrist_left"
SUBSYSTEM=="video4linux", ATTRS{serial}=="SN456", ATTR{index}=="0", SYMLINK+="openarm/cam_ceiling"
```

- `ARPHRD_CAN = 280` (`<linux/if_arp.h>`) → `ATTR{type}=="280"`.
- ✅ **LeRobot의 `port`는 그냥 `channel=` 문자열로 넘어간다**(`damiao.py:180`) → **`--robot.port=oa_fl`이 그대로 동작한다.** `canN` 형태일 필요가 **전혀 없다**(`damiao.py:171` — `/dev/`로 시작하지만 않으면 socketcan).
- ⚠️ 저가 어댑터는 **iSerial이 없거나 중복**일 수 있다 → 그러면 **USB 물리 포트 경로**로 고정한다(`KERNELS=="1-1.2:1.0"`). 케이블을 옮기면 깨지므로 **리그에 라벨을 붙여라.**
- ⚠️ Arducam은 **공장 출하 시 시리얼이 전부 동일**하다 → `ArducamUvcConfigUpdateTool`로 **시리얼을 구워야** 위 규칙이 성립한다(`16` M-19).

**LeRobot 채널 관례** (HF 공식 문서의 바이매뉴얼 예제): `can0` = follower **left** / `can1` = follower **right** / `can2` = leader left / `can3` = leader right.
⚠️ **`openarm_teleop`(C++)은 정반대다** — `leader = can0`, `follower = can2`(`openarm_bilateral_control.cpp:154-155`).
✅ **KER 리더를 쓰면 can2/can3이 비게 된다**(KER은 USB) → 어댑터 1개(2ch)로 양팔 팔로워를 다 커버.

### 2.13 셀프테스트 선례 (그대로 채택)

`openarm_can/setup/motor_sampling_check.cpp`:
- 측정: 송수신 **RTT(µs)**, 목표 Hz 대비 **실측 Hz**
- 출력: Loop Count / target·actual frequency / RTT 통계 / 모터별 Position·Velocity·Torque·**온도(°C)**
- **경고 임계값: `actual_hz < target_hz * 0.95` → "LOW" 적색 표시** ← 우리 NFR의 근거
- 통계 갱신 주기 1.0 초

`lerobot-setup-can --mode=setup|test|speed`: 인터페이스 구성 / 모터 `0x01`–`0x08` 스캔 / 속도·지연 테스트. FD 활성 확인 `ip -d link show can0 | grep fd`.
⚠️ **이건 별도 프로세스다.** 셀프테스트 중에는 GUI 백엔드가 CAN을 잡고 있으면 안 된다 → 프리플라이트(연결 전) 단계로 배치하거나, 우리가 동일 로직을 in-process로 구현한다.

**퍼블리셔 주파수 감시**(카메라 프레임 드롭 감지)는 ROS 2 `diagnostic_updater::DiagnosedPublisher`가 하던 일이다 → **런타임에 ROS 2가 없으므로 [신규구현]**. 롤링 윈도 FPS + `jitter_ms` 계산은 `15` NFR-PRF-012에 규격이 있다.

### 4.2 전원 상실 → 복구 시퀀스 (필수 절차)

**전제**: Damiao 2EC는 **이중 엔코더 + 출력축 단회전 절대위치**를 가지므로 **관절 절대각은 전원 재인가만으로 복원된다. 매 부팅 재영점은 불필요하다.**
**단서**: `m_off`(RID 54)의 flash 영속성이 1차 문서로 확정되지 않았다(§5 Q1) → **확정 전까지 부팅 시 영점 무결성 검사를 무조건 수행한다.**

```
[전원 상실 / Cat-0 E-Stop]
  └─ 토크 즉시 0 → 브레이크 없음 → 팔이 중력으로 무너짐
     (공식 안전 가이드: "if power is lost due to an emergency stop, the load being held will fall rapidly")

[전원 재인가]
  1. CAN 링크 확인 (`ip -d link show`: fd on / bitrate / dbitrate, 에러 카운터 0)   (FR-OPS-084)
  2. flock 획득 + 침입자 탐지 (바인딩 소켓 수 = 1)                                  (FR-OPS-029/030/088)
  3. 🔴 모터를 enable 하기 전에 전 관절 위치를 읽는다
     ⚠️ 그런데 LeRobot의 robot.connect() 는 set_zero_position() 을 부른다!
     → 위치 판독은 반드시 robot.connect() 이전에, 저수준 CAN(또는 is_calibrated=False 상태)으로
       해야 한다. connect() 를 먼저 부르면 낙하한 자세가 영점이 된다. (F23)
  4. 영점 무결성 검사 (FR-OPS-051):
     a. 각 관절 q_i 가 프로파일 joint_limits 안에 있는가?
     b. RID 54 m_off 를 읽어 zero_golden.yaml 의 골든값과 비교 (RO라 읽기만 가능)
     c. (선택) RID 81 xout 을 읽어 출력축 절대 위치를 교차 확인
     → 하나라도 불일치: connect/enable 금지. OA-ZRO-001/002 발행. "재영점 필요" 상태 유지.
  5. 실측 자세를 3D로 렌더링해 사용자에게 보여주고
     "이 자세가 실제 팔과 일치하는가?" 명시적 확인을 받는다  (FR-OPS-053)
     → 센서가 거짓말하는 경우를 사람이 잡는 유일한 방어선
  6. 낙하 자세가 자기충돌/환경충돌 상태인지 검사   (FR-OPS-054)
     → 그렇다면 자동 홈잉 금지. 전원 차단 후 수동 해제 안내.
  7. 🔴 팔을 rest 자세로 되돌린 뒤에만 robot.connect() 를 호출한다  (FR-OPS-055/083)
     → connect() 가 set_zero_position() 을 부르므로, 이 시점의 자세가 새 영점이 된다.
  8. 첫 send_action() 의 q_des 는 반드시 실측 q  (kp/kd 정상값)          (FR-OPS-052)
     q_des=0 이나 임의 홈 값을 먼저 때리면 안 된다.
     (반면교사: openarm_ros2 의 return_to_zero() 는 현재 위치 캡처 전에 q_des=0 을
      풀 게인으로 송신한다 → openarm_ros2#66 OPEN, PR #67 미머지)
  9. 낮은 kp (기본 정상의 30%) 로 3초간 현재 자세 홀드 → 진동/발산 없음 확인
 10. kp 램프업 → 홈으로 스무스 이동 (duration ≥ 4 s; 낙하 자세는 홈에서 멀다)
 11. → HOLD 상태 진입. CAN 명령 스트림은 이 시점부터 절대 끊지 않는다.
```

> 🔴 **3단계와 7단계의 순서가 이 시퀀스의 핵심이다.** LeRobot의 `connect()`는 "안전한 rest 자세"를 **가정**할 뿐 **검증하지 않는다**(`openarm_follower.py:133-134`). 낙하 상태에서 무심코 `connect()`를 부르면 그 순간 복구가 아니라 **영점 파괴**가 된다.

**낙하 자체를 줄이는 설계** (기구적 브레이크가 없으므로 전기/기구로 풀어야 함):
- GUI의 정지 버튼은 **Cat-2(전기적 정지: `MIT(q=qh)` 홀드, 전원·CAN 유지)** 로 매핑한다. Cat-0(전원 차단)은 물리 버튼에만 남긴다.
- 🔴 **Cat-0 E-Stop은 모터 전원을 차단하며, 그 순간 CAN 통신도 함께 끊긴다(m2 확정).** 따라서 "E-Stop 중 상태를 읽어 복구"는 불가능하다 — 위 절차 1~5(“enable 전에 위치를 읽는다”)는 **전원 재인가 이후**에만 성립한다. (Cat-0 중 CAN을 살려두는 분리 배선은 전제하지 않는다.) → §5 Q4.
- 최악을 전제로 워크셀을 설계한다: 낙하 궤적 아래에 사람·작업물 없음.

## 5. 미해결·결정필요

### Q1. `0xFE`(set_zero)가 flash에 영속되는가? [미확인 — 비차단, m1으로 설계 무관]
**설계는 이에 의존하지 않는다(m1).** SoT는 **디스크 JSON**이고, 영점은 bring-up 시 operator `set_zero`(rest 자세 정렬 → 버튼)로 재확립한다(§2.5). 영속되지 않아도 전체 하드스톱 범핑을 **매번 돌릴 필요는 없다**(rest에서의 재-set_zero면 충분). **0xAA 강제 저장에도 의존하지 않는다.** 확인 시 이득: 영속되면 bring-up이 영점 무결성 검사만으로 짧아진다. OpenArm v2의 set_zero flash 의미론은 잔여 미확인이다.
**현재 증거**: Damiao 1차 매뉴얼은 `0xFC/0xFD/0xFE` 특수 명령 자체를 문서화하지 않는다(`0x33` 읽기 / `0x55` 쓰기 / `0xAA` 저장만 기술). "쓰기는 휘발, 0xAA로 저장" 문장은 **RW 레지스터**에 대한 것이고 `m_off`는 **RO**다. 정황은 전부 "영속된다" 쪽(OpenArm이 영점을 1회성 셋업으로 취급 / i2rt가 `save_zero_position()`으로 명명 / OpenArm이 0xAA를 아예 안 보내는데도 정상 운용)이나 **어느 것도 1차 명문이 아니다.**
**확정 방법 (30분 실기)**:
```bash
# 1) 임의 자세로 이동 후 기록
openarm-can-cli -i can0 monitor --id 1,2,3,4,5,6,7,8 -d 3000   # q_before[8]
openarm-can-cli -i can0 show_param -c 1 -r 54                  # m_off_before
# 2) 그 자리에서 영점 설정 (0xFD → 0xFE → 0xFD)
openarm-can-cli -i can0 set_zero --id 1,2,3,4,5,6,7,8
openarm-can-cli -i can0 show_param -c 1 -r 54                  # m_off_after
# 3) 팔을 건드리지 말고 전원 OFF → 10초 → ON
# 4) 재확인
openarm-can-cli -i can0 monitor --id 1,2,3,4,5,6,7,8 -d 3000
openarm-can-cli -i can0 show_param -c 1 -r 54
openarm-can-cli -i can0 show_param -c 1 -r 80   # p_m  (로터측)
openarm-can-cli -i can0 show_param -c 1 -r 81   # xout (출력축)
```
| 관측 | 결론 | 명세 영향 |
|---|---|---|
| 위치 ≈ 0, `m_off` = `m_off_after` | **0xFE는 flash에 영속** | bring-up이 영점 무결성 검사만으로 짧아진다. **가장 유력** |
| 위치가 `q_before`로 복귀, `m_off` = `m_off_before` | **0xFE는 RAM만 변경** | bring-up마다 operator `set_zero`로 재영점(디스크 SoT는 그대로). **0xAA에 의존하지 않는다(m1)** |
| 둘 다 아님 / 랜덤 | 출력축 절대 엔코더 가정이 깨짐 | 🔴🔴 §4.2 전면 재검토 |

### Q2. 관절 리밋의 클램프 정본은 무엇인가? [결정필요]

**확인해야 할 것**: (a) 두 프레임 간 변환(`joint_offsets` 적용 전/후)이 정확히 무엇인지, (b) 기구적 하드스톱 실측치(`MECH_LIM_V2_*`: J1 `-80°…+200°`, J4 `0°…+140°`)와 어느 쪽이 정합하는지. **안전 엔벨로프(계층 0)는 기구 하드스톱에서, 운용 리밋(계층 3)은 태스크별 프로파일에서 온다**는 것이 유력한 정리이나, 프레임 변환을 실기로 검증하기 전에는 확정할 수 없다.

### Q3. 그리퍼 파지력 상한의 실제 물리량은 얼마인가? [결정필요]

### Q4. E-Stop의 차단 범위 [해결 — m2]
**확정(m2)**: Cat-0 물리 E-Stop은 모터 전원을 차단하며, 그 순간 **CAN 통신도 함께 끊긴다.** 따라서 "E-Stop 후 상태를 읽어 복구"는 **불가능하다는 전제로 설계한다.** 상태 판독·영점 무결성 검사(§4.2)는 **전원 재인가 이후**에만 수행한다.
- Cat-2(SW/GUI 정지)는 전원·CAN을 유지하고 `MIT(q=qh)` 홀드로 낙하를 막는다 — 이것이 기본 정지 경로다.
- Cat-0 중 CAN을 살려두는 "분리 배선"은 전제하지 않는다. 낙하 완화는 Cat-2 우선 + 워크셀 설계(낙하 궤적 아래 사람·작업물 배제)로 한다.

### Q5. VR 텔레옵 지연 [보류 — d10, 당장 평가 안 함]
**결정(d10)**: 텔레옵 지연 성능은 **당장 평가하지 않는다.** 하드 지연 상한을 명세에 못 박지 않는다 — NFR-OPS-005는 "실측 왕복 지연 상시 표시 + 사용자 설정 임계 초과 시 자동 중단"으로만 둔다. arXiv 2603.06850(**CARLA 차량 차선유지** — 도메인 불일치)을 근거로 상한을 정하지 않는다. 필요해지면 그때 우리 하드웨어·태스크로 자체 벤치를 돌려 완수율-지연 곡선을 그린다(참고: Quest2ROS 82 ms는 **Quest 2 컨트롤러** 측정치, Quest 3S 실측치 없음).

### Q6. 게인 정본은 어느 것인가? [결정필요]
3세트가 실재하고 **어깨 게인이 70 vs 240**이다. 단일 하드코딩은 즉시 어긋난다(FR-OPS-011이 프로파일로 우회).
**확인해야 할 것**: 각 세트로 동일 궤적을 돌려 (a) 추종 오차, (b) 진동/오버슈트, (c) 접촉 시 백드라이버빌리티, (d) 소비 전류를 실측 비교한다. 텔레옵·데이터수집·추론이 **서로 다른 세트를 필요로 할 가능성이 높으므로**, "단일 정본"이 아니라 "모드별 정본"이 답일 수 있다.

### Q7. Damiao RID 9 `TIMEOUT`의 공장 기본값과 단위는? [미확인 — 비차단, m4]
매뉴얼은 "설정된 주기 내에 CAN 명령을 못 받으면 인에이블 이탈"이라고만 하고 **기본값·단위(ms 추정)를 명시하지 않는다.** 0이면 워치독 비활성(무한 홀드), 0이 아니면 명령 스트림이 끊긴 뒤 그 시간 후 인에이블 이탈된다.
**결정(m4)**: RID 9를 **하드 상한 FR로 못 박지 않는다.** 대신 우리 쪽 **명령 신선도 워치독**(FR-OPS-035)이 명령 스트림을 항상 신선하게 유지하고, "홀드"는 `MIT(q=qh)` 지속 송신(FR-OPS-034)으로 구현한다 → RID 9가 발동할 상황 자체를 만들지 않는다.
**확인 가치(비차단)**: `openarm-can-cli show_param -c 1 -r 9`로 값을 판독해 두면 최소 명령 송신 주기의 안전 마진 참고가 된다.

### Q8. 🔴 **LeRobot Python 제어 루프가 500 Hz를 낼 수 있는가?** [미확인] — **이 문서의 최우선 미결**
`openarm_teleop`(C++)의 바이래터럴 스레드는 **500 Hz**다(`openarm_bilateral_control.cpp:42,72,103`). **LeRobot Python 루프의 실측 사이클 타임은 [미측정]이다.**
**왜 결정적인가**: 500 Hz가 안 나오면 **바이래터럴만 별도 실시간 프로세스로 분리**해야 하고, **그 순간 CAN 배타 소유권 문제가 부활한다**(§2.1의 단일 프로세스 전제가 무너진다). NFR-OPS-002·NFR-OPS-010·FR-OPS-063이 전부 여기 매달려 있다. ⚠️ **단, 500 Hz는 바이래터럴 힘반사 전용이며 주경로(유니래터럴/record/eval/추론)의 요건이 아니다**(m21).
**측정 규격**: `15` §2.10 M-8 / NFR-PRF-054. **판정선**: 부하 조건(카메라 5스트림 + 인코딩 + 기록 동시)에서 사이클 타임 **p99 ≤ 2.0 ms** 이고 **오버런율 ≤ 0.1%**.
**추가 함정**: LeRobot 루프의 한 사이클은 `get_observation()`(refresh 라운드) + `send_action()`(MIT 라운드)이므로 **32 프레임/사이클** → CAN-FD 이론 상한 **625 Hz**다. **Python 속도 이전에 버스부터 빠듯하다.** 프레임 수 확정은 `16` M-25(`candump -t d`로 한 사이클을 세면 즉시 판명).

### Q9. `openarm_driver`가 LeRobot과 같은 프로세스에서 CAN을 여는가? [미확인]
`openarm_control`(IK/FK)은 CAN을 열지 않는다(확인함). `openarm_ker`는 **USB만 쓴다**(확인함, `ker_stream.py:103-104` → pyusb). **`openarm_driver`는 미확인이다** — `driver.py` / `safety.py`가 있는데 내부 CAN 개방 여부를 읽지 않았다.
**왜 중요한가**: **연다면 §2.3의 이중 바인드 문제에 직접 걸린다** — 같은 프로세스 안에서도 두 개의 소켓이 열린다.
**확인해야 할 것**: `openarm_driver/driver.py` 정독. **연다면 우리 경로에서 배제한다**(`openarm_control`만 쓴다). → `16` M-24.

### Q10. 컨테이너를 쓸 것인가, 어떤 베이스인가? [결정필요]
구 명세는 `osrf/ros:humble-desktop-full`을 전제했으나 **ROS 2가 경로에서 사라져 근거가 소멸했다.** upstream에 우리 경로에 맞는 컨테이너 선례가 **없다**.
**확인해야 할 것**: (a) CUDA 의존(학습·추론)을 컨테이너에 넣을지, (b) SocketCAN 접근(`network_mode: host` / `NET_ADMIN`)과 RT 스케줄링(`SYS_NICE` + `ulimits.rtprio`)의 조합이 실제로 동작하는지, (c) MuJoCo 렌더링(IK는 렌더 불필요하나 시뮬 뷰는 필요)의 GPU 접근.
