> 요구사항 ID 영역 코드: **`PRF`** (성능/예산, Performance & Budget).
> **이 문서의 모든 요구사항은 비기능 요구사항이므로 `NFR-PRF-*` 접두사를 쓴다** — 00 §5.1의 "비기능 요구사항은 `NFR-<영역>-<3자리>`" 규약을 그대로 따른다. 영역 코드를 `NFR`로 두면 `NFR-NFR-001`이라는 퇴화형이 되므로, 이 문서의 영역 코드는 내용(성능·지연·처리량·리소스 예산)을 가리키는 `PRF`로 한다.

## 1. 범위

이 문서는 CAN 버스 예산, **LeRobot Python 제어 루프의 사이클 타임**, 텔레옵 지연 예산, 처리량(카메라·기록·디스크), 신뢰성(유실률·드롭·연속운용·복구), 실시간성(GIL·PREEMPT_RT), 리소스(CPU·GPU·메모리·USB), 그리고 이들을 확정하기 위한 **자체 벤치마크 요구사항**을 규정한다.

다루지 않는 것: 기능 동작 자체(텔레옵 매핑·IK 알고리즘·데이터셋 스키마·안전 정지 시퀀스). 이 문서는 그것들이 **얼마나 빨라야 하고, 얼마나 자주 실패해도 되는가**만 다룬다.

**이 문서의 전제 (v2 런타임)**: CAN을 소유하는 것은 **LeRobot Python API를 임베드한 단일 GUI 백엔드 프로세스**다. dora도 ROS 2도 없다.

### 🔴 이 문서에서 가장 중요한 세 문장

1. **우리 주경로는 LeRobot record(30–60 Hz)·정책 추론·실기 평가(dora식 250 Hz)이며, 이 대역은 CAN·Python 예산에 여유가 크다.** **바이래터럴 500 Hz는 힘반사(force-reflection) 전용 선택 경로**이지 주경로 요건이 아니다(D-4/m21). LeRobot Python 루프의 실측 사이클 타임은 여전히 [미측정]이고 **주경로 주파수를 확정하려면 재야 한다**(NFR-PRF-054 / M-8). **다만 500 Hz는 바이래터럴을 구현할 때에만** 도전 목표가 되며, 못 내면 바이래터럴만 별도 RT 프로세스로 분리해야 하고 그 순간 CAN 배타 소유권 문제가 부활한다.
2. **IK는 병목이 아니다** — 양팔 실측 **0.345 ms / p95 0.355 ms**, 60 Hz 예산의 **2%**. `[확정, 실측]`
3. **텔레옵 지연 허용치는 당장 확정하지 않는다(D-10 — 텔레옵 성능 평가 안 함).** 유일한 후보 근거(arXiv 2603.06850)는 **CARLA 차량 차선유지** 실험이라 도메인이 불일치했다. C-Lat/V-Lat 임계 NFR은 제거하고(§3.2), 채널 계측은 계기로만 유지한다. (정책 성공률 평가 M-6는 별개로 유지된다.)

## 2. 확정된 사실 기반

### 2.1 CAN 버스 예산 — 모든 제어 주파수 NFR의 뿌리

제어 주파수의 물리 상한은 대역폭이 아니라 **(사이클당 프레임 수) × (프레임 시간)** 이 정한다.

**프레임 시간** (표준 11비트 ID, 데이터 8바이트):

| 모드 | 비트 수 | 프레임 시간 | 설계 상수 |
|---|---|---|---|
| CAN-FD, 1 Mbps 공칭 / 5 Mbps 데이터, **BRS=1** | 138 bit | 50.80 µs | **50 µs** |
| Classic CAN 2.0, 1 Mbps | 111–135 bit | 111–135 µs | **120 µs** |

- `openarm_can`은 CAN-FD 프레임에 **`CANFD_BRS` 플래그를 실제로 세팅**한다 (`dm_motor_device.cpp`: `frame.flags = CANFD_BRS;`). 이 확인이 없으면 위 50 µs 계산은 무효다.
- 교차검증: CAN FD Frame Length Calculator (8바이트, 1M/5M, BRS) = **138 bit / 50.80 µs**.

**사이클당 프레임 수** (팔당 CAN 1채널, 모터 8개 = 7축 + 그리퍼):

| 패턴 | 구현 | read/상태 획득 | write/명령 | 프레임/사이클 |
|---|---|---|---|---|
| **A. MIT만** | `openarm_teleop`, `motor_sampling_check` | 없음 (MIT 응답을 상태로 사용) | `mit_control_all` + `recv_all` | **16** (명령 8 + 응답 8) |
| **B. refresh + MIT** | 🔴 **LeRobot 루프** (`get_observation()` + `send_action()`), `openarm_driver` | `sync_read_all_states()` — 주석: *"Reads all motor states (pos/vel/torque) in **one CAN refresh cycle**"* | `_mit_control_batch()` | **32** (refresh 8 + 응답 8 + MIT 8 + 응답 8) |

> DM 모터는 **MIT 명령에 대해서도 상태 피드백 프레임을 되돌려준다** — `motor_sampling_check.cpp`가 MIT만 보내고 위치/속도/토크/온도를 전부 표시한다. 따라서 패턴 B의 refresh 라운드는 **원리적으로 중복**이다.
> 상태 피드백 8바이트에 `data[6]=T_MOS`, `data[7]=T_Rotor`가 실려 온다 → **온도 폴링은 추가 대역을 소모하지 않는다.** (LeRobot `MotorState`도 `temp_mos`/`temp_rotor`를 갖는다 — `damiao.py:68-73`.)

**버스 부하표** (모든 주파수 NFR의 근거):

| 패턴 | 버스시간/사이클 | 100% 부하 상한 | 70% 부하 상한 | **500 Hz 부하** | 1000 Hz 부하 |
|---|---|---|---|---|---|
| **A (CAN-FD)** | 16 × 50 µs = **800 µs** | **1250 Hz** | 875 Hz | **40%** ✅ | **80%** ⚠ |
| 🔴 **B (CAN-FD) = LeRobot 루프** | 32 × 50 µs = **1600 µs** | **625 Hz** | 437 Hz | **80%** ⚠ | 160% ❌ |
| A (Classic CAN) | 16 × 120 µs = 1920 µs | **520 Hz** | 364 Hz | 96% ❌ | ❌ |

**규범적 상한**: `openarm_can` 공식 문서는 **8모터 구성에서 1000 Hz를 초과하는 제어 사이클이 CAN 연결을 불안정하게 만든다**고 경고한다. 공식 상한(1000 Hz)이 규범이다.

**🔴 따라서 확정되는 것 — 바이래터럴 500 Hz의 진짜 장벽은 두 겹이다:**

1. **버스 장벽**: LeRobot 루프는 **패턴 B(32 프레임/사이클)**이므로 CAN-FD 이론 상한이 **625 Hz**다. **500 Hz = 버스 부하 80%.** 여유가 20%밖에 없다. → **Python 속도 이전에 버스부터 빠듯하다.**
2. **Python 장벽**: LeRobot 루프의 **실측 사이클 타임이 [미측정]**이다(NFR-PRF-054 / `16` M-21).

**500 Hz를 성립시키려면 패턴 A로 내려가야 한다** — `get_observation()`을 매 사이클 부르지 말고 **MIT 응답을 상태로 사용**하는 것이다. 그러면 부하가 80% → 40%가 된다. 다만 이는 LeRobot의 `OpenArmFollower` 상위 API를 우회하고 `robot.bus`를 직접 다루는 것을 의미한다.

⚠️ **프레임 수 확정 필요**: `sync_read_all_states()`가 refresh 라운드(8+8)를 쓰는지 MIT 응답을 재사용(0)하는지는 **주석("one CAN refresh cycle")으로 추정**한 것이다. `candump -t d can0`으로 한 사이클을 세면 **즉시 판명된다** → `16` M-25. 이것이 32 vs 16 프레임, 625 vs 1250 Hz를 가른다.

**Classic CAN 2.0 폴백**은 **350 Hz 이하로 클램프**되어야 한다 (패턴 A 520 Hz 상한의 70%).

### 2.2 수신 구조의 구조적 결함 — 상태 신선도가 균일하지 않다

```cpp
// openarm_can/src/openarm/can/socket/openarm.cpp — recv_all()
int timeout_us = first_timeout_us;               // 첫 응답만 기다림 (기본 500 µs)
while (can_socket_->is_data_available(timeout_us) && can_socket_->read_canfd_frame(f)) {
    master_can_device_collection_->dispatch_frame_callback(f);
    timeout_us = 0;                              // ← 이후 select() 타임아웃 0
}
```

버스상 응답 프레임 간격은 약 50 µs인데 `select(0)`는 즉시 반환한다. **RX 큐가 순간적으로 비면 드레인 루프가 조기 종료**되고, 아직 도착하지 않은 응답은 다음 사이클의 `recv_all`이 주워간다.

함의: (a) 조인트별 상태 신선도(freshness)가 불균일해지고, (b) 뒤쪽 모터일수록 1사이클 지연된 상태로 제어될 수 있다 (500 Hz에서 1사이클 = 2.0 ms). 실제 발생 빈도는 `[미측정]` — **사이클당 소비 프레임 수를 세면 즉시 판명된다**(M-8 조건 7).

⚠️ 이 코드는 `openarm_can`(C++)의 것이다. **python-can 경로에서 LeRobot이 어떻게 드레인하는지는 `[미확인]`**이나, 동일한 "첫 응답만 기다리고 나머지는 논블로킹" 패턴이면 같은 문제가 있다. `damiao.py`의 recv 루프를 확인해야 한다.

recv 타임아웃 권장값: 최소 100 µs(수동 sleep 필요), 표준 **500 µs**, 느린 연산 2000 µs. `openarm_teleop`은 `sleep_for(200 µs)` + `recv_all(220)`을 쓰며, 이것이 **8모터 응답 왕복이 200–420 µs 안에 끝난다는 유일한 경험적 RTT 간접 근거**다 (공식 수치 없음).

### 2.4 IK solve time — **실측. 병목이 아니다** [확정]

**정본 수치 (`openarm_control` 기본 설정, 상류 튜닝값):**

| 구성 | p50 | p95 | 성공률 | 60 Hz 예산(16.7 ms) | 500 Hz 예산(2 ms) |
|---|---|---|---|---|---|
| **양팔(bimanual), 기본 설정** | **0.345 ms** | **0.355 ms** | **500/500** | **2%** ✅ | **18%** ✅ |

상류 튜닝값(그대로 채택): `--mode bimanual --max-iters 10 --dt 0.1 --damping 0.1 --posture-cost 0.01 --lm-damping 0.01`.
구현: `openarm_control/kinematics.py` — `import mink, mujoco` / `solver: str = "daqp"`(`:54`) / `mink.FrameTask` 좌우 동시(`:145-146`) / `mink.DofFreezingTask`(`:164-165`) / `mink.Configuration`(`:136`) → **양팔 단일 QP**.
⚠️ **`def solve(self) -> np.ndarray | None`(`:107`) — 인자가 없다.** README의 `solve(dt=, n_iters=)`는 **틀렸다.**

> 🔴 **결론: IK는 지연 예산에서 사실상 무시할 수 있다.** 구 명세가 "IK 솔버를 교체해야 한다"고 본 것은 MoveIt/KDL 전제 때문이었고, **그 전제가 사라졌다.**

**충돌회피를 켜면 달라진다** (별도 벤치, `openarm_mujoco` v2 `cell.xml`, nq=nv=19, Intel Core Ultra 9 285 / mujoco 3.10.0):

| 구성 | mink | p50 (ms) | p99 (ms) | EE 오차 p50 |
|---|---|---|---|---|
| bimanual, iters 10, 충돌회피 **off** | 1.1.0 | 0.702 | 0.749 | 0.68 mm |
| bimanual, iters 10, 충돌회피 **팔↔팔 321쌍** | 1.2.0 | **1.053** | **1.115** | 0.68 mm |
| bimanual, iters 5, **팔+셀 591쌍** | 1.2.0 | **1.095** | **1.325** | 0.79 mm |
| bimanual, iters 10, 591쌍 | 1.2.0 | 2.158 | 2.716 | 0.72 mm |
| bimanual, iters 10, 321쌍 (broadphase 없음) | 1.1.0 | 2.602 | 2.735 | 0.68 mm |

⚠️ 이 표의 "충돌회피 off" 수치(0.702 ms)가 정본 실측(0.345 ms)보다 **2배 크다** — 모델(`cell.xml` vs 기본 `ArmSetup`)과 mink 버전이 다르기 때문이다. **보수적으로 보려면 이 표를, 실제 배포 구성으로 보려면 정본 수치를 쓴다.** 두 수치의 차이 원인은 `[미확인]`이며, NFR-PRF-052(재현 메타데이터)가 이런 혼동을 막기 위해 존재한다.

**수렴은 max_iters 5에서 끝난다** (EE 오차 0.74 → 0.68 mm). iters 10 → 30은 시간만 3배 쓰고 오차는 0.01 mm 줄인다.
**충돌회피 비용 설계식** (mink 1.1.0, broadphase 없음, 이 CPU): **≈ 0.59 µs × (geom 쌍 수) × (max_iters)**.
**QP 백엔드** (iters 10): daqp **0.700** / quadprog 0.700 / proxqp 1.455 / **osqp 4.560 (금지)**.
solve time의 68%는 QP가 아니라 **Python QP 조립(`build_ik`)** 이다. 실제 DAQP 호출은 26%.

> ⚠ **모든 수치는 x86 데스크톱이다.** ARM/Jetson 재측정은 `[미측정]`이다.
>
> 🔴 **IK가 병목이 아니라는 것이, 리밋 문제가 없다는 뜻은 아니다.** `mink.ConfigurationLimit`은 **MJCF `jnt_range`만 본다**(MJCF right joint_1 = −80°…+200° vs LeRobot 소프트 −75°…+75°), 그리고 `kinematics.py:220-231`이 `NoSolutionFound` 시 **`limits=[]`로 재시도**한다 → **리밋을 버린 해**를 낼 수 있다. → `14` FR-OPS-043, `16` D-3.

### 2.5 텔레옵 지연 예산 — 채널을 3개로 쪼갠다

기존 노트의 "글래스-투-글래스 ≤150 ms"는 **적용 대상이 존재하지 않는 NFR**이었다. 상류 VR 경로(`dora-openarm-vr` / `-webxr`) 어디에도 **영상 리턴 경로가 구현되어 있지 않다** (enactic/dora-openarm-webxr **issue #2 "Show images by OpenArm in AR"가 OPEN**). 현행 운용은 작업자가 로봇과 같은 공간에서 **Quest 패스스루로 직접 관찰**한다.

| 채널 | 정의 | 적용 조건 | 측정 기준점 |
|---|---|---|---|
| **C-Lat** (제어채널) | 컨트롤러 물리 이동 → 모터 축 실제 반응 | **항상** | 시작: 헤드셋 물리 이벤트 / 끝: 모터 인코더 반응 |
| **V-Lat** (영상채널) | 로봇 카메라 광자 → 헤드셋 디스플레이 광자 | **원격 모드 한정** (영상 리턴 구현 시) | LED-포토다이오드 G2G 리그 |
| **P-Lat** (패스스루) | 실세계 광자 → 헤드셋 디스플레이 광자 | **로컬 모드** (작업자가 로봇 옆) | Quest 3S 패스스루 자체 지연 |

- **로컬 모드(기본)**: 체감 루프 = C-Lat + P-Lat. V-Lat 항목은 존재하지 않는다.
- **원격 모드**: 체감 루프 = C-Lat + V-Lat. V-Lat NFR은 이 모드에서만 발효된다.

**C-Lat 스테이지 예산 (v2 — 단일 프로세스, LeRobot 임베드)** — 측정치 / 계산치 / `[미확인]`을 구분한다:

| # | 구간 | 값 | 구분 | 근거 |
|---|---|---|---|---|
| ① | 손 이동 → Quest 포즈 확정 | **≈13.9 ms** | **계산** (72 Hz 1프레임) | Quest 기본 72 Hz. Quest 3S의 실제 송신 레이트는 `[미확인]` (`16` M-8) |
| ② | **One Euro 필터 위상 지연** | **우리가 정한다** | **설계 변수** | 🔴 **상류 상수를 그대로 쓰면 ≈79.6 ms** (`min_cutoff=2.0, beta=0.04`; τ = 1/(2π·f_c), beta=0.04라 손 속도 1 m/s에서도 f_c ≈ 2.04 Hz → **속도와 무관하게 ~80 ms**). **그러나 우리가 스무더를 이식하므로 우리가 상수를 노출하고 우리가 고른다** (NFR-PRF-016/017) |
| ③ | 헤드셋 → PC 전송 (Wi-Fi, UDP :5006) | **[미확인]** | 미확인 | 참고: Quest2ROS **82 ms 전체**(Quest 2 컨트롤러, ROS1/TCP) — 구성이 다르므로 대입 불가 |
| ④ | ~~IPC~~ | **≈ 0** | **소멸** | 🔴 **단일 프로세스다.** UDP 수신 → 스무딩 → IK → 안전 필터 → `send_action()`이 전부 **같은 프로세스 안의 함수 호출**이다. dora TCP+bincode도, RustDDS→FastDDS도, JTC도 **없다** |
| ⑤ | IK solve (`openarm_control`) | **p95 0.355 ms** | **측정** | §2.4 — 60 Hz 예산의 2% |
| ⑥ | 안전 필터 (클램프·점프 가드) | **[미측정]** | 미측정 | 순수 numpy 연산 → µs 단위 예상. 측정으로 확인 |
| ⑦ | `send_action()` → CAN 프레임 송신 | **0.40 ms** (8 × 50 µs) 이론 | **계산** | §2.1. ⚠️ **python-can 오버헤드 + USB(HS 2.0) 왕복은 [미측정]** — NFR-PRF-045 |
| ⑧ | 모터(DAMIAO) 명령 → 축 반응 | **[미확인]** | 미확인 | 공식 수치 없음 |
| — | **소계 ①+⑤+⑦** (스무더·네트워크·모터를 0으로 놓으면) | **≈ 14.7 ms** | — | **하한** |
| — | **소계 ①+②(상류 상수)+⑤+⑦** | **≈ 94.3 ms** | — | **상류 상수를 그대로 쓸 때의 하한** |

> 🔴 **v2에서 가장 크게 바뀐 것: ②(스무더)가 "고정 비용"에서 "설계 변수"가 됐다.**
> 구 명세는 `dora-openarm-webxr`의 하드코딩 상수(생성자에 박혀 있어 CLI로 노출조차 안 됨)를 **주어진 것**으로 취급했다. **이제 우리가 스무더를 이식하므로 우리가 상수를 정한다.**
>
> 완화 수단 (지연 절감 효과가 큰 순): (1) `min_cutoff` 2.0 → 5–10 Hz (τ 79.6 → 15.9 ms, **64 ms 절감**, 대가는 지터 증가), (2) `beta` 0.04 → 0.5–1.0 (One Euro 본래 설계 의도대로 빠른 동작에서 컷오프 상승), (3) Quest 리프레시 90 Hz (① 13.9 → 11.1 ms).
>
> **동작점은 NFR-PRF-017의 파레토 곡선(지터 RMS vs 위상 지연) 실측으로 고른다.**

> ✅ **USB 정정 (지연 예산에 유리)**: 구 명세는 USB-CAN 어댑터를 `gs_usb` / **USB Full-Speed(12 Mbps)**로 가정하고 **1 ms 스케줄 granularity가 CAN RTT 상한을 지배할 수 있다**고 경고했다. **틀렸다.** Pibiger SavvyCAN-FD-X2는 **High-Speed USB 2.0**이며 in-tree 커널 드라이버를 쓴다. USB 링크 **대역폭은 병목이 아니다**(~2–4%). ⚠️ USB **지연**은 여전히 `[미측정]`이다(NFR-PRF-045).

**V-Lat 참고치 (타 프로젝트 실측, 우리 값 아님)**: Open-TeleVision 121.50 ± 6.01 ms / ZED Mini→PICO 4U 82.00 ± 6.32 ms / PICO 4U→PICO 4U 100.50 ± 3.12 ms (XRoboToolkit 논문). **OpenArm에는 이 경로가 없다.**

**P-Lat**: Quest 3S 패스스루 photon-to-photon 지연은 **Meta 공식 수치 없음** `[미확인]`.

### 2.6 텔레옵 지연은 지금 평가 대상이 아니다 (D-10)

🔴 **D-10 확정: 텔레옵 성능(지연)은 당장 평가하지 않는다.** 따라서 텔레옵 지연을 릴리스 합격 기준(NFR)으로 못 박지 않는다. §2.5의 채널 정의(C-Lat/V-Lat/P-Lat)와 스무더 노출은 **계측 계기(instrumentation)로만** 유지하고, §3.2의 지연 임계 NFR은 제거·유보한다.

참고로 구 명세가 후보로 검토했던 arXiv 2603.06850(*Nonlinear Performance Degradation of Vision-Based Teleoperation under Network Latency*)은 **CARLA 차량 차선유지** 실험이라 양팔 매니퓰레이션과 도메인이 불일치하고 교차검증 출처가 0건이었다 → **어떤 NFR의 근거로도 쓰지 않는다.** (지연 허용치를 뒤늦게 확정해야 할 일이 생기면 §2.10 M-3/M-6 자체 벤치로 우리 도메인에서 직접 잰다.)

### 2.8 실시간성 — 🔴 **제어 루프가 Python이다**

**v2의 핵심 사실**: CAN을 소유하는 프로세스가 **Python**이고, LeRobot 제어 루프도 **Python**이다. 구 명세의 "dora 노드는 RT 경로에 둘 수 없고 `controller_manager`(C++)만 RT로 승격한다"는 전략은 **더 이상 성립하지 않는다** — RT로 승격할 C++ 프로세스가 없다.

**Python 제어 루프가 안고 있는 것:**

| 항목 | 사실 |
|---|---|
| **GIL** | 카메라 인코딩·데이터셋 쓰기·GUI 전송이 **같은 프로세스에서 돌면 제어 스레드를 블록한다.** `chrt -f`로 스레드를 RT 승격해도 **GIL을 든 다른 스레드가 놓아주지 않으면 소용없다** |
| **양팔 순차 호출** | `BimanualMixin.connect/disconnect/send_action`이 **순차 동기 호출**이다(`utils/bimanual.py:47-63`, `bi_openarm_follower.py:158-159`). 별도 스레드도 asyncio도 없다 → **팔 개수만큼 지연이 누적된다** |
| **패턴 B (32 프레임/사이클)** | `get_observation()` + `send_action()` → CAN-FD 이론 상한 **625 Hz** (§2.1) |
| **GC / 할당** | CPython GC는 real-time safe하지 않다. 제어 루프에서 할당을 최소화하고 GC를 튜닝해야 한다 (`gc.freeze()`, generational threshold) |
| **오버런 감지** | LeRobot 루프에 **없다.** 우리가 넣는다 (NFR-PRF-008) |

**함의 (명세가 반드시 반영해야 할 것):**

1. 🔴 **LeRobot Python 루프의 실측 사이클 타임이 모든 주파수 NFR의 뿌리다.** 확정 전까지 어떤 목표도 못 박지 않는다 → **NFR-PRF-054 / M-8**.
2. **RT 승격 대상은 CAN 명령을 송신하는 스레드**다(`chrt -f`, `mlockall`, `CPUAffinity`). **다만 GIL 때문에 RT 승격의 실효성 자체가 [미확인]이다** — M-8이 이를 함께 판정해야 한다.
3. **무거운 작업은 반드시 제어 루프 밖으로.** 카메라 인코딩·트랜스코딩·데이터셋 쓰기·GUI 직렬화·학습은 **별도 프로세스**(또는 GIL을 놓는 C 확장)로 분리한다. `multiprocessing` 경계가 어디에 놓이는지가 설계의 핵심이다.
4. **500 Hz가 안 나오면 바이래터럴만 별도 RT 프로세스로 분리**해야 하고, **그 순간 CAN 배타 소유권 문제가 부활한다**(`14` §2.3, `16` M-21).

**PREEMPT_RT 권장 근거** (ROS 2 문서이지만 판단 근거로 유효): *"the normal linux kernel is optimized for computational throughput and therefore is not well suited for hardware control."* RT 권한은 `/etc/security/limits.conf`의 `@realtime` 그룹(`rtprio 99`, `memlock unlimited`)으로 부여한다.

**반면교사 (C++ 상류도 오버런을 안 잡는다)**: `openarm_teleop`은 `SCHED_FIFO` prio 50 + `clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME)`을 쓰지만 **`last_elapsed_us_`를 측정만 하고 주기와 비교하지 않는다** → 오버런이 카운트되지 않고 다음 기상 시각만 조용히 밀린다. **우리는 이를 답습하지 않는다.**

> **PREEMPT_RT 필요 여부에 대한 판정**: 위 사실들은 "PREEMPT_RT를 켜라"가 아니라 **"RT 커널의 필요성은 지터 실측으로 판정하라"**를 함의한다. 판정 기준은 §2.10 M-2에 정의한다. **다만 Python + GIL에서는 PREEMPT_RT조차 GIL 경합을 고치지 못한다** — 프로세스 분리가 먼저다.

### 2.10 측정 프로토콜 규격 (M-1 ~ M-8)

> **모든 NFR은 아래 프로토콜 중 정확히 하나를 측정 방법으로 지정해야 한다.**

**🔴 M-8 — LeRobot Python 제어 루프의 실측 사이클 타임** (**최우선. 다른 모든 주파수 NFR의 선행 조건**)

**왜 재는가**: **주경로(LeRobot record 30–60 Hz · 정책 추론 · 실기 평가 250 Hz)의 주파수 목표를 확정하려면** Python 루프의 실측 사이클 타임이 필요하다. 이 대역은 CAN·Python 예산에 여유가 크므로 주경로는 대체로 통과할 것으로 예상되나, GIL 부하 조건에서의 지터·오버런을 측정해 확인한다. **별도로**, `openarm_teleop`(C++)의 바이래터럴 스레드는 **500 Hz**다(`openarm_bilateral_control.cpp:42,72,103`) — 이는 **힘반사 전용 선택 경로**이며, 그것을 구현할 경우에만 Python이 500 Hz를 낼 수 있는지가 문제가 된다(m21). 못 내면 바이래터럴만 별도 실시간 프로세스로 분리해야 하고 그 순간 CAN 배타 소유권 문제가 부활한다.

```python
# 계측 대상 (단완·양완 각각, 하드웨어 연결 상태)
#   robot.get_observation()          → t_obs
#   safety_filter(action)            → t_filter
#   robot.send_action(action)        → t_act
#   (텔레옵이면) teleop.get_action() → t_teleop  (= UDP 수신 + 스무딩 + IK)
# 각 10,000 사이클. time.perf_counter_ns() 기준.
```

| 단계 | 측정 |
|---|---|
| **1. 순수 루프 (idle)** | `get_observation()` + `send_action()`만. p50/p95/p99/max, 오버런율. **단완 / 양완 각각** (양팔은 순차 호출이라 ~2배 예상) |
| **2. 패턴 A 실험** | `get_observation()`을 **생략**하고 MIT 응답을 상태로 사용하는 루프. → 32 → 16 프레임/사이클, 버스 부하 80% → 40%. **500 Hz의 유일한 현실적 경로다.** LeRobot 상위 API를 우회해 `robot.bus`를 직접 다뤄야 한다 |
| **3. 전체 텔레옵 루프** | `teleop.get_action()`(UDP + One-Euro + IK) 포함 |
| **4. GIL 부하 조건** | 카메라 5스트림 + **무손실 PNG 프레임 쓰기**(수집 중 인코딩) + 데이터셋 쓰기 + GUI 전송이 **같은 프로세스에서** 돌 때 재측정. 🔴 **이것이 진짜 조건이다** |
| **5. 프로세스 분리 실험** | 카메라·인코딩·데이터셋을 **별도 프로세스**로 뺀 뒤 재측정 → GIL 경합의 기여도를 정량화 |
| **6. RT 승격 실효성** | `chrt -f` + `mlockall` 적용 전후 비교. **GIL 때문에 효과가 없을 수 있다** — 이것을 확인하는 것이 목적 |
| **7. 프레임 수 확정** | `candump -t d can0`으로 **한 사이클의 CAN 프레임을 센다.** 8+8+8+8=32인가, 8+8=16인가? → §2.1의 625 vs 1250 Hz 상한이 여기서 갈린다 (`16` M-25) |

**판정선**: 주경로(record/추론/평가)는 목표 주파수(30–250 Hz)에서 오버런율 ≤ 0.1%면 통과로 본다 — 예산 여유가 크므로 통과가 기본 기대다. **바이래터럴 힘반사 경로(선택)에 한해**, 조건 **4**(GIL 부하)에서 사이클 타임 **p99 ≤ 2.0 ms** 이고 **오버런율 ≤ 0.1%** 이면 **Python 500 Hz를 인정한다.**
- 500 Hz 미달이면: 조건 **5**(프로세스 분리)로 재시도 → 그래도 미달이면 **바이래터럴만 별도 RT 프로세스로 확정**하고 `01`의 CAN 소유권 이양 절을 되살린다. **주경로는 이와 무관하게 계속 동작한다.**
- 텔레옵·기록·추론 루프의 목표 주파수는 이 측정의 산출물(`f_max_python`)의 **80%**를 넘지 못하게 한다.

**전제 조건**: 측정 중 CAN 인터페이스를 **단일 프로세스가 배타 점유**해야 한다(§4 E-12). 🔴 **그리고 측정 전후로 `connect()`를 반복 호출하지 마라 — 영점이 파괴된다**(`14` §2.6). 세션 1회 connect로 전 조건을 돌린다.

**산출물**: `f_max_python`(단완/양완), 사이클 타임 분포, GIL 기여도, RT 승격 이득, 사이클당 CAN 프레임 수.

**M-1 — CAN 제어 주파수 상한 확정** (하드웨어 측 상한. M-8과 짝을 이룬다)
기존 도구를 그대로 쓴다(신규 개발 불필요).
```bash
sudo ip link set can0 type can bitrate 1000000 dbitrate 5000000 fd on restart-ms 100
sudo ip link set can0 up
for HZ in 100 200 300 400 500 600 700 750 800 900 1000; do
  timeout 30 ./motor_sampling_check --all $HZ can0 -fd | tee rtt_${HZ}.log
done
ip -s -d link show can0     # bus-error / restarts / TX·RX 프레임 (독립 검증)
```
- 도구 내장 출력: Target Hz / **Actual Hz** / **RTT(µs)** / 모터별 온도. 판정선은 도구 내장 임계 **`actual_hz ≥ 0.95 × target_hz`**.
- 도구 개조(≈20줄)로 추가 계측: **사이클당 `recv_all`이 소비한 프레임 수**(기대 8), **모터별 상태 갱신 간격**, `first_timeout_us` 스윕(100/220/300/500 µs)에 따른 actual_hz 변화.
- **산출물**: 팔당 `f_max_A`(패턴 A) / `f_max_B`(패턴 B), RTT p50/p95/p99.
- **전제 조건**: 측정 중 CAN 인터페이스를 **단일 프로세스가 배타 점유**해야 한다. SocketCAN은 배타 bind를 제공하지 않으므로(§4 F-6), 락 미획득 상태의 측정치는 **무효**다.

**M-2 — 제어 루프 지터 / PREEMPT_RT 필요성 판정** (**M-8과 통합 실행한다** — 같은 루프를 측정하는 두 관점이다)
- 측정: 기상 목표 시각 대비 실제 시각의 차이(wakeup latency) 분포, 사이클 소요시간 분포, **오버런 횟수/율**.
- 부하 조건: (a) idle, (b) 카메라 5스트림 + 인코딩 + 기록 동시 동작 **(같은 프로세스)**, (b′) **같은 부하를 별도 프로세스로 분리**한 경우, (c) 학습 프로세스 동시 동작.
- **판정**: 조건 (b)에서 **wakeup latency p99.9 > 제어 주기의 20%** 이거나 **오버런율 > 0.1%** 이면 → **① 먼저 (b′)로 프로세스 분리 효과를 본다** → 그래도 미달이면 PREEMPT_RT + 코어 격리(`isolcpus`/`nohz_full`/`rcu_nocbs`) + `chrt -f`를 **필수로 승격**한다.
- 🔴 **(b)와 (b′)의 차이가 GIL 기여도다.** Python 런타임에서는 **PREEMPT_RT보다 프로세스 분리가 먼저다** — PREEMPT_RT는 GIL 경합을 고치지 못한다.
- 참고 튜닝: governor=performance, THP off, `ulimit -l unlimited`, `/proc/sys/kernel/sched_rt_runtime_us` 950000 유지, CPython `gc.freeze()` + generational threshold.

**M-3 — C-Lat (제어채널 지연) 실측**
> 🔴 **D-10: 텔레옵 지연은 평가 대상이 아니다** → 방법 A(소프트웨어)는 **선택 진단**으로만, **방법 B/C(물리 리그)는 필수가 아니다**(NFR-PRF-014 = S / NFR-PRF-015 = C). 아래 프로토콜은 지연 허용치를 뒤늦게 확정해야 할 경우를 위한 **규격으로만** 유지한다.
- **방법 A (소프트웨어, 상대 지연)**: VR 패킷의 `t` 필드(HMD monotonic) 수신 시각 → 스무딩 출력 → IK 출력 → 안전 필터 → `send_action()` 진입 → CAN 송신까지 **각 단계에 `time.perf_counter_ns()` 타임스탬프를 전파**해 스테이지별 차이를 집계. ✅ **단일 프로세스이므로 클럭이 하나다 — 구 명세의 프로세스 간 클럭 정렬 문제가 사라졌다.** **한계: 헤드셋 내부 지연(①)은 잡히지 않는다.** 이 한계를 UI에 명시해야 한다.
- **방법 B (물리, 절대 지연)**: 컨트롤러에 LED, 로봇 J1 축에 가속도계 → 동일 오실로스코프/DAQ로 Δt 측정. **One Euro의 위상 지연이 여기서 드러난다.**
- **방법 C (주파수 응답, 권장)**: 컨트롤러를 0.2 / 0.5 / 1.0 / 2.0 Hz 사인파로 흔들고 **로봇 축의 위상 지연(deg)** 을 측정 → 지연 = phase/(360·f). **위상 지연 요소(One Euro)를 정량화하는 유일한 정직한 방법.**
- 출력 항목: 스테이지별 p50/p95/p99, 총 C-Lat p50/p99.

**M-4 — V-Lat (영상채널, 원격 모드 한정)**
- 표준 리그: **LED + 포토다이오드 G2G 측정기** (Bachhuber & Steinbach, IEEE ICIP 2016; 오픈소스 구현 `cbachhuber/G2GDelay`, Arduino + OPT101).
- 헤드셋 디스플레이에 포토다이오드를 붙이기 어려우므로 렌즈 앞 포토다이오드 지그가 필요하다 — **실현 가능성 검토 필요** `[미확인]`.
- **영상 리턴 경로가 구현되기 전에는 이 프로토콜을 실행하지 않는다.**

**M-5 — 저장/처리량 실측**
- **무손실 PNG(뎁스 TIFF/16-bit)의 실제 bpp**: 각 스트림에서 100프레임을 저장해 **평균 파일 크기**로 확정 — 중간 포맷이 PNG이므로 JPEG가 아니라 **PNG를 잰다**(현재 JPEG q90의 3~8배로 추정) `[개정 2026-07-15]`.

- 디스크: `fio`로 **지속 쓰기 대역폭 + 150 files/s 메타데이터 IOPS**를 사전 측정. 목표치를 못 내면 그 디스크로 녹화를 허용하지 않는다.

- 🔴 **인코딩 CPU 비용을 반드시 함께 잰다** — 같은 프로세스에서 돌면 제어 루프를 블록한다(M-8 조건 4/5).

**M-6 — 정책 성공률 평가 프로토콜**
- 태스크당 **N ≥ 20 시행**. 초기 상태는 사전 정의된 분포에서 무작위 추출하고 **시드를 기록**한다.
- 성공률은 **점추정 + Wilson 95% 신뢰구간**으로 보고한다. (N=20, p̂=0.5 → ±21%p; N=50 → ±13.6%p.) **N < 20의 성공률 비교는 리포트에 "통계적으로 무의미"로 표시한다.**
- 성공 판정: 사전 정의된 자동 판정기 또는 **2인 독립 라벨러 합의**. 🔴 **LeRobot에 성공/실패 라벨링도 성공률 집계도 없다** — `lerobot-eval`은 시뮬 전용(`configs/eval.py:29-31`)이고 `lerobot-rollout`은 rollout이지 eval이 아니다 → **하네스는 우리가 만든다**(`14` FR-OPS-086). ⚠️ `enactic/openeval-runner`가 이를 대체할 수 있는지 먼저 확인(`16` M-23).
- 리포트 항목: 성공률(+CI), 에피소드 길이 중앙값, 충돌 횟수, 안전정지 발동 횟수, **정책 추론 지연 p95**.

**M-7 — 리소스 프로파일링**
- GPU: 학습/추론 각각 **피크 VRAM**(`torch.cuda.max_memory_allocated`)과 스텝 시간을 정책·배치사이즈별로 기록.
- CPU: 프로세스별 코어 점유, RT 프로세스의 `chrt -p` / `VmLck` / CPU 어피니티.
- 메모리: 8시간 소킹 중 **RSS 기울기**(누수 검출).

**반복정밀도 (참고 절차, 목표치 없음)**: ISO 9283:1998 — ISO 큐브 5점(P1–P5), **≥30 사이클**, 정격 부하·정격 속도 100%, `RP = l̄ + 3·S_l`. OpenArm 공식 반복정밀도 수치는 **존재하지 않으므로**, 이 값은 제품 스펙이 아니라 **회귀 감시 지표(regression metric)** 로만 쓴다. OpenArm은 QDD 모터 + MIT 임피던스 제어이므로 산업용 로봇식 위치 반복도가 낮게 나오는 것이 정상이다.

## 5. 미해결 · 결정필요

### 5.1 `[결정필요]` — 충돌하는 사실이 실재하여 사람이 골라야 함

> **라벨 규약**: 이 표의 `Q-n`은 **이 문서 로컬 결정 라벨**이다. **전역 결정 등록부는 `16`의 `D-n`/`M-n`**이며 번호 체계가 다르다(로컬 `Q-n` ≠ 전역 `D-n`). 전역 대응: **Q-0·Q-3·Q-4 → `16` M-21**(제어 주파수/패턴), **Q-1·Q-2 → `16` D-10**(텔레옵 지연 평가), **Q-11 → `16` D-4·D-3**(게인·리밋 정본). 나머지(Q-5~Q-10·Q-12)는 전역 대응이 없는 로컬 성능 결정이다.
