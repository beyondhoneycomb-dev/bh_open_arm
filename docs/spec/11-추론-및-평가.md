# 11. 추론 실행 및 평가 (추론 설정 · 실기 평가 하네스)

요구사항 ID 영역 코드: **`INF`** (`FR-INF-nnn` / `NFR-INF-nnn`)

## 1. 범위

- **다루는 것**: 학습된 정책의 **실기(real-robot) 배포 경로** 전체 — 추론 루프(관측 → 전처리 → 정책 forward → 역정규화 → 안전 게이트 → `send_action` → CAN), 액션 청킹과 정책/제어 주파수 분리, 추론 백엔드(`sync`/`rtc`/원격 gRPC), 추론 설정 화면의 파라미터 계약, **추론 중 안전 계층(전부 우리가 만든다)**, **실기 평가 하네스(성공 라벨링·성공률 집계 — LeRobot에 없다, 우리가 만든다)**, 추론 상태머신.
- **다루지 않는 것**: 학습 파이프라인(→ 10), 데이터 수집·텔레옵(→ 02·03), IK 솔버 내부(→ 03), 캘리브레이션·영점(→ 09), 충돌감지·안전 계층의 구현 상세(→ 12; 본 문서는 그 계층이 **추론 경로 어디에 삽입되는지**를 규정한다).
- **런타임 전제**: **LeRobot Python API를 단일 GUI 백엔드 프로세스에 임베드**한다. 모드(텔레옵 / 기록 / 추론 / 수동)는 **같은 robot 객체 위의 상태 전이**이며 프로세스 재시작이 아니다. **LeRobot CLI를 모드마다 스폰하는 설계는 금지한다**(§2.9). dora-rs는 쓰지 않는다.
- **LeRobot 버전 = v0.6.1 확정**(뎁스 관측·기록이 0.6.x부터 지원되므로 D-12에 따라 **0.6.1로 상향 확정** — `16` D-11·D-12·§3.1). 본 문서의 `lerobot-rollout`(전략 5종·백엔드 sync/rtc)은 **v0.6.1에 실재하는 엔진**이며, 우리는 이를 CLI로 스폰하지 않고 **엔진으로 임베드**한다. (참조 레포가 핀한 **0.5.1에는 `lerobot-rollout`이 없고** 온보드 구동이 `lerobot-record --policy.path`였다 — 0.6.x에서 rollout 엔진으로 흡수됐다.)

## 2. 확정된 사실 기반

### 2.1 🔴 `lerobot-eval`은 실기 평가 도구가 아니다 [확정]

**`lerobot-eval`은 시뮬레이션 전용이다.** `src/lerobot/configs/eval.py:34`:
```python
@dataclass
class EvalPipelineConfig:
    env: envs.EnvConfig            # ← 기본값 없음 = 필수 필드
    eval: EvalConfig = field(default_factory=EvalConfig)
    policy: PreTrainedConfig | None = None
```
그리고 `src/lerobot/scripts/lerobot_eval.py:68` → `import gymnasium as gym`.
**robot / robot config를 전혀 받지 않는다.** → **OpenArm 실기 성공률 측정에 `lerobot-eval`은 쓸 수 없다.**

**실기 정책 구동의 단일 CLI = `lerobot-rollout`.** `src/lerobot/scripts/lerobot_rollout.py:17-34` docstring:
> `lerobot-rollout` is the single CLI for running trained policies on real robots.
> Strategies: `base` (autonomous, no recording) / `sentry` (continuous recording + auto-upload) / `highlight` (ring buffer + keystroke save) / `dagger` (human-in-the-loop) / `episodic` (episode-oriented with reset phases)
> Inference backends: `sync` (one policy call per control tick, default) / `rtc` (Real-Time Chunking for slow VLA models)

OpenArm이 **공식 예제에 그대로 등장한다** (`lerobot_rollout.py:80-88`):
```shell
lerobot-rollout \
    --strategy.type=dagger \
    --strategy.num_episodes=20 \
    --policy.path=outputs/pretrain/checkpoints/last/pretrained_model \
    --robot.type=bi_openarm_follower \
    --teleop.type=openarm_mini \
    --dataset.repo_id=user/rollout_hil_data \
    --dataset.single_task="Fold the T-shirt"
```

**그러나 `lerobot-rollout`은 rollout이지 eval 하네스가 아니다.**

**🔴 성공/실패 라벨링도, 성공률 집계도 LeRobot의 실기 경로 어디에도 없다** (직접 확인):
- `utils/constants.py:40` → `SUCCESS = "next.success"` **상수는 존재한다.**
- 그러나 `src/lerobot/rollout/` 전체 트리에서 `success` 문자열은 **1건**뿐이고 그것은 로그 문자열이다(`rollout/context.py:435` `"Rollout context assembled successfully"`). **롤아웃은 `next.success`를 한 번도 쓰지 않는다.**
- `next.success`를 실제로 **생산**하는 경로는 두 곳뿐이다:
  1. `lerobot_eval.py` — gym env의 `info`에서 (시뮬).
  2. **HIL/RL 프로세서 경로** — `processor/hil_processor.py:489-518`:
     ```python
     success = info.get(TeleopEvents.SUCCESS, False)
     ...
     new_transition[TransitionKey.DONE] = bool(terminate_episode) or (self.terminate_on_success and success)
     new_transition[TransitionKey.REWARD] = float(success)
     ```
     그 `TeleopEvents.SUCCESS`의 유일한 생산자는 **키보드 텔레옵**이다 — `teleoperators/keyboard/teleop_keyboard.py:268-284`:
     ```python
     while not self.misc_keys_queue.empty():
         key = self.misc_keys_queue.get_nowait()
         if key == "s":   success = True
         elif key == "r": terminate_episode = True; rerecord_episode = True
         elif key == "q": terminate_episode = True; success = False
     ```
     이벤트 어휘는 `teleoperators/utils.py:26-32`의 `TeleopEvents` enum — `SUCCESS` / `FAILURE` / `RERECORD_EPISODE` / `IS_INTERVENTION` / `TERMINATE_EPISODE`.
- **즉 LeRobot에는 사람 성공 라벨링의 *어휘*(`TeleopEvents`)와 *입력 원시요소*(키보드 `s`/`r`/`q`)가 존재하지만, `lerobot-rollout` 루프에는 배선되어 있지 않고, 성공률을 집계하는 코드도 없다.**

→ **실기 평가 하네스는 우리가 만든다. 이것이 이 문서의 핵심 축이다.** 우리는 LeRobot의 `TeleopEvents` 어휘와 `next.success` feature 이름을 **재사용**하고, `lerobot-rollout --strategy.type=episodic` 위에 라벨링·집계·아카이브를 얹는다(§3.7).

### 2.2 `lerobot-rollout` 제어 루프와 액션 전달 경로 [확정]

**제어 루프** (`rollout/strategies/base.py:55-77`):

### 2.3 액션 청킹 · 정책 주파수 ↔ 제어 주파수 분리 [확정]

**분리 메커니즘 = `ActionInterpolator`** (`utils/action_interpolator.py`):
```python
def add(self, action: Tensor) -> None:
    if self.multiplier > 1 and self._prev is not None:
        self._buffer = []
        for i in range(1, self.multiplier + 1):
            t = i / self.multiplier
            interp = self._prev + t * (action - self._prev)   # 선형보간
            self._buffer.append(interp)
    else:
        self._buffer = [action.clone()]                        # 첫 스텝: 보간 없음
    self._prev = action.clone()
    self._idx = 0
```
- **제어 주기 = 1 / (`fps` × `interpolation_multiplier`)** (`strategies/base.py`: `control_interval = interpolator.get_control_interval(cfg.fps)`)
- 관측 전처리·정책 forward는 `interpolation_multiplier` 틱마다 **1회만** 발생. 보간 틱은 `obs_processed` 캐시 재사용(`strategies/core.py`).
- DAgger 기록은 `record_stride = max(1, cfg.interpolation_multiplier)`로 데시메이션.
- 코드상 `multiplier` 상한 검증 **없음**(≥1만 검증). **상한은 CAN이 정한다**(§2.9).

**청크 파라미터 — 정책별 기본값**

| 정책 | `chunk_size` / horizon | `n_action_steps` (실행) | 비고 |
|---|---|---|---|
| ACT | **100** | 100 | `n_action_steps ≤ chunk_size` 강제, `n_obs_steps != 1` 미지원 |
| Diffusion | horizon **64** | **32** | `n_obs_steps = 2` |
| VQ-BeT | **5** | — | **이미지 입력 1개만 허용** → 우리 구성(뎁스 RealSense + RGB 웹캠 혼합의 다중 카메라)에서 사용 불가 |
| SmolVLA | **50** | 50 | `max_state_dim`/`max_action_dim` = **32 / 32** |
| pi0 / pi05 | **50** | 50 | 동상 |
| GR00T N1.7 | action_horizon **40** | `--execution-horizon` (예시 **8**) | state/action dim **132** |

> 🔴 **관측 차원 주의**: `use_velocity_and_torque=True`이면 `observation.state`가 **단완 24 / 양완 48차원**이다(문서 10 §2.3). **양완 48차원은 SmolVLA/pi0/pi05의 `max_state_dim=32`를 초과한다** → 이 조합의 정책은 로드 자체가 되지 않는다. 추론 설정 화면은 체크포인트의 `input_features` shape와 실기 로봇의 `observation_features` 차원이 일치하는지 **로드 전에** 검사해야 한다(FR-INF-070).

**Temporal ensembling (ACT)** — `policies/act/configuration_act.py`, `modeling_act.py`
- `temporal_ensemble_coeff: float | None = None` (기본 **비활성**)
- 가중치 **wᵢ = exp(−`temporal_ensemble_coeff` · i)**, w₀가 가장 오래된 액션. Σwᵢ=1로 정규화.
- **양수 = 과거 액션 중시 / 음수 = 최신 액션 중시 / 0 = 균등.** 원 ACT 논문 값 **0.01**.
- **활성화 시 `n_action_steps`는 반드시 1**(소스가 강제) → **추론 주파수 = 제어 주파수**가 되어 가장 비싸다.

**청크 소비 전략 3종**

| 방식 | 구현 | 재추론 시점 | 겹침 처리 |
|---|---|---|---|
| 순수 open-loop | openpi `ActionChunkBroker(policy, action_horizon)` | `action_horizon` 스텝 **전부 소진 후** | 없음 |
| Temporal ensembling | ACT `ACTTemporalEnsembler` | **매 스텝** | wᵢ=exp(−m·i) 가중평균 |
| 비동기 청킹 | LeRobot RTC / async gRPC | 큐 소진 **전에** 미리 | RTC: prefix attention inpainting / gRPC: `aggregate_fn` |

**RTC (Real-Time Chunking)** — `policies/rtc/configuration_rtc.py:38-48`, `rollout/inference/factory.py:74`
```python
@dataclass
class RTCConfig:
    enabled: bool = True
    prefix_attention_schedule: RTCAttentionSchedule = RTCAttentionSchedule.LINEAR  # ZEROS|ONES|LINEAR|EXP
    max_guidance_weight: float = 10.0     # <= 0 이면 ValueError
    execution_horizon: int = 10
    debug: bool = False
    debug_maxlen: int = 100

@InferenceEngineConfig.register_subclass("rtc")
@dataclass
class RTCInferenceConfig(InferenceEngineConfig):

```
CLI: `--inference.type=sync|rtc`, `--inference.rtc.execution_horizon=…` (nested).

> **OpenArm + VLA 조합에서 ACT temporal ensembling은 성립 불가하다.** TE는 `n_action_steps=1`을 강제하므로 정책 forward가 매 제어 틱마다 완료되어야 하는데, GR00T Orin 기준 216.5 ms/추론 = 4.6 Hz가 상한이다(§2.6). **RTC(또는 async 청킹) + `interpolation_multiplier` 조합이 사실상 유일한 선택이다.**

### 2.8 🔴 안전 — LeRobot이 주는 것은 두 개뿐이다 [확정]

**LeRobot 안전장치 전량** — `openarm_follower.py:288-304` (`send_action` 내부):
```python
goal_pos = {key.removesuffix(".pos"): val for key, val in action.items() if key.endswith(".pos")}

# [1] 절대 관절 한계 클립 — 항상 적용
for motor_name, position in goal_pos.items():
    if motor_name in self.config.joint_limits:
        min_limit, max_limit = self.config.joint_limits[motor_name]
        clipped_position = max(min_limit, min(max_limit, position))
        if clipped_position != position:
            logger.debug(f"Clipped {motor_name} from {position:.2f}° to {clipped_position:.2f}°")
        goal_pos[motor_name] = clipped_position

# [2] 현재 위치 대비 상대 변위 캡 — 기본 꺼짐
# /!\ Slower fps expected due to reading from the follower.
if self.config.max_relative_target is not None:
    present_pos = self.bus.sync_read("Present_Position")
    goal_present_pos = {key: (g_pos, present_pos[key]) for key, g_pos in goal_pos.items()}

```
`ensure_safe_goal_position` (`robots/utils.py:91-126`): `safe_diff = clamp(goal − present, ±max_relative_target)` → `safe_goal = present + safe_diff`.

**이게 전부다.**

| 있는 것 | 정확한 의미 | 함정 |
|---|---|---|
| `joint_limits` | 관절별 절대 한계 [deg], **항상 적용** | **`--robot.side` 미지정 시 기본값이 전 축 ±5°** (`config_openarm_follower.py:107-120`) → **팔이 사실상 안 움직인다** 🔴 |
| `max_relative_target` | `\|목표각 − 현재각\| ≤ 값` [deg], per-tick 캡. 실효 속도 ≈ `값 × fps` [deg/s] | **기본값 `None` = 완전히 꺼짐** 🔴. 켜면 매 틱 `sync_read` 추가로 **실효 fps 저하**(소스가 명시 경고) |

| **없는 것 → 전부 우리가 만든다** | 확인 방법 |
|---|---|
| **속도 제한** | `openarm_follower`에서 velocity는 관측(`.vel`)용으로만 존재. 제한 로직 없음 |
| **가속도 / 저크 제한** | 전무 |
| **소프트웨어 E-Stop / 토크 컷** | 없음. `disable_torque_on_disconnect=True`는 **연결 해제 시에만** 동작 |
| **NaN/Inf 검사** | 없음 |
| **통계적 이상치 거부** | 없음 |
| **폭주(runaway) 감지** | 없음 |
| **워크스페이스(카테시안) 제한 / 셀프-충돌 회피** | 없음 |

→ **안전 계층(속도·가속도·저크 클램프, 소프트 스톱 `STOP_HOLD`, 워크스페이스 가상벽, 충돌 반응)은 전부 우리가 만들고, `ctx.processors.robot_action_processor` 파이프라인에 삽입한다**(§2.2). 계층의 설계·임계·충돌 반응은 **문서 12가 소유**하고, 이 문서는 **삽입 지점과 추론 상태머신과의 결합**을 규정한다.

**🔴 토크 액션은 실행되지 않는다** — `openarm_follower.py:340`:
```python
commands[motor_name] = (kp, kd, position_degrees, 0.0, 0.0)   # vel=0, tau=0 하드코딩
```
`send_action`은 액션 dict에서 `.pos`만 추출하고(`:288`), 리턴도 `.pos`만 한다. **정책이 `.torque` 차원을 출력해도 CAN으로 나가지 않는다.** 버스 레벨 `_mit_control_batch(commands: {motor: (kp, kd, pos_deg, vel_deg_s, torque)})`(`damiao.py:492-502`)는 **이미 tau 인자를 받는다** → 막혀 있는 건 상위 함수 하나뿐이지만, **정본 경로에서는 tau=0이다.** 추론 설정 화면은 이 사실을 표시해야 한다(FR-INF-072).

**하드 상한 (모든 설정 위에 있음)**
- **명령 위치는 `±PMAX = 12.5 rad`를 절대 넘으면 안 된다.** Damiao 피드백 위치는 16-bit를 [−PMAX, +PMAX]로 매핑하며 그 밖에서 **wrap**한다. `MOTOR_LIMIT_PARAMS`(`motors/damiao/tables.py:98-112`)의 전 모터 PMAX = 12.5.
- 모터 VMAX: **DM8009 45 rad/s**(J1·J2) · **DM4340 8 rad/s**(J3·J4) · **DM4310 30 rad/s**(J5·J6·J7·gripper). 우리 속도 리밋은 `min(사용자값, 모터 VMAX)`로 클램프한다.
- 모터 TMAX: DM8009 54 Nm / DM4340 28 Nm / DM4310 10 Nm.

**명령 스트림을 끊으면 팔이 떨어진다**
- Damiao 1차 매뉴얼: *"通讯丢失防护，设定周期内没有收到 CAN 指令将自动退出'使能模式'"* (통신 상실 보호: 설정 주기 내 CAN 명령 미수신 시 **인에이블 모드 이탈**). 대응 레지스터 **RID 9 `TIMEOUT` (RW, uint32)**.
- **OpenArm에는 홀딩 브레이크가 없다.** 인에이블 이탈 = 토크 0 = **낙하**. 공식 Safety Guide: *"if power is lost due to an emergency stop, the load being held will fall rapidly."*
- → **abort/pause/fault 등 어떤 상태에서도 CAN 명령 스트림을 끊어서는 안 된다.** "홀드"는 반드시 **현재 실측 자세를 목표로 하는 MIT PD 명령(`q=qh, dq=0, tau=0`)의 지속 송신**으로 구현한다.

**E-Stop 2등급 (명세가 반드시 구분해야 함)**

| 등급 | 동작 | 낙하 |
|---|---|---|
| **Cat-2 (기본, GUI 빨간 버튼)** | 전원·CAN 유지, 즉시 `MIT(kp, kd, q=qh, dq=0, tau=0)` 홀드 | ❌ |
| **Cat-0 (물리 버튼, 전원 라인 삽입)** | 모터 전원 물리 차단 | ✅ **낙하 전제** |

→ GUI의 정지 버튼은 **반드시 Cat-2**에 매핑한다. 그렇지 않으면 "안전 버튼을 눌렀더니 팔이 떨어지는" 반직관이 발생한다.

### 2.9 CAN 소유권과 세션 라이프사이클 [확정 — v2에서 개정]

**🔴 `connect()`가 영점을 파괴한다** — `robots/openarm_follower/openarm_follower.py:152-153`:
```python
if self.is_calibrated:
    self.bus.set_zero_position()      # → CAN_CMD_SET_ZERO (damiao.py:335-340)
```
**`connect()`는 현재 물리 자세를 영점으로 확정한다.** 팔이 rest 자세가 아닌 순간 재연결하면 **그 자세가 영점이 되고** 조인트 리밋과 데이터셋 좌표계가 통째로 어긋난다.

→ **세션당 `connect()`는 1회.** 모드 전환(텔레옵 ↔ 기록 ↔ **추론** ↔ 수동)에 **disconnect/reconnect를 절대 하지 않는다.** 추론 세션 진입/이탈은 **같은 robot 객체 위의 상태 전이**다(§4).
→ **LeRobot CLI(`lerobot-rollout` 등)를 모드마다 스폰하는 설계는 금지한다.** 각 CLI는 독립 프로세스이고 시작 시 `connect()`, 종료 시 `disconnect()`를 하므로, 그 설계는 **모드 전환 때마다 영점을 재확정**한다.

**배타성은 프로세스 내부에서 해결되지만, 중재가 불필요한 것은 아니다**
- ✅ LeRobot Python API를 단일 GUI 프로세스에 임베드하면 **모드 간 배타성**은 프로세스 내부 상태머신으로 해결된다. 양팔도 한 프로세스에 버스 객체 2개(`bi_openarm_follower.py:93-94`).
- ❌ **커널이 배타성을 강제하지 않는다.** SocketCAN RAW = `socket(PF_CAN, SOCK_RAW, CAN_RAW)` + `bind()`, **배타 플래그 없음**. `cansend`, `openarm-can-cli`, 두 번째 LeRobot 인스턴스, `openarm_teleop`(C++)이 전부 무방비로 끼어들고 **에러가 나지 않는다 → 조용한 실패.**
- → **`flock` 애플리케이션 락은 여전히 필요하다.** 역할이 "모드 중재"에서 **"외부 침입자 방어"**로 축소될 뿐이다.
- 배타성의 단위가 "버스 소유"에서 **"누가 `send_action()`을 부를 권리를 갖는가"**로 내려간다 → **상태머신은 그대로 필요하다**(§4).

**CAN-FD는 `ip link`로 켜야 한다** — python-can socketcan 백엔드는 `bitrate`/`data_bitrate` 인자를 **무시한다**. 코드가 대신할 수 없으므로 GUI가 기동 시 **`ip -details link show`로 검증**해야 한다.

**제어 주파수 상한**: `openarm_can` 공식 경고 — **8모터 구성에서 1000 Hz 초과 제어 사이클은 CAN 연결을 불안정하게 만든다.**

## 4. 상태 · 오류 처리

### 4.1 추론 상태머신

**상태 9종.** 표기: `qh` = 상태 진입 시각의 **실측** 관절각(홀드 목표).
**🔴 전 상태에서 로봇 연결(`connect()`)은 유지된다.** 상태 전이는 **같은 robot 객체 위**에서 일어나며 **어떤 전이도 `disconnect()`/`connect()`를 호출하지 않는다**(FR-INF-067).

### 4.3 금지 전이 (코드 레벨 차단 + 테스트로 강제할 불변식)

- ❌ **P5 → P3 직행** (takeover 해제 후 즉시 자율 재개) — 개입 중 로봇 자세가 정책의 액션 큐·hidden state와 발산했다. 반드시 **P7 경유**.
- ❌ **P6 → P3 직행** (abort 후 즉시 재개) — 반드시 P7 경유.
- ❌ **P8 → P3 직행** (fault 후 즉시 재개) — 반드시 P7 경유.
- ❌ **P4 → P3 직행 with stale queue** — pause 시 파기한 큐를 복원해 재사용하는 것 금지.
- ❌ **세션 종료 이외의 어떤 상태 전이에서든 `disconnect()`/`connect()` 호출** — `set_zero_position()` 재호출 = **영점 파괴**(FR-INF-067).
- ❌ **세션 종료 이외의 어떤 상태에서든 CAN 명령 스트림 중단** — Damiao 통신상실 보호 → 인에이블 이탈 → **낙하**(FR-INF-044).
- ❌ **원격 서버 단절 후 큐 잔여 액션 실행** — 관측 없는 open-loop 폭주.
- ❌ **안전 게이트를 우회한 `robot.send_action()` 직접 호출** (NFR-INF-008).

| **Q10** | **[미확인]** `ActionInterpolator.interpolation_multiplier`를 **저크 완화 수단으로 쓸 수 있는가**? | 선형보간이므로 위치는 C⁰ 연속이지만 속도는 청크 경계에서 불연속이다. 실기에서 `multiplier`를 1/2/4로 바꿔가며 관절 가속도·전류 스파이크를 측정해, 우리 저크 게이트(FR-INF-073)와 역할이 겹치는지 확인한다. |
| **Q11** | **[결정필요]** 자동 성공 판정을 도입할 것인가? | LeRobot에는 분류기가 없고, enactic의 자동 판정은 dora 노드에만 있다(우리 경로에 없음). 도입하면 VLM 서빙 비용과 오판 리스크가 생기고, 도입하지 않으면 모든 평가가 사람 라벨에 묶인다. **정하려면**: 사람 라벨 N개를 먼저 모아 태스크별 성공 기준을 문장으로 확정한 뒤, VLM 판정의 precision/recall을 그 라벨에 대해 측정한다. 그 전까지 성공률 정본은 **사람 라벨**이다(FR-INF-079). |
