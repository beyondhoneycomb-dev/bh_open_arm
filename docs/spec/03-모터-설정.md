# 03. 모터 설정

> 영역 코드: **MOT** · 대상: OpenArm v2.0 (enactic) + Damiao(DM) 액추에이터 8축/팔
> 모든 수치는 1차 소스(소스코드·공식 문서·BOM)에서 인용했다. 출처 없는 수치는 쓰지 않는다.
>
> **정본 선언**: 게인 세트(5종)와 리밋 세트(5열)의 **정본 전수표는 이 문서의 §2.8 / §2.9**이며, 다른 문서(00·01 등)의 요약은 이를 참조할 뿐 대체하지 않는다. 요약이 이 문서보다 적게 세면 그 요약이 틀린 것이다.

## 1. 범위

이 문서는 GUI "모터 설정" 화면이 조작·표시해야 하는 **모든 파라미터**와 그 프로토콜 계약을 규정한다. 대상: DM 모터 CAN 프레임 인코딩, 제어 모드 4종, 모터 레지스터(RID), 게인 프로파일, 리밋 프로파일, 보호 임계(온도/전류/전압), 에러코드, 그리퍼 힘 제어, 실시간 모터 상태 모니터링, **LeRobot 런타임이 실제로 내보내는 명령의 실체**.

다루지 않는 것: CAN 물리계층 배선·종단저항, 수동 조그/티칭 UX(→ `04`), VR 텔레옵 상류 파이프라인, 데이터셋·학습.

**전제 1 — 런타임**: 정본 런타임은 **LeRobot v0.6.1**이며, GUI 백엔드가 `OpenArmFollower` / `BiOpenArmFollower`를 **단일 프로세스에 임베드**한다(→ `02` §2.0). `ros2_control` / `OpenArmHW` / MoveIt / dora-rs는 경로에 없다. **모든 주기 명령은 `OpenArmFollower.send_action()` → `DamiaoMotorsBus._mit_control_batch()`를 통과한다.** 이 문서의 프로토콜 사실(비트패킹·스케일·RID·에러코드)은 **런타임과 무관하게 유효**하지만, "누가 무엇을 실제로 보내는가"는 LeRobot 소스가 정한다(§2.13).

**전제 2 — CAN 배타성**: SocketCAN은 배타 bind를 제공하지 않으며 두 프로세스가 동일 인터페이스를 bind해도 **에러 없이 둘 다 성공**하고 로컬 루프백이 기본 ON이라 서로의 송신 프레임까지 수신한다(조용한 실패). LeRobot 임베드로 *프로세스 내부* 배타성은 해결되지만 **외부 침입자(`cansend`, `openarm-can-cli`, 두 번째 LeRobot 인스턴스)는 못 막는다** → `flock`이 여전히 필요하다(FR-MOT-006).

**전제 3 — 이 화면의 원시 CAN 접근**: RID 읽기/쓰기, `discover`, `change_id`, `change_baud`, `clear_error`는 **LeRobot API에 존재하지 않는다.** 이들은 `openarm_can` / `openarm-can-cli` 경로로만 가능하며, **LeRobot 버스가 열려 있는 동안 동시에 실행하면 §전제 2의 조용한 실패에 걸린다.** → 이 화면의 원시 CAN 조작은 **LeRobot 컨트롤 루프가 정지한 `MOTOR_CONFIG` 모드에서만** 허용한다(§4.1). 단 **`robot.disconnect()`는 하지 않는다** — CAN 소켓·flock 위생 때문이며, connect() 오버라이드(→ `02` FR-CON-061)로 재연결이 영점을 파괴하진 않지만 버스를 유지한다.

- 출처(공식 문서): https://docs.openarm.dev/setup/openarm-setup/motor-id/

> ✅ **J7 = DM4310 [확정]**. 4개 1차 소스가 일치한다 — ① 공식 하드웨어 BoM: *"J5, J6, J7, J8 motors → DM-J4310-2EC V1.1"*, ② `openarm_driver` `motor_config.types` 배열(7번째 = DM4310), ③ `openarm_teleop/src/openarm_constants.hpp` `DEFAULT_MOTOR_CONFIG` 7번째 = DM4310, ④ LeRobot 주석 `# J7 - Wrist rotation (DM4310)`. 등록값은 **DM4310**이다.
>
> ⚠ **단, `openarm_mujoco/v2/openarm_bimanual.xml`은 J7 관절을 `class="motor_DM3507"`로 선언한다**(라인 225·313). 같은 파일의 액추에이터는 `position_DM4310`(라인 362·370)이므로 **MJCF 파일 자체의 내부 불일치**다. 이것은 **시뮬레이션 자산의 버그**이지 실기 모터 타입의 불확실성이 아니다 → **MuJoCo를 모터 타입의 교차확인 소스로 쓰지 말 것.** 수정 대상은 MJCF이며 09 §2.6에서 다룬다.
>
> 등록을 틀리면 (DM3507의 T_MAX=5 Nm vs DM4310의 T_MAX=10 Nm) **MIT 토크 지령·피드백이 정확히 2배 어긋난다.** 따라서 오등록을 막는 것은 문서상의 금지가 아니라 **부팅 시 RID 21/22/23 대조 런타임 가드**다(FR-MOT-003 · FR-MOT-004).

> ⚠ **좌우 팔은 CAN ID로 구분 불가**: 양팔 모두 send 0x01–0x08 / recv 0x11–0x18 동일. 버스 스캔만으로 좌/우를 판별할 수 없다. 게다가 **채널 관례가 3자 불일치**다 — enactic `openarm_cell.yaml`은 `can0=right`, **LeRobot 공식 문서는 `can0=follower left`**, `openarm_teleop`(`openarm_bilateral_control.cpp:154-155`)은 `can0=leader`. **udev 고정 이름이 유일한 방어선**이며 상류는 이를 제공하지 않는다(→ `02` FR-CON-005·007). 재부팅/재삽입만으로 좌우가 뒤바뀔 수 있다.
>
> 🔴 **LeRobot의 `side`는 좌우를 *판별*하지 않는다 — *선언*할 뿐이다.** `OpenArmFollowerConfig.side`(`config_openarm_follower.py:55`)는 조인트 리밋 기본값을 고르는 스위치이고(`openarm_follower.py:72-84`), 실제 버스에 어느 팔이 물려 있는지는 **검사하지 않는다.** 선언이 틀리면 **왼팔에 오른팔 리밋이 적용된 채 조용히 동작한다.** → 기동 시 별도 검증이 필요하다(→ `04` FR-MAN-003: 모터 시리얼(RID 15) 교차검증 또는 J2 비대칭 스톱 프로브).

### 2.2 모터별 프로토콜 스케일 상한 vs 물리 정격 — **두 값은 다르다**

`MOTOR_LIMIT_PARAMS`(`openarm_can/include/openarm/damiao_motor/dm_motor_constants.hpp`)는 **CAN 프레임 인코딩 범위**이지 안전 토크 한계가 아니다.

| MotorType | P_MAX (rad) | V_MAX (rad/s) | **T_MAX (Nm, 패킷 스케일)** | **Peak Torque (Nm, 물리)** | Rated Torque | URDF effort | 감속비 | v2.0 사용 |
|---|---|---|---|---|---|---|---|---|
| `DM4310` | 12.5 | 30 | **10** | **7** | 3 | 7 | 10:1 | J5–J8 |
| `DM4340` | 12.5 | **8** | **28** | **27** | 9 | 27 | 40:1 | J3, J4 |
| `DM8009` | 12.5 | 45 | **54** | **40** | 20 | 40 | 9:1 | J1, J2 |
| `DM3507` | 12.5 | 50 | **5** | — | — | — | — | ❌ (v2.0 미사용. MJCF J7 class 오기로만 등장 — §2.1) |
| `DM8006` | 12.5 | 45 | 40 | — | — | — | — | ❌ (enum만) |

- T_MAX 출처: `dm_motor_constants.hpp` `MOTOR_LIMIT_PARAMS = {{12.5,50,5}(DM3507), {12.5,30,10}(DM4310), {12.5,8,28}(DM4340), {12.5,45,54}(DM8009), …}}`
- Peak/Rated Torque·감속비 출처: https://docs.openarm.dev/hardware/openarm-2.0/motor/
- URDF effort 출처: `openarm_description/assets/robot/openarm_v2.0/config/arm/joint/joint_limits.yaml` — effort가 peak torque와 정확히 일치(40/40/27/27/7/7/7)

> **함정 1 — 토크**: T_MAX까지 슬라이더를 열면 모터 정격을 초과 지령할 수 있다. **패킷 스케일은 T_MAX, 사용자 지령 클램프는 Peak Torque**로 이중 관리해야 한다(FR-MOT-033).
> **함정 2 — 속도**: DM4340의 V_MAX=8 rad/s(≈76 rpm)는 카탈로그 Max No-load Speed 52 rpm(≈5.45 rad/s)보다 크다. URDF velocity 리밋(5.4454 rad/s)이 물리 정본이다.
> **함정 3 — 검증**: RID 21/22/23(PMAX/VMAX/TMAX)로 모터 내부 저장값을 읽어 라이브러리 상수와 대조 가능하다. 불일치하면 위치·속도·토크 스케일링이 **전부** 틀린다(FR-MOT-003).

### 2.3 MIT 모드 패킷 — 비트 폭 및 스케일링 (★핵심★)

인코딩 함수 (`openarm_can/src/openarm/damiao_motor/dm_motor_control.cpp`, 원문):

```cpp
uint16_t CanPacketEncoder::double_to_uint(double x, double x_min, double x_max, int bits) {
    x = limit_min_max(x, x_min, x_max);
    double span = x_max - x_min;
    double data_norm = (x - x_min) / span;
    return static_cast<uint16_t>(data_norm * ((1 << bits) - 1));
}
```

```cpp
std::vector<uint8_t> CanPacketEncoder::pack_mit_control_data(MotorType motor_type,
                                                             const MITParam& mit_param) {
    uint16_t kp_uint = double_to_uint(mit_param.kp, 0, 500, 12);
    uint16_t kd_uint = double_to_uint(mit_param.kd, 0, 5, 12);

    LimitParam limits = MOTOR_LIMIT_PARAMS[static_cast<int>(motor_type)];
    uint16_t q_uint   = double_to_uint(mit_param.q,   -limits.pMax, limits.pMax, 16);
    uint16_t dq_uint  = double_to_uint(mit_param.dq,  -limits.vMax, limits.vMax, 12);
    uint16_t tau_uint = double_to_uint(mit_param.tau, -limits.tMax, limits.tMax, 12);
    ...
}
```

**필드 표 (구현은 이 표만 보면 된다):**

| 필드 | 비트 폭 | 스케일 범위 (min ~ max) | 모터 의존 | 바이트 배치 |
|---|---|---|---|---|
| `q` (p_des) | **16 bit** | **−P_MAX ~ +P_MAX** rad (전 기종 ±12.5) | O | D0=q[15:8], D1=q[7:0] |
| `dq` (v_des) | **12 bit** | **−V_MAX ~ +V_MAX** rad/s | O | D2=dq[11:4], D3[7:4]=dq[3:0] |
| `kp` | **12 bit** | **0 ~ 500** (고정, 모터 무관) | X | D3[3:0]=kp[11:8], D4=kp[7:0] |
| `kd` | **12 bit** | **0 ~ 5** (고정, 모터 무관) | X | D5=kd[11:4], D6[7:4]=kd[3:0] |
| `tau` (t_ff) | **12 bit** | **−T_MAX ~ +T_MAX** Nm | O | D6[3:0]=tau[11:8], D7=tau[7:0] |

- DLC 8, CAN ID = `send_can_id` (오프셋 없음).
- Damiao 공식 문서 원문 교차검증: *"The range of Kp is [0, 500], and the range of Kd is [0, 5]."* / *"Position 2 characters 16 bits, Velocity 12 bits, Kp 12 bits, Kd 12 bits"* — https://damiao.enactic.ai/en/products/hardware/dm-j4340p-2ec-v1.0/#control-frame-in-mit-mode
- `struct MITParam { double kp; double kd; double q; double dq; double tau; };` (`dm_motor_control.hpp`) — **필드 순서가 kp, kd, q, dq, tau임에 주의**(위치가 첫 인자가 아니다).

**양자화 분해능 [계산]** (16bit → 나누기 65535, 12bit → 나누기 4095):

| 필드 | DM4310 | DM4340 | DM8009 |
|---|---|---|---|
| q (rad/LSB) | 3.815e−4 | 3.815e−4 | 3.815e−4 |
| dq (rad/s per LSB) | 1.465e−2 | 3.907e−3 | 2.198e−2 |
| tau (Nm/LSB) | 4.884e−3 | 1.368e−2 | 2.637e−2 |
| kp (per LSB) | 0.1221 (전 기종 공통) | | |
| kd (per LSB) | 1.221e−3 (전 기종 공통) | | |

### 2.4 제어 모드 4종 — 명령 필드와 전환 방법

```cpp
enum class ControlMode : uint8_t { MIT = 1, POS_VEL = 2, VEL = 3, POS_FORCE = 4 };
```

| 모드 | 값 | 전송 CAN ID | 페이로드 | DLC | 명령 구조체 |
|---|---|---|---|---|---|
| **MIT** (임피던스) | 1 | `send_can_id` **+ 0x000** | §2.3 비트패킹 | 8 | `MITParam{kp, kd, q, dq, tau}` |
| **POS_VEL** (위치-속도) | 2 | `send_can_id` **+ 0x100** | `[q f32 LE][dq f32 LE]` | 8 | `PosVelParam{q, dq}` |
| **VEL** (속도) | 3 | `send_can_id` **+ 0x200** | `[dq f32 LE]` | **4** | `VelParam{dq}` |
| **POS_FORCE** (힘-위치) | 4 | `send_can_id` **+ 0x300** | `[q f32 LE][dq u16 LE][i u16 LE]` | 8 | `PosForceParam{q, dq, i}` |

**전환 방법 — 두 가지가 있고 구분해야 한다:**

1. **일시 전환**: 프레임 ID 오프셋만 바꿔 송신한다. 모터 내부 파라미터를 건드리지 않으므로 즉시·가역적이며 플래시 쓰기를 소모하지 않는다.
2. **영구 전환**: RID **10 (`CTRL_MODE`)** 쓰기. `create_set_control_mode_command()`가 `0x7FF` ID로 write-param 프레임을 보낸다. `--save` 시 플래시에 기록되어 **10,000회 쓰기 한계**를 소모한다.

**POS_FORCE 스케일링 (`dm_motor_control.hpp` 주석 원문):**

```
// P_des: position command in rad (float).
// V_des: speed limit in rad/s, scaled by 100 into uint16 (little-endian), range 0-10000
//        (values above 10000 clamp to 10000), corresponding to 0-100 rad/s.
// I_des: torque current limit per-unit value, scaled by 10000 into uint16 (little-endian),
//        range 0-10000 → 0-1.0.
//        Per-unit current value = actual current / max current (Imax printed on power-up).
```

> **i_des는 Nm이 아니다.** "최대 전류 대비 per-unit(0~1)"이다. 최대 전류(Imax)는 모터 전원 인가 시 출력된다. → §2.10 참조.

**POS_VEL / VEL 모드 사용 시 제약** (Damiao 공식): RID 11 `Damp`는 **0이 아닌 양수여야 하며 권장값 4.0**이다. 위치 제어 시 **kd=0으로 두면 모터가 진동하거나 폭주한다**(*"When controlling the position, kd cannot be set to 0. Otherwise it will cause the motor vibrate or go out of control."*).

### 2.5 명령 바이트 (enable / disable / set_zero / refresh / param)

`pack_command_data(cmd)` = `{0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, cmd}` (`dm_motor_control.cpp`)

| 명령 | CAN ID | 데이터 8바이트 |
|---|---|---|
| **Enable** | `send_can_id` | `FF FF FF FF FF FF FF` **`FC`** |
| **Disable** | `send_can_id` | `FF FF FF FF FF FF FF` **`FD`** |
| **Set Zero (영점 저장)** | `send_can_id` | `FF FF FF FF FF FF FF` **`FE`** |
| **Refresh (상태 요청)** | `0x7FF` | `[id_L][id_H]` **`CC`** `00 00 00 00 00` |
| **Read Param** | `0x7FF` | `[id_L][id_H]` **`33`** `[RID] 00 00 00 00` |
| **Write Param** | `0x7FF` | `[id_L][id_H]` **`55`** `[RID] [value 4B LE]` |

원시 CAN 교차검증 (공식 셋업 문서):
```bash
cansend can0 001#FFFFFFFFFFFFFFFC   # 모터 #1 enable
cansend can0 001#FFFFFFFFFFFFFFFD   # 모터 #1 disable
```
출처: https://docs.openarm.dev/setup/openarm-setup/configure-test/

### 2.9 리밋 프로파일 — 프레임·단위·정본

**두 개의 프레임을 구분하지 않으면 전부 틀린다:**

| 프레임 | 정의 | 사용처 |
|---|---|---|
| **F_URDF** | `openarm_description` v2.0 URDF의 관절각. **오른팔이 정본**, 왼팔은 y-반사 → 반사되는 관절은 `[lo, hi] → [−hi, −lo]` | URDF, MuJoCo, MoveIt, ros2_control, `openarm_driver`의 **사용자 좌표** |
| **F_motor** | DM 모터가 마지막 `set_zero`(0xFE)를 받은 자세를 0으로 하는 각도 | **CAN 버스 위의 실제 지령/피드백** |
| 변환 | `q_motor = q_URDF + joint_offsets` — **`openarm_driver`만 적용한다.** `ros2_control`과 LeRobot은 **offset = 0을 가정**한다 | — |

- 근거(ros2_control): `openarm_simple_hardware.cpp:285-293` — `pos_commands_`(URDF 관절각)를 그대로 `mit_control_all()`에 전달. 헤더의 `ZERO_POSITION = {0,0,0,0,0,0,0}`.
- 근거(LeRobot): `openarm_follower.py::send_action`에 offset 가산이 **어디에도 없다**. `MotorCalibration(homing_offset=0, …)`은 저장되나 사용되지 않는다(`damiao.py`: *"Damiao motors don't store calibration internally. Just cache it in memory"*).
- 근거(driver): `driver.py::send_position` — `oa.MITParam(kps[i], kds[i], target_pos[i] + self.joint_offsets[i], 0, 0)`.

> ⚠️ **위 표에서 `ros2_control`·MoveIt은 우리 런타임이 아니다** — 값의 출처(provenance)로만 인용한다. **우리 경로에서 실제로 도는 것은 LeRobot이고, LeRobot은 offset을 적용하지 않는다.**
>
> 🔴 **⇒ LeRobot 경로에서 `q_motor = q_command`다. 오프셋 계층이 없다.**
> 즉 **LeRobot의 관절각이 F_URDF와 같아지는 것은 "URDF 영점 자세에서 0xFE를 쳤다"는 전제 하에서만이다.** 그런데 `lerobot-calibrate`의 영점 자세는 **"팔을 곧게 아래로 늘어뜨린 자세"**(`openarm_follower.py:192-197`)이지 URDF 영점 자세가 아니다.

> **어느 쪽이든 GUI는 사용자 좌표(F_URDF, offset 미포함)만** 표시·입력하며, 오프셋 이중 가산/미가산을 감사 링버퍼로 감지해야 한다(FR-MOT-034, NFR-MOT-004).

**스텝 델타 vs 속도 — 서로 다른 것이다:**

> 2. 그러나 **API는 per-unit이다** — `torque_pu ∈ [0, 1]`, 정의는 `실제 전류 / 최대 전류(Imax)`. Nm이 아니다.

```python
robot.bus.sync_read_all_states() -> dict[str, MotorState]   # pos/vel/torque/temp_mos/temp_rotor  :655-674
robot.bus.sync_read("Present_Torque")                  # 개별 접근 (`_get_cached_value`, :604-616)
robot.bus._mit_control_batch(commands)                 # (kp, kd, pos_deg, vel_deg_s, torque)  :492-502
```
`MotorState`는 TypedDict다 — `damiao.py:68-73`: `{position, velocity, torque, temp_mos, temp_rotor}`.

> 🔴 **LeRobot도 ERR 니블을 파싱하지 않는다.** `_decode_motor_state()`(`damiao.py:543-568`)는 `data[1]`~`data[7]`만 읽고 **`data[0]`(= `ID | ERR<<4`)을 버린다** — `openarm_can`과 **정확히 같은 갭**이다. 즉 **런타임을 바꿔도 에러코드는 여전히 우리가 직접 디코딩해야 한다**(FR-MOT-041).

> 🔴 **LeRobot은 패킷 드랍을 은폐한다.** `_batch_refresh`가 응답을 못 받으면 **마지막 알려진 상태를 그대로 재사용**하고 `logger.warning("Packet drop: ...")`만 뱉는다(`damiao.py:696-702`). 드랍 카운터를 feature로 내보내지 **않는다.** 토크는 위치보다 노이즈가 크고 드랍의 영향을 크게 받으므로 **드랍 카운터를 관측 메타로 노출해야 한다**(FR-MOT-055).

### 2.14 제어 주파수 — 상한과 미측정 구간

| 경로 | 주파수 | 출처 |
|---|---|---|
| **openarm_can 공식 경고 — 절대 상한** | **8모터 구성에서 1000 Hz 초과 제어 사이클은 CAN 연결을 불안정하게 만든다** | openarm_can 공식 문서 |

| `openarm_driver.smooth_move` 기본 | 50 Hz × 2 s (선형보간, `np.linspace`) | `openarm_cell.yaml` `hz: 50`, `duration: 2` |
| `recv_all()` 첫 응답 타임아웃 (원시 CAN 경로) | 500 µs 기본 | `openarm.hpp` |
| **🔴 LeRobot Python 루프 실측 사이클 타임** | **[미측정] — 최우선 측정 항목** | — |

**LeRobot 루프의 타임아웃 상수** (`damiao.py:62-65`):
```python
LONG_TIMEOUT_SEC    = 0.1      # 100 ms
MEDIUM_TIMEOUT_SEC  = 0.01     #  10 ms   ← enable/disable/set_zero 사이 sleep
SHORT_TIMEOUT_SEC   = 0.001    #   1 ms   ← _mit_control_batch 응답 수집
PRECISE_TIMEOUT_SEC = 0.0001   # 100 µs
```
- `_mit_control_batch`는 전 모터에 명령을 **먼저 다 보내고** 응답을 1 ms로 일괄 수집한다(`damiao.py:513-526`) → 라운드트립이 모터 수만큼 직렬화되지 않는다.
- `sync_read_all_states()`는 pos/vel/torque를 **"in ONE CAN refresh cycle"**로 함께 읽는다(`damiao.py:655-674` docstring) → **토크 관측에 추가 CAN 비용이 없다.**
- ⚠️ `enable_torque` / `disable_torque` / `set_zero_position`은 모터마다 `MEDIUM_TIMEOUT_SEC`(10 ms) sleep을 건다 → **8모터 = 최소 80 ms.** 소프트 스톱 지연 예산에 반영해야 한다(NFR-MOT-005).
- 🔴 **Python/GIL 아래의 실측 사이클 타임은 [미측정]이다.** 이 값이 바이래터럴(500 Hz 요구) 실현 가능성을 결정한다 → §5 **Q-12**.

## 4. 상태·오류 처리

### 4.1 모터 상태 전이

> 🔴 **이 상태머신은 `robot.connect()` **이후**의 상태만 다룬다.** 연결 자체는 세션당 1회이며 `02` §4.1이 소관이다. **어떤 전이도 `robot.disconnect()`를 유발해서는 안 된다**(CAN 소켓·flock 위생). 영점 안전은 connect() 오버라이드(→ `02` FR-CON-061)로 확보되고, 0xFE는 명시 set_zero 플로우에서만 나간다.
