# 00. 개요 및 문서 규약

## 1. 이 문서 묶음이 무엇인가

OpenArm v2.0 로봇으로 **피지컬 AI 워크플로우 전체**(텔레오퍼레이션 → 데이터 수집 → 학습 → 추론 → 평가)를 하나의 GUI 플랫폼에서 수행하기 위한 **기능 명세서**다.

**이 문서는 기획서가 아니다.** 로드맵, 일정, 마일스톤, 인력, 비용, 시장 분석, 경쟁 우위는 이 묶음에 없다. 오직 *"시스템이 무엇을 할 수 있어야 하는가"*, 그 입출력과 파라미터와 제약과 인터페이스 계약만 기술한다. 기획은 이 명세가 확정된 뒤에 별도로 수행한다.

### 2.1 우리가 만드는 것

enactic의 실제 아키텍처는 **레이어드**다. 핵심 라이브러리(`openarm_can` C++, `openarm_driver`, `openarm_control`, `openarm_ker`)가 전부 **dora 의존성 0**이고, `dora-*` 노드는 그 위의 **얇은 래퍼**다(`quest_receiver.py` ≈200줄, dora 의존은 `node.send_output` 3줄뿐이고 나머지는 순수 numpy/scipy). **dora는 배선이지 로직이 아니다.** → 우리는 새 스택이 아니라 **LeRobot 위에 4개 계층**을 얹는다:

| 만드는 것 | 왜 |
|---|---|
| **VR/KER 텔레오퍼레이터** (`lerobot_teleoperator_*` 플러그인) | LeRobot에 VR/KER 텔레오퍼레이터가 **없다** |
| **안전 계층** | LeRobot 안전장치는 `joint_limits` + `max_relative_target`(기본 OFF)뿐 (§3.5) → **참조 레포에서 이식** |
| **실기 평가 하네스** | `lerobot-eval`은 **시뮬 전용**(`configs/eval.py:29-31` `env: EnvConfig`가 필수 필드). 실기 성공률 집계가 LeRobot 어디에도 없다 |
| **통합 GUI** | **PyQt6 데스크톱** — 참조 레포 `bh_indy7_lerobot_gui` 구조를 이식 |

**LeRobot CLI(`lerobot-record` 등)를 모드마다 서브프로세스로 스폰하는 설계는 금지한다.** CLI 스크립트는 각각 독립 프로세스이며 시작 시 `connect()`, 종료 시 `disconnect()`를 한다 → §3.2의 영점 파괴가 모드 전환마다 발생한다. **LeRobot Python API를 단일 GUI 백엔드 프로세스에 임베드하고, 모드는 같은 robot 객체 위 상태 전이로 처리한다**(참조 레포가 실제로 이렇게 한다). → `01-시스템-아키텍처.md`

## 3. 가장 중요한 사실 5가지

이 5가지가 명세 전체의 형태를 결정한다. 다른 것을 다 잊어도 이건 기억해야 한다. (런타임=LeRobot과 우리가 만드는 4계층은 §2·§2.1에 있다.)

### 3.1 🔴 LeRobot의 openarm은 v1 시대에 작성됐다 — 리밋·게인·그리퍼를 v2로 교체한다

**결정적 증거는 타임라인이다.** LeRobot의 openarm 통합은 **PR #2795 (2026-01-28)** 병합이다. 그 시점 OpenArm 하드웨어 최신은 **tag 1.1 (2025-10-31) = v1.x**뿐이었고, `openarm_description`의 **v2.0 자산은 2026-05-19~25**, `openarm_driver`는 2026-06에야 등장했다. → **LeRobot의 openarm 값은 v1 하드웨어 기준이다.** 모터 라인업만 v1·v2 공통이라 정확하고, **관절 리밋·PD 게인·그리퍼는 v1-era**다.

**우리는 모든 물리값을 OpenArm v2.0 정본으로 확정한다.** LeRobot 값을 그대로 정본으로 쓰면 v1 로봇을 v2 하드웨어에서 돌리게 된다. 아래가 판별선이다 — 왼쪽(LeRobot/v1-era)을 정본으로 쓰면 **오염**, 오른쪽(v2 정본)이 답이다.

| 항목 | LeRobot / v1-era (버릴 값) | **v2 정본 (쓸 값)** | 근거 |
|---|---|---|---|
| **모터 라인업** `[v1v2 공통]` | (동일 — 유일하게 정확) | J1/J2=DM8009, J3/J4=DM4340, J5/J6/J7+그리퍼=DM4310. 8모터/암, send `0x01–0x08` / recv `0x11–0x18` | `openarm_driver/configs/openarm_cell.yaml` `types` = LeRobot 매핑 일치. whats-new "Same motor lineup as 1.0" |

**참조 템플릿의 수치는 OpenArm 값이 아니다.** `bh_indy7_LeRobot`/`bh_rebot_LeRobot`은 Seeed reBot B601-DM(6+1축) + Indy7이며 **DM 모터만 공통**이다. **패턴은 이식하되**(디스크 SoT, connect≠set_zero, 그리퍼 엔드포인트 캡처, 충돌가드, tolerant 카메라), **수치는 금지**한다 — 150°/s, kp `[8,8,8,5,5,5]`, 그리퍼 −270°, 6관절, 모터 ID `0x01–0x07`, `collision_tau[20…]`는 전부 reBot 값이다.

### 3.2 🔴 `connect()`가 영점을 파괴한다 — 오버라이드하고 디스크를 SoT로 둔다

```python
# robots/openarm_follower/openarm_follower.py:152-153
if self.is_calibrated:
    self.bus.set_zero_position()      # → CAN_CMD_SET_ZERO (damiao.py:335-340)
```

`CAN_CMD_SET_ZERO`는 **현재 물리 자세를 영점으로 확정하는 모터 명령**이다. 팔이 rest 자세가 아닌 순간 재연결하면 **그 자세가 영점이 되고**, 조인트 리밋과 데이터셋 좌표계가 통째로 어긋난다. `connect()`의 독스트링도 이를 인정한다: *"We assume that at connection time, the arms are in a safe rest position."*

→ **세션당 `connect()`는 정확히 1회.** 모드 전환(텔레옵 ↔ 기록 ↔ 추론 ↔ 캘리브레이션 ↔ 수동)에 disconnect/reconnect를 **절대 하지 않는다.** 모드는 **같은 robot 객체 위의 상태 전이**이지 프로세스 재시작이 아니다. 부득이 재연결이 필요하면 **rest 자세 확인 모달을 강제**한 뒤에만 허용한다. → `01`, `14-시스템-운영.md`

**해법은 참조 레포가 이미 구현했다** — auto set_zero를 끈다. `connect()`가 자동 캘리브를 돌리지 않고(`rebot.py:247-249` *"calibrate is not auto-run"*), bring-up은 `connect_readonly()`(`rebot.py:259`, **torque-OFF** 버스 오픈, 손으로 검증)로 열며, 영점은 operator가 rest에 놓고 **명시적으로** 찍는다. **DM `set_zero`(`0xFE`)는 flash에 자동 저장되지 않아 전원 사이클마다 휘발**하므로, **SoT는 디스크 캘리브레이션 파일**(`save_calibration_atomic`: signs·그리퍼 open/close rad·captured 플래그·timestamps, 원자적 tmp→fsync→os.replace)에 둔다. 모터 flash 저장(`0xAA`)에 의존하지 않는다. → `02`, `03`

### 3.3 홀딩 브레이크가 없다 — E-Stop을 누르면 물체가 떨어진다

공식 Safety Guide 원문:

> *"OpenArm has high backdrivability suitable for bilateral control. **Be aware that if power is lost due to an emergency stop, the load being held will fall rapidly.**"*

E-Stop은 팔 내장이 아니라 **전원 라인에 삽입하는 외부 버튼**이다. 높은 백드라이버빌리티는 바이래터럴 텔레옵의 전제조건이지만, 그 대가가 이것이다. 게다가 Damiao 모터는 **CAN 명령이 끊기면 스스로 인에이블을 이탈한다**(통신상실 보호, RID 9 `TIMEOUT`) → 인에이블 이탈 = 토크 0 = 낙하.

→ **소프트 스톱(모터 토크로 자세를 유지한 채 정지)과 하드 E-Stop(전원 차단 → 낙하)은 완전히 다른 기능이며, 명세·구현·GUI에서 분리되어야 한다.** 충돌 감지의 기본 반응이 전원 차단이어서는 안 된다. **어떤 상태에서도 CAN 명령 스트림을 끊으면 안 된다.** → `12-충돌감지-및-안전.md`, `14` §4

### 3.4 토크는 `observation.state`로 평탄화된다 — 그리고 기본값이 OFF다

```python
# robots/openarm_follower/openarm_follower.py:90-99
features[f"{motor}.pos"] = float
if self.config.use_velocity_and_torque:      # ← config_openarm_follower.py:71: 기본 False
    features[f"{motor}.vel"] = float
    features[f"{motor}.torque"] = float
```

값도 실제로 채워진다(`:239-246` `sync_read_all_states()`). 버스는 tau를 버리지 않는다(`motors/damiao/damiao.py:68-73` `MotorState{position, velocity, torque, temp_mos, temp_rotor}`, `:556,566` 피드백 프레임에서 `tau_uint` 디코드). 주석: *"Reads all motor states (pos/vel/torque) in **one CAN refresh cycle**"* → **추가 CAN 비용 없음. 토크 기록은 공짜다.**

**그러나 4개의 단서가 있다:**

1. **기본값이 `False`다.** 안 켜면 **조용히** 위치-only 8차원 데이터셋이 생성된다. GUI가 강제로 `True`를 넣어야 한다.
2. **`observation.effort`는 존재하지 않는다** (LeRobot 전체 트리 grep 0건). 토크는 `feature_utils.py:68-89`에서 **`observation.state` 벡터 안으로 평탄화**된다. 단완 8모터 × (pos, vel, torque) = **24차원**, 양완 **48차원**. `names`에 `joint_1.torque` 문자열만 남는다. → `qtorque`는 별도 채널이 아니라 **`observation.state`의 부분 인덱스**다.
3. **한 벡터에 deg와 Nm이 섞인다** — `.pos` = **deg**, `.vel` = **deg/s**, `.torque` = **Nm**(변환 없음, `damiao.py:568`). 정규화 통계(mean/std) 산정 시 반드시 고려해야 한다.
4. **토크 액션은 기록만 되고 실행되지 않는다** — `openarm_follower.py:340`: `commands[motor_name] = (kp, kd, position_degrees, 0.0, 0.0)`. `send_action`은 `.pos`만 읽는다(`:288`). 단 버스 레벨 `_mit_control_batch(commands: {motor: (kp, kd, pos_deg, vel_deg_s, torque)})`(`damiao.py:492-502`)는 **이미 tau 인자를 받는다** → 막혀 있는 건 상위 함수 하나뿐이다.

→ `07-데이터-수집.md`, `08-데이터셋-관리.md`, `10-학습.md`

### 3.5 LeRobot 안전장치는 빈약하다 — 안전 계층은 참조 레포에서 이식한다

LeRobot이 제공하는 것은 **`joint_limits` + `max_relative_target`(기본 `None` = 꺼짐)** 이 전부다.

**속도 제한 없음. 가속도/저크 제한 없음. 소프트웨어 E-Stop 없음. 워크스페이스 가상벽 없음. 충돌 반응 없음.**

→ 안전 계층(속도·가속도 클램프, 소프트 스톱 `STOP_HOLD`, 워크스페이스 가상벽, 충돌 반응, 명령 신선도 워치독)은 **우리가 만든다 — 참조 레포가 이미 구현한 것을 이식한다:**

| 이식할 것 | 참조 레포 구현 | 핵심 |
|---|---|---|
| **jump-guard** | `ensure_safe_goal_position` + `_previous_q_deg`(`rebot.py:138-264`) | 스텝당 절대 델타 클램프. ⚠️ **`max_relative_target`은 속도가 아니라 스텝당 델타** — 루프가 느려지면 실효 속도 상한이 올라간다 → **속도 리밋(\|Δq\|/dt)과 점프 가드를 분리**한다 |
| **collision_guard** | `ReBotCollisionGuard`(별도 스레드, 버스 락, 토크 잔차) | `.torque` 관측이 전제(§3.4의 `use_velocity_and_torque=True`) |
| **connect_readonly** | `rebot.py:259` | torque-OFF 버스 오픈 — GUI bring-up·손 검증용 |
| **tolerant 카메라** | 죽은 카메라는 경고 후 skip | 모션 경로에 영향 없음 |

`12-충돌감지-및-안전.md`가 설계한 내용이 **그대로 유효하며**, 위치가 "dora 노드 대체"에서 **"LeRobot 위에 얹는 계층(참조 레포 이식)"**으로 바뀐다.

**게인·리밋 정본은 §3.1 매트릭스로 v2 확정됐다** — "단일 정본이 없다"는 옛 서술은 폐기한다. 운영 프로파일은 **compliant(70, 텔레옵·접촉) / stiff(230, 디지털 트윈·드라이런·평가 — sim↔real 패리티 필수) / 정책 재생(추종 정확도 우선)** 세 가지이며, **LeRobot 기본값인 240계열은 v1 유래 비정본**이라 그대로 쓰면 v1 게인으로 v2 로봇을 돌리게 된다. 명세는 이를 **이름 붙은 프로파일 저장·전환·검증 + v1 프로파일 "비정본" 라벨** 기능으로 해소한다. → `03-모터-설정.md` §2.8

## 6. 요구사항 표기 규약

### 6.1 ID

`FR-<영역>-<3자리>` (예: `FR-TEL-001`). 비기능 요구사항은 `NFR-<영역>-<3자리>`.
영역 코드는 위 표에 고정되어 있다. 문서 간 ID는 충돌하지 않는다.

### 6.2 표 형식

| ID | 요구사항 | 우선 | 근거 | 비고 |
|---|---|---|---|---|
| FR-XXX-001 | 시스템은 ~할 수 있어야 한다. | M | `파일:라인` 또는 URL | [확정] |

### 6.3 우선순위

| 코드 | 의미 |
|---|---|
| **M** | 필수 (Must) — 없으면 플랫폼이 성립하지 않는다 |
| **S** | 권장 (Should) — 없어도 동작하나 실사용에서 곧 요구된다 |
| **C** | 선택 (Could) — 있으면 좋다 |

우선순위는 **기능 자체의 필수성**을 나타낼 뿐 일정이 아니다.

### 6.4 비고 태그

| 태그 | 의미 | 다음 행동 |
|---|---|---|
| `[확정]` | 1차 소스(코드/공식 문서)로 검증됨 | 그대로 구현 |
| `[미확인]` | 출처를 찾지 못함 | 확인 후 확정하거나 측정 |
| `[결정필요]` | **충돌하는 사실이 전부 실재**하여 사람이 골라야 함 | 사람이 결정 |
| `[신규구현]` | upstream에 없어 우리가 만들어야 함 | 설계·구현 |

`[결정필요]`와 `[미확인]`은 다르다. 전자는 답이 **여러 개**라서 골라야 하고, 후자는 답을 **아직 모른다**.

### 6.5 요구사항 문장

반드시 **"~할 수 있어야 한다" / "~해야 한다"** 형태로, 검증 가능하게 쓴다.
*"~하면 좋다"*, *"~를 고려한다"*는 요구사항이 아니다.

## 7. 근거의 등급

이 명세의 모든 수치는 아래 중 하나의 등급을 갖는다. 등급 없는 수치는 명세에 넣지 않는다.

1. **1차 소스** — 코드 파일(경로·라인), 공식 문서, 데이터시트. 근거 칸에 그대로 인용한다.
2. **실측** — 우리가 측정했거나, 출처가 명시된 제3자 실측치. 측정 조건을 함께 적는다.
3. **[미확인]** — 위 둘 중 어느 것도 없음. 명세는 이 값을 **가정하지 않고**, 대신 *"측정하라"*를 요구사항으로 만든다.

**문서보다 소스를 믿는다.** LeRobot 공식 OpenArm 문서(`openarm.mdx`)는 낡았다 — record 예제가 `--repo-id / --fps / --num-episodes`를 쓰는데 실제 `RecordConfig`(draccus)는 `--dataset.repo_id / --dataset.fps / --dataset.num_episodes`다. Configuration Options 표에 **`use_velocity_and_torque`가 없다**(grep 0회). train / rollout / eval 섹션은 **0개**다. 이 명세는 문서가 아니라 소스를 인용한다.

리서치 단계에서 **20건 이상의 주장이 반증 검증**을 거쳤다. v3 검증은 나아가 **LeRobot의 openarm이 v1 시대(2026-01-28)에 작성됐음을 타임라인으로 확정**하고(§3.1), v1·v2·reBot **3축 오염**을 걸러냈다. 뒤집혀 이 명세에 **들어가지 않는 것**: dora 정본 런타임, OpenArm Dataset 정본 포맷, MoveIt IK 교체 필요, Isaac Lab archived, 텔레옵 지연 150 ms 상한, USB Full-Speed 12 Mbps, **v1 그리퍼 88 mm/링키지, LeRobot deg 리밋=v2 정본, "70=v2 게인 판별자", stale `/teleop/vr`=실기 미지원, GUI 전송 계층 오버엔지니어링, reBot 수치**. 전수는 `README.md` §"반증되어 배제된 것"과 `16-미해결-이슈.md` §9·§10.

## 8. 이 명세를 읽는 순서

구현을 시작하려면:

1. **`01-시스템-아키텍처.md`** — LeRobot 임베드 경계와 CAN 소유권을 이해하지 못하면 나머지가 무의미하다.
2. **`02` → `03` → `04`** — 로봇을 켜고(⚠️ `connect()` 1회), 모터를 설정하고, 손으로 움직인다.
3. **`05` + `06` + `07`** — VR로 조종해서 데이터를 쌓는 경로.
4. **`10` → `11`** — 쌓은 데이터로 학습하고 추론한다.
5. **`12` + `14` + `15`** — 안전, 운영, 성능. 3번 이전에 읽어야 할 수도 있다.
6. **`16`** — 무엇이 v2로 확정됐는가, 그리고 남은 소수의 비차단 자체검증.

> **이 순서는 문서 의존성이지 작업 일정이 아니다.** 이 명세는 일정을 다루지 않는다.
