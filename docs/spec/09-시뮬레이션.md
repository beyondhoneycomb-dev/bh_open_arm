# 09. 시뮬레이션 및 디지털 트윈

요구사항 ID 영역 코드: **`SIM`**

## 1. 범위

이 문서는 OpenArm v2.0 플랫폼의 **시뮬레이션 화면**과 그 백엔드를 규정한다.

**시뮬레이션 백엔드는 2단계 이중 구현이며, 사용자가 런타임에 선택한다(FR-SIM-097):**
- **🔴 1단계 = MuJoCo (정본·검증 구현).** 성숙하고 CPU 실시간이며 상류(`openarm_mujoco`·`openarm_control`)가 검증한 경로. **순수시뮬·디지털트윈·드라이런·평가의 기본 구현**이자, 실시간 텔레옵 IK 엔진의 기반(§1.1).
- **2단계 = Isaac Sim + Isaac Lab (선택 구현).** GPU 기반 학습(IL/RL)·대규모 병렬 평가·합성데이터·포토리얼 렌더. OpenArm의 Isaac 지원(`openarm_isaac_lab`)이 **아직 실험 단계**(커밋 7개·RL 태스크만·"under development")이므로 **2단계 선택 백엔드**로 둔다.

세 가지 운용 모드 — **(a) 순수 시뮬레이션**, **(b) 디지털 트윈**(실기 상태 미러링), **(c) 드라이런**(실기 전송 전 궤적 검증) — 과 실기/시뮬 전환 인터페이스, 시뮬레이션 파라미터, sim2real 갭 및 도메인 랜덤화, 정책 평가, GUI 요구사항을 다룬다.

> 🔴 **왜 MuJoCo가 1단계인가.** (1) OpenArm의 Isaac 자산(`openarm_isaac_lab`)은 RL 태스크 4종·커밋 7개·*"under development"*로 **실험 단계**이고 트윈/드라이런/평가/카메라가 없다(§2.1). (2) MuJoCo는 상류 검증됐고 CPU 실시간이며, 실시간 텔레옵 IK(mink)의 기구학 엔진이라 **어차피 프로젝트에 존재**한다(§1.1). (3) 두 백엔드는 **동일 LeRobot `Robot` ABC(§2.10)를 공유**하므로 둘 다 구현해 **런타임에 선택**하면 된다(FR-SIM-097). → **검증된 MuJoCo로 먼저 성립(1단계), Isaac은 성숙하면 선택 백엔드로 켠다(2단계).**
> **결정 이력**: 초기 검토에서 Isaac 정본안(GPU 허용 전제)을 거쳤으나, **사용자 재결정으로 "검증된 MuJoCo=1단계 정본 / Isaac=2단계 선택"으로 확정**(Isaac의 OpenArm 지원이 실험 단계임을 반영). IK는 mink 유지(§5 Q15). GPU는 2단계(Isaac)에서만 필수, 1단계(MuJoCo)는 CPU.

> 🔴 **런타임 전제 정정 (v1 폐기).**
> 이전 판은 **`dora-openarm-mujoco` 노드를 "정본 런타임"**으로 삼고 `dataflow-vr.yaml` ↔ `dataflow-vr-mujoco.yaml` 교체를 시뮬/실기 전환의 축으로 썼다. **틀렸다. dora는 쓰지 않는다.**
> 런타임은 **LeRobot**이고, 시뮬 싱크는 dora 노드가 아니라 **LeRobot `Robot` 인터페이스를 구현한 우리 플러그인**(`lerobot_robot_*`)이다(§2.10).

### 1.1 🔴 MuJoCo는 시뮬레이터가 아니다. **크리티컬 패스 위의 IK 엔진 기반**이다 [확정]

이것이 이 문서를 읽는 방식을 바꾼다.

```
VR 텔레오퍼레이터 (우리 구현)
  └─ openarm_control.Kinematics          ← 텔레옵 실시간 경로
       ├─ import mink, mujoco            (kinematics.py)
       ├─ mink.Configuration / FrameTask / ConfigurationLimit / DofFreezingTask
       └─ ArmSetup.from_args(xml=openarm_mujoco.v2.openarm_cell_xml(), …)
            └─ **v2 MJCF (cell.xml)**    ← 이 문서가 소유하는 자산
```

- `openarm_control/config.py:23-26`이 **`openarm_mujoco.v2`를 하드코딩**한다. IK가 푸는 모델은 v2 MJCF 그 자체다 (nq=19, nv=19, nu=17 — lifter 1 + 좌팔 7+2 + 우팔 7+2).
- **조인트 리밋의 실질 소스가 MJCF다**: `mink.ConfigurationLimit(setup.model)`은 **MJCF `jnt_range`만 본다**(`kinematics.py:170`). URDF도 LeRobot 리밋도 보지 않는다.
- **실측**: 양팔 IK **0.345 ms / p95 0.355 ms, 500/500 성공** (60 Hz 예산의 2%). **IK는 병목이 아니다.**

→ **함의: MJCF의 결함은 "시뮬레이션 품질 문제"가 아니라 "실기 명령 생성 경로의 결함"이다.** §2.6(J7 클래스 오기)과 §2.18(리밋 불일치)을 이 관점에서 다시 읽어야 한다.

> ⚠️ **주의 — §1.1은 시뮬레이터 정본이 Isaac으로 바뀐 뒤에도 그대로 유효하다.** MuJoCo가 시뮬레이터 정본에서 내려온 것은 **물리 시뮬 역할**이고, 여기서 말하는 것은 **IK 기구학 엔진 역할**이다(mink 유지, ①′ 결정). 즉 "MuJoCo는 시뮬레이터가 아니라 IK 엔진 기반"이라는 이 절의 논지는 오히려 새 프레이밍과 정확히 일치한다. `openarm_control`/mink/MJCF `jnt_range`(IK 리밋 소스)·§2.18·FR-SIM-080~083은 **전부 존치**한다.

**다루지 않는 것**: 실기 CAN/모터 제어 스택 자체(문서 03), 데이터셋 포맷·수집 파이프라인(문서 07/08), 학습 파이프라인·정책 아키텍처(문서 10), VR 텔레옵 입력 장치 스펙 및 IK 어댑터의 설계(문서 05 — 이 문서는 **IK가 딛고 선 자산**만 소유한다). **Isaac Sim/Lab은 §2.1·§3.10에서 시뮬레이터 정본으로 다룬다.**

## 2. 확정된 사실 기반

### 2.1 시뮬레이터 지원 범위 — 1단계 = MuJoCo (정본·검증) / 2단계 = Isaac (선택)

| 시뮬레이터 | 단계·역할 | 버전 / 자산 | 저장소 | 판정 |
|---|---|---|---|---|
| **MuJoCo (MJCF)** | 🔴 **1단계 정본(검증)** — 순수시뮬·트윈·드라이런·평가의 기본 구현 + 실시간 텔레옵 **IK 기구학 엔진**(§1.1). CPU 실시간, 상류 검증 | `openarm-mujoco==2.0.0` (mujoco≥3.6.0) | `enactic/openarm_mujoco` (master, 활발) | **정본·검증. CPU** |
| **Isaac Sim + Isaac Lab** | **2단계 선택** — GPU 학습(IL/RL)·대규모 병렬 평가·합성데이터·포토리얼 렌더. OpenArm 지원이 실험 단계라 선택 백엔드 | **핀 = Isaac Sim `5.1.0` / Isaac Lab `2.3.x`**(OpenArm 자산·LeRobot↔Isaac 브리지가 검증한 조합) · 현행 최신 = Isaac Sim **`6.0.1`**(Newton) / Isaac Lab 안정 **`2.3.2`** / 차세대 **`3.0-beta`** — **차세대 트랙, 자동 승급 금지** | `enactic/openarm_isaac_lab`(실험·커밋 7·under dev) + Isaac Lab 본체 first-party(`isaaclab_assets/robots/openarm.py`) | **선택(2단계). GPU 기반** |
| ~~dora 시뮬 노드~~ | — | — | ~~`dora-openarm-mujoco`~~ | 폐기 (dora 미사용, §2.10) |
| Gazebo / PyBullet / Genesis | — | — | — | 대상 아님 |

> **두 백엔드는 동일 LeRobot `Robot` ABC(§2.10)를 공유하므로 둘 다 구현해 런타임에 선택한다(FR-SIM-097). 1단계(MuJoCo)로 먼저 성립시키고, Isaac은 성숙하면 2단계 선택 백엔드로 켠다.**

> 🔴 **Isaac은 "죽었으니 배제"가 아니라 "2단계 선택"이다 — Isaac 연동이 실재함을 1차 소스로 확인**(구 "IL과 직교" 서술 폐기). 2단계로 켤 때 이 통합 경로를 쓴다:
> - **LeRobot 0.6.0이 Isaac을 공식 통합한다** — EnvHub `IsaacLab-Arena`(`lerobot-eval --env.type=isaaclab_arena`로 GPU 병렬 정책평가), Isaac Lab **Mimic**(소수 시연→IL 데이터 증강, Quest VR/Manus 텔레옵), **LeIsaac**(`isaaclab2lerobot`: HDF5→LeRobotDataset). 시뮬 합성/증강 데이터가 LeRobot 학습 파이프라인으로 직접 흐른다. `lerobot-eval`이 gymnasium 기반인 것은 맞으나 결론이 정정된다 — IsaacLab-Arena가 EnvHub gym 환경으로 꽂히므로 Isaac 평가 경로가 성립한다(OpenArm 양팔 임베디먼트는 커스텀 저작).
> - **OpenArm이 Isaac Lab 본체에 first-party 업스트림**됐다(reach/lift/cabinet) — enactic 7커밋 저장소보다 강한 공식 지원.
> - **Isaac Sim 6.0의 Newton 물리 = MuJoCo Warp 솔버.** 물리 계보가 MuJoCo와 공유되어, 기존 v2 MJCF 씬(§2.3)을 `convert_mjcf.py`로 USD 변환하거나 6.0 Newton에서 직접 로드할 수 있다 → **MJCF 자산 투자가 Isaac 경로에서 재사용된다.** (정본 핀 5.1.0은 PhysX 기반이므로 이 경로는 `convert_mjcf.py`→USD, Newton 직접 로드는 6.0 트랙.)

> ⚠️ **현재 `openarm_isaac_lab` 성숙도 (1차 확인 — [신규구현] 범위를 정한다).** 저장소는 **RL 태스크 4종**(reach / lift cube / open drawer / **양팔 = reach 1종만**), rsl_rl/rl_games/skrl, Isaac Sim 5.1.0 / Isaac Lab 2.3.0 핀, **커밋 7개**. README 원문: *"Imitation learning, teleoperation interfaces, and sim-to-sim / sim-to-real transfer pipelines are currently under development and will be available soon."* **카메라/센서 정의 없음.** → 이 플랫폼이 필요한 **디지털 트윈·드라이런·리플레이·평가 하네스·5카메라 렌더·VR 텔레옵**은 이 저장소에 **아직 없으며 전부 [신규구현]**이다. Isaac은 정본이되, 트윈/평가/카메라 계층은 우리가 Isaac(USD) 위에 만든다. **자산 단일 SoT는 v2 MJCF/URDF, USD는 파생물**(손수 편집 금지, FR-SIM-062 게이트).

**⚠️ USD 자산의 v1 여부는 재확인이 필요하다 [미확인].** 이전 판이 `pxr.Sdf`로 파싱해 얻은 결과(`openarm_left_finger_joint1` = **PhysicsPrismaticJoint, 0 → 0.0440 m** = v1 슬라이드 그리퍼; v2.0 pinch gripper는 **revolute, −0.7854 → 0 rad**; 링크명 `openarm_hand`/`openarm_link0`(v1) vs `ee_base_link`/`finger_inner`/`finger_outer`(v2); `openarm_bimanual_sensor.usd` = **647 B 빈 껍데기**; `physicsScene:enableGPUDynamics = False`)는 **당시 스냅샷 기준**이다. 저장소가 살아 있는 것으로 확인된 이상 **현행 커밋에서 재파싱해야 한다.** 검수 게이트(FR-SIM-062)는 이 재확인과 무관하게 유효하다 — 게이트는 자산이 v2인지를 **실행 시점에** 검사하기 때문이다.

- 아카이브된 저장소 `openarm_isaaclab_experiment`는 **사용 금지**. `openarm_simulation`은 archive이나 **후속 저장소를 가리키는 이정표**이므로 참조는 가능하다(자산 사용은 금지).

### 2.4 qpos/ctrl 레이아웃 규약 (드라이버 인터페이스)

`src/openarm_mujoco/v2/joint_resolver.py` 독스트링 원문:

```
Driver convention (8 values per arm):
  values[0:7] = joints 1–7
  values[7]   = gripper (finger_joint1; finger_joint2 is equality-constrained)

16-element combined driver: right[0:8] + left[8:16].
```

- 인덱스는 **런타임에 관절 이름으로 해석**한다 (`mujoco.mj_name2id` → `model.jnt_qposadr` / `jnt_dofadr`). 원문: *"Indices are discovered at runtime by resolving joint names, so the code is robust to changes in MJCF qpos ordering."*
- → **GUI/드라이버는 qpos 인덱스를 하드코딩하면 안 되고 반드시 `JointResolver`를 경유해야 한다.**
- 손가락 미러: `openarm_{side}_ee_finger_joint_mimic` equality (`polycoef="0 1 0 0 0"`, `solimp="0.95 0.99 0.001"`, `solref="0.005 1"`) → `finger_joint2 = finger_joint1`.

### 2.6 v2 MJCF **파일 내부 불일치** — J7 모터 클래스 [신규구현] (시뮬레이션 자산 수정)

**실기 J7은 `DM4310`으로 확정되어 있다** (03 §2.1 — 공식 BoM·`openarm_driver`·`openarm_teleop`·LeRobot 4중 일치). 불확실한 것은 모터가 아니라 **MJCF 파일**이다: 같은 파일 안에서 J7의 모터 정체가 두 갈래다.

| 위치 | J7 선언 |
|---|---|
| `v2/openarm_bimanual.xml` 관절부 | `class="motor_DM3507"` (armature 0.0049 / damping 0.01 / frictionloss 0.01) |
| `v2/openarm_bimanual.xml` 액추에이터부 | `class="position_DM4310"` (kp 30 / kv 1.5) |
| 공식 BoM (`OpenArm 2.0 BOM.xlsx`) | **J5, J6, J7, J8 = DM-J4310-2EC V1.1 ×8** |
| 공식 문서 "What's New in 2.0" | *"Same motor lineup (DM-J4310-2EC, DM4340, DM-J8009P) as 1.0"* |
| 공식 펌웨어 문서 | `DM-J3507` 펌웨어(`APP_DM3507(V3)_V5717_04.bin`)가 목록에 **존재**하기는 한다 |

`openarm_can`의 `MOTOR_LIMIT_PARAMS` (pMax, vMax, **tMax**):

```
DM3507 : {12.5, 50,  5}     ← tMax 5 Nm
DM4310 : {12.5, 30, 10}     ← tMax 10 Nm
DM4340 : {12.5,  8, 28}
DM8009 : {12.5, 45, 54}
```

**판정**: 4개 1차 소스(BoM / driver / teleop / LeRobot)가 **DM4310으로 일치**하고, MJCF 한 파일만 관절 class에서 어긋난다. 게다가 그 어긋남은 **같은 파일의 액추에이터(`position_DM4310`)와도 모순**된다. 따라서 이것은 **MJCF 자산의 오기(誤記)**다.

**영향 — 시뮬레이션에 국한된다**:
- J7의 armature / damping / frictionloss가 DM3507 값(**0.0049 / 0.01 / 0.01**)으로 잡혀 있으나, 실기는 DM4310(**0.0100 / 0.9 / 0.04**)이다 → 현재 MJCF에서 **J7 동역학이 통째로 틀리고, sim↔real 패리티가 J7에서만 깨진다.**
- **수정 대상**: `v2/openarm_bimanual.xml` 라인 225·313의 joint7 `class="motor_DM3507"` → **`class="motor_DM4310"`**. 좌·우 양팔 모두. 수정 후 J7의 `<default>` 값은 J5·J6과 동일해진다.
- → **이것은 시뮬레이션 자산 버그이지 실기 안전 이슈가 아니다.** 실기 토크 지령을 문서로 금지할 근거가 되지 않는다. 실기 측 오등록 방어는 **부팅 시 RID 21/22/23(PMAX/VMAX/TMAX) 대조 런타임 가드**가 담당한다(03 `FR-MOT-003`/`FR-MOT-004`, 12 `FR-SAF-004`).
- → 다만 **"MuJoCo가 모터 라인업을 교차확인해준다"고 써서는 안 된다.** 이 파일은 모터 타입의 교차확인 소스가 아니다(FR-SIM-007, §5 Q1).

### 2.10 시뮬 싱크 — **LeRobot `Robot` 플러그인이다. dora 노드가 아니다** 🔴 [신규구현]

**폐기된 v1 구조 (다시 쓰지 마라):** `dora-openarm-mujoco` 노드 · `dataflow-vr.yaml` ↔ `dataflow-vr-mujoco.yaml` 교체 · `mujoco-collect` / `mujoco-viewer` 노드 · `arm_{side}_observation` 출력 ID · dora 어댑터 계층.
**전부 dora 배선이며 우리 경로에 없다.** 상류의 dora dataflow는 **같은 물리 배치를 확인해 주는 참고 자료**로만 가치가 있다.

**우리 구조 — 교체의 축이 dataflow YAML에서 `Robot` 객체로 내려온다:**

```
GUI 백엔드 (단일 프로세스, LeRobot Python API 임베드)
  │
  ├─ Teleoperator            ← VR / KER (우리 구현). get_action() → 관절각 deg
  │     └─ openarm_control.Kinematics   ← v2 MJCF 기반 IK (§1.1)
  │
  └─ Robot  ─── 여기가 교체점 ────────────────────────────┐
        ├─ BiOpenArmFollower       (실기)  ← can0/can1 소유 │  같은 ABC
        └─ BiOpenArmMujoco (우리)  (시뮬)  ← CAN 미점유     │  같은 키셋
                                                           ┘
```

**시뮬 싱크가 구현해야 하는 계약 = LeRobot `Robot` 인터페이스 그 자체다.** 별도 어댑터 계층이 필요 없다 — **동일 ABC를 구현하면 드롭인이다.**

| 계약 항목 | 값 (실기·시뮬이 **동일**해야 한다) |
|---|---|
| `action_features` | `{left_joint_1.pos … left_gripper.pos, right_joint_1.pos … right_gripper.pos}` — **deg**. `use_velocity_and_torque=true`면 `.vel`(deg/s)·`.torque`(Nm)까지 → 양완 **48키** |
| `observation_features` | 위 + 카메라 5키 `{left_wrist, right_wrist, head_left, head_right, ceiling}` (`06` §2.11) |
| `send_action(action: dict)` | 시뮬은 MJCF position actuator `ctrl`에 써넣는다. **deg → rad 변환은 시뮬 싱크 내부 책임** |
| `get_observation() -> dict` | 시뮬은 `mjData.qpos` / `qvel` / `qfrc_actuator` + 오프스크린 렌더 5카메라 |
| `connect()` / `disconnect()` / `is_connected` | 시뮬은 **CAN을 열지 않는다.** `set_zero_position()` 등가물도 **있어서는 안 된다**(§2.11) |
| 단위 | LeRobot = **deg**, MJCF/MuJoCo = **rad** |

**등록**: 패키지명을 **`lerobot_robot_*`** 접두로 하고 `@RobotConfig.register_subclass("bi_openarm_mujoco")`를 달면 `register_third_party_plugins()`(`utils/import_utils.py:214-228`)가 자동 로드한다 → `--robot.type=bi_openarm_mujoco`가 `lerobot-record` / `-teleoperate` / `-rollout`에서 **LeRobot 포크 없이** 그대로 동작한다.

**이 구조가 v1보다 나은 점 (공짜로 얻는 것):**
- 상류(VR → IK)만이 아니라 **레코더 · 정책 · 데이터셋 라이터까지 전부 공유**된다. 시뮬 에피소드가 **실기와 문자 그대로 같은 스키마**로 기록된다(FR-SIM-020) → 별도 `metadata_mujoco.yaml`이 필요 없다.
- `dora-openarm-dataset-recorder`(OpenArm 포맷의 유일한 라이터)를 **재구현할 필요가 사라진다.**

### 2.11 CAN 소유권 — **문제는 축소되지만 사라지지 않는다** [확정]

> 🔴 **정정.** 이전 판은 `dataflow-vr.yaml`(dora가 CAN 소유) ↔ `dataflow_bridge_ros2_vr.yaml`(ROS 2 `OpenArmHW`가 CAN 소유)의 **상호배타**를 시뮬/실기 전환의 핵심 제약으로 삼았다. **두 그래프 모두 우리 경로에 없다.** ROS 2 `ros2_control` / `OpenArmHW` / MoveIt도 **경로에 없다.**

**✅ 축소되는 부분** — LeRobot Python API를 **단일 GUI 프로세스에 임베드**하면 모드 간 배타성이 프로세스 내부에서 해결된다. 양팔도 한 프로세스에 버스 객체 2개다(`bi_openarm_follower.py:93-94`, `utils/bimanual.py:47-63` 순차 동기 호출). **시뮬 ↔ 실기 전환은 `Robot` 객체 교체일 뿐 프로세스 재시작이 아니다.**

**❌ 사라지지 않는 부분 — 세 가지:**

1. **🔴 `connect()`가 영점을 파괴한다.** 이것이 시뮬/실기 전환 설계를 지배하는 **단 하나의 사실**이다.
   ```python
   # robots/openarm_follower/openarm_follower.py:152-153
   if self.is_calibrated:
       self.bus.set_zero_position()      # → CAN_CMD_SET_ZERO (damiao.py:335-340)
   ```
   **`connect()`는 현재 물리 자세를 영점으로 확정한다.** 팔이 rest 자세가 아닌 순간 재연결하면 **그 자세가 영점이 되고**, 조인트 리밋 · MJCF 정합 · 데이터셋 좌표계가 **통째로 어긋난다.**
   → **세션당 `connect()`는 1회.** 시뮬↔실기 전환에 실기 `disconnect()` / `connect()`를 **절대 하지 않는다.** 실기 `Robot`은 세션 내내 연결을 유지하고, 시뮬 모드에서는 **`send_action()`을 부를 권리만** 시뮬 싱크로 옮긴다.
   → **v1의 "`command: stop`으로 CAN을 해제한 뒤 시뮬로 전환" 설계는 안전 사고다. 폐기한다**(구 FR-SIM-040).

2. **커널이 배타성을 강제하지 않는다.** SocketCAN RAW = `socket(PF_CAN, SOCK_RAW, CAN_RAW)` + `bind()`, **배타 플래그 없음.** 두 프로세스가 `can0`을 bind하면 **둘 다 성공**하고 로컬 루프백이 기본 ON이라 서로의 송신 프레임까지 받는다. **에러가 나지 않는다 → 조용한 실패.**
   → **`flock` 애플리케이션 락은 여전히 필요하다.** 다만 역할이 **"모드 중재"에서 "외부 침입자 방어"로 축소**된다 — 막아야 할 대상은 `cansend`, `openarm-can-cli`, 두 번째 LeRobot 인스턴스, `openarm_teleop`(C++)이다.

3. **배타성의 단위가 내려간다.** "누가 버스를 소유하는가"에서 **"누가 `send_action()`을 부를 권리를 갖는가"**로. **상태머신은 그대로 필요하다**(§4.2).

**시뮬 싱크는 CAN을 전혀 잡지 않는다** → 순수 시뮬 · 드라이런 모드에서 실기 `Robot`은 CAN을 조용히 붙들고만 있다(명령 송신 없음). 트윈 모드에서는 실기가 관측을 내고 시뮬이 그것을 미러링한다.

**🔴 `ip link` 검증은 코드가 대신할 수 없다** — python-can socketcan 백엔드는 **`bitrate` / `data_bitrate` 인자를 무시한다.** CAN-FD를 켜는 것은 `ip link` 단계이며, GUI가 기동 시 **`ip -details link show`로 검증**해야 한다(FR-SIM-039).

### 2.18 🔴 MJCF 조인트 리밋 ≠ LeRobot 조인트 리밋 — **IK가 클램프될 해를 낸다** [확정]

**§1.1에서 확인한 대로 MJCF는 IK의 리밋 소스다.** 그리고 그 리밋이 **LeRobot의 소프트 리밋과 다르다.**

`mink.ConfigurationLimit(setup.model)`(`kinematics.py:170`)은 **MJCF `jnt_range`만** 본다.

| 조인트 | MJCF right (rad → deg) | LeRobot right (deg) | 판정 |
|---|---|---|---|
| joint1 | −1.3963 … 3.4907 → **−80° … +200°** | **−75 … +75** | 🔴 **MJCF가 압도적으로 넓다** |
| joint2 | −0.17453 … 3.3161 → **−10° … +190°** | −9 … +90 | 🔴 MJCF가 넓다 |
| joint3 | ±1.5708 → ±90° | ±85 | 약간 넓다 |
| joint4 | 0 … 2.4435 → 0° … +140° | 0 … +135 | 약간 넓다 |
| joint5 | ±1.5708 → ±90° | ±85 | 약간 넓다 |
| joint6 | ±0.7854 → **±45°** | ±40 | 약간 넓다. **절대값 자체가 좁다** — VR 자세 1:1 매핑 시 상시 리밋 충돌 |
| joint7 | ±1.5708 → ±90° | ±80 | 약간 넓다 |
| gripper | right finger1 −0.7854…0 / left finger1 0…0.7854 | −65 … 0 | 🔴 **규약·스케일 자체가 다르다** |

**귀결 3가지 (전부 [신규구현] 요구사항으로 등재):**

1. **IK가 LeRobot이 클램프할 해를 정상적으로 낸다.** → `ArmSetup` 생성 **직후**, `Kinematics(...)` 생성 **전에** `setup.model.jnt_range[jid]`를 **LeRobot 리밋으로 덮어써야 한다.** `ConfigurationLimit`이 **생성 시점에 model을 읽으므로 순서가 결정적이다**(FR-SIM-080).
2. **폴백이 리밋을 통째로 버린다.** `kinematics.py:220-231`: `NoSolutionFound` → **`limits=[]`로 재시도**. 조인트 리밋을 포기한 해가 나올 수 있다. → **어댑터 레벨 클램프가 1차 방어선**이어야 한다(`12` FR-SAF-016).
3. **단위**: `openarm_control` = **rad**, LeRobot = **deg**. 변환 누락 시 **조용한 오동작**(57.3배 오차).

**상류의 검증된 IK 튜닝값** — `dora-openarm-vr/config/dataflow-mujoco.yaml`이 `openarm_control`을 bimanual로 실제 사용하며 넘기는 인자다. **그대로 베낀다:**
```
--mode bimanual --max-iters 10 --dt 0.1 --damping 0.1 --posture-cost 0.01 --lm-damping 0.01
```
⚠️ `--tick-hz` 기본값 **500.0**의 help 문구가 *"Dora tick rate in Hz; must match the dataflow timer"*다 — **dora 잔재**다. 속도 캡 변환식이 `rad_per_sec / (max_iters × dt × tick_hz)`(`kinematics.py:261-269`)이므로 **`tick_hz`가 틀리면 속도 캡이 통째로 틀린다.** **우리 실제 루프 주기로 반드시 바꿔야 한다.**

⚠️ **`Kinematics.solve()`는 인자를 받지 않는다** — `def solve(self) -> np.ndarray | None` (`kinematics.py:107`). README의 `solve(dt=, n_iters=)`는 **틀렸다.** `dt`/`max_iters`는 `IKParams`에만 있다. **문서가 아니라 소스를 따른다.**

### 2.17 참고 — 하드웨어 요구사항 (1단계 MuJoCo = CPU / 2단계 Isaac = GPU)

### 3.10 Isaac Sim/Lab 시뮬 경로 [2단계 선택]

> 🔴 **2단계 구현.** Isaac은 검증된 1단계(MuJoCo)가 성립한 뒤 켜는 **선택 백엔드**다(OpenArm Isaac 지원이 실험 단계, §2.1). 그 안에서도 **자산 파이프라인·평가·데이터 브리지는 M/S**(2단계를 켤 때 먼저 필요), **RL 학습·GR00T/Cosmos는 "나중 연동"으로 C**(선후관계: 평가·데이터 브리지 → RL/Cosmos).
> 🔴 **Isaac RL 정책 산출물 계약(함정).** Isaac Lab RL(rsl_rl/rl_games/skrl)은 `.pt`/JIT/`.onnx`로 **LeRobot 정책이 아니다** → **ONNX 익스포트 + PreTrainedPolicy 래핑** 또는 **gRPC 정책서버(:8080) 서빙**을 경유해야 하며, Isaac 네이티브 체크포인트를 LeRobot 정책으로 **직접 로드하는 것은 차단**한다. (반면 LeRobot 네이티브 RL = HIL-SERL(gym-hil MuJoCo 기판)은 산출 정책이 LeRobot 정책이라 추론에 드롭인 — §신규 4절.)
> **환경 API = manager-based 필수**(DR EventManager·ADR·Mimic 전제). **롤아웃마다 `{Isaac Sim 버전, Isaac Lab 버전, 물리 백엔드}`를 메타로 기록**하고 **백엔드(MuJoCo/PhysX/Newton) 간 동정값·DR 중심값 교차 이식은 금지**(경고 후 차단).

### 4.2 액추에이션 싱크 모드 전이 — **`send_action()` 호출 권리의 이동** (CAN 재연결 없음)

> 🔴 **전제 정정.** 이 표의 축은 더 이상 "CAN 소유권"이 아니다. **실기 `Robot`은 세션 내내 `connect()` 상태를 유지**하며(FR-SIM-040), 전환은 **`send_action()`을 부를 권리가 어느 싱크에 있는가**로만 표현된다. `disconnect()`/`connect()`는 **어떤 전이에서도 일어나지 않는다.**

**세션 시작 시 1회**: `flock` 획득 → `ip -details link show`로 CAN-FD 검증(FR-SIM-039) → 실기 `Robot.connect()` **1회** (→ `set_zero_position()`이 여기서 **단 한 번** 발동, 반드시 rest 자세에서).

| 현재 | 이벤트 | 다음 | `send_action()` 권리 | 게이트 |
|---|---|---|---|---|
| `SIM_ONLY` | 트윈 진입 | `TWIN` | **없음** (관찰 전용) 또는 실기(REAL+TWIN) | 실기 `is_connected` + 모터 스케일 대조 통과(RID 21/22/23 ↔ `MOTOR_LIMIT_PARAMS`, 03 FR-MOT-003/004) + **`use_velocity_and_torque = true`**(FR-SIM-025b) |
| `SIM_ONLY` | 드라이런 실행 | `DRYRUN` | **시뮬** | 클램프 정본 선택(FR-SIM-031/032) |
| `DRYRUN` | 전 항목 통과 | `SIM_ONLY` (게이트 해제 플래그 set) | 시뮬 | — |
| `DRYRUN` | 항목 실패 | `SIM_ONLY` (게이트 유지) | 시뮬 | 위반 리포트 표시, **`REAL` 전이 차단** |
| `SIM_ONLY` / `TWIN` | 실기 실행 | `REAL` | **실기** | **드라이런 통과 필수**(FR-SIM-072) |
| `REAL` / `TWIN` | 시뮬 복귀 | `SIM_ONLY` | **실기 → 시뮬로 이동** | 🔴 **`disconnect()` 금지.** 실기 `Robot`은 연결을 유지한 채 명령만 멈춘다. 멈추는 방식은 **명령 스트림 중단이 아니라 `STOP_HOLD` 유지**여야 한다(`12` §2.3 — 스트림을 끊으면 인에이블 이탈 → **낙하**) |
| any | 세션 종료 | — | 없음 | rest 자세 확인 → `disconnect()` → `flock` 해제. **세션당 `disconnect()`도 1회** |
