# 08. 데이터셋 관리·시각화·품질

## 1. 범위

이 문서는 **이미 기록된 데이터셋**을 대상으로 하는 브라우징·에피소드 뷰어·편집·통계·버전/계보·HF Hub·외부 포맷 상호운용을 규정한다.

**정본이자 유일한 포맷은 LeRobot v3.0이다.** `07-데이터-수집.md`가 LeRobot으로 직접 기록하므로 **파생 포맷도, 변환 단계도 존재하지 않는다**(§2.1). 이 문서에서 "변환"은 **외부 데이터셋을 들여오는 import**와 **외부 소비자를 위한 export**만을 뜻한다.

다루지 않는 것: 수집 루프 자체(녹화 상태머신·카메라 파이프라인·텔레옵 → `07`), 학습 실행과 하이퍼파라미터(→ `09`), 실기 평가 세션 오케스트레이션(→ `10`), 정책 서빙. 데이터셋을 **쓰는(write)** 주체는 `07`이며, 여기서는 쓰인 결과물의 **읽기·검증·편집·파생·공유**만 규정한다.

요구사항 ID 영역 코드: **`DAT`**.

## 2. 확정된 사실 기반

인용은 `huggingface/lerobot` **v0.6.0**(공개 최신, 2026-07-06) 로컬 스냅샷과 `enactic/openarm_dataset` 소스를 **직접 열람**한 결과다. [개정 2026-07-15] `pyproject.toml:28`이 자칭하는 `0.6.1`은 **PyPI·git 태그가 없는 유령 버전**이므로 정본은 **v0.6.0**으로 고정하고, 재현 시 **commit SHA로 핀**하는 것을 권장한다(→ `07 §2`).

### 2.1 포맷 계층이 없다 — LeRobot v3.0이 유일한 정본

```python
# src/lerobot/datasets/dataset_metadata.py:58
CODEBASE_VERSION = "v3.0"
```

`LeRobotDataset.create()`에 버전 인자가 없다. **v2.1로 기록하는 옵션은 존재하지 않는다.**

> **폐기**: v1 명세의 `openarm_dataset v0.3.0 ⊋ LeRobot`(1차 = OpenArm, 파생 = LeRobot) 계층은 **성립하지 않는다.** 그 전제는 "OpenArm 포맷으로 기록한 뒤 LeRobot으로 변환한다"에 서 있었는데, **OpenArm 포맷으로 기록할 수단이 우리 런타임에 없다**(→ `07 §2.1`).
> 따라서 v1의 다음 요구사항은 **전부 삭제되었다**: 변환 손실 매트릭스·손실 리포트, 무손실 확장 exporter(`capture_ts`/`observation.velocity`/`observation.effort`), 역변환 금지 조항, 교집합 창 커버리지 검사, 벤더 스무딩(`fps=250.0` 하드코딩) 우회, 에피소드 id 매핑표. **벤더 리샘플러·스무더·교집합 창은 우리 경로를 통과하지 않는다.**

→ **우리는 LeRobot으로 기록하므로 이 컨버터에 먹일 입력이 없다.** 이 도구의 유일한 용도는 **레거시 OpenArm 데이터셋을 LeRobot으로 들여오는 import**다(FR-DAT-040).

### 2.3 🔴 GR00T는 LeRobot 네이티브 정책이다 — 변환이 필요 없다

LeRobot v0.6.0의 등록된 정책 타입 20종에 **`groot`가 포함된다**(`act`, `diffusion`, `pi0`, `pi05`, `pi0_fast`, `smolvla`, `groot`, `tdmpc`, `vqbet`, `eo1`, `evo1`, `xvla`, `wall_x`, …). 즉 **`lerobot-train --policy.type=groot`가 우리 LeRobot v3.0 데이터셋 위에서 그대로 돈다.**

→ **`--format gr00t`(= LeRobot v2.1 + `meta/modality.json`) 변환 경로는 우리에게 불필요하다.** v1 명세의 "GR00T 때문에 v2.1이 필요하다"는 근거는 **소멸했다.** (`policies/groot/utils.py`가 `modality stats`를 다루지만 이는 **체크포인트 측 embodiment 통계**이지 데이터셋 포맷 요구가 아니다.)

> **귀결**: v2.1을 산출할 이유가 **하나도 남지 않았다.** v2.1은 `lerobot ≥0.5`에서 로드조차 되지 않으므로(`BackwardCompatibilityError`), 시스템은 v2.1 산출 자체를 차단한다(FR-DAT-042).

**에피소드-per-파일이 아니라 chunk/file 패킹**이다. → **에피소드 하나를 삭제하려면 chunk를 다시 써야 한다.** 이것이 편집 연산이 비싼 이유이며, 우리가 `lerobot-edit-dataset`을 재구현하지 않는 이유다(FR-DAT-019).

**feature 스키마 전수** (→ `07 §2.2`, 동일 표를 여기서 반복하지 않는다):
`action`(float32, `(D,)`), `observation.state`(float32, `(D,)`), `observation.images.<cam>`(video, `(H,W,3)`), **뎁스 사용 시 `observation.images.<cam>_depth`(video, `(H,W,1)`, `is_depth_map=true`, RealSense `use_depth`)**, `timestamp`(float32), `frame_index`/`episode_index`/`index`/`task_index`(int64).

**`D`는 구성에 따라 8 / 16 / 24 / 48이다** (`use_velocity_and_torque` × 단완/양완). **토크는 `observation.state`의 부분 인덱스이며 `observation.effort`는 존재하지 않는다**(전체 트리 grep 0건). 매핑 규칙의 정본은 **`07 §2.3.2`**이며, 소비자는 인덱스를 하드코딩하지 말고 `meta/info.json`의 `names`에서 조회해야 한다(FR-DAT-005).

`meta/episodes/*.parquet` 컬럼: `episode_index`, `length`, `tasks`, `dataset_from_index`/`dataset_to_index`(전역 프레임 경계), `data/chunk_index`·`data/file_index`, `videos/{video_key}/chunk_index`·`file_index`.

### 2.5 시간 의미론 — `timestamp`는 관측 시각이 아니다

```python
# src/lerobot/datasets/dataset_writer.py:207-208
frame_index = self.episode_buffer["size"]
timestamp   = frame_index / self._meta.fps
```

**`timestamp`는 `frame_index / fps`의 균등 격자**다. 캡처 지터·카메라 도착 오프셋·CAN 지연은 **이 필드에 남지 않는다.**

이것은 변환 손실이 아니라 **LeRobot 포맷의 설계 자체**다. 우리가 LeRobot으로 직접 기록하므로 **잃을 원본이 애초에 없다** — 대신 **처음부터 기록하지 않으면 영원히 없다.**

| | LeRobot v3.0 `timestamp` |
|---|---|
| 정의 | **격자 좌표** — `frame_index / fps` |
| 담기는 것 | 프레임 순서 |
| 담기지 않는 것 | 벽시계 시각, dt 지터, 스트림 간 도착 오프셋 |
| 뷰어 시간축으로서 | **정확하다** (프레임 = 격자점이 정의상 참) |
| 지연·드롭 분석용으로서 | **무용하다** |

→ **뷰어의 1차 시간축은 `timestamp`(= 격자)로 충분하고 정직하다.** 그러나 품질·지연 분석은 `07 FR-REC-045`의 **벽시계 사이드카**를 요구하며, 그 스키마는 **[결정필요]**(`07 §5 Q1`, 본 문서 §5-1).

> ⚠️ **`float32` 함정**: `DEFAULT_FEATURES`의 `timestamp`는 **float32**다(`constants.py:80-87`). 유닉스 시각(≈1.772×10⁹ s)은 **float32로 표현 불가능하다** — 이 구간의 ULP가 **128 s**다. 즉 **이 필드에 벽시계를 담을 의도가 애초에 없다.** 벽시계는 반드시 별도 필드/사이드카여야 한다.

`tolerance_s = 1e-4` — LeRobot은 로드 시 `timestamp` 간격이 `1/fps ± tolerance` 안에 있는지 검사한다(`lerobot_dataset.py:55,170-174`). 네이티브 기록물은 정의상 통과한다.

### 2.6 통계(stats) — 계산 규칙과 소비처

```python
# src/lerobot/datasets/compute_stats.py:27,40,53,71-73,120-129
DEFAULT_QUANTILES = [0.01, 0.10, 0.50, 0.90, 0.99]
def __init__(self, quantile_list=None, num_quantile_bins: int = 5000): ...
self._quantile_keys = [f"q{int(q * 100):02d}" for q in self._quantile_list]   # q01,q10,q50,q90,q99
self._histograms = [np.zeros(self._num_quantile_bins) for _ in range(vector_length)]
stats = {"min": …, "max": …, "mean": …, "std": stddev, "count": np.array([self._count])}
```

- 알고리즘: **running(스트리밍) 통계.** `mean`·`mean_of_squares`를 증분 갱신 → `std = sqrt(max(0, mean_of_squares - mean²))`.
- 분위수는 **정확값이 아니라 히스토그램 근사**다(`num_quantile_bins = 5000`).
- 이미지 feature는 `is_depth_map`이 아니면 **255.0으로 나눠 [0,1] 스케일**로 통계 산출(`compute_stats.py:537`).
- 저장 위치: `meta/stats.json`.

**정규화 소비처 — 어떤 통계가 없으면 어떤 정책이 못 도는가** (전부 소스 확인):

| 정책 | STATE / ACTION 정규화 | 필요한 통계 키 | 출처 |
|---|---|---|---|
| ACT | `MEAN_STD` | `mean`, `std` | `configuration_act.py:90-92` |
| Diffusion | **`MIN_MAX`** | `min`, `max` | `configuration_diffusion.py:109-110` |
| pi0 | `MEAN_STD` | `mean`, `std` | `configuration_pi0.py:74-75` |
| SmolVLA | `MEAN_STD` | `mean`, `std` | `configuration_smolvla.py:35-36` |
| **pi0.5** | **`QUANTILES`** | **`q01`, `q99`** | `configuration_pi05.py:75-77` (*"Pi0.5 uses quantiles for state/action"*) |
| **GR00T** | **`IDENTITY`** | — (체크포인트의 embodiment modality stats 사용) | `configuration_groot.py:263-268` |

`NormalizationMode` enum 전체: `MIN_MAX`, `MEAN_STD`, `IDENTITY`, `QUANTILES`, `QUANTILE10` (`configs/types.py:34-39`).

→ **`q01`/`q99`는 선택 항목이 아니다.** LeRobot에 `scripts/augment_dataset_quantile_stats.py`(분위수 통계 보강)가 별도로 존재한다는 사실 자체가, 분위수 없는 데이터셋이 실전에서 만들어진다는 증거다. 우리는 처음부터 전 분위수를 채운다(FR-DAT-026).

#### 2.6.1 🔴 토크 데이터셋 고유의 통계 위험 — `std ≈ 0` 채널

`use_velocity_and_torque=True`이면 `observation.state`에 `.vel`·`.torque` 채널이 들어온다(→ `07 §2.3`). 그런데:

- **정지가 지배적인 관절의 `.vel` 채널** → 값이 거의 항상 0 → **`std ≈ 0`**.
- **비접촉 구간이 지배적인 `.torque` 채널** → 중력 보상분 근처에 상수처럼 머문다 → **`std` 매우 작음**.

`MEAN_STD` 정규화(`ACT`, `pi0`, `SmolVLA`)는 `(x - mean) / std`를 계산하므로 이 채널들에서 **수치가 폭발한다.** `MIN_MAX`(Diffusion)도 `max - min ≈ 0`이면 같은 문제를 갖는다.

**LeRobot의 `Normalize` 모듈이 `std`에 epsilon floor를 적용하는지는 [미확인]이다** — 소스로 확인하지 못했다(§5-4). 적용하지 않는다면 우리가 stats 산출 단계에서 방어해야 한다(FR-DAT-027).

또한 **한 `observation.state` 벡터에 deg(위치)·deg/s(속도)·Nm(토크)이 혼재한다**(→ `07 §2.4`). 세 정규화 모드 모두 **채널별**이므로 스케일 혼재 자체는 수학적으로 흡수되지만, **통계 리포트를 사람이 읽을 때는 단위를 표시해야 한다**(FR-DAT-016).

### 2.9 버전·계보의 축

- **HF Hub**: `dataset.push_to_hub(tags=…, private=…)` 내부에서 `hub_api.create_tag(repo_id, tag=CODEBASE_VERSION, revision=branch, repo_type="dataset")` → 데이터셋 레포에 **`v3.0` 태그를 부착**하고, 기존 태그가 있으면 **`delete_tag` 후 재생성**한다(`lerobot_dataset.py:622-623`). 로드 시 `self.revision = revision if revision else CODEBASE_VERSION`.
  → **git revision(브랜치/태그/커밋 SHA)이 데이터셋 버전 관리의 축이다.**
- **`repo_id` 자동 스탬핑**: 기록 시 `_%Y%m%d_%H%M%S`가 자동 부착된다(`configs/dataset.py` `stamp_repo_id()`, → `07 §2.8`). **데이터셋 식별자는 사용자가 입력한 이름이 아니다.**
- **학습 실행 → 데이터셋**: `DatasetConfig{repo_id, root, episodes: list[int]|None, revision, eval_split, …}`가 체크포인트의 `train_config.json`에 직렬화된다(`--config_path=<ckpt>/train_config.json --resume=true`로 복원).
  → **체크포인트 → (데이터셋 revision + 사용 에피소드 목록)은 이미 복원 가능하다.**
  → **역방향(에피소드 → 이 에피소드를 쓴 체크포인트 목록)은 없다. 우리가 만든다**(FR-DAT-034).
- **버전 함정**: **LeRobot v2.1 산출물은 lerobot ≥0.5에서 로드 불가**(`BackwardCompatibilityError`: major 2 < 3). 승격 경로는 `scripts/convert_dataset_v21_to_v30.py`(**단방향**, v3.0→v2.1 다운그레이드는 **없다** — 직접 확인). §2.3에 따라 **v2.1을 산출할 이유가 없다.**
- **환경** [개정 2026-07-15] (B1·M9): 정본 lerobot은 **v0.6.0**(0.6.x 계열)이며, 그 `requires-python` 실제값은 **[미확인]**이다 — 07/08은 Python ≥3.12로 가정하나 확정된 소스 대조가 없고, `09`(mujoco 3.10)·`11`(openeval 3.11)과 **단일 환경 공존이 가능한지도 [미확인]**이다(임의 확정 금지).

### 2.10 통계 재계산 트리거 (정본 표)

정규화 통계는 학습 입력이다. 아래 연산 중 하나라도 발생하면 **stats는 즉시 무효(stale)로 마킹되어야 하며, 무효 상태의 데이터셋으로 학습을 시작해서는 안 된다.**

| 연산 | `observation.*` / `action` 통계 | `task_index` 통계 | 비고 |
|---|---|---|---|
| 에피소드 삭제 (`delete_episodes`) | **무효** | 무효 | 모집단 변경 + **에피소드 인덱스 재번호** |
| 에피소드 추가 (record `--resume`) | **무효** | 무효 | |
| 병합 (`merge`) | **무효** | 무효 | task_index 재매핑 동반 |
| 분할 (`split`) | **train split만으로 fit → 동일 stats를 val/test·추론에 적용** | train 기준 | 🔴 [개정 2026-07-15] 방향 정정(H6): 이전 초안의 "split별 별도 산출 + 전역 stats를 val에 상속 = 누수"는 **역전이었다.** 정규화 통계는 **train으로만 fit**하고 val/test·실기 추론에 그대로 적용한다 — val/test를 **자기 분포로 재정규화하는 것**이 검증셋 누수다. split-local stats는 **진단용으로만** 산출한다 |
| feature 제거 (`remove_feature`) | 해당 키만 무효 | 유효 | |
| 태스크 라벨 수정 (`modify_tasks`) | **유효** | **무효** | 수치 분포 불변, 정수 매핑만 변경 |
| 이미지→비디오 (`convert_image_to_video`) | 유효 | 유효 | 이미지 통계는 [0,1] 스케일이라 재인코딩과 무관하지 않을 수 있음 → **[미확인]** |

## 4. 상태·오류 처리

### 4.1 데이터셋 상태 전이

| 상태 | 의미 | 진입 조건 | 이탈 |
|---|---|---|---|
| `SCANNING` | 루트 스캔·버전 판별 중 | 브라우저 오픈 | → `READY` / `INVALID` |
| `READY` | 로드 가능, stats 유효 | 스키마 검증 통과 + stats 해시 일치 | → `EDITING` / `IMPORTING` / `STATS_STALE` |
| `STATS_STALE` | 데이터는 유효하나 통계 무효 | §2.10 트리거 | 재계산 → `READY` |
| `EDITING` | 편집 트랜잭션 오픈 | 편집 시작 | 커밋 → 새 버전 `READY`(+원본 `READY` 유지) / 어보트 → `READY` |
| `IMPORTING` | 레거시 OpenArm → LeRobot 변환 중 | import 시작 | → `VERIFYING` / `FAILED` |
| `VERIFYING` | 산출물 로드·재매핑 검증 중 | 편집/import 완료 | 통과 → `READY` / 실패 → `INVALID` |

### 4.3 편집 트랜잭션 원자성

편집 커밋은 다음을 만족해야 한다.

1. 새 버전에 대한 쓰기가 **완료되고 `finalize()`가 성공한 뒤에만** 계보 레코드가 커밋된다.
2. **사이드카 재매핑(FR-DAT-021)이 전수 대조를 통과한 뒤에만** 새 버전이 `READY`가 된다. 실패하면 `INVALID`이며 원본은 그대로 유지된다.
3. 커밋 실패 시 새 버전 디렉터리를 제거하고 원본 상태를 유지한다.
4. stats는 §2.10의 트리거에 따라 **커밋과 동시에 stale로 마킹**되며, 재계산 완료 전까지 새 버전은 학습 입력으로 노출되지 않는다.

## 5. 미해결 · 결정필요

### 1. [결정필요] 벽시계 타임스탬프를 파생물이 아니라 **원본**에 어떻게 남길 것인가

**v1과 문제가 뒤집혔다.** v1에서는 "원본(OpenArm)에 있는 실측 시각이 변환 중 소실된다"가 문제였다. 이제 **변환이 없으므로 소실될 원본도 없다** — 대신 **LeRobot 포맷이 애초에 벽시계를 담지 않으므로, 처음부터 기록하지 않으면 영원히 없다**(§2.5).

`timestamp` 필드는 float32이고 유닉스 시각은 float32로 **표현 불가능하다**(ULP = 128 s). 따라서 벽시계는 반드시 별도 필드/사이드카여야 하며, dtype 선택이 남는다:
- **float64(초)**: ULP ≈ 238 ns. 타이밍 분석에는 충분하나 ns 정수 정밀도는 아니다.
- **int64(ns)**: 정확. LeRobot의 부동소수 feature 관례에서 벗어난다.
- **별도 사이드카 파일**: LeRobot 스키마 무오염. **HF Hub 업로드 시 따라가지 않을 수 있다** → 파생물 수신자는 타이밍 분석 불가.

**결정하려면 확인할 것**: (a) 커스텀 feature를 실은 데이터셋이 `LeRobotDataset` 로드를 통과하는지, (b) [확인됨] `dataset_to_policy_features`(`feature_utils.py:169-212`)가 `observation`/`action` 접두가 아닌 feature를 `else→continue`로 배제하므로 `lerobot-train`이 이를 학습에서 자동 제외한다(→ v3-7), (c) 파생물 단독으로 타이밍 분석을 해야 하는 소비자가 실제로 있는지. **남은 것은 사이드카 vs 커스텀 feature의 운반성이며, 기록 1회 + 로드 1회로 확인한다.** (→ `07 §5 Q1`과 동일 항목. **두 문서가 같은 결정에 걸려 있다.**)

### 2. [결정필요] 원본 불변(CoW) vs upstream의 파괴적 편집 (FR-DAT-022)

upstream `lerobot-edit-dataset --operation.type modify_tasks`는 **in-place**로 명시되어 있다. CoW를 택하면 upstream 편집기를 **그대로 호출할 수 없고**(사본을 만든 뒤 사본에 호출해야 하며), v3.0 chunk 패킹 때문에 사본 비용이 데이터셋 전체 크기다.

→ **정하려면**: (a) 에피소드당 실측 디스크 크기(비디오 지배, NFR-DAT-007의 실측이 선행), (b) 예상 편집 빈도 × 데이터셋 수로 스토리지 증가분 산출, (c) 편집 후 원본 복구가 필요했던 실제 사례 빈도를 파일럿으로 확인.
**유력안**: 파괴적 편집을 허용하되 **편집 전 스냅샷을 강제**하고, 스냅샷을 N일 후 GC. 완전 CoW보다 저렴하고 완전 in-place보다 안전하다. 그러나 이는 결정이 필요한 사항이다.

### 3. [결정필요] 성공/실패 라벨 저장소 (FR-DAT-020, → `07 §5 Q3`)

LeRobot 네이티브 record는 성공 라벨을 기록하지 않는다. 사이드카로 두면 **`delete_episodes`/`split`/`merge`가 에피소드 인덱스를 재번호할 때 라벨이 어긋날 위험**이 생긴다(FR-DAT-021, NFR-DAT-004).

→ **정하려면**: `lerobot-edit-dataset`이 재번호를 수행할 때 **원본 인덱스 → 새 인덱스 매핑을 산출물로 내놓는지** 확인해야 한다. 내놓지 않는다면 우리가 **에피소드 내용 해시로 재식별**해야 하며(NFR-DAT-004의 측정 절차가 곧 그 구현이다), 그 비용이 CoW 결정(§5-2)에도 영향을 준다.

### 4. [미확인] LeRobot `Normalize`가 `std`에 epsilon floor를 적용하는가 (FR-DAT-027, `OA-DAT-005`)

`use_velocity_and_torque=True`면 `observation.state`에 `std ≈ 0`인 채널(정지 관절의 `.vel`, 비접촉 구간의 `.torque`)이 반드시 생긴다(§2.6.1). `MEAN_STD`(ACT/pi0/SmolVLA)는 `(x-mean)/std`를 계산한다.

→ **확인 방법**: `src/lerobot/policies/normalize.py` 상당 모듈의 division 경로를 읽는다. floor가 없으면 (a) stats 산출 시 std floor 주입, (b) `QUANTILES` 정책(pi0.5)만 사용, (c) 해당 채널을 학습 입력에서 제외 중 하나를 택해야 한다. **이 결정은 토크를 기록하는 우리 플랫폼의 정체성에 직결된다** — 토크 채널이 학습에서 쓸모없어지면 기록의 의미가 절반으로 준다.

### 5. [확정] 관절 리밋 오버레이 — v2 URDF가 기계 상한 정본 (FR-DAT-013)

세 값이 실재하고 서로 다르다. 정본은 **v2 URDF**로 확정한다(GROUND-v3 d3, → v3-6):
- **기계 상한(정본)** = `openarm_description` **`openarm_v2.0/.../joint_limits.yaml`(rad)**: joint2 **−10°/+190°**(v1 ±100°에서 변경 — 유일한 차이), joint1 −80°/+200°, joint3 ±90°, joint4 0°/140°, joint5 ±90°, joint6 ±45°, joint7 ±90°.
- **소프트 클램프(비정본)** = `config_openarm_follower.py`의 deg 값(예: joint_1 −75°/+75°)은 LeRobot **v1-era 보수 기본값**이지 기계 상한이 아니다.
- **`side` 미지정 시 ±5°로 떨어지는 초안전 기본값**(→ `07 §2.6`)은 오퍼레이터 실수 신호일 뿐 정본 리밋이 아니다.

→ 뷰어는 **기계 상한(v2 URDF)을 정본 오버레이로** 그리고, 데이터셋 기록 시 실제 적용된 소프트 클램프 값이 다르면 **둘을 구분해 함께 표시**한다. 데이터셋 메타에 실제 사용된 `joint_limits`를 기록하는 것이 선결 조건이다(→ `07 FR-REC-010`). (상세 → `03-모터-설정.md`)

### 6. [미확인] `lerobot-dataset-viz`가 v3.0을 읽는가 (FR-DAT-009, §2.8)

`lerobot-dataset-viz`는 LeRobot v0.6.0의 `[project.scripts]`에 등재되어 있으나, Rerun의 v3.0 판독 이슈(rerun-io#11678)가 이 버전에서 해소되었는지 **확인하지 못했다.**

→ **확인 방법**: 네이티브 v3.0 데이터셋 1개에 `lerobot-dataset-viz`를 실행한다. 동작하면 FR-DAT-009의 자체 뷰어 요구는 **우선순위를 낮출 수 있다**(우리 고유 요구 — 토크 채널 시각화, 품질 리포트 오버레이, 라벨 편집 — 은 여전히 남는다).

### 7. [미확인] `convert_image_to_video`가 이미지 통계를 무효화하는가 (§2.10)

이미지 통계는 [0,1] 스케일로 산출되므로(`compute_stats.py:537`) 이론상 재인코딩과 무관해야 하지만, 손실 압축이 픽셀 분포를 바꾸므로 `mean`/`std`가 미세하게 이동할 수 있다.

→ **확인 방법**: 동일 에피소드에 대해 이미지 상태와 비디오 변환 후의 이미지 통계를 대조하여 상대 오차를 산출한다. 임계치를 넘으면 §2.10 표에서 "유효"를 "무효"로 바꿔야 한다.

### 8. [미확인] 분위수 정확값 vs 히스토그램 근사의 편차 (FR-DAT-029)

LeRobot은 5000 bin 히스토그램으로 q01/q99를 근사한다. **pi0.5가 이 값을 정규화 스케일로 직접 쓴다.** 그리고 우리 `observation.state`는 **deg와 Nm이 혼재**하므로 채널별 동적 범위가 극단적으로 달라 bin 해상도의 영향이 채널마다 다를 수 있다.

→ **확인 방법**: 우리 데이터셋 1개에 대해 exact 분위수와 LeRobot 근사값의 상대 오차를 **채널 타입별(pos/vel/torque)로 분해**해 산출한다. 임계치를 넘으면 exact 분위수로 `stats.json`을 덮어쓰는 것을 검토한다.
