# 02. 로봇 연결·온보딩·캘리브레이션

## 1. 범위

**다룬다**: 하드웨어 인벤토리(v2.0 스펙·모터 배치·CAN 채널·어댑터), SocketCAN 인터페이스 구성(`ip link` CAN-FD 셋업 + udev 고정), CAN 버스 배타 소유권 획득(`flock`), **세션 연결 계약**(`connect()` 1회, rest 자세 강제), 첫 연결 마법사(버스 스캔 → 모터 발견 → ID/타입 대조 → 펌웨어/에러코드 확인 → rest 자세 확인 → connect → 영점 캘리브레이션 → 헬스체크), 영점 오프셋의 저장 위치·포맷·재캘리브레이션 트리거, 로봇 프로파일(좌/우 팔, CAN ID 맵, 모터 타입 배열, 게인·리밋 프로파일)의 임포트/익스포트, 연결 실패 모드의 감지와 복구.

**다루지 않는다**: 모드 전환 상태머신과 컨트롤 루프 소유권(→ `01-시스템-아키텍처.md`, `SYS`), 모터 CAN 프로토콜 비트패킹·제어모드·RID 레지스터 전체·그리퍼 제어(→ `03-모터-설정.md`, `MOT`), 조그·Freedrive(→ `04-수동-동작.md`, `MAN`), 카메라 검출·역할 할당·캘리브레이션(→ `06-카메라-서브시스템.md`, `CAM`), 진단 토픽·실패 모드 전수표(→ `14-시스템-운영.md`, `OPS`).

이 문서의 요구사항 영역 코드는 **`CON`** 이다.

## 2. 확정된 사실 기반

### 2.0 런타임 전제 — LeRobot Python API 임베드 [확정]

정본 런타임은 **LeRobot v0.6.1**이다. GUI 백엔드가 **LeRobot Python API를 단일 프로세스에 임베드**하고, `OpenArmFollower` / `BiOpenArmFollower` 객체가 CAN 소켓을 **세션당 한 번** 연다.

```python
from lerobot.robots.openarm_follower    import OpenArmFollower,   OpenArmFollowerConfig
from lerobot.robots.bi_openarm_follower import BiOpenArmFollower, BiOpenArmFollowerConfig

cfg   = OpenArmFollowerConfig(port="oa_fl", side="left", id="...", use_velocity_and_torque=True)
robot = OpenArmFollower(cfg)
robot.connect(calibrate=True)     # ← CAN 소켓 open. 세션당 1회. 이후 절대 재호출하지 않는다.
...
robot.disconnect()                # ← 세션 종료에서만.
```

- OpenArm은 LeRobot의 **1급 로봇**이다 — `robots/utils.py:63-68`이 `openarm_follower` / `bi_openarm_follower`를 하드코딩 분기로 인스턴스화한다.
- 양팔도 **한 프로세스**다 — `bi_openarm_follower.py:93-94`가 `OpenArmFollower` 객체 2개를 만들고, `utils/bimanual.py:47-63`의 `BimanualMixin.connect()`가 **순차 동기 호출**로 둘 다 연결한다. 별도 스레드도 asyncio도 없다.
- **LeRobot CLI(`lerobot-record` / `lerobot-teleoperate` / `lerobot-rollout` / `lerobot-calibrate`)를 모드마다 서브프로세스로 스폰하는 설계는 금지한다.** 이들은 `pyproject.toml [project.scripts]`가 정의하는 **각각 독립 프로세스**이며, 각 스크립트가 시작 시 `robot.connect()`, 종료 시 `robot.disconnect()`를 한다 → §2.0.1의 영점 파괴가 모드 전환마다 발생한다.
- **`ros2_control` / `OpenArmHW` / MoveIt / dora-rs는 이 경로에 없다.** 다만 하위 라이브러리 `openarm_control`(IK), `openarm_driver`, `openarm_ker`는 사용한다.

#### 2.0.1 🔴 `connect()`가 영점을 파괴한다 — 이 문서에서 가장 중요한 사실

`robots/openarm_follower/openarm_follower.py:133-158`:

```python
@check_if_already_connected
def connect(self, calibrate: bool = True) -> None:
    """
    We assume that at connection time, the arms are in a safe rest position,
    and torque can be safely disabled to run calibration if needed.
    """
    self.bus.connect()                              # :140
    if not self.is_calibrated and calibrate:
        self.calibrate()                            # :145  ← 이 안에도 set_zero_position()이 있다
    for cam in self.cameras.values():
        cam.connect()
    self.configure()
    if self.is_calibrated:
        self.bus.set_zero_position()                # :152-153  ← 🔴
    self.bus.enable_torque()                        # :155
```

`motors/damiao/damiao.py:335-340`:
```python
def set_zero_position(self, motors: str | list[str] | None = None) -> None:
    """Set current position as zero for selected motors."""
    target_motors = self._get_motors_list(motors)
    for motor in target_motors:
        self._send_simple_command(motor, CAN_CMD_SET_ZERO)   # → CAN 0xFE
        time.sleep(MEDIUM_TIMEOUT_SEC)
```

→ **`connect()`는 호출 시점의 물리 자세를 무조건 영점으로 확정한다.** 팔이 rest 자세가 아닌 순간 `connect()`가 불리면 **그 자세가 영점이 되고**, 조인트 리밋·데이터셋 좌표계·티칭 포인트가 통째로 어긋난다. 복구는 재캘리브레이션뿐이다.

**🔴 그러나 이 auto-set-zero는 피할 수 없는 제약이 아니라 오버라이드 대상이다.** 참조 구현(`bh_indy7_LeRobot`/`bh_rebot_LeRobot`, 같은 Damiao DM 모터 기반)이 이미 상위 클래스의 auto-set-zero를 서브클래스에서 제거하는 패턴을 구현했다 → **우리는 이를 OpenArm에 이식한다(§2.0.4).** 그 오버라이드 위에서 다음을 강제한다:
1. **`connect()` 오버라이드로 auto-set-zero를 제거한다** → 재연결이 영점을 파괴하지 않는다(§2.0.4, FR-CON-061). "세션당 `connect()` 1회"는 이제 **영점 안전의 필수요건이 아니라** flock·CAN 소켓 위생 차원의 권장사항으로 완화된다. 모드 전환은 여전히 같은 robot 객체 위의 **상태 전이**로만 구현한다(CLI 스폰 금지, `send_action()` 호출권 인계).
2. **영점 확립은 명시적 operator 플로우로만.** `connect_readonly()`(torque-OFF)로 손 검증 → rest 자세 정렬 → 명시적 set_zero → 디스크 영속(§2.0.4). rest 자세 확인 모달은 **connect가 아니라 이 명시적 set_zero 단계**에 건다(FR-CON-063).
3. **`0xFE`는 명시적 set_zero 플로우 외 어디서도 전송되지 않는다** (→ `03` FR-MOT-045).

> ⚠️ `is_calibrated`는 **모터가 아니라 메모리 캐시**를 본다 — `damiao.py:858-860` `return bool(self.calibration)`, `:749-758` *"Damiao motors don't store calibration internally / Just cache it in memory."* 즉 캘리브레이션 파일이 존재하기만 하면 `connect()`가 `set_zero_position()`으로 직행한다.

#### 2.0.2 LeRobot 캘리브레이션 ≠ enactic 영점 캘리브레이션 [확정]

**두 절차는 서로 다른 영점 자세를 만든다.** 혼용하면 조용히 어긋난다.

| | `lerobot-calibrate` (`OpenArmFollower.calibrate()`, `openarm_follower.py:165-215`) | `openarm-can-zero-position-calibration` (enactic) |
|---|---|---|
| 영점 자세 | **"팔을 곧게 아래로 늘어뜨리고 그리퍼 닫음"** — 소스 프롬프트 원문: *"Arm hanging straight down / Gripper closed"* (`:192-197`) | 기계 하드스톱 범핑 / OpenArm Cell **Zero-Position Calibration Jig** |
| 로봇 자동 이동 | ❌ 없음. 사람이 손으로 자세를 잡고 ENTER | ✅ **있다** (*"The robot will move automatically when calibration starts."*) |
| 산출물 | `MotorCalibration(homing_offset=0, range_min=-90, range_max=90)` — **전 관절 고정 ±90°**, JSON 파일로 저장 (`:203-213`) | 모터 플래시 영점(0xFE) |
| 저장 위치 | LeRobot 캘리브레이션 JSON (**메모리 캐시 + 파일**). 모터 내부에는 저장 안 됨 | **모터 플래시** (비휘발성, 10,000회 제한) |
| 0xFE 전송 | ✅ `bus.set_zero_position()` (`:199`) | ✅ |

- LeRobot의 `range_min=-90 / range_max=90`은 **실측 가동범위가 아니라 하드코딩 상수**다 — 소스 로그 원문: *"Setting range: -90° to +90° for safety by default for all joints"* (`:201`). **가동범위 스캔을 하지 않는다.**
- → **영점 자세를 프로파일에 명시적으로 기록해야 한다**(`zero_method` ∈ {`lerobot_hanging`, `enactic_jig`, `enactic_bumping`}). 같은 관절각이 절차마다 다른 물리 자세를 뜻하기 때문이다(FR-CON-031).

#### 2.0.4 🔴 우리의 연결·영점 모델 — 참조레포 디스크 SoT 패턴 [확정]

§2.0.1의 auto-set-zero는 **오버라이드 대상**이다. 참조 구현 `bh_indy7_LeRobot`/`bh_rebot_LeRobot`(Seeed reBot + Neuromeka Indy7, **같은 Damiao DM 모터**)이 상위 클래스의 "매 launch마다 auto-zero" 기본 동작을 서브클래스에서 제거했다 — 그 **패턴을 OpenArm에 이식한다**(수치가 아니라 구조를 이식한다). 핵심: **LeRobot `OpenArmFollower`를 상속**(→ `03` FR-MOT-058의 토크 경로 확장과 **같은 서브클래스**)해 connect()의 자동 `set_zero_position()`을 제거한다.

`bh_indy7_LeRobot/src/lerobot_plugin_rebot/rebot.py:247-266` (이식 레퍼런스 — 패턴만):
```python
def connect(self, calibrate: bool = True) -> None:
    # calibrate는 자동 실행하지 않는다 — set-zero + 그리퍼 엔드포인트 캡처는
    # 명시적 operator 플로우다. (auto-set-zero 제거)
    self.arm_backend.connect()          # torque ON, 그러나 set_zero 없음
    self._previous_q_deg = list(...)    # jump-guard 시드
    ...
def connect_readonly(self) -> None:     # 버스 오픈 + 모터 등록 + 피드백 워밍업
    self.arm_backend.open_readonly()    # enable_all 절대 안 함 → torque OFF
```

**세 가지 [신규구현 = 참조레포 패턴 이식]:**

1. **connect() 오버라이드** — 서브클래스 connect()는 `set_zero_position()`을 **호출하지 않는다.** → **재연결이 영점을 파괴하지 않는다**(FR-CON-061). 상위 기본 동작의 위험(§2.0.1 = 연결 순간의 임의 자세를 영점으로 확정)을 구조적으로 제거한다.
2. **connect_readonly()** — 버스를 **torque-OFF로 오픈**(모터 등록 + 피드백 워밍업만, `enable_torque` 절대 안 함). GUI bring-up에서 **de-energised 상태로 손으로 팔을 움직여** 방향·영점을 검증한다. operator가 명시적으로 토크를 인가하기 전까지 무전원(FR-CON-062). 참조 `rebot.py:259-266`.
3. **명시적 operator set_zero 플로우** — 손으로 rest 자세 정렬 → 버튼 → **전 모터 disable → settle → 모터별 0xFE → readback 검증 → 디스크 저장.** enabled 모터에 set_zero를 보내면 조용히 skip되므로 **disable 선행이 필수**다. 참조 `transport.py:97-102,624-635`, `rebot.py:414-426`(FR-CON-063).

**영점·캘리브레이션의 SoT 이원화 [확정]** — DECISIONS M-1의 "디스크가 SoT"는 **관절 zero를 제외한 전부**에만 성립한다:

| 데이터 | SoT | 근거 |
|---|---|---|
| **관절 zero(영점)** | **모터 NV** (0xFE = `set_zero_position`, 별도 save 불필요) | cmjang·LeRobot·motorbridge 모두 0xFE 뒤 save 호출 없음. 참조 `schema.py:153-158` *"the zero lives in the motor, not as a JSON offset"* |
| **관절 부호 sign · 그리퍼 open/close rad · captured 플래그 · timestamps · zero_method** | **디스크 JSON** | 참조 `ReBotCalibration`(`schema.py:150-173`), `save_calibration_atomic`(`atomic_io.py:47-79`) |

- **레지스터 flash store(0xAA)에는 의존하지 않는다** — firmware별 신뢰성 편차가 있다(참조 실측: rebot B601 실기에서 `store_parameters()`가 무시되어 power-cycle 후 모터 ID가 복귀 — `docs/v1/work_log/2026-05-21…:19,30`). **0xFE(영점)만 신뢰**하고 나머지는 디스크에 둔다.
- **원자적 저장**: `mkstemp → write → flush → os.fsync → os.replace` (POSIX 원자적 — 옛 파일 또는 새 파일만 보이며 partial write는 절대 안 보인다). 경로 `~/.cache/huggingface/lerobot/calibration/openarm/<id>.json`.
- ⚠️ **0xFE의 전원 사이클 영속성은 [미확인]**이다(→ §5 Q-6). DM 프로토콜에 별도 `SAVE_PARAM`(0xAA)이 존재하므로 자명하지 않다. **전원 사이클마다 영점이 사라질 수 있다고 가정**하고, **매 전원 세션 시작에 명시적 operator set_zero 플로우로 영점을 재확립**한다. 디스크 JSON은 zero를 제외한 캘리브레이션 메타의 SoT다.

**operator 캘리브레이션 플로우(온보딩 마법사가 강제 — FR-CON-063):**
```
1. connect_readonly()                         # torque OFF, 읽기만
2. (관절 방향 확인) 필요 시 sign 갱신 → 디스크
3. 손으로 URDF-영점 rest 자세 정렬 → set_zero
   → disable_all → settle → 모터별 0xFE → readback 검증 → 디스크 저장
4. 손으로 그리퍼 open 끝단 → capture_gripper_endpoint("open")
   손으로 그리퍼 close 끝단 → capture_gripper_endpoint("close")
   → 각각 현재 native rad 캡처 → 디스크 저장 (그리퍼 정규화 SoT, §2.9.1)
5. enable_torque()                            # present-pose 홀드로 안전 인게이지
```

> 이것은 **공식 절차와 정합**한다: enactic 공식 경로도 기계식 지그(CAD 정의 자세) 또는 하드스톱 범핑으로 URDF-영점 자세를 잡은 뒤 `openarm-can-cli … set_zero --arm`(0xFE)을 **팔 단위(per-CAN)**로 친다(§2.9). 우리는 여기에 **디스크 JSON SoT + connect_readonly + auto-set-zero 오버라이드**를 더한다.

### 2.5 SocketCAN 설정 — 🔴 `ip link` CAN-FD 셋업은 **필수이고 코드가 대신할 수 없다** [확정]

**python-can의 socketcan 백엔드는 `bitrate` / `data_bitrate` 인자를 무시한다.**

`can/interfaces/socketcan/socketcan.py:694-703` — `SocketcanBus.__init__` 시그니처에 **`bitrate`도 `data_bitrate`도 없다**:
```python
def __init__(self, channel: str = "", receive_own_messages: bool = False,
             local_loopback: bool = True, fd: bool = False,
             can_filters: CanFilters | None = None,
             ignore_rx_error_frames=False, **kwargs) -> None:
```
두 인자는 `**kwargs`로 흘러가 `BusABC`에 전달되고 **소켓에 아무 영향을 주지 않는다.** 실제로 소켓에 작용하는 것은 `fd`뿐 (`:765-769` → `setsockopt(SOL_CAN_RAW, CAN_RAW_FD_FRAMES, 1)`).

그런데 LeRobot은 이 인자들을 **넘긴다** — `motors/damiao/damiao.py:174-186`:
```python
kwargs = {"channel": self.port, "bitrate": self.bitrate, "interface": self.can_interface}
if self.can_interface == "socketcan" and self.use_can_fd and self.data_bitrate is not None:
    kwargs.update({"data_bitrate": self.data_bitrate, "fd": True})
self.canbus = can.interface.Bus(**kwargs)
```
그리고 OpenArm config는 `socketcan`을 못 박는다 — `config_openarm_follower.py:59-64`: `can_interface="socketcan"`, `use_can_fd=True`, `can_bitrate=1000000`, `can_data_bitrate=5000000`.

> 🔴 **함의**: 링크를 CAN 2.0으로 올려놓은 채 LeRobot이 `fd=True`로 열면 **python-can은 조용히 성공하고 통신만 깨진다.** 어떤 에러도 나지 않는다.
> → **`ip link`로 FD를 켜는 단계는 선택이 아니라 필수이고, GUI가 기동 시 `ip -details link show`로 반드시 검증해야 한다**(FR-CON-008).

```bash
# 필수 — 링크를 CAN-FD로 올린다 (이걸 안 하면 위 함정에 걸린다)
sudo ip link set can0 down
sudo ip link set can0 type can bitrate 1000000 dbitrate 5000000 fd on
sudo ip link set can0 txqueuelen 1000          # 기본 10은 8모터 고주기에 얕다
sudo ip link set can0 up

# LeRobot 내장 동등물 (내부적으로 위 ip link를 실행 — lerobot_setup_can.py:96-121)

# 검증 (이것이 FR-CON-008의 판정 근거)

```

- **`openarm-can-configure-socketcan-4-arms`는 쓰지 않는다.** `FD_MODE` 기본값이 **`false`**여서(`:24`) `-fd`를 빠뜨리면 CAN 2.0으로 올라가는데 LeRobot은 `use_can_fd=True`가 기본이라 조용히 어긋난다. LeRobot이 동등물을 내장한다.
- `restart-ms`(버스오프 자동 복구)와 `txqueuelen`은 **LeRobot도 enactic 스크립트도 건드리지 않는다** → 우리가 추가한다(FR-CON-003).
- systemd 유닛으로 부팅 시 자동 구성 (상류가 제공하지 않는다 — 부재 확인, FR-CON-004).

#### 2.5.1 udev 고정은 선택이 아니다 [확정]

**동일 VID/PID 어댑터 2개를 꽂으면 `can0`~`can3` 할당이 비결정적이다** — USB 열거 순서가 부팅마다 경합한다. 좌우 팔이 뒤바뀐 채 기동될 수 있다.

- 커널 소스 `drivers/net/can/usb/gs_usb.c:1355` → `netdev->dev_id = channel;` → 멀티채널 장치는 채널별 netdev에 **서로 다른 `dev_id`**를 심고 `/sys/class/net/<if>/dev_id`로 노출한다. **한 어댑터 안의 2채널도 udev로 구분 가능하다.**
- `ARPHRD_CAN = 280` → `ATTR{type}=="280"`.

```udev
# /etc/udev/rules.d/80-openarm-can.rules
# 어댑터는 serial(또는 USB 포트 경로)로, 채널은 dev_id로 구분한다.
SUBSYSTEM=="net", ACTION=="add", ATTR{type}=="280", \
  ATTRS{serial}=="<ADAPTER_A_SERIAL>", ATTR{dev_id}=="0x0", NAME="oa_fl"
SUBSYSTEM=="net", ACTION=="add", ATTR{type}=="280", \
  ATTRS{serial}=="<ADAPTER_A_SERIAL>", ATTR{dev_id}=="0x1", NAME="oa_fr"
SUBSYSTEM=="net", ACTION=="add", ATTR{type}=="280", \
  ATTRS{serial}=="<ADAPTER_B_SERIAL>", ATTR{dev_id}=="0x0", NAME="oa_ll"
SUBSYSTEM=="net", ACTION=="add", ATTR{type}=="280", \
  ATTRS{serial}=="<ADAPTER_B_SERIAL>", ATTR{dev_id}=="0x1", NAME="oa_lr"
```

**함정 3개:**
1. **`NAME=`을 `can0`~`can3`으로 주지 마라.** 커널이 이미 `canN`을 할당하는 중이라 udev와 이름을 두고 경합한다(순서에 따라 승자가 바뀐다). **`can`으로 시작하지 않는 이름**을 써서 네임스페이스를 분리한다. `IFNAMSIZ = 16`이므로 길이는 충분하다.
2. LeRobot의 `port`는 `channel=` 문자열로 그대로 넘어가므로 **`--robot.port=oa_fl`이 그대로 동작한다**(`damiao.py:180`).
3. **`ATTRS{serial}`이 없거나 두 어댑터가 같은 시리얼을 보고할 수 있다.** 그러면 **USB 물리 포트 경로**로 고정한다 — 고정 리그에서는 오히려 이쪽이 안정적이다:
   ```udev
   SUBSYSTEM=="net", ACTION=="add", ATTR{type}=="280", \
     KERNELS=="1-1.2:1.0", ATTR{dev_id}=="0x0", NAME="oa_fl"
   ```
   경로는 `udevadm info -a -p /sys/class/net/can0 | grep KERNELS`로 확인. **"어느 허브 구멍에 꽂았나"로 고정되므로 케이블을 옮기면 깨진다 — 리그에 라벨을 붙인다.**

> ⚠️ **`openarm-can-cli … discover --full-scan` 실행 후 인터페이스가 10 Mbps로 남는다.** 공식 문서가 명시하는 부작용이며 반드시 `can_configure`(또는 위 `ip link`)로 복구해야 한다.

### 2.6 CAN 소유권 — **배타 bind는 존재하지 않는다 (조용한 실패의 메커니즘)**

Linux 커널 SocketCAN 공식 문서 원문 (https://docs.kernel.org/networking/can.html):

> *"**Multiple sockets can be opened at the same time**, on different or the same protocol module and they can listen/send frames on different or the same CAN IDs."*
> *"To meet multi user needs **the local loopback is enabled by default**."*
> bind 시점 기본 필터: *"The filters are set to exactly one filter receiving everything."*

**소켓을 여는 두 스택 모두 배타 옵션을 쓰지 않는다:**
- python-can (LeRobot 경로): `socketcan.py:582` `socket(PF_CAN, SOCK_RAW, CAN_RAW)` → `:601` `sock.bind((channel,))`
- `openarm_can` (C++ 진단 경로): `can_socket.cpp:40` `socket(PF_CAN, SOCK_RAW, CAN_RAW)` → `:69` `bind(...)`. `CAN_RAW_LOOPBACK`/`RECV_OWN_MSGS`/`FILTER` **어느 것도 설정하지 않는다**(커널 기본값 그대로).

**따라서 프로세스 A, B가 같은 인터페이스를 동시에 bind하면**:
1. **`bind()`가 둘 다 성공한다.** 예외도, errno도, 로그도 없다.
2. 루프백 기본 ON → A는 B의 명령 프레임(0x01–0x08)까지 수신한다. A는 recv id(0x11–0x18)로 디스패치하므로 조용히 버린다 → **에러 없음.**
3. 모터가 B의 질의에 응답한 프레임을 **A가 먼저 소비**할 수 있다. A는 자기 상태 캐시를 갱신하고 **"정상"으로 보인다.** B는 프레임을 뺏겨 타임아웃 → LeRobot은 **마지막 알려진 상태를 재사용**하고(`damiao.py:696-702`) `logger.warning`만 뱉는다 → **낡은 상태로 제어한다.**
4. 둘 다 MIT 명령을 쓰면, 게인이 다르므로(§2.10) 모터는 두 제어기 사이에서 **진동한다.**
5. **어느 계층도 이걸 감지하지 못한다.**

⇒ **배타성은 커널이 주지 않는다. 애플리케이션이 만들어야 한다.** 이것이 FR-CON-010 / FR-CON-011이 존재하는 이유다.

### 2.7 CAN 소유권 — 문제는 **축소되지만 사라지지 않는다** [확정]

**✅ 축소되는 부분**: LeRobot Python API를 단일 GUI 프로세스에 임베드하면(§2.0) **모드 간 배타성은 프로세스 내부에서 해결된다.** robot 객체가 하나이고, 양팔도 같은 프로세스의 버스 객체 2개다. **프로세스 간 CAN 소유권 중재 프로토콜(IPC)은 불필요하다.**

**❌ 사라지지 않는 부분**: 커널이 배타성을 강제하지 않으므로(§2.6) **외부 침입자는 여전히 무방비로 끼어든다.**

| 침입자 | CAN 여는 방식 | 위험 |
|---|---|---|
| `cansend` / `cangen` | `socket(PF_CAN, SOCK_RAW, CAN_RAW)` + `bind()` | **송신** — 모터 명령 스트림 오염 |
| `openarm-can-cli` (`discover`, `can_configure`, `set_zero`, `clear_error`) | `openarm_can::CANSocket` (`can_socket.cpp:40,69` — 동일 소켓, **락 없음**) | **송신**. "진단용"처럼 보여도 `discover`/`can_configure`는 송신한다. `--full-scan`은 **인터페이스 비트레이트까지 바꿔놓는다** |
| **두 번째 LeRobot 인스턴스** (CLI 스폰 포함) | `can.interface.Bus(...)` | **송신 + `connect()`가 `set_zero_position()` 호출 → 영점 파괴**(§2.0.1) |
| `openarm_teleop` (C++, 500/1000 Hz 바이래터럴) | `openarm_can` | **송신.** 게다가 채널 관례가 정반대(§2.2) |
| `candump` | 읽기 전용 | 물리적으로는 안전. 다만 안전을 강제하는 것은 커널이 아니라 candump의 선의(善意)다 |

→ **`flock` 애플리케이션 락은 여전히 필요하다.** 역할이 "모드 간 중재"에서 **"외부 침입자 방어"**로 **축소**될 뿐이다.

```python
# 세션 진입점에서 사용 채널 전부에 대해 획득. 실패 시 기동 거부.
import fcntl, os

class CanBusLock:
    """/run/lock/openarm-<iface>.lock 에 대한 배타 광고 락.

    커널은 SocketCAN RAW 다중 bind를 막지 않는다. 이 락은 '규칙을 지키는 프로세스들끼리'의
    합의일 뿐이며, 규칙을 모르는 프로세스(cansend, openarm-can-cli)는 막지 못한다.
    막지 못하는 것을 최소한 '탐지'하기 위해 FR-CON-011의 침입자 감시와 함께 쓴다.
    """
    def __init__(self, channel: str):
        self.path = f"/run/lock/openarm-{channel}.lock"
        self.fd = None

    def acquire(self) -> None:
        self.fd = os.open(self.path, os.O_CREAT | os.O_RDWR, 0o666)
        try:
            fcntl.flock(self.fd, fcntl.LOCK_EX | fcntl.LOCK_NB)
        except BlockingIOError:
            os.close(self.fd); self.fd = None
            raise RuntimeError(f"{self.path} is held by another process")
        os.write(self.fd, f"{os.getpid()}\n".encode())
```

**프로세스가 죽으면 커널이 flock을 자동 해제한다** — stale lock이 없다. PID 파일보다 우월한 이유다.

**배타성의 단위가 내려간다**: "버스를 누가 소유하는가"에서 **"누가 `send_action()`을 부를 권리를 갖는가"**로. 텔레옵 루프와 정책 루프는 **둘 다 `send_action()`을 호출**하며 같은 robot 객체를 공유해도 두 루프가 동시에 돌면 명령이 인터리브된다. **LeRobot에 중재자는 없다.** → 모드 상태머신은 그대로 필요하다(→ `01-시스템-아키텍처.md`, `SYS`).

### 2.9 영점(zero) 캘리브레이션 — 확정 절차

**문서가 명시하는 제약 (전부 강제 대상)**:
- 실행 전 사람이 팔을 **물리적으로 영점 자세에 근접**시켜야 한다.

- ⚠️ **모터 파라미터 플래시 쓰기는 약 10,000회 하드 리밋**이다. *"it should not be run frequently in scripts."* 초과 시 모터 플래시 영구 손상.
- v2.0 신규: **OpenArm Cell의 Zero-Position Calibration Jig**가 그리퍼를 CAD 정의 각도로 물리 구속해 조립 공차를 제거한다.

**저수준 등가 API**: `OpenArm::set_zero_all()`, `DMDeviceCollection::set_zero(int i)`, `GripperComponent::set_zero()`, `openarm-can-cli … set_zero --id N` — 전부 **0xFE 명령**(§2.3)이다.

>
> **GUI는 사용자 좌표계(F_URDF, 오프셋 미포함) 값만 표시·입력한다. 이중 가산/미가산은 감사 링버퍼로 감지한다**(FR-CON-033, `03` NFR-MOT-004).

- **좌/우가 부호 반전 비대칭**이다. 좌우를 같은 위젯/같은 값으로 다루면 안 된다.
- 그리퍼(J8)만 `POS_FORCE`, 나머지 7축은 MIT — **단, 이는 `openarm_driver` 경로의 이야기다.** LeRobot의 `send_action()`은 **그리퍼를 포함한 8모터 전부를 MIT로** 보낸다(`openarm_follower.py:342` `self.bus._mit_control_batch(commands)`, `motor_index`에 `"gripper": 7` 포함). → 그리퍼 힘 제어(POS_FORCE)를 쓰려면 **우리가 우회해야 한다** (→ `03` FR-MOT-046).

#### 2.9.1 🔴 그리퍼 정규화 = 엔드포인트 rad 캡처 [확정 — 참조레포 패턴]

**v2 그리퍼는 pinch 그리퍼**(revolute −45°–0°, 2지 mimic, 인핸드 카메라 내장, 교체형 핑거)이며 **로드셀이 없다.** v1의 병렬링크·88 mm 개구량 서술은 v2에 무효다. 그리퍼 open/close 범위는 **개구량(mm)을 실측하거나 모터각:핑거각 비율을 확정할 필요 없이** 참조레포 패턴으로 정규화한다:

- **`capture_gripper_endpoint(direction)`** — operator가 그리퍼를 물리적으로 open/close 끝단에 놓고 **현재 native rad를 그대로 캡처**(힘·로드셀 측정 전무)해 디스크에 영속한다. 참조 `motorbridge_backend.py:245-259`, `rebot.py:442-460`.
- **정규화**: `norm∈[0,1]`은 캡처된 두 rad 사이 **선형 보간**이다 — `rad(norm) = open_rad + clamp01(norm)·(close_rad − open_rad)`. 개구량 mm·2:1 매핑을 몰라도 제어가 성립한다.
- **set_zero가 그리퍼 캡처를 무효화한다**: 0xFE는 그리퍼 모터도 재-zero하므로, 캡처된 엔드포인트 rad는 재-zero 후 더 이상 물리 open/close를 가리키지 않는다. → set_zero 시 `*_captured=False`로 떨어뜨리고 **재캡처를 요구**한다(참조 `rebot.py:419-426`).
- **힘 상한**은 POS_FORCE의 `torque_pu`(per-unit 전류 상한, `[0,1]`)로 별도 관리한다 — Nm/N으로 노출하지 않는다(→ `03` FR-MOT-046·047). **파지력을 로드셀로 N 측정하지 않는다.**

→ 이 패턴이 온보딩 캘리브레이션(§2.0.4 step 4)과 그리퍼 제어(→ `04` FR-MAN-016)의 정본이다. FR-CON-064.

#### (b) 관절 리밋 — 🔴 `side` 미지정 = **±5°**

`send_action()`은 `config.joint_limits`로 **조용히 클리핑**한다 (`openarm_follower.py:290-297` — `logger.debug`로만 남는다):

| `side` 값 | joint_1 | joint_2 | joint_4 | gripper | 소스 |
|---|---|---|---|---|---|
| **`None` (기본!)** | **(−5, +5)** | **(−5, +5)** | **(0, +5)** | **(−5, 0)** | `config_openarm_follower.py:109-120` |
| `"right"` | (−75, +75) | (−9, +90) | (0, 135) | (−65, 0) | `:34-43` `RIGHT_DEFAULT_JOINTS_LIMITS` |
| `"left"` | (−75, +75) | (−90, +9) | (0, 135) | **(−65, 0)** ⚠️ | `:23-32` `LEFT_DEFAULT_JOINTS_LIMITS` |

- **단위는 deg**, 프레임은 **캘리브레이션 후 모터 프레임**(§2.9).
- `side`는 생성자에서만 반영된다 — `openarm_follower.py:72-84`. `None`이면 경고 로그만 뜨고 ±5°로 간다: *"Set config.side to either 'left' or 'right' to use pre-configured values for joint limits."*
- ⚠️ **좌팔 그리퍼 리밋 `(-65.0, 0.0)`은 우팔과 동일하다 — 미러링 누락 버그다.** 좌팔 개방 방향은 `+`이므로 **좌팔 그리퍼가 열리지 않는다** (→ `03` FR-MOT-051).
- **URDF 기계 리밋은 이보다 넓다** (joint_1 `−80…+200°`). LeRobot 값은 파일 주석이 인정하듯 *"the default joint limit values are small for safety"* — **의도적 보수 기본값**이며 단순 오류가 아니다.

> ⇒ 온보딩은 **`side`를 반드시 명시**하고(FR-CON-056), 어느 리밋 세트를 운영 클램프 정본으로 쓸지 프로파일에 박는다(FR-CON-041). → §5 Q-1.

#### (c) 🔴 LeRobot에는 속도·가속도·저크 제한이 **없다** [확정]

LeRobot이 제공하는 안전장치는 **`joint_limits` + `max_relative_target`이 전부**다. `robots/` 트리 전체 grep 결과 `velocity_limit` / `acceleration_limit` / `jerk` **0건**.

`max_relative_target: float | dict[str, float] | None = None` (`config_openarm_follower.py:75`) — **기본값 `None` = 꺼짐**:
- 켜면 `send_action()`이 매 스텝 `bus.sync_read("Present_Position")`을 한 번 더 하고(`openarm_follower.py:301`, 주석 원문 *"/!\ Slower fps expected due to reading from the follower."*), `ensure_safe_goal_position()`(`robots/utils.py:91-118`)이 **`safe_goal = present + clamp(goal − present, ±max_relative_target)`**로 자른다.
- **단위는 deg**(goal_pos가 deg이므로). **스텝당 위치 델타 상한**이지 속도 상한이 아니다 — dt에 무관하다.
- 스칼라 또는 모터명→값 dict. dict일 때 키 집합이 액션과 정확히 일치하지 않으면 `ValueError`(`utils.py:99-100`).

→ **속도·가속도 클램프, 소프트 E-Stop, 워크스페이스 가상벽, 충돌 반응은 전부 우리가 만든다** (→ `04` FR-MAN-011, `12-충돌감지-및-안전.md`).

| v1 BOM: `Emergency Stop` ×1 (Monotaro). 배선 원문: *"The system includes two emergency stop (E-stop) buttons — one for the leader pair and one for the follower pair."* → **2팔당 1개, 전원 차단 방식** | 동일 |

> **[미확인]** v2.0 표준 구성에 하드웨어 E-Stop 회로가 포함되는지, 인터록 사양과 차단 범위(모터 전원만인지 전체인지)는 공식 문서에 **회로도·사양이 제시되지 않는다.** → §5 Q-4.

⇒ **소프트 스톱(전 모터 `disable_all()` / 토크 0)과 하드 E-Stop(전원 차단 → 낙하)은 완전히 다른 기능이다.** 온보딩은 두 경로를 각각 확인시키고 UI에서 분리해야 한다. 그리고 **`disable_all()`조차 파지물 낙하를 유발한다** — 백드라이버빌리티가 높기 때문이다.

| 경로 | 주파수 | 출처 |
|---|---|---|
| **`openarm_can` 공식 경고 — 절대 상한** | **8모터 구성에서 1000 Hz 초과 = CAN 연결 불안정** | openarm_can 공식 문서 |
| `openarm_teleop` 리더-팔로워 바이래터럴 (C++) | 스레드 **500 Hz** (`openarm_bilateral_control.cpp:42,72,103`), `#define FREQUENCY 1000.0` (`openarm_constants.hpp:42`) | `enactic/openarm_teleop` |
| CAN 버스 시간 [계산] | 8모터 × 2프레임 × ~52 µs ≈ **832 µs/사이클** → 1000 Hz에서 **버스 사용률 ≈83%**(지터 위험), 750 Hz ≈62%, 500 Hz ≈42% | CAN-FD 1 M/5 M, 8B 페이로드 |

**LeRobot 루프의 CAN 타임아웃 상수** (`motors/damiao/damiao.py:62-65`):
```python
LONG_TIMEOUT_SEC    = 0.1      # 100 ms
MEDIUM_TIMEOUT_SEC  = 0.01     #  10 ms   ← enable/disable/set_zero 사이의 sleep
SHORT_TIMEOUT_SEC   = 0.001    #   1 ms   ← _mit_control_batch 응답 수집
PRECISE_TIMEOUT_SEC = 0.0001   # 100 µs
```
`_mit_control_batch`는 전 모터에 명령을 **먼저 다 보내고** 응답을 `SHORT_TIMEOUT_SEC`(1 ms)로 일괄 수집한다(`damiao.py:513-526`). `sync_read_all_states()`도 *"in ONE CAN refresh cycle"*로 pos/vel/torque를 함께 읽는다(`damiao.py:655-674`) → **토크 관측에 추가 CAN 비용이 없다.**

> 🔴 **측정하지 않은 것**: Python/GIL 아래에서 `sync_read_all_states()` + `_mit_control_batch()` 왕복이 실제로 몇 Hz를 내는가.
> - 30–60 Hz 기록/텔레옵은 여유로울 것으로 보이나 **실측 전에는 단정하지 않는다.**
> - 바이래터럴이 요구하는 **500 Hz**가 안 나오면 바이래터럴만 별도 실시간 프로세스로 분리해야 하고, **그 순간 CAN 배타 소유권 문제가 부활한다**(§2.7).
> - `BimanualMixin`은 **순차 동기 호출**(`utils/bimanual.py:47-63`)이라 팔 개수만큼 지연이 누적된다.
> → §5 **Q-5**에 최우선 측정 항목으로 등재한다.

헬스체크 실측 선례: `openarm_can/setup/motor_sampling_check.cpp` — RTT(µs)·목표 대비 실측 Hz를 측정하고 **`actual_hz < target_hz * 0.95` 시 "LOW" 경고**. 이 임계를 그대로 채택한다.

### 3.2 CAN 배타 소유권 (조용한 실패 차단 · 외부 침입자 방어)

> **역할 축소 [확정]**: LeRobot Python API를 단일 프로세스에 임베드하므로(§2.0) **모드 간 배타성은 프로세스 내부에서 해결된다.** 아래 요구사항의 목적은 **외부 침입자 방어**로 축소되며, **프로세스 간 CAN 소유권 이양 프로토콜(IPC)은 삭제되었다.**

## 4. 상태·오류 처리

### 4.1 온보딩 상태 전이

> 🔴 **운용 모드 전환은 이 상태머신이 아니라 `CONNECTED` 내부의 컨트롤 루프 교체로 표현된다**(→ `SYS`). `disconnect()`는 세션 종료에서만 한다 — 단, 우리 오버라이드 connect()는 재연결 시 **0xFE를 내보내지 않으므로**(FR-CON-061), 재연결 자체가 영점을 파괴하지는 않는다. 영점 재확립이 필요하면 `CONNECTED`/`READY` → `READONLY`(torque-OFF) → `ZEROING`(명시 set_zero)로 되돌아간다.

| 상태 | 진입 조건 | 허용 전이 | 이탈 시 동작 |
|---|---|---|---|
| `DISCONNECTED` | 초기 상태 / CAN 락 미보유 | → `IF_VERIFIED` | — |
| `IF_VERIFIED` | `ip -details link show`로 **UP + `fd on` + bitrate 1M/dbitrate 5M 확인**(FR-CON-008) | → `LOCKED`, → `DISCONNECTED` | — |
| `LOCKED` | `flock` 획득 + 침입자 프리플라이트 통과 | → `DISCOVERED`, → `DISCONNECTED`(락 해제) | 락 해제 |
| `DISCOVERED` | 8모터 응답(원시 CAN 진단 경로), ID/타입 대조 통과 | → `VERIFIED`, → `FAULT` | — |
| `VERIFIED` | 펌웨어·PMAX/VMAX/TMAX 일치, ERR 코드 전무, **LeRobot 설정 검증 통과**(`side`, `use_velocity_and_torque`) | → `READONLY`, → `FAULT` | — |
| `READONLY` | **`connect_readonly()` 성공 — torque-OFF 버스 오픈**(FR-CON-062). 손으로 방향·영점 검증 | → `ZEROING`, → `VERIFIED`(취소) | — |
| 🔴 `ZEROING` | **rest 자세 모달 확인 후 명시적 set_zero**(FR-CON-063) — **0xFE는 이 상태에서만 나간다** + 그리퍼 엔드포인트 캡처(FR-CON-064) → 디스크 저장 | → `CONNECTED`, → `READONLY`(취소) | — |
| 🔴 `CONNECTED` | **`enable_torque()`(0xFC) 성공. present-pose 홀드로 인게이지.** 우리 오버라이드 connect()는 **0xFE를 내보내지 않는다** | → `READY`, → `READONLY`(재영점: torque-OFF 후 재캘리브레이션), → `SAFE_HOLD`, → `SESSION_END` | — (**`disconnect()`는 세션 종료에서만**) |
| `READY` | 헬스체크 전 항목 PASS, 프로파일 저장됨 | → 운용 모드 루프(`SYS`), → `READONLY`(재영점), → `SAFE_HOLD` | — (**버스는 계속 잡고 있다**) |
| `SAFE_HOLD` | 소프트 스톱 발동 / 워치독 발동 | → `READY`(사용자 명시적 재개), → `SESSION_END` | — |
| `SESSION_END` | 사용자가 세션 종료 | → `DISCONNECTED` | **`robot.disconnect()` — 세션당 1회** |
| `FAULT` | 아래 F1–F7 중 하나 | → `SAFE_HOLD`(복구 가능), → `SESSION_END` | 즉시 `bus.disable_torque()` 시도 |

> **`FAULT` → `READY` 직접 전이는 금지한다.** `SAFE_HOLD`를 거쳐 사용자의 명시적 재개를 요구한다.
> **`SAFE_HOLD` → `READY`는 자동 전이 금지.**
> **`FAULT`에서도 `disconnect()`하지 않는다** — 토크만 끄고 버스는 유지해 CAN 소켓 churn·flock 재획득을 피한다(오버라이드로 재연결이 영점을 파괴하진 않지만, 세션 위생상 버스를 유지한다).

| **F8** | 🔴 **영점 파괴 (임의 자세에서 0xFE 발동)** | `set_zero_position()`이 명시적 set_zero 플로우(FR-CON-063) 외의 경로에서 호출됨. 또는 실측 관절각이 기대 자세와 계통적으로 어긋남 | **복구 불가.** 영점 재캘리브레이션(명시 플로우, rest 자세)만이 방법 → **예방(FR-CON-061 오버라이드 + FR-CON-063 명시 플로우)이 유일한 대응.** 우리 서브클래스는 connect()에서 auto-set-zero를 제거하므로 재연결로는 발생하지 않는다. | ⚠️ LeRobot **기본** connect()는 무조건 `set_zero_position()`을 부르고 끄는 **설정 플래그가 없다** → **메서드 오버라이드로 제거**(FR-CON-061) |

### 4.3 온보딩 중 안전 규칙 (무조건)

1. 🔴 **`connect()`는 오버라이드해 auto-set-zero를 제거한다**(FR-CON-061). 0xFE는 **명시적 set_zero 플로우(rest 자세 모달 후)에서만** 나간다 — 임의 자세 0xFE = 영점 파괴 = 복구 불가.
2. 🔴 **모드 전환에 disconnect/reconnect를 쓰지 않는다**(CAN 소켓·flock 위생). LeRobot CLI를 모드마다 스폰하지 않는다.
3. **enactic 영점 캘리브레이션(지그/범핑) 시작 = 로봇이 자동으로 움직인다.** 모달 확인 없이 실행 금지.
4. **`enable_torque()`(0xFC)로 팔이 즉시 present-pose 홀드에 들어간다.** 고지·동의 없이 실행 금지.
5. **토크 차단(`disable_torque()`, 0xFD)도 파지물을 떨어뜨린다.** 백드라이버빌리티가 높고 **홀딩 브레이크가 없다.** 파지 중 정지는 소프트 스톱이 아니라 **중력보상 홀드** 경로여야 한다(→ `12-충돌감지-및-안전.md`).
6. **버스오프에서 자동 재활성화 금지.**
7. **`flock` 없이 CAN 소켓을 여는 코드 경로를 만들지 않는다.**
