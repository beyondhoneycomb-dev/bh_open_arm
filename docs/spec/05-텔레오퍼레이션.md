# 05. 텔레오퍼레이션 (Quest 3S VR · KER)

## 1. 범위

**다루는 것**: LeRobot `Teleoperator` 플러그인으로서의 텔레옵 경로 전 구간 — 플러그인 계약과 등록, 액션 키셋·데이터셋 정합, Quest 3S 포즈 수신, 좌표변환, 스무딩, 클러치/스케일링, **teleoperator 내부 IK**(`openarm_control`), 그리퍼 매핑, KER 텔레오퍼레이터, 바이래터럴 힘 피드백의 가능/불가능 경계, 텔레옵 상태머신, 안전 게이트, 헤드셋 영상 리턴, GUI 노출 파라미터.

**다루지 않는 것**: 로봇 연결·캘리브레이션·영점(`02-로봇-연결-및-온보딩.md`), 모터 게인/리밋 프로파일의 정본 관리(`03-모터-설정.md`), 카메라 배치·시간동기·ZED SBS 분할(`06-카메라-서브시스템.md`), 에피소드 기록 규약과 `push_to_hub`(`07-데이터-수집.md`), 데이터셋 스키마(`08-데이터셋-관리.md`), 충돌 감지 정책 일반(`12-충돌감지-및-안전.md`), CAN 링크 구성(`01-시스템-아키텍처.md`).

**전제 (런타임)**: 정본 런타임은 **LeRobot**이다. dora-rs는 쓰지 않는다. 다만 dora 노드가 감싸던 **하위 라이브러리**(`openarm_control`, `openarm_driver`, `openarm_ker`)와 **공개된 프로토콜/알고리즘**(`dora-openarm-vr`의 `quest_receiver.py`, Apache-2.0)은 그대로 쓴다. dora 노드는 배선(wiring)이지 로직이 아니다.

**전제 (우리 산출물)**: 이 문서의 핵심 산출물은 pip 패키지 **`lerobot_teleoperator_openarm_vr`** 와 **`lerobot_teleoperator_openarm_ker`** 다. **LeRobot 포크는 불필요하다.**

## 2. 확정된 사실 기반

### 2.1 아키텍처 — 텔레옵이 앉는 자리

```
GUI 백엔드 (단일 프로세스, LeRobot Python API 임베드, 세션당 connect() 1회)
  ├─ OpenArmFollower / BiOpenArmFollower        ← can0/can1 소유
  ├─ Teleoperator (우리가 만든다)
  │    ├─ OpenArmVR   : Quest 포즈 → 클러치 → 스케일 → IK → 관절각 deg dict
  │    └─ OpenArmKER  : USB 인코더 → 관절각 deg dict (IK 불필요)
  ├─ 텔레옵 루프 (우리 소유. lerobot_teleoperate.py:188-234가 참조 구현)
  ├─ 안전 계층 (우리가 만든다)
  └─ LeRobotDataset (기록)
```

**LeRobot CLI(`lerobot-teleoperate` / `lerobot-record`)를 모드마다 스폰하는 설계는 금지한다.** 각 CLI는 독립 프로세스이고 시작 시 `robot.connect()`, 종료 시 `disconnect()`를 한다. `connect()`는 `is_calibrated`일 때 `bus.set_zero_position()`을 호출해(`robots/openarm_follower/openarm_follower.py:152-153` → `motors/damiao/damiao.py:335-340`, `CAN_CMD_SET_ZERO`) **현재 물리 자세를 영점으로 확정**한다. 모드 전환마다 재연결하면 rest가 아닌 자세가 영점이 되어 조인트 리밋과 데이터셋 좌표계가 통째로 어긋난다. → **텔레옵 상태머신의 어떤 전이도 `disconnect()`/`connect()`를 호출하지 않는다** (§4.2 금지 전이 5).

단, 우리 텔레오퍼레이터는 **CLI에서도 무수정으로 동작해야 한다**(§2.3). CLI는 디버깅·회귀 검증 경로로 유지하며, 운용 경로가 아니다.

### 2.3 LeRobot `Teleoperator` 계약 — 포크 불필요

**등록 3요소:**

1. `@TeleoperatorConfig.register_subclass("openarm_vr")` — draccus `ChoiceRegistry` 제공 (`teleoperators/config.py:22-31`).
2. **pip 배포명이 `lerobot_teleoperator_`로 시작**해야 한다. `register_third_party_plugins()`(`utils/import_utils.py:214-228`)가 `("lerobot_robot_", "lerobot_camera_", "lerobot_teleoperator_", "lerobot_policy_", "lerobot_env_")` 접두사를 가진 배포판을 자동 import한다. 호출 지점: `lerobot_teleoperate.py:284`, `lerobot_record.py:545`, `lerobot_calibrate.py:105`, `lerobot_train.py:767`.
3. **클래스명 규칙**: `make_teleoperator_from_config()`(`teleoperators/utils.py:36-118`)의 if/elif 체인에 없는 타입은 `make_device_from_device_class(config)`로 폴백한다(`utils.py:114-118`). 이것은 **config 클래스명에서 `Config` 접미사를 떼어** device 클래스명을 만든다(`import_utils.py:149-178`). → `OpenArmVRConfig` → 클래스 **`OpenArmVR`**.

**추상 멤버 (`teleoperators/teleoperator.py:87-208`) — 전부 구현 필수:**

| 멤버 | 종류 | 시그니처 | 라인 |
|---|---|---|---|
| `action_features` | property | `-> dict` | 87-98 |
| `feedback_features` | property | `-> dict` | 100-111 |
| `is_connected` | property | `-> bool` | 113-120 |
| `connect` | method | `(self, calibrate: bool = True) -> None` | 122-131 |
| `is_calibrated` | property | `-> bool` | 133-137 |
| `calibrate` | method | `() -> None` | 139-147 |
| `configure` | method | `() -> None` | 171-177 |
| `get_action` | method | `() -> RobotAction` | 179-188 |
| `send_feedback` | method | `(feedback: dict[str, Any]) -> None` | 190-203 |
| `disconnect` | method | `() -> None` | 205-208 |

- `RobotAction = dict[str, Any]` (`types.py:40`) — **flat dict. 키는 자유다.**
- `Teleoperator.__init__`(`teleoperator.py:29-56`)가 `HF_LEROBOT_CALIBRATION / TELEOPERATORS / self.name / f"{self.id}.json"`을 mkdir·로드한다. 모터가 없는 소스도 이 경로를 갖는다.
- `TeleoperatorConfig` 상속 필드: `id: str | None`, `calibration_dir: Path | None` (`config.py:22-31`).
- `@check_if_not_connected` / `@check_if_already_connected` (`utils/decorators.py:22-41`) — `is_connected`를 보고 `DeviceNotConnectedError`를 던진다.
- **`get_action(self)`는 인자가 없다.** 로봇 관측이 들어오지 않는다 (§2.6에서 처리).

**`action_features` 규약이 저장소 안에 2종 공존한다:**
(a) flat `{key: type}` — `openarm_leader.py:67-76`, `phone`. (b) `{"dtype","shape","names"}` — `keyboard_ee`, `gamepad`.
데이터셋 피처 생성(`aggregate_pipeline_dataset_features`)과 맞는 것은 **(a)** 다. **우리는 (a)를 쓴다.**

### 2.4 🔴 IK는 teleoperator 안에 있어야 한다 [결정 — 확정]

**데이터셋 action names = `robot.action_features`다** (`lerobot_record.py:394-401`이 `initial_features=create_initial_features(action=robot.action_features)`로 만든다). 그리고 `build_dataset_frame`(`utils/feature_utils.py:131-132`)이 `values[name] for name in ft["names"]`로 **teleop의 `get_action()` 출력 dict를 그 이름으로 인덱싱**한다.

→ **`get_action()`은 팔로워 액션 키셋과 동일한 키로, 관절각(degree)을 반환해야 한다.**

**기각한 대안 (EE 포즈를 내고 LeRobot processor가 IK):**
- 동작은 하지만 **CLI를 못 쓴다.** `lerobot_teleoperate.py:240-273`이 `make_default_processors()`를 부르고 그것은 identity 3종이다(`processor/factory.py:59-63`). `record()`도 `@parser.wrap()` CLI 경로에서는 항상 identity로 폴백한다(`lerobot_record.py:383-392`). **CLI에 processor 주입 지점이 없다.**
- 이 경로를 택하면 LeRobot의 placo 기반 `RobotKinematics`(`model/kinematics.py:59` `require_package("placo")`)에 묶인다.
- 게다가 `InverseKinematicsEEToJoints`(`robots/so_follower/robot_kinematic_processor.py:272-366`)는 관측의 `.pos` 키를 **필터 없이 전부 긁는다**(`:318-321`). `bi_openarm_follower` 관측은 `left_*`/`right_*` 16키이므로 **q_raw가 오염된다.** `GripperVelocityToJoint`는 `q_raw[-1]`을 그리퍼로 가정한다(`:417`). 양팔에서 그대로는 깨진다.

→ **IK-in-teleoperator를 택한다.** `lerobot-teleoperate` / `lerobot-record` / `lerobot-rollout`이 전부 무수정으로 돈다.

### 2.5 🔴 액션 키셋 — `.vel` / `.torque` 키를 반드시 함께 내야 한다

플랫폼 정체성상 `use_velocity_and_torque = True`가 강제된다(기본값은 **False**, `config_openarm_follower.py:71`). 그러면:

```python
# robots/openarm_follower/openarm_follower.py:90-99
features[f"{motor}.pos"] = float
if self.config.use_velocity_and_torque:
    features[f"{motor}.vel"] = float
    features[f"{motor}.torque"] = float
```
`_motors_ft`는 **`observation_features`와 `action_features` 양쪽**에 쓰인다(`:105-121`).

→ **`robot.action_features` = 단완 24키 / 양완 48키**이고, `build_dataset_frame`이 그 48개 이름으로 우리 `get_action()` dict를 인덱싱한다. **`.vel` / `.torque` 키가 없으면 `KeyError`로 record가 런타임 사망한다.**

VR/KER 텔레오퍼레이터에는 토크 소스가 없다. **값 규약:**

| 키 | 값 | 근거 |
|---|---|---|
| `{motor}.pos` | IK 해 (deg) | 실제로 실행되는 유일한 차원 (`openarm_follower.py:288` `goal_pos`가 `.pos`만 추출) |
| `{motor}.vel` | **0.0** | `send_action`이 vel을 0으로 하드코딩한다 (`:340`) |
| `{motor}.torque` | **0.0** | 동일. `action_features`가 `.torque`를 선언해도 **무시된다** |

```python
# openarm_follower.py:340
commands[motor_name] = (kp, kd, position_degrees, 0.0, 0.0)   # vel=0, tau=0 하드코딩
```
→ 데이터셋의 `action` 벡터에서 `.vel`/`.torque` 열은 **VR/KER 텔레옵 세션에서 항상 0**이다. 이것은 손실이 아니라 **실제로 명령된 값의 정직한 기록**이다. 힘 정보는 `observation.state`(팔로워 실측)에 남는다.

**키셋 (양완, `bi_openarm_follower.py:99-103`이 `left_`/`right_` 접두사를 붙인다):**
```
left_joint_1.pos  left_joint_1.vel  left_joint_1.torque
... (joint_1..joint_7, gripper) × (pos, vel, torque) × (left, right) = 48키
```
단위: `.pos` = **deg**, `.vel` = **deg/s**, `.torque` = **Nm** (`damiao.py:568`).

**리더·팔로워 플래그는 반드시 단일 스위치로 묶는다.** 팔로워만 True면 위 `KeyError`가 난다.

### 2.6 텔레옵 루프 — 구조·주파수·관측 접근

`lerobot_teleoperate.py:188-234` (`teleop_loop`):
```python
while True:
    loop_start = time.perf_counter()
    obs = robot.get_observation()
    if robot.name == "unitree_g1":          # :197-198  ← openarm은 하드코딩 배제
        teleop.send_feedback(obs)
    raw_action = teleop.get_action()        # 루프당 정확히 1회
    teleop_action = teleop_action_processor((raw_action, obs))      # CLI에서는 identity
    robot_action_to_send = robot_action_processor((teleop_action, obs))  # identity
    _ = robot.send_action(robot_action_to_send)
    dt_s = time.perf_counter() - loop_start
    precise_sleep(max(1 / fps - dt_s, 0.0))
```
- **`TeleoperateConfig.fps: int = 60`** (`lerobot_teleoperate.py:141`). → **스텝 예산 16.7 ms.**
- **`get_action()`은 블로킹하면 안 된다.** 포즈 수신은 별도 스레드에서 돌고 `get_action()`은 최신 샘플 스냅샷만 읽는다.
- **`send_feedback()`은 openarm에서 절대 호출되지 않는다** (§2.13).

**관측 접근 문제와 해법:** `get_action()`에는 `obs`가 안 들어오는데 IK는 `sync(measured16_rad)`가 필요하다(`openarm_control/kinematics.py:174`).

| 경로 | 해법 | 결과 |
|---|---|---|
| **우리 GUI 루프 (운용 경로)** | 루프가 `teleop.sync_state(obs)`(우리가 추가하는 **비추상** 메서드)를 `get_action()` 직전에 호출 | 실측 폐루프 |
| **LeRobot CLI (검증 경로)** | 호출 지점이 없다 → IK가 직전 해에서 **개루프 누적 적분** (`kinematics.py:235` `integrate_inplace`) | 팔로워가 클램프·포화되면 드리프트 |

→ CLI 경로는 **개루프임을 명시**하고, GUI 경로에서만 `sync_state()`를 보장한다.

### 2.7 Quest 3S 포즈 소스 — 상류 프로토콜이 전부 공개돼 있다 [확정]

> **v2에서 VR 텔레옵은 실기로 동작한다** — 근거는 v2 전용 튜토리얼 `tutorial/data-collection-vr.mdx`이고, MuJoCo 시뮬(`dataflow-vr-mujoco.yaml`)과 **실기 로봇**(`dataflow-vr.yaml`) 둘 다다. 시작 시 팔이 초기 자세로 이동하면 **트리거를 잡아 컨트롤러를 팔에 "정렬(alignment)"** 한 뒤 제어한다. 이 정렬은 **좌표 기준 정렬이지 모터 `set_zero`가 아니다** (§2.8, FR-TEL-026 — VR "영점"은 트리거 정렬이다).
> ⚠️ 공식 오버뷰 페이지 `teleop/vr`("NVIDIA Isaac Labs 시뮬 프로토타입 / ~20 FPS TCP / coming soon")는 **v1 문구가 v2 트리에 그대로 복붙된 죽은 페이지**다(v2·v1 바이트 동일). **정본이 아니며 근거로 쓰지 않는다.**
> Quest **3S**는 공식 문서·레포 어디에도 없다(전부 "Meta Quest 3", WebXR 경로만 PICO 4 추가). 3S 지원은 **우리 플랫폼 결정**이며 3S 실동작은 자체 검증한다(U-1).

#### 경로 A — Quest APK → UDP (`dora-openarm-vr`, Apache-2.0)

`src/dora_openarm_vr/quest_receiver.py:15-58`이 프로토콜을 전부 문서화한다.

| 항목 | 값 |
|---|---|
| 클라이언트 | Quest **3** 대상 **APK** (사이드로드, Google Drive 배포, **소스 비공개**) |
| 전송 | **UDP :5006**, 개행 종료 UTF-8 **평문 JSON**, 버퍼 4096 B (`udp_receiver.py:26,51-58`) |
| JSON 키 | `t, lc, rc, rf, lt, rt, lg, rg, lsx, lsy, rsx, rsy, a, b, x, y, v, vl, vr` |
| 확정 시맨틱 | `t`=타임스탬프 · `lt`/`rt`=트리거 아날로그 · `lg`/`rg`=그립 아날로그 · `lsx/lsy/rsx/rsy`=썸스틱 · `a/b/x/y`=버튼 · `v`/`vl`/`vr`=유효성 ∈ {**0=OK, 1=STALE, 2=INVALID**} |
| `lc`/`rc`/`rf` | 좌/우 컨트롤러 포즈 · `rf`는 레퍼런스 포즈로 추정 (NECK 레퍼런스 차감이 존재하므로). **정확한 시맨틱 [미확인]** |
| Unity LH→RH | `p = [x, y, -z]`, `q = [qw, -qx, -qy, qz]` |
| 프레임 체인 | 레퍼런스 포즈 차감(NECK) → `R_FRAME` 회전 → `FRAME_OFFSET_NECK = [0.1, 0, 1.2]` |
| 출력 | `float32[7] = [px, py, pz, qw, qx, qy, qz]` — **MuJoCo 월드 프레임**, 쿼터니언 **scalar-first** |
| 트리거→그리퍼 | 우 `(-1.57/2)*(1-trigger)` ∈ `[-0.785, 0]` rad · 좌 `(+1.57/2)*(1-trigger)` ∈ `[0, +0.785]` rad (`:71-76`) |

**⚠️ 리프터 계통 오차.** `openarm_control`은 `cell.xml`을 로드하고 리프터 DOF를 `mink.DofFreezingTask`로 **freeze**한다 (실측 `freeze_dofs = [0, 8, 9, 17, 18]` = 리프터 + 좌우 finger 2개씩, `kinematics.py:154-168`). → 선택지: ① 텔레옵 중 리프터를 물리적으로 `q_lift = 0`에 고정한다(가장 단순, 권장). ② `--xml`로 리프터 없는 씬(`openarm_bimanual_xml()`)을 쓴다(키프레임 `home` 존재 여부 **[미확인]**). ③ private `_IKSolver` configuration에 접근해 리프터 qpos를 주입한다. **LeRobot 경로에 리프터 제어 노드가 없다**(상류에서는 dora `cell-lifter` 노드가 담당했다).

### 2.9 IK = `openarm_control` (mink + MuJoCo + daqp) [확정, 실측 완료]

`pip install openarm_control` 한 줄이 `openarm-mujoco`(v2 MJCF+에셋), `mink`, `mujoco`, `daqp`, `pyyaml`을 전부 끌어온다. **ROS 불필요, dora 불필요.** 파일 4개가 전부다: `{__init__, config, kinematics, poses}.py`.

**공개 API (`kinematics.py:62-119`) — 소스 그대로:**

```python
class Kinematics:
    def __init__(self, setup: ArmSetup, ik_params: IKParams | None = None) -> None: ...
    def fk(self, side: str, joints: np.ndarray) -> np.ndarray: ...          # float[8] rad → float32[7]
    def fk_bimanual(self, right, left) -> tuple[np.ndarray, np.ndarray]: ...
    def set_target(self, side: str, pose: np.ndarray) -> None: ...          # float32[7] world
    def sync(self, values16: np.ndarray) -> None: ...                       # float32[16] rad
    def ready(self) -> bool: ...
    def solve(self) -> np.ndarray | None: ...                               # ★ 인자 없음 (:107)
    def set_gripper(self, side: str, value: float) -> None: ...
```

> 🔴 **`solve()`는 인자를 받지 않는다** (`kinematics.py:107`). **README의 `solve(dt=..., n_iters=...)`는 틀렸다.** `dt`/`max_iters`는 `IKParams`에만 있다. 문서 말고 소스를 따른다.

**16-벡터 레이아웃** (`openarm_mujoco/v2/joint_resolver.py:20-24`): `right[0:7]` = joint1..7, `right[7]` = gripper, `left[8:15]` = joint1..7, `left[15]` = gripper. **전부 rad.**

**`IKParams` 전 필드와 기본값 (`kinematics.py:46-59`):**
```python
position_cost: float = 1.0        orientation_cost: float = 1.0
lm_damping: float = 0.01          damping: float = 0.25      # 전역 Tikhonov
solver: str = "daqp"              posture_cost: float = 0.01 # 0 = 비활성
diag_reg: float = 0.0             dt: float = 0.1            # 적분 스텝(가상 시간)
max_iters: int = 5                velocity_limits: dict[str, float] | None = None   # None = VelocityLimit 비활성
```

**상류가 이미 검증한 튜닝값** (공식 VR dataflow `dora-openarm-vr/config/dataflow-mujoco.yaml`이 `openarm_control`을 bimanual로 실제 사용):
```
--mode bimanual --max-iters 10 --dt 0.1 --damping 0.1 --posture-cost 0.01 --lm-damping 0.01
```
VR 노드의 출력 포즈 프레임(`FRAME_OFFSET_NECK`, MJCF 월드)이 `set_target()` 규약과 정확히 일치한다. **"VR 포즈 → openarm_control IK"는 상류가 이미 검증한 계약이다.**

**내부 구성 (`_IKSolver.__init__`, `kinematics.py:128-183`):**

| 메커니즘 | 라인 | 내용 |
|---|---|---|
| 양팔 단일 QP | `:136,145-146` | 하나의 `mink.Configuration` + 좌/우 `mink.FrameTask` 동시 |
| 여유자유도 | `:175-176` | `mink.PostureTask(cost=posture_cost)`, 타깃 = **`home` keyframe 고정** |
| 조인트 리밋 | `:170` | `mink.ConfigurationLimit(setup.model)` — **MJCF `jnt_range`만 본다** |
| 속도 리밋 | `:172-173` | `velocity_limits`가 not None일 때만 `mink.VelocityLimit` |
| DOF 동결 | `:154-168` | `mink.DofFreezingTask` — 리프터·손가락 |
| LM / Tikhonov | `:143`, `:178` | task별 `lm_damping`, 전역 `damping` |

**명시적 널스페이스 프로젝션 API는 없다.** `PostureTask` 타깃이 항상 `home`으로 고정되고(`:138,176`) `_posture_task`는 private이라 **공개 API로 팔꿈치 자세를 못 바꾼다.**

### 2.10 🔴 IK 안전 3중 함정 — 어댑터가 반드시 막아야 한다

#### (a) 조인트 리밋 불일치 — `mink.ConfigurationLimit`은 MJCF만 본다

MJCF: `openarm_bimanual.xml:269-313`(right) / `:181-225`(left). LeRobot: `config_openarm_follower.py:22-43`.

| 조인트 | MJCF right (rad → deg) | LeRobot right (deg) | 판정 |
|---|---|---|---|
| joint1 | −1.3963 … 3.4907 → **−80° … +200°** | **−75 … +75** | MJCF가 압도적으로 넓음 |
| joint2 | −0.17453 … 3.3161 → −10° … +190° | −9 … +90 | MJCF가 넓음 |
| joint3 | ±1.5708 → ±90° | ±85 | 약간 넓음 |
| joint4 | 0 … 2.4435 → 0° … +140° | 0 … +135 | 약간 넓음 |
| joint5 | ±1.5708 → ±90° | ±85 | 약간 넓음 |
| joint6 | ±0.7854 → **±45°** | ±40 | 약간 넓음 |
| joint7 | ±1.5708 → ±90° | ±80 | 약간 넓음 |
| gripper | 우 finger1 −0.7854…0 / 좌 finger1 0…+0.7854 | **−65 … 0** | **규약·스케일 자체가 다름** (§2.11) |

→ **IK가 LeRobot이 클램프할 해를 정상적으로 낸다.**
**해결 (순서가 중요):** `ArmSetup.from_args(...)` 생성 → **`setup.model.jnt_range[jid] = (lo_rad, hi_rad)`를 LeRobot 리밋으로 덮어쓰기** → `Kinematics(setup, IKParams(...))` 생성. `ConfigurationLimit`이 **생성 시점에 model을 읽으므로** 순서를 뒤집으면 무효다.

#### (b) 폴백이 리밋을 통째로 버린다

```python
# openarm_control/kinematics.py:220-231
try:    vel = mink.solve_ik(..., limits=self._limits, ..., safety_break=False, ...)
except mink.exceptions.NoSolutionFound:
    try: vel = mink.solve_ik(..., limits=[], ...)      # 🔴 리밋을 버리고 재시도
    except mink.exceptions.NoSolutionFound:
        print("Warning: IK solver failed ...")          # → None 반환
        return None
```
`DofFreezingTask`(constraints)는 유지되지만 **`ConfigurationLimit`·`VelocityLimit`이 사라진다.**
→ **어댑터 레벨 클램프가 1차 방어선**이어야 한다. LeRobot의 `max_relative_target`은 기본 `None`(꺼짐)이고 `joint_limits` 클램프만으로는 속도를 못 막는다.

#### (c) 단위 — `openarm_control` = **rad**, LeRobot = **deg**

MJCF `<compiler angle="radian">` (`cell.xml:18`). LeRobot `MotorNormMode.DEGREES` (`openarm_follower.py:54-56`), 액션도 deg (`damiao.py:440`에서 `np.radians()` 변환). **변환 누락 시 조용한 오동작.**

#### (d) 속도 캡 변환식 — `IKParams`를 직접 만들 때의 함정

기본 캡 (rad/s, `openarm_control/config.py:40-48`): `[1.57, 1.57, 3.14, 3.14, 12.6, 12.6, 12.6]` (J1·J2=DM8009, J3·J4=DM4340, J5·J6·J7=DM4310).

```python
# kinematics.py:261-269
def _convert_velocity(rad_per_sec, dt, max_iters, tick_hz) -> float:
    return rad_per_sec / (max_iters * dt * tick_hz)
```
`register_ik_args`/`ik_params_from_args` 경로는 rad/s를 이 식으로 변환해 `IKParams.velocity_limits`에 넣는다. **`IKParams`를 직접 구성하면서 rad/s를 그대로 넣으면 캡이 `max_iters × dt × tick_hz` 배 어긋난다.**
`velocity_limits` 키는 **MJCF 조인트 이름** `openarm_left_joint1` … `openarm_right_joint7` (`kinematics.py:375`).
🔴 **`--tick-hz` 기본값은 500.0이고 help 문구가 *"Dora tick rate in Hz; must match the dataflow timer"*다** (`kinematics.py:361-366`). **dora 잔재다. 우리 루프 fps(기본 60)로 반드시 바꿔야 한다.** 500을 남기면 반복당 캡이 8.3배 과하게 조여져 팔이 기어간다.

#### (e) 기타

- `solve()`는 `integrate_inplace`로 **누적 적분**한다(`:235`). seed는 "직전 solve 결과"이지 로봇 실측이 아니다. → **매 사이클 `sync()` 필수**(§2.6).

### 2.11 그리퍼 매핑

**v2 그리퍼는 pinch 그리퍼다 [v2 확정].** 2지 mimic(finger2 = finger1 ×1), **인핸드 카메라 내장, 교체형 핑거**, 컴팩트한 단순 액추에이션. **v1의 링키지 병렬·88 mm 스펙은 무효다** (whats-new-in-2.0 대조표). 팔로워의 **네이티브(드라이버) 그리퍼 제어 모드는 POS_FORCE**(`gripper_posforce_limits` — 속도항은 **DM4310 vMax = 30 rad/s 이하**로 제한하고, 힘항은 **무차원 `torque_pu ∈ [0,1]`**(드라이버 내부 `/4.5` 스케일, TODO 잔존)다. 상류의 '50 rad/s, 1.0 Nm' 표기는 근거가 없다 — 속도 50은 DM4310 vMax 30을 초과하고 '1.0 Nm'은 실측 근거가 없다. **Nm/N 파지력 표기는 실기 파지력 캘리브레이션 이후에만** 쓴다 → [개정 2026-07-15])이고, 손으로 잡은 **open/close 엔드포인트를 rad로 디스크에 영속 캡처**한다(참조레포 `capture_gripper_endpoint` 패턴 — 정본 관리는 `03-모터-설정.md` / 캡처 플로우는 `02-로봇-연결-및-온보딩.md`). 아래 텔레옵 매핑은 트리거 → rad → deg 지령을 **이 캡처된 가동 범위 안으로** 넣는 것이며, LeRobot 팔로워 경로에서는 그 지령이 MIT 배치로 실행된다(§2.5).

**트리거 → 그리퍼 각 (`quest_receiver.py:71-76`):**


| ⚠️ 주석/코드 불일치 | 주석은 ±1.57을 의도하나 `/2.0` 때문에 ±0.785만 낸다 |

### 2.14 안전 — 하드웨어 사실과 LeRobot의 빈약함

**🚨 LeRobot 안전장치는 `joint_limits` + `max_relative_target`(기본 **`None` = 꺼짐**)이 전부다. 속도 제한 없음. 가속도/저크 제한 없음. 소프트웨어 E-Stop 없음.** → 안전 계층은 전부 우리가 만든다.

**🚨 OpenArm에는 홀딩 브레이크가 없다.** 인에이블 이탈 = 토크 0 = **중력 낙하**. 공식 Safety Guide: *"if power is lost due to an emergency stop, the load being held will fall rapidly."*

**🚨 모터의 통신 상실 보호는 "안전 홀드"가 아니라 "안전 낙하"다.**
Damiao 매뉴얼 원문: *"通讯丢失防护，设定周期内没有收到 CAN 指令将自动退出'使能模式'"* (설정 주기 내에 CAN 명령을 받지 못하면 자동으로 인에이블 모드를 벗어난다). 레지스터 **RID 9 `TIMEOUT`** (RW, uint32).

> 🔴 **런타임 변경이 이 위험을 키운다.** dora 경로는 500 Hz(2 ms 주기)로 명령을 보냈다. **LeRobot 텔레옵 루프 기본값은 60 Hz = 16.7 ms 주기**이고, 기록 fps를 30으로 낮추면 **33 ms**다. **RID 9 `TIMEOUT` 공장 기본값이 이보다 짧으면 매 프레임 인에이블이 풀려 팔이 떨어진다.** RID 9 기본값·단위는 **[미확인]**이며 **기동 전 확인이 필수다** (§5 U-3). 확인 전까지는 fps를 낮추는 어떤 변경도 안전 검토 없이 하지 않는다.

→ **어떤 상태에서도 CAN 명령 스트림을 끊지 않는다.** "홀드"는 반드시 **현재 실측 자세를 목표로 하는 `send_action({..., f"{m}.pos": q_measured_deg})`의 지속 송신**으로 구현한다 (→ `_mit_control_batch(kp, kd, q, 0.0, 0.0)`).

**정지 유형 — 반드시 분리 (IEC 60204-1, [개정—검토권고 B3]):**

> ⚠️ **Cat-2는 controlled stop이지 비상정지 범주가 아니다** — 소프트웨어 홀드를 "Cat-2 E-Stop"·"SW E-Stop"으로 부르지 않는다. 인증 하드웨어가 없으므로 "safety-rated" 표현도 쓰지 않는다.

| 유형 | 명칭 | 동작 | 낙하? |
|---|---|---|---|
| **operational hold (기본)** | SW 정지 명령 (비인증 controlled stop) | 즉시 `send_action(q = q_measured)` 홀드. **전원·CAN 유지** | ❌ |
| **emergency stop Cat-0** | HW E-Stop (전원 라인 삽입 외부 버튼) | 모터 전원 물리 차단 | ✅ **낙하** |

> **GUI의 정지 버튼은 operational hold에 매핑한다.** 그렇지 않으면 "정지 버튼을 눌렀더니 팔이 떨어지는" 반직관이 발생한다. 진짜 비상정지(전원 차단→낙하)는 별도 HW 버튼이다.

**관절 리밋 — 3세트가 전부 실재. 정본은 v2 URDF [v2 확정]:**

| 출처 | joint1 | joint4 | joint6 | 성격 |
|---|---|---|---|---|
| **MJCF `jnt_range` = v2 URDF `joint_limits.yaml`** (`mink.ConfigurationLimit`이 강제) | −80° … +200° | 0° … +140° | ±45° | **기계 정본** |
| LeRobot `config_openarm_follower.py:22-43` | **−75° … +75°** | 0° … +135° | ±40° | *"the default joint limit values are small for safety"* — **보수적 소프트 클램프** |
| `openarm_driver/configs/openarm_cell.yaml` (드라이버 운영 리밋, 좌우 비대칭) | 우 `[-1.3, 2.0]` / 좌 `[-2.0, 1.396]` rad | `[0, 2.4435]` rad | ±0.7854 rad | **운영 리밋** (LeRobot 경로에서는 미사용) |

→ **정본은 v2 URDF `joint_limits.yaml`(= MJCF `jnt_range`)다** (GROUND d3 — j2 = −10°…+190° 등 v2 값이며 v1 ±100°에서 변경됨). LeRobot deg(±75° 등)와 드라이버 per-side 리밋은 **URDF에서 파생된 보수적 운영/소프트 클램프**일 뿐 **별개의 정본이 아니다.** **텔레옵 경로는 IK QP(MJCF) → LeRobot `joint_limits`로 실제로 두 번 클램프되므로**, 두 지점을 URDF 파생 **단일 운영 세트**로 통일하지 않으면 **IK가 낸 합법 해가 팔로워에서 다시 잘린다.** §2.10(a)의 `jnt_range` 덮어쓰기(FR-TEL-044)가 이 통일을 강제한다. 남는 자유도는 정본 선택이 아니라 **"운영 클램프를 URDF 대비 얼마나 보수적으로 좁힐지"라는 안전 튜닝값**뿐이다.

**⚠️ joint6이 ±0.7854 rad(±45°)로 매우 좁다.** 컨트롤러 자세를 1:1로 매핑하면 상시 리밋에 걸린다 → **회전 스케일을 위치 스케일과 분리해야 한다**(FR-TEL-029).
**⚠️ joint4(팔꿈치)는 하한 0** → 음의 굽힘 불가.
**⚠️ 관절 명령은 `±PMAX = 12.5 rad`를 절대 넘으면 안 된다.** Damiao 피드백 위치는 16-bit를 `[−PMAX, +PMAX]`로 매핑하고 그 밖에서 **wrap**한다.

**모터 토크 상한** (`motors/damiao/tables.py:98-112`, `(pmax rad, vmax rad/s, tmax Nm)`): DM4310 `(12.5, 30, 10)` · DM4340 `(12.5, 8, 28)` · DM8009 `(12.5, 45, 54)`. OpenArm 매핑: J1·J2 = DM8009(±54 Nm), J3·J4 = DM4340(±28 Nm), J5·J6·J7·gripper = DM4310(±10 Nm).

**CAN 패킷 드랍**: `_batch_refresh`가 **마지막 알려진 상태를 그대로 재사용**하고 `logger.warning`만 뱉는다 (`damiao.py:696-702`). 토크는 위치보다 노이즈가 크고 드랍의 영향을 크게 받는다 → **드랍 카운터를 GUI에 노출해야 한다.** LeRobot은 이를 feature로 내보내지 않는다.

**추적 소실 — 상류 거동:**

| 경로 | 유효성 모델 | 소실 시 동작 |
|---|---|---|
| UDP (APK) | `v`/`vl`/`vr` ∈ {0=OK, 1=STALE, 2=INVALID} | INVALID → 포즈 발행 중단 + 스무더 `reset()`. **정지 신호는 나가지 않는다.** STALE은 **그대로 통과** |
| WebXR | **없음** | `inputSources.length < 2`면 프레임을 버릴 뿐. `reset()`이 호출되지 않는다 |

> 🚨 **STALE(1)은 하류에서 "정상 정지 상태"와 구분할 수 없다** (HMD가 마지막 양호 포즈를 계속 보내므로). → **보수적으로 소실 취급한다** (`treat_stale_as_lost = true`).
> 🚨 **하트비트/타임아웃 코드가 상류 어디에도 없다.**

### 2.15 CAN 소유권 (텔레옵 관점)

- **LeRobot Python API를 단일 GUI 프로세스에 임베드하면** 모드 간 배타성은 프로세스 내부에서 해결된다. 양팔도 한 프로세스에 버스 객체 2개(`bi_openarm_follower.py:93-94`, `utils/bimanual.py:47-63` 순차 동기 호출).
- **그러나 커널이 배타성을 강제하지 않는다.** SocketCAN RAW = `socket(PF_CAN, SOCK_RAW, CAN_RAW)` + `bind()`, 배타 플래그 없음. `cansend`, `openarm-can-cli`, 두 번째 LeRobot 인스턴스, `openarm_teleop`(C++)이 전부 무방비로 끼어들고 **에러가 나지 않는다.** → **`flock` 애플리케이션 락이 여전히 필요하다.** 역할이 "모드 중재"에서 **"외부 침입자 방어"**로 축소될 뿐이다.
- 배타성의 단위가 "버스 소유"에서 **"누가 `send_action()`을 부를 권리를 갖는가"**로 내려간다. **상태머신은 그대로 필요하다.**
- **`bitrate`/`data_bitrate` 인자는 python-can socketcan 백엔드에서 무시된다** → `ip link`로 CAN-FD를 켜는 단계가 **필수**이고 코드가 대신할 수 없다. 텔레옵 기동 전 `ip -details link show`로 검증한다. (상세는 `01-시스템-아키텍처.md`.)

## 4. 상태·오류 처리

### 4.1 텔레옵 상태 전이표 (11상태)

표기: `qh` = 진입 시각의 **실측** 관절각(홀드 목표, deg) · `Δt_hb` = 마지막 유효 VR 프레임 이후 경과 시간 · `f` = 루프 fps(기본 60).
**전제: `robot.connect()`는 세션당 1회 이미 끝났다. 어떤 상태도 재연결하지 않는다.**

| # | 상태 | 모터 출력 (매 틱, **항상 송신**) | 진입 조건 | 이탈 조건 → 다음 상태 |
|---|---|---|---|---|
| **S0** | **IDLE** | 텔레옵 비활성. 팔로워는 `send_action(q=qh)` 홀드 유지. `disable_torque()`는 **rest 자세에서만** 허용 (낙하) | 부팅 후 connect 완료 / `stop` 완료 | 사용자 `start` → **S10** |
| **S10** | **RECOVERY_CHECK** | **enable 금지.** 명령 송신 없음 (읽기만) | S0에서 start / E-Stop 해제 / 전원 재인가 후 | ① 전 관절 실측 위치 읽기 ② `joint_limits` 범위 검사 ③ 영점 유효성 대조 ④ 낙하 자세 자기충돌 검사 ⑤ **3D로 실측 자세 표시 + 사용자 명시 확인** → 전부 통과 → **S1** · 하나라도 불일치 → **S0** (재영점 필요 경보) |
| **S1** | **HOMING** | 저속 램프(`duration ≥ 4 s`). **첫 명령의 `q_des`는 반드시 실측 `q`.** 낮은 kp(정상의 30%)로 홀드 → kp 램프업 | S10 통과 | 홈 도달 → **S2** · 델타 위반 / CAN 오류 → **S8** |
| **S2** | **HOLD** | `send_action(q = qh)` **지속 송신** | S1 완료 / S5·S6·S7 해제 | VR 링크 OK **&&** 클러치 성립 → **S3** · `stop` → **S9** · E-Stop → **S8** |
| **S3** | **ALIGNING** | `align_target += clip(q_target − align_target, ±(align_rate/f))` → `send_action` | S2에서 클러치 성립 / S5·S6·S7 복귀 | 전 관절 `\|q_target − q\| < align_threshold(0.1 rad)` → **S4** · 클러치 해제 → **S6** · `Δt_hb > hb_timeout` → **S5** · IK 실패 → **S7** · E-Stop → **S8** |
| **S4** | **FOLLOWING** | `send_action(q_ik)` (LeRobot 리밋 클램프 + `max_relative_target` + 워크스페이스 클램프 + 속도 클램프 통과) | S3에서 정렬 수렴 | ① 발산(`any(\|q_new − q_last\| > align_threshold)`) → **S3** (강등) ② 클러치 해제 → **S6** ③ `Δt_hb > hb_timeout` 또는 validity INVALID/STALE → **S5** ④ IK 잔차/폴백/워크스페이스 위반 → **S7** ⑤ 토크 상한 초과 → **S7** ⑥ E-Stop → **S8** ⑦ `pause` → **S6** |
| **S5** | **LINK_LOST** | 🔴 **명령 송신을 절대 멈추지 말 것.** 감속 후 `send_action(q = qh)` 홀드 | `Δt_hb > hb_timeout` (기본 **100 ms**) · validity **INVALID(2)** · validity **STALE(1)** (`treat_stale_as_lost = true`) | VR 복귀 **&&** 사용자 **명시적 재-engage** → **S3**. **자동 재개 금지** |
| **S6** | **PAUSED / CLUTCH_OUT** | `send_action(q = qh)` 홀드. **기준점 파기** + One Euro 스무더 `reset()` | 클러치 해제(grip < 임계) · 사용자 pause | 재-engage(grip > 임계) → **S3** (**반드시 재정렬을 거친다. S4 직행 금지**) |
| **S7** | **IK_FAULT** | `send_action(q = qh)` 홀드 + HUD 경고. **마지막 유효 관절각 유지** | ① `solve() → None` ② EE 잔차 > `ik_residual_max_m`(0.03 m) 또는 회전 잔차 > `ik_residual_max_rad`(0.15 rad) ③ limit-free 폴백 발동 ④ 해가 LeRobot 리밋에 클램프됨 ⑤ 워크스페이스 가상벽 위반 | 잔차가 임계 × 0.5 아래로 `ik_recover_ms`(기본 **50 ms**) 연속 복귀 → **S3** · 사용자 abort → **S2** |
| **S8** | **ESTOP** | **operational hold(기본)**: `send_action(q = qh)` 홀드 — 전원·CAN 유지, **낙하 없음** · **emergency stop Cat-0(물리 버튼)**: 전원 차단 → **낙하** | SW 정지 버튼 / 컨트롤러 버튼 조합 / HW E-Stop / 스텝 점프 위반 / CAN 버스 오류 | 물리적 리셋 + 사용자 확인 → **S10** |
| **S9** | **STOPPING** | 홈으로 저속 이동 → (선택) `disable_torque()` → **낙하** | 사용자 `stop` | 완료 → **S0** |

**⚠️ `disable_torque()`는 물리적으로 팔을 떨어뜨린다.** 브레이크가 없다. → **낙하 없이 세션만 끝내는 "홈 홀드 종료"(disable 없이 홈에서 `send_action` 홀드 유지)를 기본 종료 거동으로 제공하고, `disable`은 명시적 옵션으로만 노출한다.**

## 5. 미해결·결정필요

### 5.1 [결정필요] — 충돌하는 사실이 전부 실재하여 사람이 골라야 함

> **라벨 규약**: `Q-n`은 **이 문서 로컬 결정 라벨**이다. **전역 결정 등록부는 `16`의 `D-n`/`M-n`**이며 번호 체계가 다르다(로컬 `Q-n` ≠ 전역 `D-n`). 전역 대응: **Q-1·Q-7 → `16` M-22**(VR 경로/영점), **Q-2 → `16` D-3**(관절 리밋 정본), **Q-3 → `16` D-5**(그리퍼 스펙), **Q-4 → `16` M-16**(관절/EE 속도 한계), **Q-6 → `16` M-21**(바이래터럴·제어 주파수). Q-5·Q-8은 전역 대응이 없는 로컬 결정이다. `[미확인]`은 §5.2의 `U-n`으로 별도 관리한다.

| # | 질문 | 무엇을 확인/측정해야 하는가 |
|---|---|---|
