# 01. 시스템 아키텍처 및 런타임 경계

> 요구사항 ID 영역 코드: **`SYS`** (`FR-SYS-nnn` / `NFR-SYS-nnn`)
> 이 문서는 전체 기능 명세의 기반 문서다. 다른 영역 문서(텔레옵·데이터셋·학습·GUI·운영)는 여기서 확정한 **프로세스 경계 · 세션 생명주기 · CAN 소유권 · 모드 상태머신**을 전제로 한다.

---

## 1. 범위

이 문서가 다루는 것:

- **런타임 결정**: LeRobot Python API를 임베드한 **단일 헤드리스 백엔드 프로세스(FastAPI)**. dora-rs는 쓰지 않는다.
- **세션 생명주기**: `connect()` 1회 원칙과 그 근거(영점 파괴). 이것이 이 문서의 **최상위 제약**이다.
- **CAN 소유권**: `flock` 배타 락(역할 = 외부 침입자 방어), `ip link` 링크 상태 검증, 채널↔팔 매핑.
- **모드 상태머신과 배타성**: 모드는 프로세스 재시작이 아니라 **같은 robot 객체 위의 상태 전이**다.
- **LeRobot 확장 경로**: 서드파티 플러그인 메커니즘(포크 불필요).
- **우리가 만들 구성요소 전수표**와 각각의 근거.
- **아키텍처 리스크**: 바이래터럴 500 Hz 미달 시 CAN 배타 소유권 문제의 부활.
- **경계 계약**: 브라우저(웹 SPA) ↔ 백엔드(실시간 **WebSocket** + 비실시간 **HTTP REST**), 백엔드 ↔ 원격 추론 서버(gRPC). 백엔드 내부(제어 루프 ↔ robot ↔ CAN)는 인프로세스 함수 호출이다.

이 문서가 다루지 **않는** 것: IK 파라미터 튜닝(→ `05`), VR 좌표변환·클러치(→ `05`), 모터 프로토콜·게인·리밋 전수(→ `03`), 데이터셋 스키마(→ `07`/`08`), 안전 계층 알고리즘(→ `12`), 학습 하이퍼파라미터(→ `10`), GUI 화면(→ `13`). 단 **경계에 걸리는 계약**(플래그 동기화, WS/REST 계약, 큐/워치독, 스레드·프로세스 소유권)은 여기서 규정한다.

**아키텍처 = 웹 단일.** 프론트엔드는 브라우저에서 도는 **웹 SPA**, 백엔드는 **헤드리스 FastAPI 서비스**다. 브라우저 ↔ 백엔드는 **단일 WebSocket(실시간 제어·텔레메트리·카메라) + HTTP REST(비실시간 CRUD)** 로 연결되며(§2.16), gRPC는 **백엔드 ↔ 원격 추론(async PolicyServer :8080)** 홉 하나에만 쓴다. PyQt6 데스크톱안은 폐기됐다. connect 1회·모드=상태전이·flock CAN 배타 소유·CLI 스폰 금지는 **백엔드 프로세스 내부의 불변식으로 그대로 유효**하다.

## 2. 확정된 사실 기반

### 2.1 런타임 = LeRobot. dora는 쓰지 않는다. 하위 라이브러리는 쓴다 [확정]

enactic의 실제 아키텍처는 **레이어드**다. 핵심 라이브러리(`openarm_can` C++/apt, `openarm_driver` pip, `openarm_control` pip, `openarm_ker` pip, `openarm_teleop` C++)가 **전부 dora 의존성 0**이고, `dora-*` 노드 다수는 그 위의 **얇은 래퍼**다. `dora-openarm`의 본체는 `dora.Node()` + `SingleArmDriver`가 전부이며, `dora-openarm-vr`의 `quest_receiver.py`(≈200줄)는 dora 의존이 `node.send_output` 3줄뿐이고 나머지는 순수 numpy/scipy다.

> **dora는 배선(wiring)이지 로직이 아니다.** Apache-2.0이므로 로직은 그대로 이식 가능하다.

**LeRobot에서 OpenArm은 1급 로봇이다** — `make_robot_from_config`에 하드코딩 분기가 있다(`robots/utils.py:63-68`: `openarm_follower`, `bi_openarm_follower`). `lerobot_calibrate.py` / `lerobot_teleoperate.py` / `lerobot_record.py` / `lerobot_rollout.py`가 전부 openarm을 import한다.

**우리가 사용하는 하위 라이브러리 (dora 없이 직접):**

| 라이브러리 | 용도 | CAN 접근 | 비고 |
|---|---|---|---|
| `openarm_control` (pip, `0.1.0`) | **FK/IK** (mink + MuJoCo + daqp) | 없음 | `05`/`04`가 소비. 테스트 디렉터리 **없음** |
| `openarm_ker` (pip, `0.2.0`) | KER 리더암 스트림 | **없음 (USB)** | §2.7 |
| `openarm_mujoco` (pip) | MJCF v2 자산 | 없음 | `openarm_control` 의존 |
| `openarm_driver` (pip) | 고수준 CAN 드라이버 | **있음** | ⚠️ **정본 경로에서 쓰지 않는다.** LeRobot `DamiaoMotorsBus`와 **CAN 이중 바인드**가 되므로 같은 프로세스에서 함께 열어서는 안 된다 |
| `openarm_teleop` (C++) | 바이래터럴 참조 구현 | **있음** | §2.11. 별도 프로세스이며 LeRobot과 **동시 실행 불가** |

### 2.2 🔴 `connect()`가 영점을 파괴한다 — 이 문서의 최상위 제약 [확정]

```python
# lerobot/robots/openarm_follower/openarm_follower.py:152-153
if self.is_calibrated:
    self.bus.set_zero_position()      # → CAN_CMD_SET_ZERO
```

```python
# lerobot/motors/damiao/damiao.py:335-340
def set_zero_position(self, motors: str | list[str] | None = None) -> None:
    """Set current position as zero for selected motors."""
    target_motors = self._get_motors_list(motors)
    for motor in target_motors:
        self._send_simple_command(motor, CAN_CMD_SET_ZERO)
```

`CAN_CMD_SET_ZERO`는 **현재 물리 자세를 모터의 영점으로 확정**하는 명령이다. `connect()`의 독스트링도 이를 전제한다 — `openarm_follower.py:133-134`: *"We assume that at connection time, the arms are in a safe rest position."*

**결과**: 팔이 rest 자세가 **아닌** 순간에 재연결하면 **그 자세가 영점이 된다.** 관절 리밋(`joint_limits`)과 데이터셋 좌표계가 통째로 어긋나고, 이미 수집한 에피소드와 새 에피소드가 **다른 좌표계**를 갖게 된다. 이 오류는 예외를 내지 않는다 — 조용히 잘못된 데이터를 만든다.

> **⇒ 세션당 `connect()`는 정확히 1회. 모드 전환(수동 ↔ 텔레옵 ↔ 기록 ↔ 추론)에 disconnect/reconnect를 절대 하지 않는다. 모드는 같은 robot 객체 위의 상태 전이다.** (FR-SYS-001 ~ 004)

**연관 사실 — 홀딩 브레이크가 없다.** 인에이블 이탈 = 토크 0 = **중력 낙하**. 공식 Safety Guide: *"if power is lost due to an emergency stop, the load being held will fall rapidly."* E-Stop은 전원 라인에 삽입되는 **외부** 물리 버튼이며 소프트웨어 계층이 아니다. (상세 → `12`)

그리고 LeRobot은 record와 추론을 **의도적으로 분리**한다 — `lerobot_record.py:193-197`: *"For policy-based deployment, use lerobot-rollout instead."* / `:434-436`: *"lerobot-record is for data collection only."*

**⇒ 금지. 우리는 LeRobot Python API를 단일 백엔드 프로세스에 임베드한다.** (FR-SYS-002)

**임베드할 API 표면 (소스 확인):**

### 2.4 CAN 소유권 — 축소되지만 사라지지 않는다 [확정]

**✅ 축소되는 부분**: LeRobot Python API를 단일 프로세스에 임베드하면 **모드 간 배타성은 프로세스 내부에서 해결된다.** 양팔도 한 프로세스에 버스 객체 2개다 — `bi_openarm_follower.py:93-94`:

```python
self.left_arm  = OpenArmFollower(left_arm_config)
self.right_arm = OpenArmFollower(right_arm_config)
```

`utils/bimanual.py:47-63`의 `BimanualMixin`은 `connect` / `disconnect`를 **동일 프로세스·동일 스레드·순차 동기 호출**로 처리한다. `send_action`도 순차다(`bi_openarm_follower.py:158-159`).

**❌ 사라지지 않는 부분 — 커널이 배타성을 강제하지 않는다.**

SocketCAN RAW = `socket(PF_CAN, SOCK_RAW, CAN_RAW)` + `bind()`. **배타 플래그가 없다.**

| 스택 | 소켓 생성 | 배타 제어 |
|---|---|---|
| python-can socketcan (LeRobot이 쓰는 것) | `socketcan.py:582` `socket(PF_CAN, SOCK_RAW, CAN_RAW)` → `:601` `sock.bind((channel,))` | ❌ 없음 |
| `openarm_can` (C++) | `can_socket.cpp:40` 동일 → `:69` `bind()` | ❌ 없음 |

Linux 커널 문서(https://docs.kernel.org/networking/can.html) 원문:
> - "**Multiple sockets can be opened at the same time** … they can listen/send frames on different or the same CAN IDs."
> - "To meet multi user needs **the local loopback is enabled by default**."
> - "The reception of the CAN frames on the same socket that was sending the CAN frame … is therefore **disabled by default**." (= `CAN_RAW_RECV_OWN_MSGS` off)

이 두 기본값의 조합이 **조용한 실패**를 만든다: `bind()`는 둘 다 성공하고(`EADDRINUSE` 없음), 두 프로세스의 MIT 명령이 물리 버스에서 인터리빙되며, 각자 상대방의 송신 프레임까지 자기 RX 큐에서 받아 상태 추정이 오염된다. **에러도 로그도 나지 않는다.**

**침입 가능한 주체 (전부 무방비):** `cansend` · `openarm-can-cli` · 두 번째 LeRobot 인스턴스 · `openarm_teleop`(C++) · `openarm_driver` 직접 사용 스크립트.
(`candump`는 읽기만 하므로 물리적으로 안전하나, 그건 커널 강제가 아니라 candump의 선의다.)

> **⇒ `flock` 애플리케이션 락은 유지한다. 다만 역할이 "모드 중재" → "외부 침입자 방어"로 축소된다.**
> **⇒ 배타성의 단위가 "버스 소유"에서 "누가 `send_action()`을 부를 권리를 갖는가"로 내려간다. 상태머신은 그대로 필요하다.** (FR-SYS-005 ~ 009)

`flock`은 fd 닫힘 시 커널이 자동 해제하므로 stale lock이 없다 — PID 파일보다 우월하다. `/var/lock`(= `/run/lock`, `1777`)은 일반 사용자도 쓸 수 있다.

### 2.5 `ip link`로 CAN-FD를 켜는 것은 **필수**다 — 코드가 대신할 수 없다 [확정]

LeRobot config는 CAN-FD를 켜라고 지시한다 — `config_openarm_follower.py:59-64`:

```python
can_interface: str = "socketcan"
use_can_fd: bool = True
can_bitrate: int = 1000000        # Nominal bitrate (1 Mbps)
can_data_bitrate: int = 5000000   # Data bitrate for CAN FD (5 Mbps)
```

그리고 `damiao.py:169-193`이 이를 `can.interface.Bus(**kwargs)`로 넘긴다.

**그러나 python-can socketcan 백엔드는 `bitrate` / `data_bitrate`를 무시한다** — `can/interfaces/socketcan/socketcan.py:694-703`의 `SocketcanBus.__init__` 시그니처에 **두 인자가 모두 없다**:

```python
def __init__(
    self,
    channel: str = "",
    receive_own_messages: bool = False,
    local_loopback: bool = True,
    fd: bool = False,
    can_filters: CanFilters | None = None,
    ignore_rx_error_frames=False,
    **kwargs,          # ← bitrate / data_bitrate가 여기로 흘러가 BusABC까지 가고 소켓에 아무 영향도 주지 않는다
) -> None:
```

소켓에 실제로 작용하는 것은 `fd`뿐 — `socketcan.py:765-769`:
```python
if fd:
    self.socket.setsockopt(constants.SOL_CAN_RAW, constants.CAN_RAW_FD_FRAMES, 1)
```

**함의**: 링크를 CAN 2.0으로 올려놓고 LeRobot이 `fd=True`로 열면 **python-can은 조용히 성공하고 통신만 깨진다.**

링크 설정의 정본은 `lerobot-setup-can`이다 — `lerobot_setup_can.py:96-121`:
```python
subprocess.run(["sudo", "ip", "link", "set", interface, "down"], ...)
cmd = ["sudo", "ip", "link", "set", interface, "type", "can", "bitrate", str(bitrate)]
if use_fd:
    cmd.extend(["dbitrate", str(data_bitrate), "fd", "on"])
subprocess.run(cmd, ...)
subprocess.run(["sudo", "ip", "link", "set", interface, "up"], ...)
```

> ⚠️ enactic의 `openarm-can-configure-socketcan-4-arms` 스크립트는 **`FD_MODE` 기본값이 `false`**이고 can0~can3을 똑같이 설정할 뿐 역할을 배정하지 않는다. **이 스크립트는 쓰지 않는다.** LeRobot이 동등물을 내장한다.

> **⇒ 백엔드는 기동 시 `ip -details link show <ch>`를 파싱해 `fd on` · `bitrate 1000000` · `dbitrate 5000000`을 검증하고, 불일치면 기동을 거부해야 한다.** (FR-SYS-006)

### 2.6 어댑터 · 채널 매핑 [확정 / 일부 미확인]

**Pibiger SavvyCAN-FD-X2 (USB2CANFD-X2)** — 2채널, **High-Speed USB 2.0**, 소프트웨어 제어 120 Ω 종단 내장, 1 µs 타임스탬프, 절연 2500 V. 벤더가 배포하는 `For_Linux_SocketCAN/` 트리에 **커널 모듈 소스가 없다** → **in-tree 드라이버로 동작**한다는 뜻. **BOM 수량 2 → 총 4채널.**

**USB 링크 대역폭은 병목이 아니다.** 2채널 × 8모터 × 2프레임 × 40 B @ 1 kHz ≈ **10.2 Mbps** vs High-Speed USB 2.0 **480 Mbps** → **사용률 ≈ 2–4%.** (이전 노트의 "gs_usb / USB Full-Speed 12 Mbps" 가정은 **틀렸다**. 이 정정은 지연 예산을 유리한 방향으로 바꾼다.)

**진짜 걱정거리는 대역폭이 아니라 지연이다** — `[미확인, 실측 필요]`:
- USB RTT 추정 300–500 µs (HS 마이크로프레임 125 µs × 2 + MCU + CAN TX/RX)
- 2채널이 **같은 USB 파이프 + 같은 Cortex-M4 펌웨어**를 통과 → **head-of-line blocking** 가능
- 어댑터 1개 = **단일 장애점**. USB 재열거 한 번에 두 팔이 같이 죽는다
- 💡 완화: 양팔을 **서로 다른 어댑터**에 나눠 붙이면 둘 다 해소된다. BOM에 어댑터가 2개 있으므로 추가 비용 0

**CAN 버스 시간 (채널당, CAN-FD 11-bit ID / BRS on / 8 B):** 프레임 ≈ 50–55 µs. 사이클당 8모터 × 2프레임 ≈ **832 µs**.

| 제어 주파수 | 주기 | CAN 버스 사용률 |
|---|---|---|
| 1000 Hz | 1000 µs | **≈ 83%** ⚠️ 지터 위험 |
| 750 Hz | 1333 µs | ≈ 62% ✅ |
| 500 Hz | 2000 µs | ≈ 42% ✅ |

**채널 매핑 — 스크립트가 정하지 않는다. 우리 config가 정한다.**

LeRobot 공식 관례(HF 문서 바이매뉴얼 예제):

| 채널 | 역할 |
|---|---|
| `can0` | follower **left** |
| `can1` | follower **right** |
| `can2` | leader **left** (CAN 리더암 사용 시에만) |
| `can3` | leader **right** (동상) |

> 🔴 **`openarm_teleop`(C++)의 기본값은 정반대다** — `openarm_bilateral_control.cpp:154-155`: `leader_can_interface = "can0"`, `follower_can_interface = "can2"`. 두 스택을 한 리그에서 쓰려면 반드시 CLI 인자로 명시하라.

**udev 고정은 선택이 아니다.** 동일 VID/PID 어댑터 2개를 꽂으면 `can0`~`can3` 할당은 **부팅 시 USB 열거 순서에 따라 비결정적**이다. 규칙 작성 시 함정 3개:
1. **`NAME=`을 `canN`으로 주지 마라** — 커널이 이미 `canN`을 할당 중이라 경합한다. `oa_fl` / `oa_fr` 같은 비-`can` 접두사를 쓴다(`IFNAMSIZ=16`).
2. LeRobot의 `port`는 그냥 `channel=` 문자열이다(`damiao.py:180`, `:171`은 `/dev/`로 시작하지만 않으면 socketcan) → **`--robot.port=oa_fl`이 그대로 동작한다.**
3. 한 어댑터의 두 채널은 **같은 `ATTRS{serial}`을 공유**한다 → serial만으로 구별 불가. `ATTR{dev_id}`(gs_usb라면 `gs_usb.c:1355` `netdev->dev_id = channel`로 보장) 또는 USB 포트 경로(`KERNELS==`)를 병용해야 한다.

`[미확인]` 실물 드라이버(`ethtool -i can0`), `dev_id` 노출 여부, 고유 iSerial 보고 여부 — **전부 실기에서 확정해야 한다.**

### 2.7 KER 리더암은 **USB**다 — CAN 채널을 0개 소비한다 [확정]

`enactic/openarm_ker/src/openarm_ker/ker_stream.py`:

- ⚠️ `openarm_mini`는 KER이 **아니다** — Feetech 서보 시리얼 버스(`config_openarm_mini.py:26-27`).

### 2.8 토크 관측 — `observation.state`로 평탄화된다 [확정]

플랫폼 정체성(힘/컴플라이언스 데이터 수집)은 LeRobot 위에서 **유지된다.**

```python
# openarm_follower.py:90-99
features[f"{motor}.pos"] = float
if self.config.use_velocity_and_torque:
    features[f"{motor}.vel"]    = float
    features[f"{motor}.torque"] = float
```
값도 실제로 채워진다(`:239-246` `sync_read_all_states()` → `state.get("torque", 0.0)`). 버스는 tau를 버리지 않는다(`damiao.py:68-73` `MotorState{position, velocity, torque, temp_mos, temp_rotor}`, `:556,566` 피드백 프레임에서 `tau_uint` 디코드). 독스트링: *"Reads all motor states (pos/vel/torque) in **one CAN refresh cycle**"* → **추가 CAN 비용 없음.**

**아키텍처에 직결되는 단서 4개:**

| # | 단서 | 아키텍처 대응 |
|---|---|---|
| 1 | `use_velocity_and_torque: bool = **False**` (`config_openarm_follower.py:71`) | 기본이 OFF. 안 켜면 **조용히** 위치-only 8차원 데이터셋이 생성된다 → **백엔드가 강제로 True를 넣어야 한다** |
| 2 | **`observation.effort`는 존재하지 않는다** (전체 트리 grep 0건) | 토크는 `feature_utils.py:68-89`에서 **`observation.state` 벡터 안으로 평탄화**된다. 단완 8모터 × (pos,vel,torque) = **24차원**, 양완 **48차원**. `names`에 `joint_1.torque` 문자열만 남는다 |
| 3 | **리더·팔로워 플래그를 반드시 동시에 켠다** | `lerobot_record.py:394-401`이 action feature를 `robot.action_features`로 만들고, `feature_utils.py:131-132`의 `build_dataset_frame`이 `values[name] for name in ft["names"]`로 teleop dict를 인덱싱한다 → **팔로워만 True면 `values["joint_1.torque"]` KeyError로 런타임 사망.** → **단일 스위치로 묶어라** |
| 4 | **토크 액션은 기록만 되고 실행되지 않는다** | `openarm_follower.py:340`: `commands[motor_name] = (kp, kd, position_degrees, **0.0, 0.0**)` — vel/tau 하드코딩 0. `send_action`은 `.pos`만 읽는다(`:288`). 단 버스 레벨 `_mit_control_batch(commands: {motor: (kp, kd, pos_deg, vel_deg_s, torque)})`(`damiao.py:492-502`)는 **이미 tau 인자를 받는다** → **막혀 있는 건 상위 함수 하나뿐** |

**단위 [확정]**: `.pos` = **deg**, `.vel` = **deg/s**, `.torque` = **Nm(변환 없음)** (`damiao.py:568`).
→ **한 `observation.state` 벡터에 deg와 Nm이 섞인다.** 정규화 통계(mean/std) 산정 시 반드시 고려해야 한다. (→ `10`)

**패킷 드랍이 조용히 삼켜진다**: `_batch_refresh`가 드랍 시 **마지막 알려진 상태를 그대로 재사용**한다(`damiao.py:696-702`, `logger.warning("Packet drop: …")`). 토크는 위치보다 노이즈가 크고 드랍에 민감하다 → **드랍 카운터를 관측 메타로 노출**해야 한다. LeRobot은 이를 feature로 내보내지 않는다.

### 2.9 IK = `openarm_control` (mink + MuJoCo + daqp) — 병목이 아니다 [확정, 실측 완료]

`enactic/openarm_control/src/openarm_control/kinematics.py`:
- `import mink, mujoco` / `solver: str = "daqp"`(`:54`) / `mink.FrameTask` 좌우 동시(`:145-146`) / `mink.DofFreezingTask`(`:164-165`) / `mink.Configuration`(`:136`) → **양팔 단일 QP**
- **`def solve(self) -> np.ndarray | None`(`:107`) — 인자 없음.** (README의 `solve(dt=, n_iters=)`는 **틀렸다**)

**실측**: 양팔 기본설정 **0.345 ms / p95 0.355 ms, 500/500 성공**. 60 Hz 예산의 **2%**. → **IK는 병목이 아니다.**

**상류가 이미 검증한 계약**: 공식 VR dataflow(`dora-openarm-vr/config/dataflow-mujoco.yaml`)가 `dora-openarm-ik`(= `openarm_control`)를 **bimanual 모드로 실제 사용**하고, VR 노드의 출력 포즈 프레임(`FRAME_OFFSET_NECK`, MJCF 월드)이 `set_target()` 규약과 **정확히 일치**한다. 상류 튜닝값 `--mode bimanual --max-iters 10 --dt 0.1 --damping 0.1 --posture-cost 0.01 --lm-damping 0.01`을 **그대로 베낀다.**

**아키텍처가 부담해야 하는 4가지:**

1. **LeRobot도 자체 IK를 갖고 있다** — `model/kinematics.py`의 `RobotKinematics`(**placo** 기반) + `robots/so_follower/robot_kinematic_processor.py`의 등록 스텝 8개. **`openarm_control`을 택하면 LeRobot의 EE processor 경로와 EE 공간 데이터셋 기록을 포기하는 것**이다. 무비용이 아니라 **분기**다.
2. **IK는 teleoperator 안에 있어야 한다** — 데이터셋 action names = `robot.action_features` = `{joint_1.pos, …, gripper.pos}` (**deg**)이고 `build_dataset_frame`이 그 이름으로 teleop 출력을 인덱싱한다 → **`get_action()`은 관절각(deg)을 반환해야 한다.** EE 포즈를 내고 processor가 IK를 도는 대안은 동작하지만, `lerobot_teleoperate.py:240-273`이 `make_default_processors()`(identity 3종)를 부르고 `record()`도 CLI 경로에서 identity로 폴백하므로(`lerobot_record.py:383-392`) **자체 루프가 필수**가 된다. 우리는 어차피 자체 루프를 쓰지만(§2.3), **IK-in-teleoperator가 단순하고 상류 계약과 일치**하므로 이쪽을 택한다.
3. **조인트 리밋 불일치** — `mink.ConfigurationLimit`은 MJCF `jnt_range`만 본다. MJCF right joint_1 = **−80°…+200°**, LeRobot 소프트 리밋 = **−75°…+75°** → **IK가 LeRobot이 클램프할 해를 낸다.** `ArmSetup` 생성 후 `model.jnt_range`를 LeRobot 리밋으로 **덮어쓴 뒤** `Kinematics(...)`를 만들어야 한다(**순서 중요**).
4. **폴백이 리밋을 버린다** — `kinematics.py:220-231`: `NoSolutionFound` → **`limits=[]`로 재시도**. 조인트 리밋을 포기한 해를 낼 수 있다 → **어댑터 레벨 클램프가 1차 방어선**이어야 한다.

**단위**: `openarm_control` = **rad**, LeRobot = **deg**. 변환 필수.

### 2.12 LeRobot 안전장치는 빈약하다 — 안전 계층은 우리 것이다 [확정]

### 2.16 웹 단일 아키텍처 = 헤드리스 FastAPI 백엔드 + 웹 SPA [확정]

**프론트엔드는 브라우저에서 도는 웹 SPA, 백엔드는 헤드리스 파이썬 서비스(FastAPI)다.** 둘은 같은 프로세스가 아니라 **네트워크 경계(WebSocket + HTTP REST)** 로 분리된다. 참조 구현(`bh_indy7_LeRobot/src/bh_indy7_lerobot_gui`)의 PyQt6 데스크톱 GUI는 **UI로 재사용하지 않는다** — 그러나 그 **백엔드 로직 패턴**(robot 객체 임베드, 모드 전이, 안전가드, 캘리브레이션 SoT, 이중 추론, tolerant 카메라)은 UI와 무관하게 그대로 이식한다.

```
[브라우저: 웹 SPA (React 등)]                    [원격 추론 서버 (선택)]
  Three.js + urdf-loader (v2 URDF)                 LeRobot async PolicyServer
        │  ▲                                              ▲
   WS   │  │ WS  (텔레메트리: 관절/토크/상태, 카메라)      │ gRPC :8080
 (제어) │  │ REST(설정·데이터셋·잡 CRUD)                   │ (백엔드↔추론 홉만)
        ▼  │                                              │
┌──────────────────────────────────────────────────────────────┐
│  헤드리스 백엔드 프로세스 (FastAPI / uvicorn)                    │
│   · LeRobot Python API 임베드 · connect 1회 · 모드=상태전이      │
│   · 안전 필터 계층 · flock CAN 배타 락 · robot 객체 소유         │
│   · cv2 프레임 → JPEG/뎁스 인코딩 → WS 바이너리 카메라ID 멀티플렉스 │
└──────────────────────────────────────────────────────────────┘
        │  (SocketCAN RAW, CAN-FD 1M/5M — 인프로세스)
        ▼
   [can0..can3 → OpenArm 팔로워/리더]
```

- **브라우저 ↔ 백엔드 실시간 = 단일 WebSocket.** 제어 명령(브라우저→백엔드) + 텔레메트리(관절/토크/모드/락/드랍, 백엔드→브라우저) + 카메라 프레임(바이너리 JPEG, 카메라ID 멀티플렉싱)을 **하나의 WS**에 실어 나른다. WebRTC 별도 스트림으로 분리하지 않는다(오버엔지니어링 배제 — D-2 "프로토콜 분리 금지" 준수). (FR-SYS-031·032·038)
- **브라우저 ↔ 백엔드 비실시간 = HTTP REST.** 설정 get/set, 데이터셋 목록/조회, 잡(기록·학습·추론) 시작·정지 등 CRUD. 표준 FastAPI 패턴이며 **프로토콜 분리가 아니다** — 실시간 단일 채널(WS)과 무상태 CRUD(REST)의 표준 역할 분담이다. (FR-SYS-037)
- **gRPC-web은 브라우저 경계에 쓰지 않는다.** 브라우저에서의 **양방향 스트리밍이 gRPC-web에서 제약**(서버→클라 스트리밍만 가능, 클라→서버 스트리밍 불가)이라 텔레옵 양방향에 부적합 → **WS 채택.**
- **gRPC는 백엔드 ↔ 원격 추론(LeRobot async PolicyServer :8080) 홉 하나에만** 남는다. 브라우저 경계가 아니다(§2.17, FR-SYS-034). 온보드 추론은 백엔드가 **`lerobot-rollout` 엔진을 인프로세스로 임베드**(LeRobot 0.6.x, CLI 스폰 없음 — `lerobot-record --policy.path`는 0.5.1 폐기 패턴), 원격 추론은 gRPC PolicyServer(:8080). **InferenceMode = LOCAL / ASYNC** 선택.
- **3D 뷰포트는 브라우저에서 Three.js + `urdf-loader`(v2 URDF)로 렌더**하고, 관절 상태를 WS 텔레메트리로 피드한다(외부 Rerun/Foxglove 창 없음).
- **카메라 라이브 프리뷰 스트리밍은 웹 선택의 실재 비용이다.** 데스크톱이면 프레임이 인프로세스로 위젯에 그려졌지만, 웹에서는 **백엔드가 cv2 프레임을 JPEG로 인코딩**(뎁스는 16-bit PNG 또는 컬러맵)**해 WS 바이너리 프레임으로 다중 카메라를 멀티플렉싱** 전송해야 한다. 이것이 데스크톱 대비 추가되는 유일한 큰 구현이다(픽셀 레벨·해상도·코덱 → `06`, FR-SYS-038).

> **connect 1회 / 모드=상태전이 / flock CAN 배타 소유 / CLI 스폰 금지는 전부 그대로 유효하다 — 백엔드 프로세스 내부의 불변식이다.** 웹 전환은 UI 전송 계층만 바꿀 뿐, §2.2~§2.4의 세션·CAN 규율에는 영향이 없다.

> **직렬화·와이어 프레이밍·백프레셔가 이제 실재한다.** 데스크톱 인프로세스에서는 없던 비용이다. 브라우저↔백엔드 경계에 텔레메트리 latest-wins, 카메라 프레임 드롭, WS 재연결/재동기화를 명시 설계해야 한다(FR-SYS-032). 반면 백엔드 내부(제어 루프 ↔ robot ↔ CAN)는 여전히 인프로세스 함수 호출이다. 이전 판의 전송 계층 `[결정필요]` 절(WebRTC/Foxglove/rosbridge 비교)은 삭제됐고, 그 자리를 이 확정된 WS+REST 계약이 대체한다.

## 4. 상태·오류 처리

### 4.1 모드 상태머신 — 같은 robot 객체 위의 상태 전이

**핵심 불변식**: `CONNECTED` 진입 시 `connect()` 1회. 세션 종료 시 `disconnect()` 1회. **그 사이의 모든 모드 전이는 CAN 버스를 놓지 않는다.**

```
IDLE ──[flock 획득 → 링크 검증 → connect()]──> CONNECTED
                                                   │
                    ┌──────────────┬───────────────┼──────────────┬──────────────┐
                    ▼              ▼               ▼              ▼              ▼
                 MANUAL         TELEOP          RECORD        INFERENCE     CALIBRATE
              (GUI 조그)   (VR/KER → send_action)  (TELEOP+   (정책 →      (토크 OFF,
                                                  데이터셋)   send_action)   수동)
                    └──────────────┴───────────────┴──────────────┴──────────────┘
                                                   │
                                          [SAFE_HOLD / FAULT]
                                                   │
                             ──[disconnect()]──> IDLE  (세션 종료 시에만)

SIM  : CONNECTED와 무관. CAN 락을 획득하지 않는다. IDLE에서 직접 진입.
BILATERAL(500 Hz 미달 시) : 🔴 유일하게 버스를 놓는 모드 — §4.2 (T-BI)
```

| 모드 | CAN 락 | robot 객체 | `send_action()` 호출권 | 비고 |
|---|---|---|---|---|
| `IDLE` | 미보유 | 미연결 | 없음 | — |
| `CONNECTED` | 보유 | **연결** | 없음 (유휴) | 관측만 폴링. 모든 모드의 허브 |
| `MANUAL` | 보유 | 연결 | **GUI 조그** | 안전 필터 경유 (FR-SYS-017) |
| `TELEOP` | 보유 | 연결 | **VR / KER 텔레오퍼레이터** | `get_action()` → 안전 필터 → `send_action()` |
| `RECORD` | 보유 | 연결 | **TELEOP과 동일 루프** | 기록은 텔레옵 **위에 얹히는** 모드지 대체가 아니다. 같은 루프에 `LeRobotDataset.add_frame()`이 추가된다 |
| `INFERENCE` | 보유 | 연결 | **정책** | 액션 청크 → 업샘플 → 안전 필터 → `send_action()` |
| `CALIBRATE` | 보유 | 연결 | **없음 (토크 OFF)** | 사람이 팔을 잡는다. 모든 것과 배타 |
| `SIM` | **미보유** | **미연결** | 없음 (MuJoCo 내부) | CAN을 만지지 않는다 |
| `SAFE_HOLD` | 보유 | 연결 | **없음** (중력보상 홀드) | 토크 0이 아니다 (아래 경고) |
| `FAULT` | 보유 | 연결 | **없음** | 사람 개입 전까지 자동 복구 금지 |
| `BILATERAL` 🔴 | **자식 프로세스로 이양** | **연결 해제** | 자식 프로세스 | **NFR-SYS-003 미달 시에만.** §4.2 (T-BI) |

> **`SAFE_HOLD`는 토크 0이 아니다.** OpenArm은 QDD 백드라이버블 관절이며 **홀딩 브레이크가 없다.** 공식 안전 가이드: *"if power is lost due to an emergency stop, the load being held will fall rapidly."* 정지는 **중력보상 홀드를 거쳐 감속 하강**해야 한다. (→ `12`)

### 4.2 전이 시퀀스

#### (T1) `IDLE` → `CONNECTED` — 세션 개시 (세션당 1회)

| 단계 | 동작 | 성공 판정 | 실패 시 |
|---|---|---|---|
| 1 | 사용 채널 전부에 `flock(LOCK_EX\|LOCK_NB)` 획득 | 전 채널 획득 | 실패 → **기동 거부**. 락 파일의 PID 표시. **CAN 소켓을 열지 않는다** |
| 2 | **링크 검증**: `ip -details link show <ch>` → `fd on` + `bitrate 1000000` + `dbitrate 5000000` + `ERROR-ACTIVE` | 전부 통과 | 불일치 → 거부 + `lerobot-setup-can --mode=setup` 안내. 락 해제 |
| 3 | **침입자 프리플라이트**: `/proc/net/can/rcvlist_all`에서 해당 ifindex RX 리스너 수 확인 | 우리 것 외 0 | >0 → **경고**(candump 가능성). 사용자 확인 후 진행 |
| 4 | **🔴 rest 자세 확인 모달** — `connect()`가 현재 자세를 영점으로 확정한다는 사실 고지 + 관절별 현재 자세 표시 | 사용자 승인 | 거부 → 락 해제, `IDLE` 유지 |
| 5 | config 조립: `side` 명시(FR-SYS-013), `use_velocity_and_torque=True` **팔로워·리더 동시**(FR-SYS-012), `push_to_hub=False`(FR-SYS-015) | — | — |
| 6 | `robot.connect(calibrate=True)` → CAN 소켓 open → (캘리브레이션됨) `set_zero_position()` → `enable_torque()` | `robot.is_connected` | 예외 → 락 해제, `FAULT` |
| 7 | 모드 = `CONNECTED` | — | — |

> 양팔은 `BimanualMixin.connect()`가 좌 → 우 **순차 동기 호출**한다(`utils/bimanual.py:47-63`). 한쪽 실패 시 다른 쪽이 이미 연결된 상태로 남을 수 있다 → **부분 연결 상태를 명시적으로 처리**해야 한다.

#### (T2) `CONNECTED` ↔ 임의 모드 — **버스를 놓지 않는다**

| 단계 | 동작 |
|---|---|
| 1 | 현재 활성 제어 루프에 **정지 신호** → **`join()`으로 종료 확인** (겹침 0 보장, FR-SYS-020) |
| 2 | `send_action()` 뮤텍스가 해제되었는지 확인 |
| 3 | 새 모드의 제어 루프 시작 (새 텔레오퍼레이터 `connect()` / 정책 로드 / 데이터셋 라이터 open) |
| 4 | 모드 갱신 |

**`connect()` / `disconnect()`를 호출하지 않는다. CAN 소켓은 열린 채로 유지된다. 영점은 건드려지지 않는다.**

#### (T-BI) 🔴 `CONNECTED` → `BILATERAL` — **유일하게 버스를 놓는 전이** (NFR-SYS-003 미달 시에만)

| 단계 | 동작 | 실패 시 |
|---|---|---|
| 1 | 활성 루프 정지 + join. `SAFE_HOLD` 경유 | — |
| 2 | **rest 자세 이동 강제** — 복귀 시 `connect()`가 영점을 다시 찍기 때문 | 미도달 → 전이 거부 |
| 3 | `robot.disconnect()` → 소켓 close | — |
| 4 | 잔여 바인딩 검증(`/proc/net/can/rcvlist_all` == 0) | ≠0 → `FAULT`. 자동 재시도 금지 |
| 5 | `flock`을 자식 프로세스(`openarm_teleop` 또는 자체 RT 바이너리)에 **이양** — ⚠️ 채널 관례가 정반대임에 주의(`leader=can0`, `follower=can2`) → **CLI 인자로 명시** | — |
| 6 | 복귀 시 역순 + **🔴 (T1) 4단계 rest 자세 모달 재실행** | — |

> **이 전이가 존재하는 것 자체가 아키텍처 부채다.** NFR-SYS-003이 500 Hz 달성을 확인하면 이 전이는 **삭제된다.**

### 4.3 모드 배타 행렬

`✗` = 동시 불가 · `✓` = 동시 가능 · `△` = 조건부

| | MANUAL | TELEOP | RECORD | INFERENCE | CALIBRATE | BILATERAL | SIM | 관측(read-only) |
|---|---|---|---|---|---|---|---|---|
| **MANUAL** | — | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ | ✓ |
| **TELEOP** | ✗ | — | **△** | ✗ | ✗ | ✗ | ✓ | ✓ |
| **RECORD** | ✗ | **△** | — | ✗ | ✗ | ✗ | ✓ | ✓ |
| **INFERENCE** | ✗ | ✗ | ✗ | — | ✗ | ✗ | ✓ | ✓ |
| **CALIBRATE** | ✗ | ✗ | ✗ | ✗ | — | ✗ | ✓ | △ (상태 폴링만) |
| **BILATERAL** | ✗ | ✗ | ✗ | ✗ | ✗ | — | ✓ | △ (`candump`만) |
| **SIM** | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | — | ✓ |
| **관측(read-only)** | ✓ | ✓ | ✓ | ✓ | △ | △ | ✓ | — |

**△ (TELEOP × RECORD)**: **RECORD는 TELEOP의 상위집합이다.** 별개의 두 루프가 아니라 **같은 루프에 데이터셋 라이터가 붙은 것**이다. LeRobot의 `record_loop`도 teleop을 **요구**한다(`lerobot_record.py:193-197`). 두 개의 독립 루프를 동시에 돌리는 것은 금지.

**배타 사유:**

| 조합 | 충돌 자원 | 근거 |
|---|---|---|
| **텔레옵 × 추론** | **`send_action()` 호출권** | 같은 관절에 VR 셋포인트와 정책 액션이 동시 도착 → 모터가 마지막 도착 셋포인트를 따르며 두 소스 사이에서 진동. **LeRobot에 중재자가 없다** |
| 임의 모드 × MANUAL | `send_action()` 호출권 | GUI 조그와 자동 명령이 동시에 나가면 안 됨 |
| 임의 모드 × CALIBRATE | **토크 상태 + 모터 플래시** | 캘리브레이션은 토크를 끄고 모터 파라미터를 **플래시에 쓴다**(1만 회 제한) |
| 임의 모드 × BILATERAL | **CAN fd** | 별도 프로세스가 CAN을 소유한다. 커널이 이중 바인드를 막지 않는다(§2.4) |
| SIM × 모든 것 | 없음 | MuJoCo는 CAN을 만지지 않는다 |

### 4.4 경계별 계약

| 경계 | 전송 | 페이로드 | 주기 | 백프레셔 | 워치독 |
|---|---|---|---|---|---|

## 5. 미해결 · 결정필요

### Q1. 🔴 `[미확인]` **LeRobot Python 제어 루프가 바이래터럴 500 Hz를 낼 수 있는가?** — 바이래터럴 채택 시의 최대 미결 리스크
`openarm_teleop`(C++)은 바이래터럴을 **500 Hz** 스레드로 돈다(`openarm_bilateral_control.cpp:42,72,103`). LeRobot Python 루프(`sync_read_all_states()` + `_mit_control_batch()`, GIL 아래)의 실측 사이클 타임은 **아무도 측정한 적이 없다.**
> **⚠️ 500 Hz는 바이래터럴 힘반사에만 해당한다.** 주경로(유니래터럴 텔레옵, LeRobot record 30–60 Hz, dora eval 250 Hz, 추론)는 500 Hz를 요구하지 않는다. 바이래터럴은 **선택적 특수 모드**다.
- **달성 시**: 바이래터럴이 단일 백엔드 프로세스 안에 들어온다. CAN 소유권은 §2.4 그대로.
- **미달 시**: 바이래터럴만 별도 RT 프로세스로 분리 → **프로세스 간 CAN 배타 소유권 문제 부활** + **모드 전환 시 disconnect/reconnect = 영점 파괴 위험 재도입**(§4.2 T-BI).
**측정 방법**: FR-SYS-023. 단완/양완 × 카메라 유무 × 안전 필터 유무로 p50/p95/p99를 낸다. 이 측정이 §4.1의 `BILATERAL` 행과 §4.2의 (T-BI) 전이의 존폐를 결정한다.

### Q2. `[결정필요]` 리더암을 무엇으로 할 것인가 — CAN 리더암 vs KER(USB)?
- **CAN 리더암**(`openarm_leader`): LeRobot 1급 지원. **CAN 채널 2개 소비**(can2/can3). **바이래터럴 가능**(모터가 있다).
- **KER**: **CAN 채널 0개 소비**(USB, ESP32-S3). 어댑터 1개로 충분 → BOM 축소. **그러나 LeRobot 텔레오퍼레이터가 없어 우리가 만들어야 하고**(난이도 낮음), **모터가 없으므로 바이래터럴이 원천 불가능하다.**
→ **이 결정은 Q1과 결합되어 있다.** 바이래터럴이 요구사항이면 KER은 탈락한다.

### Q3. `[미확인]` **Quest 3S에서 상류 APK가 동작하는가?**
`dora-openarm-vr`의 APK는 **Quest 3 대상 빌드**이며 소스가 비공개다(Google Drive 배포). Quest 3S 동작 여부는 확인되지 않았다. **프로토콜(UDP :5006 평문 JSON)은 완전 공개되어 있으므로 수신부는 문제없다** — 리스크는 APK 자체의 기기 호환성뿐이다.
**대안**: `dora-openarm-webxr`(pip 설치 가능, WebXR 브라우저, **APK 불필요**, HTTPS :8443, Quest 3 + PICO 4 명시). 클라이언트 `static/ar.js`가 오픈소스(Apache-2.0)다.
**확인 방법**: 실기에 APK를 사이드로드해 `nc -lu 5006`으로 JSON 수신 확인.

### Q4. `[미확인]` USB-CAN 어댑터의 실물 드라이버·`dev_id`·iSerial
in-tree `gs_usb` 장치표에 **Pibiger VID/PID가 없다**(`gs_usb.c:1647-1658` 확인). 벤더는 "Linux SocketCAN 호환"만 주장하고 커널 모듈 소스를 배포하지 않는다 → **in-tree 드라이버로 붙는다는 뜻이지만 어느 것인지 모른다.** `peak_usb`는 거의 확실히 아니다(PEAK VID `0x0c72`).
**확인 방법**: `lsusb` → `ethtool -i can0` → `readlink -f /sys/class/net/can0/device/driver` → `cat /sys/class/net/can0/dev_id` → `udevadm info -a -p /sys/class/net/can0`. **udev 규칙(FR-SYS-008)이 여기에 종속된다.**

### Q5. `[미확인]` 2채널 어댑터의 head-of-line blocking
can0/can1의 URB가 같은 USB 파이프 + 같은 Cortex-M4 펌웨어를 통과한다. 펌웨어가 두 채널을 직렬 처리하면 한쪽 팔의 버스트가 다른 팔의 지연을 밀어올린다. 대역폭은 무관하다(2–4%).
**완화책**: 양팔을 서로 다른 어댑터에 나눠 붙인다(BOM에 2개 있으므로 추가 비용 0). **단 4채널 텔레옵(CAN 리더암) 구성에서는 어댑터가 이미 다 쓰인다** → Q2와 결합.
**확인 방법**: 양채널 동시 부하 시 지연 분포 비교 (FR-SYS-025).

### Q6. `[미확인]` `observation.state`에 deg와 Nm이 섞인 벡터의 정규화
단완 24차원 / 양완 48차원 벡터에 **deg(위치)** · **deg/s(속도)** · **Nm(토크)**가 섞인다. `dataset_to_policy_features()`는 `observation.state`를 통째로 `FeatureType.STATE`로 넘긴다(`feature_utils.py:168-173`) → **차원만 맞으면 통과하지만** 스케일이 극단적으로 다른 채널이 한 벡터에 섞이는 문제가 `compute_stats`의 mean/std에 어떻게 작용하는지는 미검증이다.
**확인 방법**: 실데이터로 채널별 통계를 뽑아 검토. (→ `10`)

### Q7. `[미확인]` `enactic/openeval-runner` / `openeval-web`
2026-07 활발히 push되는 신규 저장소. 평가 하네스로 보이나 소스를 조사하지 않았다. **우리가 만들 실기 평가 하네스(§2.13)와 중복될 가능성이 있다.**
**확인 방법**: 소스 조사. (→ `11`)

### Q8. `[미확인]` `openarm_control`의 성숙도
릴리스 `0.1.0` 하나, **테스트 디렉터리 없음**(직접 확인). IK는 우리 크리티컬 패스 위에 있다(실측 0.345 ms로 성능은 검증됐으나 **정확성 회귀 테스트가 없다**).
**확인 방법**: 자체 회귀 테스트 작성(FK↔IK 왕복, 리밋 준수, 특이점 근방 거동).
