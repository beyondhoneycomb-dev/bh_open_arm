# 06. 카메라 서브시스템 및 영상 화면

## 1. 범위

카메라 등록·바인딩, RGB 웹캠과 뎁스 카메라의 혼합 구성, USB 대역폭 예산, 시간 동기화, intrinsic/hand-eye 캘리브레이션, 녹화용 인코딩(RGB·뎁스), **웹 프리뷰 스트리밍(백엔드 cv2 프레임 → JPEG 인코딩 → 단일 WebSocket 바이너리)**, 웹 GUI 카메라 화면, 프레임 드롭·연결 끊김 감지와 복구를 다룬다.

다루지 않는 것: 에피소드 라이프사이클과 데이터셋 쓰기(`07-데이터-수집.md`, 영역 `REC`), 3D 뷰포트·패널 레이아웃 일반 **및 브라우저↔백엔드 단일 WebSocket 전송·멀티플렉싱 프레이밍 규약**(`13-GUI-화면-명세.md`, 영역 `GUI`), VR HMD로의 스테레오 렌더링과 텔레옵 지연 예산(`05-텔레오퍼레이션.md`, 영역 `TEL`), 정책의 시각 백본 선택(`10-학습.md`).

경계 규칙 하나만 여기서 못 박는다: **카메라 프레임을 만드는 것, 시각(timestamp)을 붙이는 것, 인코딩하는 것(녹화용 LeRobot 인코딩 + 웹 프리뷰용 JPEG 인코딩·카메라ID 태깅), 화면에 그릴 프레임을 산출하는 것은 이 문서다. 그 프레임이 어느 에피소드의 몇 번째 행에 들어가는지는 `07`, 그 JPEG 바이너리를 실어 나르는 단일 WebSocket 전송·멀티플렉싱 프레이밍 일반은 `13`이다.**

> 🔴 **v2 정본 전제 (이전 판을 뒤집는다).** 이전 판은 카메라를 **"고정 4슬롯(Arducam B0495 ×3 + ZED mini ×1) = 소프트웨어 계약"** 으로, 뎁스를 **"정본 경로에 0개, RealSense는 legacy 옵션"** 으로 서술했다. **둘 다 오염이다.**
> - **소프트웨어 계약은 빈 dict다.** `OpenArmFollowerConfig.cameras = field(default_factory=dict)` — 고정 슬롯이 없고 **전량 사용자 설정**이다(§2.1). Arducam B0495 ×3 + ZED mini는 **OpenArm Cell 하드웨어 참조설계일 뿐** 우리 소프트웨어 계약이 아니다(§2.11에 각주로 격하).
> - **뎁스는 지원 대상이다(D-12: "뎁스 필요").** 뎁스 표준 경로는 **RealSense(`intelrealsense`)**, RGB 웹캠은 **OpenCV(로지텍 등 USB 웹캠)**, **같은 dict에 자유롭게 혼합 등록**한다. 단 **뎁스 관측·기록은 LeRobot ≥ 0.6.x가 필요**하다(0.5.1은 async/record 뎁스 미구현 — §2.4).
> - **ZED은 LeRobot 표준 카메라 클래스에 없다**(opencv/intelrealsense/reachy2/zmq 뿐). ZED SBS 스플리터는 정본 경로의 요구가 아니라 **참조 하드웨어(Cell)를 쓸 때만 필요한 서드파티/커스텀 경로**로 격하한다(§2.11).

## 2. 확정된 사실 기반

### 2.1 카메라 계약 = 빈 dict, 전량 사용자 설정 [확정]

`robots/openarm_follower/config_openarm_follower.py`(L73-74):
```python
# Camera configurations
cameras: dict[str, CameraConfig] = field(default_factory=dict)
```
→ **openarm 로봇에는 표준·고정 카메라 셋이 없다.** 카메라 종류·개수·해상도·fps·뎁스 여부는 **전량 사용자가 `--robot.cameras`(양완이면 `--robot.{left,right}_arm_config.cameras` 포함)로 등록**한다. HF openarm 공식 record 예시(https://huggingface.co/docs/lerobot/openarm)에도 `--robot.cameras`가 아예 없다 — 카메라는 전량 사용자 지정이다.

**따라서 이 문서는 "몇 대가 어디에 붙는가"를 소프트웨어 계약으로 규정하지 않는다.** 규정하는 것은 (a) 어떤 카메라 클래스를 어떻게 등록하는가, (b) RGB와 뎁스를 어떻게 섞는가, (c) 등록 → 데이터셋 키 → GUI 라벨의 유도 규칙, (d) 대역폭·동기·캘리브레이션·인코딩·프리뷰의 계약이다. 물리 배치(슬롯 이름, 몇 대)는 **사용자 배포별 구성**이며, 대표 예시(OpenArm Cell 참조 하드웨어)는 §2.11에 각주로 둔다.

**슬롯 키는 사용자가 정하고, LeRobot 카메라 dict 키로 그대로 등록한다.** 데이터셋 키의 정본은 `observation.images.<cam>`이다(§2.12). 양완(`bi_openarm_follower`)에서:
- 팔별 `--robot.{left,right}_arm_config.cameras`에 넣은 키에는 `left_`/`right_`가 **자동으로 붙고**,
- 최상위 `--robot.cameras`에 넣은 키는 **접두사 없이 그대로** 간다.
(`bi_openarm_follower.py:99-112`, `config_bi_openarm_follower.py:35-38` 주석 명시.)

**카메라 이름 목록은 데이터셋 생성 시점의 1급 인자다** — 출처는 **LeRobot `robot.observation_features`의 `_cameras_ft`**(= 등록한 `cameras` 키셋)이다. 카메라 키가 곧 데이터셋 스키마다(§2.12).

### 2.2 LeRobot 카메라 클래스 · 설정 계약 [확정]

**LeRobot이 제공하는 카메라 클래스는 4종이 전부다** — `make_cameras_from_configs`(`cameras/utils.py`):
```
type: opencv           -> OpenCVCamera        # 폰/내장/USB 웹캠(로지텍 등)  — RGB 전용
type: intelrealsense   -> RealSenseCamera     # Intel RealSense              — 뎁스 지원
type: reachy2_camera   -> Reachy2Camera       # Reachy2
type: zmq              -> ZMQCamera            # 네트워크 카메라
else                   -> make_device_from_device_class(cfg)  # 서드파티 플러그인
```
공식 docs 카메라 표(https://huggingface.co/docs/lerobot/cameras)와 일치: **RealSenseCamera = Intel RealSense (with depth)**. **ZED 전용 클래스도, 스테레오 SBS 분할 유틸도 없다**(`cameras/utils.py`의 함수는 `make_cameras_from_configs`, `get_cv2_rotation` 둘뿐).

**`OpenCVCameraConfig`** (`cameras/opencv/configuration_opencv.py`):
```python
index_or_path: int | Path          # /dev/videoN 또는 인덱스
color_mode: ColorMode = RGB        # RGB / BGR
rotation: Cv2Rotation = NO_ROTATION
warmup_s: int = 1
fourcc: str | None = None          # 4자("MJPG"/"YUYV"), 미지정 시 링크 포맷이 드라이버 재량
backend: Cv2Backends = ANY         # V4L2=200, DSHOW=700, MSMF=1400, …
# base CameraConfig:
fps: int | None = None             # ← None이면 shape 결정 불가 → 반드시 지정
width: int | None = None           # ← 동상
height: int | None = None          # ← 동상
```
**명시적 제약(L56): "Only 3-channel color output (RGB/BGR) is currently supported."** → **OpenCV 웹캠은 뎁스 불가, RGB 3채널만.** `use_rgb`/`use_depth` 필드가 없으므로 `_cameras_ft`가 각각 `True`/`False`로 폴백해 **RGB 1키만** 낸다.

**`RealSenseCameraConfig`** (`cameras/realsense/configuration_realsense.py`, `@register_subclass("intelrealsense")`):
```python
serial_number_or_name: str         # 시리얼 또는 이름 (둘 중 하나 필수)
color_mode: ColorMode = RGB
use_depth: bool = False            # ← 뎁스 키를 내는 유일한 스위치
use_rgb: bool = True               # (0.6.x) 뎁스 전용 카메라(use_rgb=False)도 가능
rotation, warmup_s = 1
# base CameraConfig: fps/width/height (전부 기본 None → 반드시 지정)
```
주석: **"Depth stream (if enabled) will use the same FPS as the color stream"** — 컬러/뎁스가 **동일 fps 락스텝**(같은 rs pipeline). 즉 뎁스↔RGB 시간 정렬은 카메라 내부에서 보장된다(§2.6에서 다시 씀).

→ **뎁스 키(`observation.images.<cam>_depth`)를 내는 카메라 클래스는 `intelrealsense` 하나뿐이다.** 로지텍 웹캠으로는 뎁스를 얻을 수 없다. RGB 웹캠 = OpenCV, 뎁스 = RealSense, **같은 dict에 혼합**한다.

**혼합 등록 예 (프로그램/ CLI 공통 형태)** — `il_robots` 문서 형식:
```bash
--robot.cameras="{ front: {type: opencv, index_or_path: /dev/video10,
                           width: 640, height: 480, fps: 30},
                   wrist: {type: intelrealsense, serial_number_or_name: 233522074606,
                           width: 640, height: 480, fps: 30, use_depth: true} }"
```
탐색: `lerobot-find-cameras opencv` / `lerobot-find-cameras realsense` (OpenCV·RealSense만 auto-discovery; ZMQ/Reachy2는 수동).

### 2.3 프레임 접근 모드 — 캡처/프리뷰 격리의 계약 [확정]

| 메서드 | 동작 | 블로킹 | 용도 |
|---|---|---|---|
| `read()` | 하드웨어가 새 프레임 줄 때까지 대기 | Yes | 단순 스크립트 |
| `async_read(timeout_ms=200)` | 백그라운드 스레드의 최신 프레임, 버퍼 비면 timeout까지 대기, 없으면 `TimeoutError` | timeout | **캡처 루프** |
| `read_latest(max_age_ms)` | 논블로킹 peek, 최신 프레임이 `max_age_ms`보다 오래되면 `TimeoutError` | **No** | **UI 시각화 / 로깅 / 모니터링** |

뎁스도 동일한 3모드가 있다(**단 async/latest 뎁스는 0.6.x부터** — §2.4): `read_depth()` / `async_read_depth(timeout_ms=200)` / `read_latest_depth(max_age_ms=500)` → 전부 `(H,W,1) uint16` mm(거리, 픽셀 `0`=측정 불가).

**이 표가 캡처↔프리뷰 격리의 유일한 근거다**: 캡처 루프는 `async_read()`로 최신 미소비 프레임을 받고, GUI는 **논블로킹** `read_latest()`로 peek한다. 그래서 프리뷰가 느려도 캡처 루프를 잠식하지 않는다(FR-CAM-041/042, NFR-CAM-006).

### 2.4 🔴 뎁스 = RealSense, 그리고 LeRobot 버전 게이트 (가장 중요) [확정]

뎁스의 존재 여부를 가르는 것은 **하드웨어 유무가 아니라 LeRobot 버전**이다.

**LeRobot 0.5.1** (참조 레포가 핀한 버전, git 3c15fd8) — **뎁스가 관측·기록 경로에 없다.**
- `cameras/realsense/camera_realsense.py` L531 & L574: `# NOTE(Steven): Missing implementation for depth for now` — `async_read()`·`read_latest()`는 **컬러만** 반환.
- 뎁스는 오직 동기 `read_depth(timeout_ms=200)`(L323)만 준다: "`np.uint16` (raw depth values in millimeters)". 스트림 활성화(L284-291)는 `rs.format.z16`(16-bit mm).
- `openarm_follower.py`(0.5.1): `get_observation`이 `read_latest()`만 호출(L244), `_cameras_ft = (H,W,3)`(L104). → **뎁스 키 없음, RGB만 기록.**

**LeRobot 0.6.1** (commit e40b58a, 2026-07-08) — **뎁스 파이프라인 엔드투엔드 신설.**
- `async_read_depth(timeout_ms=200) -> (H,W,1) uint16`(L635, "millimeters, Pixel 0 = no measurement"), `read_latest_depth(max_age_ms=500)`(L657).
- `openarm_follower.py`(0.6.1) `_cameras_ft`(L100-111):
  ```python
  if getattr(cfg, "use_rgb",  True):  features[cam]           = (cfg.height, cfg.width, 3)
  if getattr(cfg, "use_depth", False): features[f"{cam}_depth"] = (cfg.height, cfg.width, 1)
  ```
  `get_observation`(L252-258): `obs[cam] = cam.read_latest()`; `if cam.use_depth: obs[f"{cam}_depth"] = cam.read_latest_depth()`. 동일 패턴이 `so_follower.py`·`omx_follower.py`에도 있어 **0.6.x의 표준 뎁스 배선 관용구**다.

**→ 결론 [확정]**: 정본 런타임은 **LeRobot v0.6.1로 확정**이다(`02` §2.0 · `03` §2, `pyproject.toml:28 version="0.6.1"`). 이 버전이 뎁스 관측→데이터셋 파이프라인을 엔드투엔드로 제공하므로(위 `async_read_depth`/`read_latest_depth`·`{cam}_depth` feature) D-12의 뎁스 요구가 **표준 경로로 충족**된다 — 뎁스가 전 명세의 전제이자 이 버전이 요구하는 기능이므로 상향은 확정 전제다. 참조 레포가 핀했던 **0.5.1은 뎁스를 async/record 경로에서 지원하지 않으므로**(동기 `read_depth`만) 우리 정본 경로가 아니다. (0.5.1 위에서 뎁스를 쓰려면 `get_observation` 오버라이드 + `read_depth()` 폴링 + `{cam}_depth` feature 수동 등록 + 뎁스 비디오 인코더를 직접 배선해야 하지만, 이는 채택하지 않는 대안으로만 남긴다.) 이 버전 결정은 확정 사항이다(FR-CAM-083).

**뎁스 비디오 인코딩** (`configs/video.py::DepthEncoderConfig`, 0.6.x):
- `vcodec="hevc"`(HEVC **Main 12**), `pix_fmt="gray12le"`(12-bit grayscale), `extra_options={"x265-params":"lossless=1"}` — **무손실**.
- 양자화: `DEPTH_QUANT_BITS=12`, `DEPTH_QMAX=4095`, `depth_min=0.01 m`, `depth_max=10.0 m`, `shift=3.5`, `use_log=True`(로그 양자화).
- 뎁스는 중간 산물이 `.tiff`, 인코딩 후 **디코딩은 pyav 백엔드만 지원** → 뎁스를 쓰면 학습 파이프라인 `video_backend=pyav` 강제(§2.8, 제약).

### 2.5 슬롯 등록 · 데이터셋 키 유도 [확정]

카메라 feature 생성 규칙 (`feature_utils.py::hw_to_dataset_features` L88-137): 모든 카메라 tuple → `f"observation.images.{key}"`, `dtype="video"`(use_videos)|`"image"`, `names=["height","width","channels"]`. **뎁스 `(H,W,1)`도 길이 3이라 image로 분류**되어 `observation.images.{cam}_depth` 비디오 feature가 된다(`pipeline_features.py` L113-120). 정책 입력 전치(`dataset_to_policy_features` L189-197): `(h,w,c)→(c,h,w)` → 뎁스는 `(1,H,W)`.

| 등록 위치 | 예 | 결과 데이터셋 키 |
|---|---|---|
| 팔별 `--robot.left_arm_config.cameras='{wrist: …}'` | 좌완 손목 | `observation.images.left_wrist` |
| 팔별 `--robot.right_arm_config.cameras='{wrist: …}'` | 우완 손목 | `observation.images.right_wrist` |
| 최상위 `--robot.cameras='{front: …}'` | 정면 RGB | `observation.images.front` |
| 최상위 `--robot.cameras='{overhead: {type: intelrealsense, …, use_depth: true}}'` | 상방 RealSense | `observation.images.overhead` + `observation.images.overhead_depth` |

⚠️ **함정 4가지:**
1. **최상위 키가 팔별 키와 충돌하면 `ValueError`** (`bi_openarm_follower.py:44-52`). 슬롯 이름 편집은 허용하되 충돌은 저장 전에 선제 차단한다(FR-CAM-009).
2. **팔별에 넣은 키에는 `left_`/`right_`가 자동으로 붙는다.** 팔과 무관한 카메라(정면·상방·뎁스 등)는 반드시 **최상위 `--robot.cameras`**에 등록한다. 그러지 않으면 키가 `left_<name>`이 되어 좌완 카메라로 오해석된다.
3. 🔴 **`width`/`height`/`fps`를 반드시 명시한다.** `CameraConfig`의 셋은 **전부 기본 `None`**(`cameras/configs.py:59-63`). 미지정 시 `_cameras_ft`의 shape가 `(None, None, 3)`이 되어 `hw_to_dataset_features`가 예외로 죽는다(FR-CAM-081).
4. **`repo_id`에 타임스탬프가 자동으로 붙는다** — `stamp_repo_id()`(`configs/dataset.py:75-83`)가 `_%Y%m%d_%H%M%S`를 덧붙인다. GUI가 데이터셋 이름을 보여줄 때 **stamped 이름을 읽어와야** 한다(`--resume` 경로에서는 미호출).

**성공/실패 라벨은 LeRobot 네이티브에 없다.** `lerobot-record`가 만드는 자동 feature는 `timestamp`/`frame_index`/`episode_index`/`index`/`task_index` 5종뿐(`utils/constants.py:80-87`)이며 **`success`가 없다.** → 에피소드 성공 라벨링은 **[신규구현]**(FR-CAM-057).

### 2.6 시간 동기화

**(a) RealSense 간 HW 동기 (뎁스 카메라를 복수 쓸 때 표준)**
- 옵션 `rs2_option_inter_cam_sync_mode` — "0=default, 1=Master, 2=Slave".
- 배선: "pins 5 (SYNC) and pins 9 (Ground) need to be connected". 외부 트리거는 "**100 µs positive pulse** at the nominal frame rate(30 Hz→33.33 ms), high impedance, 1.8 V CMOS."
- 데이지체인 실측: 패시브 배선 3 m 미만, 최대 6대(NYU FRL).
- "RGB to depth hardware sync is only supported with the **same frame rate for all streams**" → HW 동기 시 전 스트림 동일 fps 강제(FR-CAM-018).
- **크로스벤더 하드 동기는 불가.** RealSense끼리만. RealSense↔OpenCV 웹캠은 하드 동기 불가.
- **RealSense 한 대의 컬러+뎁스는 장치 내부에서 동일 fps 락스텝**이므로(§2.2), 그 쌍은 별도 동기 배선 없이 시간 정렬이 보장된다 — 뎁스를 쓰는 편의 이점.

**(b) 소프트 동기 (ROS 2 `message_filters` `ApproximateTime`)**
```python
ApproximateTimeSynchronizer(fs, queue_size, slop, queue_offset=False,
                            allow_headerless=False, sync_arrival_time=False)
```
docstring: `allow_headerless`/`sync_arrival_time`은 "avoid this as much as you can, since the delays are unpredictable." `ExactTime`은 "exactly the same timestamp" → HW 동기 없이 사용 불가.
**`slop`의 물리적 하한**: 30 fps에서 프레임 간격 33.3 ms → HW 동기 없는 두 카메라의 위상차 최대 **±16.7 ms**(반 프레임). `slop`을 이보다 작게 잡으면 매칭이 대량 실패한다(NFR-CAM-002). 권장 수치 1차 출처 없음 → 실측 확정(§5-Q4).

**(c) LeRobot는 프레임에 캡처 시각을 실어 주지 않는다 → `capture_ts`는 [신규구현]**
- LeRobot `Camera` 추상은 `connect / read / async_read / read_latest / disconnect`이고 반환값은 **ndarray뿐**이다. **프레임에 `capture_ts`를 부착할 상류 지점이 없다.**
- `openarm_follower.get_observation()`은 카메라를 **순차 호출**해 obs dict에 담는다 → 여러 스트림의 실제 grab 시각은 한 사이클 안에서 순차적으로 벌어져 있고, 그 차이는 어디에도 기록되지 않는다.
- → **우리 `Camera` 서브클래스 안에서 grab 직후 `CLOCK_MONOTONIC`을 찍어 사이드카로 내보내는 것이 유일하게 올바른 지점이다**(FR-CAM-014). 소비자(레코더/GUI)가 수신 시각으로 찍으면 순차 호출 지연이 타임스탬프에 섞여 동기 오차 분석이 무의미해진다.

**(d) LeRobot의 `timestamp`는 실측값이 아니다**
`dataset_writer.py:206-210`: `timestamp = frame_index / self._meta.fps` — **합성 타임스탬프**. 멀티카메라 지연·지터가 데이터셋에 기록되지 않는다. 읽기 허용오차 `tolerance_s=1e-4`(`lerobot_dataset.py:55,170-174`)는 합성값끼리의 검사라 실 지터를 못 잡는다. → **실측 캡처 타임스탬프를 별도 사이드카로 우리가 기록**한다(FR-CAM-016).

### 2.7 캘리브레이션

**Intrinsic**
- **OpenCV 웹캠(로지텍 등)**: UVC/V4L2 원시 장치 → 공장 intrinsic 조회 표준 API 없음 → **체커보드/ChArUco intrinsic 캘리브레이션 필수**(온보딩 단계, FR-CAM-024).
- **RealSense**: 공장 캘리브레이션 intrinsic을 펌웨어에 보유 → `camera_info`(librealsense `rs2_intrinsics`)로 조회 가능. 뎁스↔컬러 정렬(align)도 SDK가 제공. → 뎁스를 쓰면 intrinsic·정렬이 표준으로 제공되는 이점.
- 재캘리브레이션: 체커보드/ChArUco + `cv2.calibrateCamera`.

**Hand-eye (AX = XB)** — OpenCV `calibrateHandEye`
- 입력: `R_gripper2base`, `t_gripper2base`, `R_target2cam`, `t_target2cam` → 출력: `R_cam2gripper`, `t_cam2gripper`.
- 메서드: 분리해 **TSAI** / **PARK** / **HORAUD**, 동시해 **ANDREFF** / **DANIILIDIS**(dual quaternion).
- 구성: 손목 카메라 = **eye-in-hand**, 정면/상방 = **eye-to-hand**.
- ⚠️ 알려진 결함: TSAI가 eye-to-hand에서 잘못된 값을 낸다(opencv#20974), "Most hand-eye calibration methods don't work"(opencv#24871) → **복수 메서드 교차검증 + 재투영 오차 리포트 필수**(FR-CAM-026).
- `easy_handeye2`(ROS 2): `calibration_type: eye_in_hand|eye_to_hand`, ArUco, 결과 **YAML 영속화** 후 TF 발행. 권장 샘플 수 미명시 → [미확인](§5-Q9).

**뎁스 인코딩** (§2.4): `raw depth → clip[depth_min,depth_max] → 12bit 로그 양자화 → gray12le → HEVC Main 12(무손실) → MP4`. **뎁스는 기본 무손실이라 RGB(손실 AV1)보다 파일이 크다** — 뎁스 카메라를 여러 대 붙이면 스토리지·인코딩 부하가 RGB 다중보다 급증한다(NFR-CAM-005, 대역폭 §2.9).

**2단계 저장 — 이것이 우리의 녹화 경로다 (FR-CAM-032)**
`streaming_encoding`의 upstream 기본값 `False`를 **그대로 유지한다.** LeRobot은 수집 중 프레임을 이미지 라이터 스레드로 디스크에 흘리고(`num_image_writer_processes=0`, `num_image_writer_threads_per_camera=4`), **에피소드 종료 시점(`save_episode()`)에 배치 인코딩**한다.
1. **수집 중**: 원본 프레임을 디스크에 기록 → 캡처 루프에 인코딩 부하를 싣지 않아 드롭 위험을 캡처 단계에서 제거.
2. **에피소드 종료 후**: **배치 트랜스코딩**으로 LeRobot 비디오 포맷으로 변환. 변환 전후 **프레임 수·타임스탬프 일치를 검증**한 뒤에만 원본을 정리.

> 🔴 **중간 산물은 무손실 PNG다** (`IMAGE_FILE_PATTERN="frame-{frame_index:06d}.png"`, `datasets/utils.py:82-98`). JPEG q90이 아니다(그것은 dora 노드의 출력 인코딩이었고 우리 경로에 없다). 뎁스 중간 산물은 `.tiff`(`DEPTH_FILE_PATTERN`). → `15-비기능-요구사항.md` `FR-NFR-028`의 디스크 예산과 `14-시스템-운영.md` F14 디스크 임계는 **PNG(+뎁스 tiff/HEVC12) 기준으로 재산정 대상**이다([결정필요] §5-Q13).

**백프레셔 (verbatim)**: "bounded queue(default 30). When the encoder can't keep up: queue fills → new frames are **dropped** (not blocked) → warning `"Encoder queue full for {camera}, dropped N frame(s)"` → at episode end, **total dropped frames per camera are reported**."
**드롭 허용 기준 (verbatim)**: "~**5%** … system is likely overloaded", "~**2%** … can be considered **expected**."

### 2.9 대역폭 예산 — 다중 카메라의 1차 병목

**산식(비압축 기준)**: `Mbps = W × H × Bpp × 8 × fps / 1e6`
- RGB8/YUYV 2 Bpp, 뎁스 z16 2 Bpp.
- 640×480 YUYV @30 = **147.5 Mbps**; 1280×720 RGB @30 = **663 Mbps** (Clearpath "> 660 Mbps").
- **RealSense 한 대 RGB+뎁스**(640×480@30, 각 2 Bpp) = `2 × 147.5` = **295 Mbps**.

| 항목 | 값 | 출처 |
|---|---|---|
| USB3 이론 대역폭 | 5000 Mbps | RealSense 멀티카메라 문서 |
| USB3 **실효** 대역폭 | **3200–3600 Mbps** | nomagic.ai |
| D415 ×4 컬러+뎁스 풀해상도 | **3539 Mbps → 실패** | 동상 |
| D415 ×4, 640×360 | **882 Mbps → 성공** | 동상 |
| 호스트 허브 공유 | "ports may **share the same 5000Mbps bandwidth**" | 동상 |
| 대책 | "Purchase a **PCIe card** … separate bandwidth for each port" | 동상 |
| 전형 증상 | `"Frame did not arrive in time"` — **대역폭 또는 전력** 부족 양쪽 | librealsense#1873 |

읽어낼 것:
- **뎁스를 켜면 카메라당 대역폭이 대략 2배가 된다**(컬러+뎁스). 뎁스 카메라 수 × (RGB+뎁스) 로 예산이 급증하므로, 뎁스 카메라 대수 상한을 대역폭·전력·스토리지로 판정한다(FR-CAM-010/011, NFR-CAM-004).
- **USB 링크에서 MJPEG로 협상하면 실제 대역은 크게 낮다.** 그러나 협상 포맷은 드라이버 재량(`OpenCVCameraConfig.fourcc` 기본 `None`)이므로 `v4l2-ctl --list-formats-ext`로 확인하고 `fourcc`를 명시해야 예산이 확정된다(FR-CAM-071). (RealSense 뎁스 z16은 무압축.)
- **허브 공유는 흔한 병목이다.** 여러 카메라를 한 루트 허브에 몰면 실효 상한을 공유한다. `lsusb -t`로 컨트롤러별 합을 판정한다(FR-CAM-008, NFR-CAM-004).
- **프리뷰 raw 전송은 성립하지 않는다 — 반드시 저해상도 JPEG로 인코딩해 보낸다** — 1280×720 RGB@30 = >660 Mbps/카메라. 웹 단일 GUI에서 프리뷰는 브라우저↔백엔드 **WebSocket을 건너므로**(§2.10) 원본을 그대로 실으면 카메라 한 대만으로 링크를 포화시킨다. 이 >660 Mbps는 USB 캡처 예산이 아니라 **프리뷰를 인코딩하지 않고 네트워크로 보낼 때의** 전송 경로 수치이며, 그래서 백엔드 JPEG 인코딩이 웹 선택의 **필수 비용**이다(§2.10). (녹화 스트림은 이 전송 경로를 타지 않는다 — 원본은 디스크에 흐르고 배치 인코딩된다, §2.8.)

### 2.10 프리뷰 = 웹 스트리밍 (JPEG-over-WebSocket)

> 🔴 **웹 단일 GUI로 확정 (DECISIONS-v3 말미).** 아키텍처는 **헤드리스 파이썬 백엔드(FastAPI) + 웹 SPA 프론트**다. 백엔드가 CAN·LeRobot·안전계층·`robot` 객체를 소유(connect 1회, 모드=상태전이, `flock`)하고, 프론트는 브라우저에서 3D(Three.js + urdf-loader)와 카메라 프리뷰를 그린다. **프리뷰는 이제 프로세스 경계·네트워크를 건넌다** — 이전 판(PyQt6 데스크톱 인프로세스 위젯 렌더)은 무효다. 카메라 프레임을 브라우저로 실어 나르는 것이 **웹을 택한 대가의 실재하는 유일한 큰 구현**이다.

**전송 계약 (D-2 프로토콜 통일):**
- **단일 WebSocket 하나**가 실시간 양방향을 전부 나른다 — 텔레옵 명령 + 텔레메트리(관절/토크/상태) + **카메라 프레임(바이너리 JPEG)**. 카메라 프레임은 **카메라ID·스트림종류(RGB/뎁스) 태그로 멀티플렉싱**해 같은 WS에 실어 여러 카메라를 구분한다. **별도 WebRTC/RTSP/Foxglove 스트림을 분리하지 않는다**(사용자가 오버엔지니어링으로 배제).
- **gRPC-web은 브라우저 양방향 스트리밍 제약으로 부적합하여 WS를 택했다.** gRPC는 **백엔드↔원격 추론(LeRobot async PolicyServer :8080) 홉에만** 쓰이며 브라우저 경계에는 없다.
- 비실시간 CRUD(카메라 설정 get/set, 캘리브레이션·데이터셋 조회 등)는 표준 FastAPI **HTTP REST**로 나눈다 — 이는 프로토콜 분리가 아니라 실시간/비실시간의 자연스러운 구분이다.

**인코딩 계약:**
- **RGB 프리뷰**: 백엔드가 **논블로킹** `read_latest(max_age_ms)`로 최신 프레임을 peek → **저해상도로 다운스케일 → `cv2.imencode('.jpg', …)`** → WS 바이너리 프레임으로 전송. 프리뷰는 표시 전용이므로 해상도·품질·FPS를 녹화와 독립으로 낮춘다(FR-CAM-047/087).
- **뎁스 프리뷰**: `(H,W,1)` uint16 mm를 **16-bit PNG** 또는 **컬러맵(`cv2.applyColorMap`) JPEG**로 인코딩해 같은 WS에 태그해 보낸다. (녹화용 뎁스는 이 경로가 아니라 무손실 HEVC12로 별도 인코딩된다 — §2.4/§2.8.)
- 캡처 루프(`async_read()`)와 프리뷰 인코딩(`read_latest()` → JPEG)은 서로를 블로킹하지 않는다(§2.3). 프리뷰 인코딩·WS 전송이 밀리면 **프리뷰 프레임을 드롭**할 뿐 캡처·녹화를 막지 않는다(NFR-CAM-006).

- **표시 지연의 최대 기여자는 여전히 카메라+USB 캡처(~100 ms)다** — 여기에 백엔드 JPEG 인코딩·WS 전송·브라우저 디코드/페인트가 더해진다. 프리뷰 스택을 최적화해도 ~100 ms 항은 고정 비용이다(NFR-CAM-001).

### 2.11 참조 하드웨어 각주 — OpenArm Cell (Arducam B0495 + ZED mini)

> ⚠️ **이 절은 소프트웨어 계약이 아니라 하드웨어 참조설계다.** OpenArm Cell(v2.0 표준 평가 인클로저)의 공식 BOM은 카메라로 **Arducam B0495 ×3**(좌 EE / 우 EE / 천장) + **Stereolabs ZED mini(ZED-121210) ×1**(헤드)를 쓴다. 그러나 §2.1이 못 박듯 **우리 소프트웨어는 이 구성에 얽매이지 않는다** — 카메라 dict는 빈 dict에서 사용자가 채운다. 아래는 Cell 참조 하드웨어를 실제로 쓰려는 사용자를 위한 정보이며, **정본 계약이 아니다.**

- **Arducam B0495**: AR0234 1/2.6" 2.3 MP, **글로벌 셔터**, YUYV 무압축, USB 3.0 UVC(드라이버 불필요), 최대 80 fps @ 960×600(네이티브 모드), M12 대각 FOV 95°, **뎁스 없음**. 외부 트리거 모드 존재(Arducam 포럼).
- **ZED mini(ZED-121210)**: 1/3" 4 MP ×2, 2×(1280×720)@60 fps를 **side-by-side 한 프레임(2560×720)** 으로 출력, 베이스라인 63 mm, USB 3.0 Type-C(`2b03:f682`). 규격상 뎁스 0.10–15 m지만 **뎁스는 ZED SDK로만 얻는다.**
- **udev 안정화**: Arducam 3대는 공장 출하 시 UVC 시리얼이 전부 동일 → `ArducamUvcConfigUpdateTool`로 `CELL<n>_CAM_{LEFT,RIGHT,CEILING}`을 구워 넣어야 심링크가 성립. ZED-M은 `2b03:f682`의 V4L2 장치.
- **USB 토폴로지**: EE 2대 + 헤드 1대 → **Belkin AVC018BTBK 허브**, **Sanwa KB-USB-RCC305 리피터**로 연장 + 별도 급전; 천장은 **Sanwa KB-USB-RLC305** 단독.

**이 참조 하드웨어를 우리 LeRobot 경로에 쓸 때의 제약 (왜 각주로 격하했나):**
1. **ZED mini는 LeRobot 표준 카메라 클래스에 없다**(opencv/intelrealsense/reachy2/zmq 뿐). ZED을 `type: opencv`로 그냥 열면 `2560×720` SBS 프레임이 **단일 텐서로 통째** 기록되어(종횡비 3.56:1) 정책 입력·스테레오 페어로 부적합하다. 좌우 분할을 하려면 **`lerobot_camera_*` 서드파티 플러그인 또는 ZMQ 브리지를 우리가 만들어야 한다**(FR-CAM-078, [신규구현·선택]). **정본 뎁스 경로는 ZED이 아니라 RealSense다.**
2. **Arducam B0495는 뎁스를 내지 않는다.** Cell 참조 하드웨어만으로는 **뎁스 스트림이 0개**다 — D-12의 뎁스 요구를 채우려면 이 구성에 **RealSense를 추가**하거나 ZED SDK를 도입해야 한다. 우리 표준은 **RealSense를 dict에 등록**하는 쪽이다(§2.2, §2.4).
3. **웹캠 튜닝 컨트롤은 OpenCV 웹캠 전용이다**(웹 패널이 값을 REST로 보내고 **백엔드가 cv2 `CAP_PROP`로 적용** — 참조 레포 `camera_controls.py`의 `CAMERA_CONTROLS` 로직을 UI와 무관하게 이식: brightness/contrast/saturation/sharpness/gain/backlight, auto_exposure+exposure, auto_wb+wb_temperature, auto_focus+focus, pan/tilt/zoom). **RealSense는 이 경로로 제어 불가**(librealsense SDK 별도) → RealSense는 config(해상도/fps/use_depth) 수준만 노출(FR-CAM-047, §3.7).

### 2.12 참조 레포 패턴 — tolerant 카메라 연결 [이식]

참조 레포(사용자 작업물 `bh_indy7_LeRobot`)는 카메라를 **관용적으로(tolerant) 연결**한다 — 이 패턴을 이식한다.
`lerobot_plugin_rebot/rebot.py` `_connect_cameras_tolerant`(L280-290):
```python
# Each camera opens independently — a dead /dev/video* is skipped with a
# warning and must never fail the arm connect (observation and recording
# degrade; motion does not).
for cam_name, cam_cfg in self.config.cameras.items():
    try:
        cam = self._make_camera(cam_cfg); cam.connect()
        self.cameras[cam_name] = cam
    except Exception as exc:
        logger.warning("reBot camera %r connect failed: %s", cam_name, exc)
```
- **카메라별 독립 오픈. 죽은 카메라는 경고 후 skip하고, 팔 connect를 실패시키지 않는다** — 관측·기록은 degrade하되 **모션은 영향받지 않는다.**
- 참조 레포는 `_make_camera`가 `OpenCVCamera` 하드코딩(L277-278)이고 `cameras: dict[str, OpenCVCameraConfig]`(config.py L133)이며 `observation_features`가 `(H,W,3)`만(follower_base.py L86-94), `get_observation`이 `async_read()`만(L111-120) — **v1-era로 뎁스가 없다.** 우리는 이 3곳을 확장한다:
  - `_make_camera`: OpenCV 하드코딩 → `cfg.type`으로 **OpenCV/RealSense 디스패치**(`make_cameras_from_configs`처럼).
  - `cameras` 타입: `dict[str, OpenCVCameraConfig]` → `dict[str, CameraConfig]`(RealSense 허용).
  - `observation_features`/`get_observation`: `use_depth`면 `{cam}_depth=(H,W,1)` 추가 + `read_latest_depth()`(0.6.x) 배선 — 0.6.1 `openarm_follower` 패턴 그대로.

## 4. 상태 · 오류 처리

### 4.1 카메라 상태 전이

```
UNBOUND ──(시리얼/경로 바인딩)──> BOUND ──(open + 프로파일 적용)──> OPENING
                                                                   │
                                         ┌────(프레임 도착)─────────┘
                                         ▼
                                     STREAMING ◄──────────────┐
                                         │                     │
                    (드롭률>2% | 링크 USB2 | 동기오차 초과)     │ (N프레임 연속 정상)
                                         ▼                     │
                                     DEGRADED ─────────────────┘
                                         │
                          (프레임 타임아웃 | USB 분리 | open 실패)
                                         ▼
                                       LOST ──(재열거 백오프)──> RECOVERING
                                         ▲                          │
                                         └────(재바인딩 실패)────────┘
                                                                    │
                                                      (시리얼 일치 + 프로파일 복원)
                                                                    ▼
                                                               STREAMING
```

| 상태 | 진입 조건 | GUI 표시 | 녹화 영향 |
|---|---|---|---|
| `UNBOUND` | 시리얼/경로 미배정 | 회색 타일 | 프리플라이트 실패 |
| `BOUND` | 배정, 미개방 | 회색 타일 | — |
| `OPENING` | `open()` 후 첫 프레임 대기 | 스피너 | 시작 차단 |
| `STREAMING` | 프레임 정상 수신 | 정상 | 정상 |
| `DEGRADED` | 드롭률 > 2%, 또는 USB2 링크, 또는 동기 오차 > `slop` | 노란 테두리 + 사유 | 에피소드 경고 플래그 |
| `LOST` | 프레임 타임아웃 초과 또는 장치 분리 | 빨간 테두리, 마지막 프레임 freeze + "LOST" | **녹화 중이면 에피소드 중단** |
| `RECOVERING` | 재열거 백오프 진행 중 | 빨간 테두리 + 재시도 횟수 | 녹화 재개 불가(에피소드 폐기 후 재기록) |

> **tolerant 연결(FR-CAM-084)과의 관계**: 수집 시작 시점에 죽어 있던 카메라는 `UNBOUND`/`LOST`로 남되 **팔 connect·모션을 실패시키지 않는다**(관측·기록만 degrade). 프리플라이트(FR-CAM-063)가 이를 사용자에게 표면화한다.

**녹화 중 `LOST` 발생 시의 규칙**: 카메라 1대라도 `LOST`가 되면 해당 에피소드는 **불완전 에피소드**다. 조용히 이어붙이지 말고 **폐기하거나 실패로 라벨링**한 뒤 재기록한다. 프레임 수가 맞아도 시간축이 깨진 데이터가 더 나쁘다.

### Q1. [확정·종결] 뎁스용 LeRobot 버전 — v0.6.1로 확정
**종결.** 정본 런타임은 **LeRobot v0.6.1**로 확정됐다(`02` §2.0 · `03` §2, `pyproject.toml:28 version="0.6.1"` — `02`:17·`03`:16과 일치). v0.6.1이 뎁스 관측·기록 파이프라인(`async_read_depth`/`read_latest_depth`, `{cam}_depth` feature)을 표준으로 제공하므로 D-12의 뎁스 요구가 표준 경로로 충족된다(§2.4, FR-CAM-083). 참조 레포가 핀했던 **0.5.1은 뎁스를 async/record 경로에서 지원하지 않으므로**(동기 `read_depth`만) 우리 경로가 아니고, 그 위의 직접 배선(옛 옵션 (b): `get_observation` 오버라이드 + `read_depth()` 폴링 + `{cam}_depth`·뎁스 인코더)은 채택하지 않는 대안으로만 남긴다. GROUND-v3 §5의 '뎁스용 0.6.x 상향 여부' 잔여 open은 이로써 닫힌다.

### Q2. [미확인] `slop`의 실제 권장값
30 fps 프레임 간격 33.3 ms에서 유도한 하한 16.7 ms는 산술값이다. **정하려면**: 실제 카메라 세트를 돌리며 슬롯 쌍별 `capture_ts` 차이 분포(q99)를 측정하고 매칭 성공률 99% 이상을 유지하는 최소 `slop`을 실측한다(FR-CAM-022).

### Q3. [미확인] 등록 카메라 세트가 목표 fps를 유지하는가
LeRobot record 루프의 실측 사이클 타임은 [미측정]이다. 우리 구성은 한 프로세스에 CAN 사이클(양팔 16모터) + 카메라 N스트림 grab(+뎁스) + 이미지 라이터 스레드가 얹힌다. **정하려면**: NFR-CAM-003/006 절차에 `record_loop()` 사이클 타임 히스토그램(p50/p95/p99)을 추가 계측한다. 달성 fps가 목표의 0.95배 미만이면 grab을 별도 스레드로 뺀다(`async_read()`의 백그라운드 스레드 계약이 1차 방어선 — §2.3).

### Q4. [미확인] hand-eye 권장 샘플 포즈 수
easy_handeye2 README에 명시 없음. **정하려면**: 5→30 포즈로 늘려가며 5개 메서드 간 결과 편차·재투영 오차가 수렴하는 지점을 측정한다.

### Q5. [미확인] 뎁스 정책 학습 실효성
0.6.1이 뎁스를 데이터셋에 기록하는 것은 소스로 확인했으나, 표준 정책(ACT/SmolVLA/π0)이 `observation.images.<cam>_depth`를 실제 입력으로 소비하는지(전처리/정규화)는 정책별 image key 매핑 확인이 필요하다. **정하려면**: 정책별 image key 매핑을 소스로 확인하고, 뎁스 포함/미포함 기준선을 실기 태스크로 비교한다.

### Q6. [미확인] 총 대역폭 상한 실측
뎁스를 켜면 카메라당 대역폭이 배증하고, 링크가 MJPEG로 협상되는지 YUYV인지가 실제 대역을 좌우한다(FR-CAM-071). **정하려면**: PCIe USB 확장 카드 유/무 두 조건에서 등록 세트를 10분 동시 녹화해 슬롯별 드롭률·달성 FPS를 측정한다(NFR-CAM-003/004). 합격 기준은 드롭률 2% 미만.

### Q7. [결정필요] 중간 산물 PNG/tiff 디스크 예산 재산정
2단계 저장의 중간 산물은 **무손실 PNG**(뎁스는 tiff → HEVC12 무손실)다. `15` `FR-NFR-028`·`14` F14의 디스크 임계는 JPEG q90 전제라 무효다. **정하려면**: 등록 세트 10분 수집으로 PNG(+뎁스) 실제 바이트/에피소드를 측정하고 배치 트랜스코딩 RTF(NFR-CAM-005)를 잰다. 두 수치로 예산을 다시 짠다.
