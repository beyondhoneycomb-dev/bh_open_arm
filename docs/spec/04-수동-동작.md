# 04. 수동 모터 동작 (Manual Jogging / Freedrive / 티칭)

---

## 1. 범위

이 문서는 오퍼레이터가 **직접 로봇을 움직이는 모든 경로** — 관절 조그(joint jog), 카테시안 조그(Cartesian jog), Freedrive(중력보상 hand-guiding), 티칭(teaching)/재생(replay), 홈 복귀 — 의 기능 요구사항을 정의한다. 정책 추론·VR 텔레옵·데이터 수집은 다루지 않는다(별도 문서).

모터 레지스터 설정(RID 쓰기, 보드레이트, ID 변경, 영점 캘리브레이션 절차 자체)은 **모터 설정 문서**의 영역이며, 여기서는 수동 동작이 그 결과에 **의존하는 지점**(영점 프레임, 게인 프로파일, 리밋 정본)만 전제로 인용한다. 충돌 감지 알고리즘의 내부(GMO 관측기 설계)도 **충돌·안전 문서**의 영역이며, 여기서는 **조그/Freedrive 중의 연동 계약**만 규정한다.

요구사항 ID 영역 코드: **`MAN`**.

---

## 2. 확정된 사실 기반

### 2.1 자유도 · 액추에이터 · 물리 한계

OpenArm v2.0은 팔당 **7-DOF + 그리퍼(J8)** = 8모터. 좌/우 팔은 **각각 별도 CAN 버스**(enactic 4개 소스 모두 `can0`=right, `can1`=left).

| 관절 | 모터 | send/recv CAN ID | pMax [rad] | vMax [rad/s] | **tMax [Nm]** (패킷 스케일) | **Peak Torque [Nm]** (물리) | URDF effort |
|---|---|---|---|---|---|---|---|
| J1 | DM8009 | 0x01 / 0x11 | 12.5 | 45 | 54 | **40** | 40 |
| J2 | DM8009 | 0x02 / 0x12 | 12.5 | 45 | 54 | **40** | 40 |
| J3 | DM4340 | 0x03 / 0x13 | 12.5 | 8 | 28 | **27** | 27 |
| J4 | DM4340 | 0x04 / 0x14 | 12.5 | 8 | 28 | **27** | 27 |
| J5 | DM4310 | 0x05 / 0x15 | 12.5 | 30 | 10 | **7** | 7 |
| J6 | DM4310 | 0x06 / 0x16 | 12.5 | 30 | 10 | **7** | 7 |
| J7 | DM4310 | 0x07 / 0x17 | 12.5 | 30 | 10 | **7** | 7 |
| J8 (grip) | DM4310 | 0x08 / 0x18 | 12.5 | 30 | 10 | **7** | — |

> 출처: `openarm_can/include/openarm/damiao_motor/dm_motor_constants.hpp` (`MOTOR_LIMIT_PARAMS`), `openarm_driver/configs/openarm_cell.yaml` (`motor_config`), `openarm/website/docs/hardware/openarm-2.0/motor.mdx` (BoM: J5–J8 = DM-J4310-2EC V1.1), `openarm_description/.../openarm_v2.0/config/arm/joint/joint_limits.yaml`.

**tMax ≠ Peak Torque.** tMax는 CAN 12-bit 토크 필드의 **스케일 상한**이지 물리 한계가 아니다. URDF effort 리밋이 물리 peak torque와 정확히 일치하므로, **토크 지령 클램프의 정본은 URDF effort**다. tMax까지 열면 모터 정격을 초과 지령한다.

토크 12-bit 분해능(잔차 임계값 하한을 결정): DM8009 ≈ **0.0264 Nm/LSB**, DM4340 ≈ **0.0137**, DM4310 ≈ **0.0049**.

페이로드: 정격 **4.1 kg** / 피크 **6.0 kg** (엔드이펙터 무게 포함). 각 관절에 **기계적 스토퍼** 존재. **기계식 브레이크는 없다** — 토크를 끊으면 중력으로 낙하한다.

### 2.3 좌표 프레임 규약

| 프레임 | 정의 | 누가 쓰나 |
|---|---|---|
| **F_URDF** | URDF 관절각(**rad**). **사용자 좌표 — 본 명세의 정본** | URDF, MuJoCo/MJCF, `openarm_control`(IK) |
| **F_motor** | 마지막 `set_zero`(CAN `0xFE`)를 받은 자세를 0으로 하는 모터각 | CAN 버스 위의 실제 지령/피드백 |
| **LeRobot API 경계** | **deg.** 액션·관측 `.pos`가 전부 deg (`.vel`=deg/s, `.torque`=**Nm 원단위**) | `robot.send_action()` / `robot.get_observation()` |
| 변환 | 🔴 **LeRobot은 `joint_offsets`를 적용하지 않는다** — `send_action()`(`openarm_follower.py:288-341`)에 offset 가산이 **없다**. 가산하는 것은 `openarm_driver`뿐(우리 런타임이 아님) | — |

```yaml
# openarm_driver/configs/openarm_cell.yaml
joint_offsets:
  right_arm: [0.0, -0.506145,  1.570796, -1.745329, 0.0,  0.331612, -1.570796, 0.0]
  left_arm:  [0.0,  0.506145, -1.570796, -1.745329, 0.0, -0.331612,  1.570796, 0.0]
```
→ **오프셋 이중 가산/미가산이 곧 관절각 전체의 오프셋 오류**다. GUI는 사용자 좌표(F_URDF)만 다루고, 가산은 파이프라인의 정확히 한 지점에서만 일어나야 한다.

**월드/베이스 축** (MJCF 1차 확인): base_link의 월드 대비 회전은 **단위행렬**(quat 속성 없음). 축 규약 **X=전방, Y=좌, Z=상** (우수). base 원점 = `(0.185, ∓0.031, 1.34 + q_lift)` [m], `q_lift ∈ [0, 0.3]`(Cell 리프터).
→ **base 프레임과 world 프레임은 회전이 동일하고 위치만 오프셋**이다. 카테시안 조그의 `base` / `world` 프레임은 **회전 축이 같고 원점만 다르다**.
> 출처: `openarm_mujoco/v2/openarm_bimanual.xml:174,262`, `cell.xml:91,95`.

**EE 제어점**: `right_ee_control_point` / `left_ee_control_point` site는 `ee_base_link` 원점(`pos="0 0 0"`) = **joint7 축 원점 = 손목**이다. 핑거 피벗은 여기서 z −0.068 m, 파지점(TCP)은 더 아래. → **기본 EE 타깃은 파지점이 아니라 손목이다.** 조작감에 직결된다.

**좌우 미러 규약**: `q_left = [-1, -1, -1, +1, -1, -1, -1] ⊙ q_right` (**joint4만 동부호**). 수치 검증 오차 0.0, `default_bimanual.yaml`의 `reflect.include.joints: [1,2,3,5,6,7]`(joint4 부재)와 독립 일치.

### 2.4 🔴 수동 동작은 **별도 프로세스가 아니라 같은 robot 객체 위의 모드**다 [확정]

**수동 동작 세션은 CAN을 새로 열지 않는다.** 정본 런타임은 LeRobot이며(→ `02` §2.0), `OpenArmFollower` / `BiOpenArmFollower` 객체가 **세션당 한 번** `connect()`로 버스를 연다. 조그·Freedrive·티칭·재생은 전부 **그 객체 위의 컨트롤 루프 교체**다.

> 🔴 **수동 동작 진입/이탈에 `robot.disconnect()` / `robot.connect()`를 호출해서는 안 된다.**
> LeRobot **기본** `OpenArmFollower.connect()`는 `if self.is_calibrated: self.bus.set_zero_position()`(`openarm_follower.py:152-153`)을 무조건 실행하며 → CAN `0xFE` → 현재 물리 자세가 영점으로 확정된다. 이를 끄는 **설정 플래그가 없으므로 `OpenArmFollower`를 상속해 connect()에서 auto-set-zero를 제거**한다(→ `02` FR-CON-061). 오버라이드 후에는 재연결이 영점을 파괴하지 않으나, 조그 세션은 여전히 CAN 소켓·flock 위생상 같은 robot 객체 위의 모드 전이로만 구현한다.
> → 수동 동작 모드는 **`send_action()` 호출권을 넘겨받는 상태 전이**로만 구현한다.

**그래도 CAN 배타성 문제는 사라지지 않는다.** **SocketCAN은 배타 bind를 제공하지 않는다** — 두 프로세스가 같은 인터페이스를 bind하면 **둘 다 성공하고**, 로컬 루프백이 기본 ON이라 서로의 송신 프레임까지 받는다. **에러가 나지 않는다** → 조용한 실패.

LeRobot을 단일 프로세스에 임베드하면 *프로세스 내부* 배타성은 해결되지만, **외부 침입자는 못 막는다**:
`cansend` · `openarm-can-cli`(`discover`/`can_configure`도 **송신한다**) · **두 번째 LeRobot 인스턴스**(→ 영점 파괴까지 동반) · `openarm_teleop`(C++ 바이래터럴).

→ **`flock` 애플리케이션 락이 필요하다**(→ `02` FR-CON-010). 역할은 "모드 중재"가 아니라 **"외부 침입자 방어"**다.

**배타성의 단위가 내려간다**: "누가 버스를 소유하는가" → **"누가 `send_action()`을 부를 권리를 갖는가."** 조그 루프와 텔레옵 루프와 정책 루프는 **전부 같은 `send_action()`을 호출**한다. 같은 객체를 공유해도 두 루프가 동시에 돌면 명령이 인터리브된다 — **LeRobot에 중재자는 없다.** 전이 시 이전 루프의 종료를 join으로 확인한 뒤 새 루프를 시작해 **겹침 0을 보장**해야 한다(FR-MAN-001).

⚠️ `openarm_can`(C++)과 LeRobot `DamiaoMotorsBus`(python-can)는 **완전히 독립된 두 스택**이다 — LeRobot은 `openarm_can`을 **링크하지 않는다**. 즉 **서로의 존재를 알 방법이 구조적으로 없다.** 원시 CAN 진단(RID·`discover`·`clear_error`)은 **LeRobot 컨트롤 루프를 정지시킨 `MOTOR_CONFIG` 모드에서만** 허용한다(→ `03` FR-MOT-007). **그때도 `disconnect()`는 하지 않는다.**

### 2.5 지령 계약 — MIT 모드

```cpp
struct MITParam { double kp; double kd; double q; double dq; double tau; };
// 12-bit 인코딩: kp ∈ [0, 500], kd ∈ [0, 5]  (모터 무관 고정)
// q  : 16-bit, [-pMax, +pMax]
// dq : 12-bit, [-vMax, +vMax]
// tau: 12-bit, [-tMax, +tMax]
```
> 출처: `openarm_can/src/openarm/damiao_motor/dm_motor_control.cpp` `pack_mit_control_data()`.

수동 동작의 모든 관절 지령은 **MIT 위치 지령**으로 내려간다.

#### 🔴 LeRobot 경로가 실제로 보내는 것 — `send_action()`의 실체 [확정]

`robots/openarm_follower/openarm_follower.py:288-343`:
```python
goal_pos = {k.removesuffix(".pos"): v for k, v in action.items() if k.endswith(".pos")}   # :288
# :290-297  config.joint_limits로 조용히 클리핑 (logger.debug 뿐)
# :299-304  max_relative_target (기본 None = 꺼짐)
commands[motor_name] = (kp, kd, position_degrees, 0.0, 0.0)   # :340  ← 🔴 dq=0, tau=0 하드코딩
self.bus._mit_control_batch(commands)                          # :342  ← 8모터 전부 (그리퍼 포함)
```

**여기서 확정되는 4가지 — 수동 동작 설계를 지배한다:**

| # | 사실 | 함의 |
|---|---|---|
| 1 | **액션 단위는 deg** (`{joint_1.pos: <deg>}`), MIT 인코딩 직전 `np.radians()` 변환 (`damiao.py:439-441`) | 사용자 좌표(rad) ↔ LeRobot(deg) 변환이 필요하다. 관측 `.torque`는 **Nm 원단위**(`damiao.py:568`) |
| 2 | **`tau`가 항상 0** (`:340`) | 🔴 **Freedrive(중력보상), 가상벽 반발 토크, 바이래터럴이 이 경로로는 불가능하다.** `send_action()`을 우회해야 한다 → `03` FR-MOT-058 |
| 3 | **그리퍼도 MIT로 나간다** (`motor_index`에 `"gripper": 7`, `:308-317`) | **POS_FORCE 그리퍼 힘 제어가 LeRobot 경로에 없다** → `03` FR-MOT-046 |
| 4 | **버스 레벨은 이미 tau를 받는다** — `damiao.py:492-502` `_mit_control_batch(commands: {motor: (kp, kd, pos_deg, vel_deg_s, torque)})` | **막힌 것은 상위 함수 하나뿐이다.** `DamiaoMotorsBus`는 손댈 필요 없다 |

**게인 전환 지점은 이미 열려 있다** — `send_action(action, custom_kp=None, custom_kd=None)`(`openarm_follower.py:268-273`, `:322-337`)이 모터명→게인 dict를 받는다. 조그/Freedrive/재생별 게인 프로파일은 이 인자에 건다.

#### 🔴 LeRobot에는 속도·가속도·저크 제한이 **없다** [확정]

`robots/` 트리 전체 grep: `velocity_limit` / `acceleration_limit` / `jerk` → **0건.**
LeRobot이 제공하는 안전장치는 **`joint_limits`(위치 클램프) + `max_relative_target`이 전부**다.

`max_relative_target: float | dict[str, float] | None = None` (`config_openarm_follower.py:75`) — **기본 `None` = 꺼짐**:
- **스텝당 위치 델타 상한, 단위 deg, dt 무관.** `ensure_safe_goal_position()`(`robots/utils.py:91-118`)이 `safe_goal = present + clamp(goal − present, ±cap)`로 자른다.
- 활성화 시 매 스텝 `bus.sync_read("Present_Position")`이 **추가로 돈다** — 소스 주석 원문: *"/!\ Slower fps expected due to reading from the follower."*
- ⚠️ **이것은 속도 제한이 아니다.** dt와 무관하므로 루프 주파수가 올라가면 실효 속도도 함께 올라간다. `openarm_driver`의 `joint_delta_position_limits`와 정확히 같은 함정이다(§2.2).

> **⇒ 조그 속도 클램프, 가속도/저크 제한, 소프트 E-Stop, 가상벽은 전부 우리가 만든다**(FR-MAN-011, FR-MAN-012, FR-MAN-036, FR-MAN-052).

⚠ **Damiao 공식 경고**: *"When controlling the position, **kd cannot be set to 0. Otherwise it will cause the motor vibrate or go out of control.**"* → Freedrive 이탈 후 위치 제어 복귀 시 **kd=0인 상태로 위치 지령을 보내면 안 된다.**

**제어 주파수 상한**: `openarm_can` 공식 경고 — **8모터 구성에서 1000 Hz 초과 제어 사이클은 CAN 연결을 불안정하게 만든다.** 🔴 `[개정 2026-07-15]` LeRobot 기본 조그 루프는 **패턴 B = 32프레임/사이클**(`_batch_refresh`가 REFRESH 라운드를 얹음 — 15 §2.1 정본): [계산] CAN 버스 시간 ≈**1664 µs/사이클** → CAN-FD 실질 상한 **≈625 Hz**, 500 Hz ≈**80%**, **1000 Hz 불가.** 고주파는 **패턴 A**(get_observation 매사이클 미호출) = 16프레임 → ≈1250 Hz. 종전 '2프레임 832 µs → 1 kHz ≈83%'는 패턴 A만 가정한 과소추정이며, **16 vs 32 확정은 candump 실측**(NFR-MAN-001).
🔴 **LeRobot Python 루프의 실측 사이클 타임은 [미측정]이다** — `sync_read_all_states()` + `_mit_control_batch()` 왕복이 GIL 아래서 몇 Hz를 내는지 알려져 있지 않다. 조그 루프 주파수는 **실측으로 확정**한다(NFR-MAN-001).

**LeRobot 타임아웃 상수** (`damiao.py:62-65`): `LONG=100 ms` / `MEDIUM=10 ms` / `SHORT=1 ms`(`_mit_control_batch` 응답 수집) / `PRECISE=100 µs`.
⚠️ **`disable_torque()`는 즉시 정지 경로가 아니다** — 모터마다 순차 송신 후 최대 1 ms 블로킹 대기(`damiao.py:279-294`, `:355-357`) → 단완 최대 ~8 ms, 양팔 ~16 ms. **즉시 정지(Cat 2)는 MIT 위치 홀드 프레임 1개**로 구현한다(→ `03` NFR-MOT-005).

### 2.7 카테시안 조그 백엔드 — `openarm_control` (mink + MuJoCo + daqp) [확정, 실측 완료]

**자체 IK를 작성하지 않는다. IK 솔버를 교체하지도 않는다.** `openarm_control`을 쓴다. 상류 공식 VR dataflow가 이미 이것을 bimanual 모드로 실사용한다.

**🔴 실측 [확정]: 양팔 기본설정 0.345 ms / p95 0.355 ms, 500/500 성공.** 60 Hz 예산의 **2%**, 500 Hz 예산(2 ms)의 **17%**. → **IK는 병목이 아니다.**

```python
from openarm_control import Kinematics, IKParams
kin = Kinematics(setup, IKParams(damping=0.1, posture_cost=0.01, dt=0.1, max_iters=10))
kin.sync(values16)                   # float32[16] = right[0:8] + left[8:16], **rad** — 현재 관절각 주입
kin.set_target("right", pose_r)      # pose = float32[7] = [px,py,pz, qw,qx,qy,qz] — **MuJoCo world 프레임**
result = kin.solve()                 # → float32[16] (right[8] + left[8]), **rad** | None
```

> ⚠️ **`solve()`는 인자를 받지 않는다** — `kinematics.py:107`: `def solve(self) -> np.ndarray | None`.
> **README/docstring의 `solve(dt=0.1, n_iters=5)`는 틀렸다**(`kinematics.py:27`의 예제 주석). `dt` / `max_iters`는 **`IKParams` 생성자로만** 준다(`:47-58`).

| 파라미터 | 라이브러리 기본값 (`kinematics.py:47-58`) | **상류 튜닝값 (그대로 채용)** |
|---|---|---|
| `position_cost` / `orientation_cost` | 1.0 / 1.0 | 동일 |
| `lm_damping` (per-task LM) | 0.01 | `--lm-damping 0.01` |
| `damping` (전역 Tikhonov) | 0.25 | **`--damping 0.1`** |
| `posture_cost` (중립자세 태스크) | 0.01 | `--posture-cost 0.01` |
| `dt` (적분 스텝) | 0.1 | `--dt 0.1` |
| `max_iters` | 5 | **`--max-iters 10`** |
| `solver` | `daqp` | 동일 |
| limits | `mink.ConfigurationLimit` 항상 + `VelocityLimit`은 **opt-in(기본 비활성)** | 기본 비활성 |
| 속도 캡 (opt-in) | `ARM_JOINT_VELOCITY_LIMITS_RAD_S` (`config.py:39-40`) | — |

출처: `enactic/dora-openarm-vr/config/dataflow-mujoco.yaml`의 `dora-openarm-ik` 노드 인자 = `--mode bimanual --max-iters 10 --dt 0.1 --damping 0.1 --posture-cost 0.01 --lm-damping 0.01`. **상류가 실기에서 검증한 레퍼런스 설정이다.**

- **타깃 프레임은 MuJoCo world**다 (mink `FrameTask.set_target(transform_target_to_world)`). base는 월드에서 회전 없이 평행이동만 되어 있으므로 **회전은 동일, 위치만 오프셋**.
- **비구동 DOF는 `mink.DofFreezingTask`로 freeze**(리프터·핑거 포함) — `kinematics.py:164-165`.
- **`mink.CollisionAvoidanceLimit`은 사용하지 않는다** → QP IK 단계에 충돌 회피 제약이 **없다**.
- **단위: `openarm_control`은 전부 rad. LeRobot은 deg.** 변환 필수(→ `03` §2.12).

#### ⚠️ 반드시 다뤄야 하는 4가지

1. 🔴 **조인트 리밋 불일치 — IK가 LeRobot이 클램프할 해를 낸다.**
   `mink.ConfigurationLimit`은 **MJCF `jnt_range`만 본다.** MJCF right joint_1 = **−80°…+200°**, LeRobot 소프트 리밋 = **−75°…+75°**.
   → IK가 리밋 밖의 해를 내고, `send_action()`이 그것을 **조용히 클리핑**한다(`openarm_follower.py:290-297`, `logger.debug`뿐) → **EE가 타깃에 도달하지 않는데 오퍼레이터는 이유를 모른다.**
   **해법**: `ArmSetup` 생성 후 **`model.jnt_range`를 LeRobot 리밋으로 덮어쓴 뒤** `Kinematics(...)`를 만든다. **순서가 중요하다** — `Kinematics` 생성 시점에 limit 객체가 만들어진다.

2. 🔴 **IK 폴백이 리밋을 버린다.**
   `kinematics.py:218-234` — `NoSolutionFound` → **`limits=[]`로 재시도**한다:
   ```python
   except mink.exceptions.NoSolutionFound:
       try:
           vel = mink.solve_ik(self._config, tasks, self._dt, self._solver_name,
                               limits=[],  # ← 조인트 리밋을 포기한다
                               constraints=constraints, safety_break=False, **self._solver_params)
       except mink.exceptions.NoSolutionFound:
           print("Warning: IK solver failed (constrained and unconstrained). Skipping step.")
           return None
   ```
   → **조인트 리밋을 포기한 해가 나올 수 있다.** **어댑터 레벨 클램프가 1차 방어선이어야 한다**(FR-MAN-028).

3. **IK 실패가 조용하다.** 최종 실패는 `print()` 한 줄과 `None` 반환뿐 → **스텝 스킵 = 조용히 안 움직인다.** 오퍼레이터는 "버튼이 안 먹는다"고만 인식한다 → 그대로 노출해서는 안 된다(FR-MAN-025).

4. **LeRobot도 자체 IK를 갖고 있다** — `model/kinematics.py`의 `RobotKinematics`(**placo** 기반) + `robots/so_follower/robot_kinematic_processor.py`의 등록 스텝 8개. **`openarm_control`을 택하면 LeRobot의 EE processor 경로와 EE 공간 데이터셋 기록을 포기하는 것**이다. 무비용이 아니라 **분기**다. (본 명세는 `openarm_control`을 택한다 — 상류 검증 계약이 있고 실측이 있다.)

**리스크**: `openarm_control` 릴리스는 **`0.1.0` 하나**, **테스트 디렉터리 없음**(직접 확인).

⚠ **MoveIt은 우리 경로에 없다.** (참고: OpenArm 공식 MoveIt은 KDL 플러그인 + `kinematics_solver_timeout: 0.005`로 7-DoF 여유자유도 실시간 조그에 부적합하다. 그러나 이제 **런타임 자체가 ROS 2가 아니므로 이 비교는 더 이상 결정에 관여하지 않는다.**)

⚠ **리프터 높이가 IK 모델에 반영되지 않는다**: `mink.Configuration`이 `keyframe home`(lifter qpos=0)으로 초기화되고 리프터 qpos를 갱신하는 코드가 없으며 `DofFreezingTask`가 리프터를 freeze한다. 리프터를 올리면 base가 최대 **0.3 m** 상승하지만 IK는 base z=1.34로 계산 → **최대 0.3 m 계통 오차.**

**7-DoF 여유자유도(널스페이스)**: mink의 `posture_cost`는 "중립 자세 태스크"의 가중치일 뿐, **엘보 자세를 직접 지정하는 API가 아니다.** 엘보 각(swivel angle) 제어는 **신규 구현**이며, 실현 경로는 (a) `posture_cost`의 타깃 자세 벡터를 엘보 슬라이더로 조작, (b) 별도 nullspace task 추가 중 하나다.

### 2.8 Freedrive (중력보상 hand-guiding) — 🔴 **LeRobot에 API가 없다. 전부 [신규구현]이다.**

**LeRobot이 제공하는 것 (전수, `damiao.py`):**
```python

bus.enable_torque(motors=None, num_retry=0)    # :296-307  → 0xFC
bus.torque_disabled(motors=None)               # :323-333  contextmanager (자동 재-enable)
```
- **`freedrive` / `gravity_comp` 심볼은 LeRobot 전체 트리에 없다** (grep: `robots/unitree_g1/` 외 0건).
- `send_action()`은 **`tau`를 항상 0으로 하드코딩**한다(`openarm_follower.py:340`) → **중력보상 토크를 실을 경로가 없다.**

> 🔴 **⇒ 중력보상 Freedrive는 `send_action()`을 우회해야만 가능하다.**
> 버스 레벨 `_mit_control_batch(commands: {motor: (kp, kd, pos_deg, vel_deg_s, torque)})`(`damiao.py:492-502`)는 **이미 tau 인자를 받으므로** `DamiaoMotorsBus`는 수정할 필요가 없다. **막힌 것은 `OpenArmFollower.send_action()` 하나뿐이다** (→ `03` FR-MOT-058).

**프로토콜 근거 (Damiao 공식):**
> *"when kp=0 and kd is not 0, a given v_des will achieve uniform rotation. **When kp=0 and kd=0, a given t_ff will achieve a given torque output.**"*

**레퍼런스 구현이 이미 존재한다** — `openarm_teleop/control/gravity_compasation.cpp` (**이식 대상**):
```cpp
Dynamics arm_dynamics(urdf_path, root_link, leaf_link);   // root: openarm_body_link0
arm_dynamics.Init();
arm_dynamics.GetGravity(arm_joint_positions.data(), grav_torques.data());
// MITParam{kp, kd, q, dq, tau} = {0, 0, 0, 0, τ_grav}   ← 순수 토크 피드포워드
openarm->get_arm().mit_control_all(cmds);
```
⚠️ **`openarm_control`(Python)에는 중력보상도 토크도 없다** — 파일 전체가 `{__init__, kinematics, poses, config}.py` 4개이고 `torque|tau|MITParam|bilateral` grep **0건**. **"`openarm_control`을 쓰면 힘제어가 된다"는 거짓이다.** 동역학(중력·코리올리)은 우리가 `pinocchio` 또는 KDL 바인딩으로 재구현하거나 MuJoCo(`mj_rne`)로 얻어야 한다.

**세 가지 Freedrive 경로:**

| 경로 | 지령 | LeRobot API로 가능? | 특성 |
|---|---|---|---|
| **(A) 순수 백드라이브** | `bus.disable_torque()` (0xFD) | ✅ **가능** | QDD 백드라이버빌리티에 의존. **DM4340은 감속비 40:1이라 무겁고, 중력 낙하 위험.** 브레이크 없음. **파지물을 떨어뜨린다.** |

### 2.9 티칭 · 재생 — 🔴 **LeRobot에 보간기도 안전 체커도 없다**

**LeRobot이 제공하는 것**: `robot.send_action(action)` 한 방. **궤적 보간기(`smooth_move` 동등물)가 없다.** `robots/` 트리에 `interpolate` / `smooth_move` / `trajectory` 심볼 없음.
**LeRobot의 안전장치 전량**: `joint_limits`(위치 클램프, 조용함) + `max_relative_target`(스텝 델타, 기본 OFF). **속도·가속도·저크 제한 없음.**

→ **보간기와 안전 체커는 우리가 만든다.** 아래는 **이식 레퍼런스**(`openarm_driver` — 우리 런타임이 아님):

```python
# openarm_driver/driver.py — 이식 레퍼런스. LeRobot 경로에는 없다.
def smooth_move(self, position, hz, duration):
    num_steps = int(hz * duration)
    for smoothed_position in self._interpolate(
            np.array([np.array(self.last_command), np.array(position)]), num_steps=num_steps):
        self.send_position(smoothed_position)      # → 우리 경로에서는 robot.send_action(...)
        time.sleep(1.0 / hz)
# _interpolate = np.linspace  → **선형 보간**
```
현재 위치 → 목표 위치 **선형보간, 기본 50 Hz × 2 s**. `move_to_start_position()` / `move_to_stop_position()`.

**안전 체커 (`openarm_driver/safety.py`)** — 조그·재생 가드레일의 **이식 레퍼런스**(LeRobot에는 없다):

| 체커 | 위반 시 |
|---|---|
| `JointPosChecker` | `is_safe=False, force_stop=False` + **클리핑된 위치 반환**(계속 진행) |
| `JointDeltaPosChecker` | `is_safe=False, **force_stop=True**` → **`RuntimeError` 발생 (즉시 정지)** |
| `CompositeChecker` / `NullChecker` | 조합/무효화. `Checker` ABC로 커스텀(충돌·작업공간) 확장 가능 |

⚠️ **LeRobot의 `joint_limits` 클리핑은 `JointPosChecker`와 같은 성격이지만 더 조용하다** — `logger.debug`로만 남기고 `is_safe` 같은 반환값이 **없다**(`openarm_follower.py:290-297`). 클리핑 발생을 우리가 별도로 검출·카운트해야 한다(FR-MAN-013, `03` FR-MOT-060).

🔴 **우리 그리퍼 제어 정본 = 엔드포인트 rad 캡처(참조레포 패턴, → `02` §2.9.1 / `03` FR-MOT-062).** operator가 그리퍼를 물리 open/close 끝단에 놓고 **현재 native rad를 캡처**해 디스크에 영속하고, `norm∈[0,1]`을 두 캡처 rad 사이 **선형 보간**으로 정의한다. → **모터각:핑거각 비율(2:1 의혹)과 개구량(mm)을 확정할 필요 없이** 제어가 성립한다. ros2_control 항등 매핑·URDF/MECH 불일치는 제어 기준이 아니라 참고 정보로만 쓴다.

❌ **"그리퍼 파지력 상한 1.0 Nm"은 근거가 없다. 그리고 로드셀로 파지력(N)을 측정하지 않는다.** 힘 상한은 POS_FORCE **`torque_pu`(per-unit 전류 상한 `[0,1]`)** 로만 다룬다:
1. API는 per-unit이다 — `gripper_component.hpp`: *"torque_pu: per-unit current limit [0, 1]"*, `limit_torque_pu_ = std::clamp(torque_pu, 0.0, 1.0)`.
2. 드라이버는 `torque_pu = gripper_posforce_limits[1] / 4.5`를 넘기며 바로 위에 `# TODO: Now We should multiply 10 to convert Nm to pu?`(driver.py:206-210)가 있다 — **우리 경로는 이 상수에 의존하지 않고** `torque_pu`를 직접 노출한다.
3. `50.0 rad/s`는 DM4310 **vMax=30 rad/s를 초과** → vMax로 클램프한다.
→ Nm/N 라벨 금지, 속도 상한만 vMax 이내로 클램프. 로드셀 pu↔N 캘리브레이션은 **하지 않는다.**

### 2.12 조그 중 안전 — 정지 카테고리와 충돌 감지

| Cat (IEC 60204-1) | 동작 | OpenArm 대응 |
|---|---|---|
| **0** | 즉시 전원 차단 | `disable_all()` (0xFD) → ⚠ **브레이크 없음 → 중력 낙하.** 최후 수단 |
| **1** | 제어 감속 후 전원 차단 | 감속 궤적 → 정지 확인 → `disable_all()` |
| **2** | 제어 정지, **전원 유지** | `(kp_orig, kd_orig, q_now, 0, τ_grav)` = **현재 위치 홀드. 기본 정지 전략** |

**충돌 감지 연동**: 업스트림에 사후 충돌 감지가 없다. 잔차(GMO) 기반 감지를 우리가 구현하며, 조그 중 임계값 기본값은 URDF effort의 ±10%: **J1/J2 ±4.0, J3/J4 ±2.7, J5–J7 ±0.7 Nm** (토크 LSB 대비 충분한 여유: J5의 경우 0.7/0.0049 ≈ 143 LSB). 확정 조건 **연속 5샘플(5 ms @1 kHz)**, 해제 히스테리시스 **0.7배**, 관측기 이득 **K=90** [1/s].

⚠ **DM 모터 ERR 코드가 파싱되지 않는다**: 피드백 D[0] 상위 니블에 ERR(8=과전압, 9=저전압, A=과전류, B=MOS 과열, C=코일 과열, D=통신두절, E=과부하)이 실려오지만 `openarm_can`의 `parse_motor_state_data()`는 `data[1]`~`data[7]`만 읽고 **`data[0]`을 버린다.** → `err = (data[0] >> 4) & 0xF`를 **우리가 직접 디코딩**해야 한다.

⚠ **그리퍼는 토크·속도 피드백을 주지 않는다** (ros2_control에서 `vel_states_ = 0; tau_states_ = 0;` 하드코딩) → 그리퍼 관절에 대한 토크 기반 충돌 감지는 **불가**.

**데드맨**: 업스트림에 소프트웨어 데드맨/E-Stop이 **없다**. **LeRobot에도 없다** — 안전장치는 `joint_limits` + `max_relative_target`(기본 OFF)이 전부이며 소프트웨어 E-Stop이 존재하지 않는다. `dora-openarm-vr` README의 트리거는 *"정렬용"*이지 안전 기능으로 규정되어 있지 않다. UR/Franka 관례상 조그는 **hold-to-move**(누르는 동안 이동, 떼면 즉시 정지)가 기본이며, Freedrive는 **별도 hold-to-activate 버튼**이다.

## 4. 상태 · 오류 처리

### 4.1 수동 동작 세션 상태 전이

> 🔴 **이 상태머신은 `robot.connect()` 이후의 상태만 다룬다.** 연결은 세션당 1회이며 `02` §4.1 소관이다. **모드 전이는 같은 robot 객체 위 `send_action()` 호출권 인계로만** 구현한다(CAN 소켓·flock 위생). 영점 안전은 connect() 오버라이드(→ `02` FR-CON-061)로 확보되며, 재영점이 필요하면 명시 set_zero 플로우(FR-MAN-005)로만 0xFE를 보낸다.
> 모든 전이는 **같은 robot 객체 위에서 `send_action()` 호출권을 넘기는 것**이다.
>
> ⚠️ **`τ_grav`가 포함된 지령은 FR-MAN-030([신규구현])이 선행되어야 가능하다.** LeRobot의 `send_action()`은 `tau`를 항상 0으로 하드코딩하므로, 그 전까지 `HOLD`/`FAULT`의 실제 지령은 `(kp, kd, q_now, 0, **0**)`이며 **중력보상이 없다.**

| 상태 | 진입 조건 | MIT 지령 | 이탈 |
|---|---|---|---|
| `DISCONNECTED` | 초기 | — | `flock` 획득 + 좌우 검증(FR-MAN-003) + **rest 자세 확인** → `robot.connect()` → `HOLD` (**`connect()`가 0xFE + 0xFC를 자동 수행한다**) |
| `IDLE` | 모터 토크 차단(`bus.disable_torque()`) | 없음 | `bus.enable_torque()` + 게인 프로파일 로드 → `HOLD`. **버스는 계속 잡고 있다** |
| `HOLD` | 모터 enable, 현재 위치 홀드 | `(kp, kd, q_now, 0, τ_grav)` | 데드맨 press → `JOG_*` / Freedrive press → `FREEDRIVE` / 재생 시작 → `REPLAY` / 원시 CAN 요청 → `MOTOR_CONFIG` |
| `MOTOR_CONFIG` | 원시 CAN 진단(RID·discover·clear_error) | 없음 (LeRobot 루프 정지) | 완료 → `IDLE`. **`disconnect()`는 하지 않는다** (→ `03` FR-MOT-007) |
| `JOG_JOINT` | 데드맨 hold + 관절 버튼 hold | `(kp, kd, q_target(t), 0, 0)` | 버튼/데드맨 release → `HOLD` · 리밋 도달 → `HOLD`(해당 방향 잠금) · 충돌 → `FAULT` |
| `JOG_CART` | 데드맨 hold + 카테시안 버튼 hold | IK 해 → `(kp, kd, q_ik(t), 0, 0)` | 상동 · **IK 실패 → `HOLD` + 사유 표시**(스텝 스킵 금지) |
| `FREEDRIVE` | Freedrive 데드맨 hold | `(0, kd_fd, ·, 0, τ_grav + τ_fric + τ_wall)` | release → **`kd` 복원 → `HOLD`** (kd=0으로 위치지령 금지) · 폴트 → `FAULT` |
| `REPLAY` | 궤적 사전 검증 통과 + 데드맨 hold | `(kp, kd, q_traj(t), 0, 0)` | 완료 → `HOLD` · 데드맨 release → **즉시 `HOLD`(궤적 중단)** · 충돌 → `FAULT` |
| `FAULT` | 충돌 / ERR 니블 / 통신두절 / 리밋 위반 | `(kp, kd, q_now, 0, τ_grav)` (Cat 2 래치) | 오퍼레이터 명시적 ACK → `HOLD`. **자동 재개 금지** |
| `ESTOP` | E-Stop 입력 | Cat 2(기본) 또는 Cat 0(이중 확인 시) | E-Stop 해제 + ACK → `IDLE` |

MoveIt SRDF의 `home`이 의도적인지 Setup Assistant 산출물의 부산물인지 판단. **하드스톱 자세를 "홈"으로 쓰지 않는다는 원칙을 채택하면 J4=π/2로 확정된다.** |
| 2 | **[결정필요] 게인 프로파일 정본은?** `compliant`(kp 어깨 70) vs `stiff`(kp 어깨 230, = MuJoCo 액추에이터) vs LeRobot/teleop 240 계열(v1 유래). 조그·Freedrive·재생 각각 어느 프로파일을 기본으로 쓸 것인가. | 실기에서 두 프로파일로 동일 조그 궤적을 실행하여 추종 오차·진동·역구동성(backdrivability)을 비교. **sim2real을 중시하면 `stiff`(MuJoCo 패리티), 사람 근처 안전을 중시하면 `compliant`.** 조그와 Freedrive에 서로 다른 프로파일을 쓰는 것도 유효한 선택이다. |
| 3 | **[결정필요] 관절 리밋 클램프의 정본은 URDF인가 LeRobot 보수값인가?** URDF J1 `[-80°, +200°]` vs LeRobot J1 `(-75°, +75°)`. 둘 다 실재하며 LeRobot은 **의도적 보수 안전 기본값**이다. | 이 명세는 **URDF를 기계 리밋 정본(1차 클램프)** 으로, LeRobot 값 같은 보수 박스를 **운영 리밋(2차 클램프)** 으로 분리했다(FR-MAN-007). 남은 결정은 **기본 운영 프로파일의 값**이다. **의존성**: 운영 리밋을 URDF 전 범위로 넓히는 것은 (a) 각 관절의 기계 스토퍼 실위치 확인, (b) 자기간섭(self-collision) 확인을 **통과한 뒤에만** 가능하다. 그 전까지는 LeRobot 수준의 보수 박스가 기본값이다. ⚠️ 이 리밋은 **IK에도 반영해야 한다** — `mink.ConfigurationLimit`은 MJCF `jnt_range`만 보므로 `model.jnt_range`를 운영 리밋으로 덮어써야 한다(FR-MAN-020). |
| 4 | **[결정필요] Freedrive의 중력보상 품질을 확정할 수 없다.** `friction.yaml`이 **0바이트 빈 파일**이고, 유일한 실측 마찰 계수(`openarm_teleop/config/follower.yaml`)는 **v1 튜닝값**이다. | v2.0 실기에서 **여기 궤적(exciting trajectory) 기반 마찰 식별**을 수행하여 관절별 `Fc/k/Fv/Fo`를 재획득해야 한다(FR-MAN-035). 식별 전까지 NFR-MAN-004(드리프트 ≤0.5°/min)는 검증 불가. **코드가 `tanh(0.1·k·ω)`를 쓴다는 사실**(문서 수식과 불일치)을 식별 스크립트에 반드시 반영할 것. |
| 5 | **[해결] 그리퍼 힘·정규화 = 참조레포 패턴.** `gripper_posforce_limits`의 "1.0 Nm" 주석은 무효로 취급한다. **로드셀로 N을 측정하지 않는다** — 힘 상한은 `torque_pu ∈ [0,1]`(per-unit)로만 노출(FR-MAN-016). open/close는 **엔드포인트 rad 캡처**의 norm[0,1] 선형매핑(FR-MAN-016, → `03` FR-MOT-062). driver의 `/4.5` 상수(소스 TODO)에 의존하지 않는다. | 잔여 비차단: 속도 상한 50 rad/s > DM4310 vMax(30 rad/s)는 vMax로 클램프. pu↔N 변환은 우리 경로에 불필요하다. |
| 6 | **[비차단] 그리퍼 모터각 ↔ 핑거관절각 매핑 (2:1 의혹) — 제어 경로와 무관.** 우리 제어 정본은 엔드포인트 rad 캡처의 선형매핑이므로 이 비율·개구량(mm)을 몰라도 성립한다(FR-MAN-016). | 표시용 mm가 필요하면 그리퍼 개폐 시 모터 엔코더 각과 개구량(mm)을 동시 측정(1차 소스에 수치 없음; `general.mdx`는 `v2_mech_stop_conf.png` 이미지). **v1의 44 mm를 전용하면 안 된다.** 그러나 제어의 선행조건은 아니다. |
| 7 | **[미확인] 데드맨 릴리즈 → 정지 도달 지연 (NFR-MAN-002).** 20 ms P99는 목표치이지 실측이 아니다. UI → 제어 루프 → CAN 프레임 경로 전체의 지연이 측정된 바 없다. **그리고 LeRobot Python 루프의 사이클 타임 자체가 미측정이다.** | 조그 버튼 릴리즈 시각과 CAN 버스상 정지 프레임 타임스탬프를 **candump 하드웨어 타임스탬프로 동시 측정**(어댑터가 1 µs 분해능 타임스탬프를 제공한다). **Python/GIL 경로이므로 P99 tail을 반드시 볼 것.** 정지 경로가 `disable_torque()`(단완 ~8 ms, 양팔 ~16 ms 블로킹)인지 **Cat 2 홀드 프레임 1개**인지에 따라 결과가 갈린다 — 후자로 설계해야 한다. |
| 8 | ✅ **[해결] mink IK solve time — 실측 완료.** `openarm_control` 양팔 기본설정 **0.345 ms / p95 0.355 ms, 500/500 성공**(`--max-iters 10 --dt 0.1 --damping 0.1`). 500 Hz 예산(2 ms)의 17%. **IK는 병목이 아니다.** | 남은 것은 **LeRobot Python 루프의 CAN 왕복 사이클 타임**(항목 14)이다. 그쪽이 진짜 병목 후보다. 시스템은 그럼에도 solve time을 상시 계측한다(NFR-MAN-003). |
| 9 | **[미확인] 티칭 반복 정밀도의 물리적 상한 (NFR-MAN-005).** DM4340(40:1) / DM8009(9:1)의 **백래시 스펙(arcmin)이 공식 문서에 없다.** 엔코더 14-bit가 로터측인지 출력측인지도 미명시(이중 엔코더인 것만 확정). | 동일 티칭 포인트를 **양방향(+/− 접근)으로 각 10회 재생**하여 EE 위치 산포와 히스테리시스를 실측. 백래시는 접근 방향 의존 오차로 드러난다. 재생 시 항상 같은 방향에서 접근하는 것(unidirectional approach)이 백래시를 우회하는 표준 기법이다. |
| 10 | **[미확인] DM 모터의 정격 온도 임계값.** T_MOS/T_Rotor의 경고·차단 온도를 데이터시트에서 찾지 못했다. 08 노트의 70/85 °C, 80/100 °C는 **추정치**다. | `openarm-can-cli show_param`으로 RID 2(`OT_Value`)의 공장 기본값을 실기에서 읽는다. |
| 11 | **[미확인] 조그·Freedrive의 실효 충돌 임계값.** ±0.1·τ_max는 **문헌 규칙**이지 OpenArm 실측값이 아니다. | 충돌 없는 상태에서 대표 조그 궤적을 반복 실행하여 잔차의 max·σ를 수집하고 `max + 3σ`(또는 nominal max의 105–110%)로 관절별 임계값을 도출하는 **튜닝 마법사**를 돌린다. Freedrive는 별도 세트가 필요하다(FR-MAN-037). |
