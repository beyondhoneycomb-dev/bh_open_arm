## 1. 범위

- **다루는 것**: 정책 모델(policy) 학습 서브시스템과 GUI "학습 / 학습 설정" 화면. 지원 정책 레지스트리와 지원 등급, `lerobot-train` 설정 파라미터 전량(기본값·범위·검증규칙), **관측 차원(state_dim) 정합성**, **단위 혼재(deg/Nm) 정규화 통계 처리**, 학습 잡 실행·큐·GPU 중재, 모니터링·로그, 체크포인트·재개, 데이터셋→체크포인트 계보, 원격 학습, 오프라인 평가.

- **런타임 전제**: 학습 스택의 정본 진입점은 **`lerobot-train`(draccus 기반 CLI)** 이다. **데이터셋 정본 포맷은 처음부터 `LeRobotDataset` v3.0**이며, `openarm_dataset` 포맷에서의 변환 단계는 **정본 경로에 없다**(§2.8). 이 문서의 모든 파라미터명은 LeRobot 소스의 실제 필드명을 그대로 쓴다. **공식 문서(`openarm.mdx`)는 낡았다 — 소스가 정본이다**(§2.5).

### 2.3 🔴 관측·액션 차원 — `use_velocity_and_torque`가 state_dim을 3배로 만든다 [확정]

**이것이 이 문서에서 가장 조용하게 깨지는 항목이다.**

feature 생성 (`robots/openarm_follower/openarm_follower.py:90-99`):
```python
@property
def _motors_ft(self) -> dict[str, type]:
    features: dict[str, type] = {}
    for motor in self.bus.motors:
        features[f"{motor}.pos"] = float
        if self.config.use_velocity_and_torque:
            features[f"{motor}.vel"] = float
            features[f"{motor}.torque"] = float
    return features
```
`observation_features` = `{**self._motors_ft, **self._cameras_ft}`, `action_features` = `self._motors_ft`.

**평탄화 — `observation.effort`는 존재하지 않는다** (`utils/feature_utils.py:68-89`):
```python
joint_fts = {key: ftype for key, ftype in hw_features.items()
             if ftype is float or (isinstance(ftype, PolicyFeature) and ftype.type != FeatureType.VISUAL)}
if joint_fts and prefix == ACTION:
    features[prefix] = {"dtype": "float32", "shape": (len(joint_fts),), "names": list(joint_fts)}
if joint_fts and prefix == OBS_STR:
    features[f"{prefix}.state"] = {"dtype": "float32", "shape": (len(joint_fts),), "names": list(joint_fts)}
```
`.pos`/`.vel`/`.torque`가 **하나의 `observation.state` 벡터와 하나의 `action` 벡터로 concat**되고, `names`에 `joint_1.torque` 같은 문자열만 남는다. 전체 트리 grep 결과 `observation.effort` / `OBS_EFFORT` 상수는 **0건**이다.

**결과 차원 (모터 8개 = `joint_1..joint_7` + `gripper`)**

| 구성 | `use_velocity_and_torque` | `observation.state` / `action` shape | `names` 배열 |
|---|---|---|---|
| 단완 `openarm_follower` | `False` (**기본값**, `config_openarm_follower.py:71`) | **(8,)** | `joint_1.pos … gripper.pos` |
| 단완 `openarm_follower` | **`True`** | **(24,)** | `joint_1.pos, joint_1.vel, joint_1.torque, …, gripper.torque` |
| 양완 `bi_openarm_follower` | `False` | **(16,)** | `left_*`/`right_*` 접두 |
| 양완 `bi_openarm_follower` | **`True`** | **(48,)** | 동상 |

**→ 정책의 state_dim 설정이 여기 맞춰져야 한다.** ACT/Diffusion은 데이터셋 shape에서 자동 유도하므로 조용히 통과한다. 그러나:
- **위치-only(8/16)로 학습한 정책에 pos+vel+torque(24/48) 데이터셋을 먹이거나 그 반대이면 학습이 조용히 깨진다.** 차원이 다르면 로드 시 shape mismatch가 나지만, **동일 차원의 다른 의미**(예: 양완 위치-only 16 vs 단완 pos+vel+torque 24 — 차원은 다르나, `rename_map`을 잘못 걸면 인덱스가 밀린다)는 예외 없이 통과한다.
- **SmolVLA / pi0 / pi05는 `max_state_dim = max_action_dim = 32`다. 양완 48차원은 이 상한을 초과한다.** 24차원(단완)은 들어가고 48차원(양완)은 **못 들어간다.** → 양완 + 토크 관측 + VLA 조합은 **성립하지 않는다.** GR00T(132)만 수용한다. 이것은 정책 선택을 실질적으로 제약한다.

**리더·팔로워 플래그를 반드시 동시에 켠다** — `lerobot_record.py:394-401`이 dataset action feature를 `robot.action_features`(팔로워)로 만들고, `feature_utils.py:131-132`의 `build_dataset_frame`이 teleop이 준 dict를 그 이름으로 인덱싱한다:
```python
frame[key] = np.array([values[name] for name in ft["names"]], dtype=np.float32)
```
→ **팔로워만 `True`이면 `values["joint_1.torque"]`에서 `KeyError`로 런타임 사망.**

**토크 액션은 기록만 되고 실행되지 않는다** — `openarm_follower.py:340`:
```python
commands[motor_name] = (kp, kd, position_degrees, 0.0, 0.0)   # vel=0, tau=0 하드코딩
```
`send_action`은 `.pos`만 읽는다(`:288`). `action_features`가 `.torque`를 선언해도 **무시된다.** → 데이터셋 `action` 벡터의 `.vel`/`.torque` 성분은 **"리더가 그렇게 움직였다"는 측정 기록**이지 **"팔로워에게 그렇게 명령했다"가 아니다.** 정책이 액션의 토크 차원을 출력하도록 학습하는 것은 가능하나, **그 출력은 실기에서 버려진다**(문서 11).

**단위 [확정]**: `.pos` = **deg**, `.vel` = **deg/s**, `.torque` = **Nm(변환 없음)** (`motors/damiao/damiao.py:568`). 모터별 토크 스케일 `MOTOR_LIMIT_PARAMS`(`motors/damiao/tables.py:98-112`): DM8009 ±54 Nm(J1·J2), DM4340 ±28 Nm(J3·J4), DM4310 ±10 Nm(J5·J6·J7·gripper).

**CAN 추가 비용은 없다** — `damiao.py:655-674` `sync_read_all_states()` docstring: *"Read ALL motor states (position, velocity, torque) from multiple motors in ONE refresh cycle."* → **토크 기록은 공짜다.** 남은 질문은 "기록할 것인가"가 아니라 **"정책 입력으로 쓸 것인가"**뿐이다(§5-Q1).

### 2.4 단위 혼재 — 한 벡터 안에 deg와 Nm이 섞인다. 정규화가 이를 어떻게 처리하는가 [확정]

`observation.state`(24/48차원)에는 **deg(±135 규모), deg/s(±수백 규모), Nm(±10~54 규모)** 가 한 벡터에 섞인다.

**정규화는 차원별(element-wise)이다** — `processor/normalize_processor.py:349-362`:
```python
if norm_mode == NormalizationMode.MEAN_STD:
    mean, std = stats["mean"], stats["std"]
    denom = std + self.eps          # eps: float = 1e-8  (:97, :443)
    return (tensor - mean) / denom
```
MIN_MAX(`:364-384`)와 QUANTILES(`:386-...`)도 마찬가지로 벡터 stats를 element-wise로 적용하며, **분모가 0이면 `eps`로 치환**한다:
```python
denom = torch.where(denom == 0, torch.tensor(self.eps, ...), denom)   # MIN_MAX / QUANTILES
```

→ **스케일이 다른 채널이 섞여 있다는 사실 자체는 정규화가 흡수한다.** 학습이 깨지는 것은 스케일 차이가 아니라 **퇴화 차원(degenerate dimension)** 이다:

| 위험 | 메커니즘 | 결과 |
|---|---|---|
| **거의 움직이지 않는 관절의 `.vel`** | 데이터셋 전체에서 σ ≈ 0 | MEAN_STD: `denom = σ + 1e-8` → 정규화 후 **잔여 노이즈가 ±10⁶ 규모로 증폭**된다. 손실이 그 채널에 지배당한다. |
| **정지 구간이 지배적인 `.torque`** | 중력 보상분만 있어 분산이 작음 | 동상. 특히 gripper 토크(±10 Nm 스케일인데 실제 변동은 수백 mNm) |
| **QUANTILES(pi05/GR00T)의 q01 == q99** | 상수 채널 | `denom → eps` → 동일 폭발 |
| **MIN_MAX(Diffusion)의 min == max** | 상수 채널 | `denom → eps` → 정규화값이 항상 −1 |

**→ 학습 전에 `meta/stats.json`을 열어 채널별 σ(또는 q99−q01, max−min)를 검사하고, 임계 이하 채널을 명시적으로 처리(제외 / 상수 대체 / 별도 스케일)해야 한다. LeRobot은 이 검사를 하지 않는다.** [신규구현]

### 2.5 `lerobot-train` 설정 파라미터 전량 — GUI "학습 설정" 화면의 정본 스키마

### 2.7 정규화 통계 — 기록 시점에 분위수까지 생성된다 [확정]

**`meta/stats.json` 키**: `min`, `max`, `mean`, `std`, `count` + 분위수 `q01`, `q10`, `q50`, `q90`, `q99`.
- `datasets/compute_stats.py:27` → `DEFAULT_QUANTILES = [0.01, 0.10, 0.50, 0.90, 0.99]`, 키 이름 `:53` → `f"q{int(q * 100):02d}"`.
- `compute_episode_stats(episode_data, features, quantile_list=None)` (`:486-489`)는 **`quantile_list is None`이면 `DEFAULT_QUANTILES`로 대체**한다 → **`lerobot-record`로 새로 기록한 v3.0 데이터셋은 분위수를 처음부터 포함한다.** (v1 명세의 "벤더 변환기가 q01/q99를 쓰는지 미확인"은 **경로에서 사라졌다.**)
- 분위수는 **정확값이 아니라 히스토그램 근사**다 — `RunningQuantileStats`(`:30`), `num_quantile_bins: int = 5000`(`:40`).
- 이미지 feature는 `is_depth_map`이 아니면 **255.0으로 나눈 [0,1] 스케일** 기준으로 통계 산출.

**분위수가 없는 기존 데이터셋의 보강 경로가 in-tree에 있다** — `scripts/augment_dataset_quantile_stats.py`:
```bash
python src/lerobot/scripts/augment_dataset_quantile_stats.py --repo-id=<user>/<ds>
```
docstring: *"Most datasets created before the quantile feature was added do not contain quantile statistics (q01, q10, q50, q90, q99)… 1. Loads an existing LeRobot dataset in v3.0 format 2. Checks if it already contains quantile statistics 3. If missing, computes… 4. Updates the dataset metadata."*
(`normalize_processor.py`의 QUANTILES 분기가 q01/q99 부재 시 내는 에러 메시지도 이 스크립트를 지목한다.)

**정책별 요구 통계**

| 정책 | 필요한 통계 | 부재 시 |
|---|---|---|
| ACT | mean/std (VISUAL·STATE·ACTION 전부) | `ValueError("MEAN_STD normalization mode requires mean and std stats…")` |
| Diffusion | VISUAL mean/std + STATE·ACTION **min/max** | `ValueError("MIN_MAX … requires min and max stats…")` |
| pi0 | STATE·ACTION mean/std | 동상 |
| **pi05** | STATE·ACTION **q01/q99** | `ValueError("QUANTILES … requires q01 and q99 stats, please update the dataset with the correct stats using the augment_dataset_quantile_stats.py script")` |
| GR00T | `use_percentiles=True` → q01/q99 | 동상 |

**체크포인트 동봉**: 학습에 사용된 정규화 통계는 학습 산출물과 함께 **불변으로 보관**되어야 하며, 배포 시 서빙 측 정규화가 학습 시 통계와 동일함을 검증해야 한다. LeRobot이 이를 체크포인트에 자동 저장하는지는 [미확인] → 우리 계보 레코드(§2.8)가 `stats.json` 해시를 독립적으로 보관한다 [신규구현].

### 2.8 데이터셋 → 학습 계보 (lineage) [확정 — v2에서 전면 개정]

**포맷 결정**: 수집·보관·학습 **전 구간 정본 = `LeRobotDataset` v3.0**. `lerobot-record`가 직접 v3.0을 쓴다. **중간 포맷도, 변환 단계도 없다.**

- `openarm_dataset` pip는 **라이터가 없다**(리더/컨버터 전용) → 우리 수집 경로의 산출물이 될 수 없다.
- 따라서 v1 명세의 변환 손실 매트릭스(ns 타임스탬프 재합성, `qvel`/`qtorque` 폐기, 스무딩 `fps=250.0` 하드코딩, 교집합 무경고 절단, 에피소드 id 재번호)는 **경로에서 사라졌다.** 이 항목들은 이 문서에서 삭제한다.
- `qtorque` ↔ LeRobot 매핑 질문도 사라졌다 — 토크는 **처음부터 `observation.state`의 부분 인덱스**다(§2.3).
- **`timestamp`(및 `index`/`episode_index`/`frame_index`/`task_index`)는 정책 feature에서 구조적으로 배제된다 [확정]** — `dataset_to_policy_features`(`feature_utils.py:169-212`)가 feature를 VISUAL/ENV/STATE(`observation*`)/ACTION(`action*`)으로만 분류하고 **그 외는 `else→continue`로 제외**한다. `timestamp`는 `observation`/`action`으로 시작하지 않으므로 **정책 입력·출력에 절대 들어가지 않는다.** → **기록해 두어도 학습이 이상해지지 않는다**(순수 메타데이터: 시간정렬·`delta_timestamps`·`tolerance_s` 용도, `lerobot_dataset.py:52,160-163`). **D-7(타임스탬프는 기록하되 학습 배제)** 이 프레임워크상 안전함을 소스가 보장한다.
- `openarm_dataset` 포맷은 **OpenEval 연동 시에만** 등장한다(문서 11 §2.10).

**남는 계보 축**
- **데이터셋 버전** = `--dataset.repo_id` + git `revision`. LeRobot은 로드 시 `revision = revision or CODEBASE_VERSION`.
- **에피소드 → 체크포인트 역추적**은 LeRobot이 제공하지 않는다 → 우리 계보 저장소 [신규구현].
- **`--dataset.push_to_hub`(record) / `--policy.push_to_hub`(train) 기본값이 `True`다** → 명시적으로 false 하지 않으면 사내 데이터·모델이 HF Hub로 나간다 🔴.
- **다중 데이터셋 학습은 `NotImplementedError`다**(`train.py:246-247`) → 여러 수집 세션을 합치려면 `lerobot-edit-dataset`(기능 범위 [미확인]) 또는 자체 병합이 선행되어야 한다.

### 2.9 오프라인 지표는 온라인 성공률을 예측하지 못한다 (요구사항의 근거) [확정]

- robomimic 공식 스터디: **"the best validation policy is 50 to 100% worse than the best performing policy"**, **"unlike supervised learning, the best validation loss does not correspond to the best performing policy"**. 권고: *"try each and every model directly on the robot."*
- ACT 원저자 README: **"if your ACT policy is jerky or pauses in the middle of an episode, just train for longer! Success rate and smoothness can improve way after loss plateaus."** / 실데이터는 **"at least 5000 epochs or 3-4 times the length after the loss has plateaued"**.
- Isaac-GR00T README: 비결정적 이미지 증강 때문에 **동일 설정 재학습 시 5–6% 성능 편차**.

→ (a) val loss 최소 체크포인트 자동 선택을 기본값으로 삼아서는 안 된다. (b) 손실 기준 조기 종료를 자동 수행해서는 안 된다. (c) 체크포인트 다수를 보존하고 각각을 **실기 롤아웃으로 채점**해야 한다. (d) 단일 롤아웃 세트로 두 체크포인트를 비교해서는 안 된다.

**⚠️ `lerobot-eval`은 시뮬 전용이다** — `configs/eval.py:34` `env: envs.EnvConfig`가 **기본값 없는 필수 필드**이고 `lerobot_eval.py:68`이 `import gymnasium as gym`이다. **실기 성공률 측정 도구가 아니다.** 실기 채점 하네스는 우리가 만들며, 그 요구사항은 **문서 11이 소유한다.** 이 문서는 "체크포인트 → 성공률 표"를 **소비**할 뿐이다.

**데이터 요구량 (근거 있는 것만)**

| 근거 | 수치 |
|---|---|
| LeRobot `act.mdx` | "Often achieves high success rates with **just 50 demonstrations**" |
| LeRobot `smolvla.mdx` | "We recommend recording **~50 episodes**" = 5개 큐브 위치 × 10 에피소드. **"We tried similar dataset with 25 episodes, and it was not enough"** |
| Data Scaling Laws (arXiv 2410.18647, ICLR'25) | **"The diversity of environments and objects is far more important than the absolute number of demonstrations"** |

> "태스크당 N 에피소드"라는 단일 숫자는 근거가 없다. 근거 있는 형태는 **"변형(variation)당 ~10 에피소드 × 변형 5개"** 구조다.

### 2.10 학습 잡과 하드웨어 배타성 [확정 — v2에서 축소]

**학습(`lerobot-train`)은 CAN을 만지지 않는다.** 순수 오프라인 GPU 잡이다. 이 문서에서 CAN 관련 요구사항은 **학습 잡이 실기 롤아웃 잡과 GPU·로봇을 공유할 때의 스케줄링 제약**으로만 남는다.

- 실기 롤아웃(문서 11)은 GUI 백엔드 프로세스가 **세션당 `connect()` 1회**로 CAN을 잡고 있는 동안 수행된다. 학습 잡이 같은 머신의 GPU를 점유하면 롤아웃의 정책 forward와 **VRAM·SM을 경합**한다.
- SocketCAN RAW는 **배타 bind를 제공하지 않는다** → 외부 침입자(`cansend`, `openarm-can-cli`, 두 번째 LeRobot 인스턴스, `openarm_teleop`)를 막는 `flock` 애플리케이션 락이 필요하다. 이 락의 소유·해제 요구사항은 **문서 11·12가 소유**하며, 학습 축에서는 **"롤아웃 잡을 큐잉할 때 락 가용성을 확인한다"** 로만 나타난다.

## 4. 상태 · 오류 처리

### 4.1 학습 잡 상태 전이

| 현재 상태 | 이벤트 | 다음 상태 | 부수 효과 |
|---|---|---|---|
| — | 잡 생성 | `QUEUED` | 설정 스냅샷 불변 기록, 계보 레코드 생성 |
| `QUEUED` | 사전검증 실패 | `REJECTED` | 위반 항목·수정법 표시. 프로세스 미기동 |
| `QUEUED` | GPU 확보 | `PREFLIGHT` | 차원 정합성(FR-TRN-062/064), 통계 존재·퇴화 검사(FR-TRN-020/067), VRAM 추정, gated 모델 권한 검사 |
| `PREFLIGHT` | 통과 | `RUNNING` | `lerobot-train` 프로세스 기동, 로그 스트림 개시 |
| `PREFLIGHT` | 실패 | `REJECTED` | GPU 반환 |
| `RUNNING` | `step == steps` | `COMPLETED` | 최종 체크포인트 저장, GPU 반환 |
| `RUNNING` | 사용자 중단 | `CANCELLED` | SIGTERM → 마지막 체크포인트 보존, 중단 step 기록, GPU 반환 |
| `RUNNING` | 프로세스 비정상 종료 | `CRASHED` | exit code·시그널·직전 로그 보존, GPU 반환 |
| `CRASHED` / `CANCELLED` | 재개 요청 | `QUEUED` | `--config_path=<ckpt>/…/train_config.json --resume=true` |
| `COMPLETED` | 실기 평가 요청 | (평가 세션은 문서 11이 소유) | 체크포인트 + `names` 배열 + 정규화 통계 해시 전달 |

### 4.2 오류 조건과 복구

| 조건 | 감지 | 복구 동작 |
|---|---|---|
| **GPU OOM** | exit code **137**(OOM-killer), `dmesg`, stderr의 `torch.cuda.OutOfMemoryError` | `CRASHED` → 마지막 체크포인트 재개 제안 + **배치 축소안**(예: ½) 및 `gradient_checkpointing`/LoRA/`train_expert_only` 제안. **자동 재시도는 사용자 확인 후에만.** |
| **차원 불일치** (체크포인트 state_dim ≠ 데이터셋 state_dim) | PREFLIGHT — `meta/info.json` shape vs 체크포인트 `input_features` | `REJECTED`. 관측 구성(위치-only / pos+vel+torque)을 명시하고 데이터셋 또는 정책 변경 요구 |
| **정책 상한 초과** (양완 48차원 + SmolVLA/pi0/pi05) | PREFLIGHT — `max_state_dim` 비교 | `REJECTED`. GR00T 또는 ACT/Diffusion으로 변경, 혹은 `use_velocity_and_torque=false` 데이터셋 요구 |
| **퇴화 채널** (σ/Δq/Δminmax < 임계) | PREFLIGHT — `meta/stats.json` 파싱 | 경고 + 사용자 선택 요구(제외 / 상수 대체 / 진행). 확인 없이 학습 시작 금지 |
| **q01/q99 부재** (pi05/GR00T) | PREFLIGHT — `meta/stats.json` 키 검사 | `REJECTED` → `augment_dataset_quantile_stats.py` 실행 제안 |
| `output_dir` 이미 존재 (`resume=false`) | 사전검증 (`FileExistsError` 선제 차단) | `REJECTED`. 덮어쓰기 / 새 경로 / 재개 선택 요구 |
| optimizer만 지정, scheduler 누락 | 사전검증 | `REJECTED`. 둘 다 요구 |
| `eval_steps>0` + `eval_split==0` | 사전검증 | `REJECTED`. `eval_split` 설정 요구 |
| `dataset.repo_id`가 list | 사전검증 | `REJECTED`. 사전 병합(FR-TRN-071) 안내 |
| VQ-BeT + 카메라 ≥2 | 사전검증 | `REJECTED`. 정책 변경 요구 |
| gated 모델 권한 없음 (`Cosmos-Reason2-2B`) | PREFLIGHT — HF API 조회 | `REJECTED`. HF 승인 링크 + `HF_TOKEN` 설정 안내 |
| **의도치 않은 Hub 푸시** | PREFLIGHT — `policy.push_to_hub` / `save_checkpoint_to_hub` 값 검사 | 폐쇄망 프로파일에서 **`REJECTED`**. 기본값이 `True`이므로 명시적 false를 강제 |
| 디스크 부족 | `statvfs()` 폴링 | 경고 임계 → 신규 잡 차단. 정지 임계 → 진행 중 잡을 다음 `save_freq`에서 정상 종료 |
| 프로세스는 살아있으나 step 미진행 | `samples_per_s ≈ 0` 지속(예: 5분) | `STALLED` 경고. dataloader 워커 데드락/디스크 I/O 의심 → 진단 정보(`num_workers`, `dataloading_s`) 제시 |

## 5. 미해결 · 결정필요

**Q1. 토크·속도 성분을 정책 입력으로 쓸 것인가?** `[결정필요]` ← **이 문서에서 가장 큰 미결**
- **질문이 좁아졌다.** 이전 명세에서는 "변환기가 `qtorque`를 폐기하므로 무손실 확장 변환기를 만들어야 하는가"가 섞여 있었다. 이제 **기록은 공짜다** — `use_velocity_and_torque=True` 한 줄이면 되고 CAN 추가 비용이 0이다(`sync_read_all_states`가 한 refresh 사이클). 따라서 남는 것은 **순수 실험 문제**다.
- 선택지: **(a)** 항상 켜서 기록하되 정책 입력에는 `.pos`만 쓴다, **(b)** 24/48차원 전체를 정책 입력으로 쓴다.
- 비용: (b)를 택하면 **양완에서 SmolVLA/pi0/pi05가 탈락**한다(`max_state_dim=32` < 48). GR00T(132) 또는 ACT/Diffusion(상한 없음)만 남는다. 이것은 무시할 수 없는 대가다.
- **정하려면**: FR-TRN-073/074 — **동일 데이터셋**(24/48차원으로 기록)에서 `.pos`-only 부분 벡터로 학습한 정책과 전체 벡터로 학습한 정책을 만들고, **접촉 풍부 태스크**(예: 삽입, 문 열기, 천 접기)의 **동일 롤아웃 세트에서 성공률 차이**를 측정한다. 차이가 유의하지 않으면 (a)를 택하고 VLA 선택지를 유지한다.
- **기록 자체는 무조건 켠다** — 켜지 않으면 이 실험을 나중에 할 수 없다(데이터를 다시 모아야 한다).

**Q2. `use_velocity_and_torque`를 데이터 수집 표준으로 강제할 것인가?** `[결정필요]`
- 강제 시: 모든 데이터셋이 24/48차원이 되고, **양완 VLA 경로가 닫힌다**(Q1의 (b)를 택하지 않아도, 정책이 `.pos`만 소비하도록 부분 선택하는 경로(FR-TRN-074)가 반드시 구현되어야 한다).
- 강제하지 않으면: 위치-only 데이터셋과 토크 데이터셋이 섞이고 **병합이 불가능**해진다(차원이 다름). 다중 데이터셋 학습이 `NotImplementedError`이므로 병합이 유일한 경로인데, 차원이 다르면 병합할 수 없다.
- → **한 조직 안에서 두 구성을 섞는 것은 금지해야 한다.** 하나를 골라 전 세션에 강제한다. **정하려면**: Q1을 먼저 답한다. Q1이 (a)여도 기록은 켜므로, 실질적으로 "강제 ON + `.pos` 부분 선택 경로 구현"이 유력하다.

**Q3. 필수(M) 지원 정책 목록을 무엇으로 확정할 것인가?** `[결정필요]`
- 제안: ACT / Diffusion (M) — **차원 상한이 없어 24/48 어느 구성에서도 동작**한다. SmolVLA는 단완(24)에서만 (M), 양완(48)에서는 사용 불가.
- **정하려면**: 카메라 구성(뎁스 RealSense + RGB 웹캠 혼합, 고정 슬롯 없는 사용자 설정 dict)과 양완/단완 표준을 확정하고, 각 정책을 우리 데이터셋 50 에피소드로 1회씩 돌려 §2.2의 `[미확인]` 칸(학습 시간·VRAM)을 채운다.

**Q4. Diffusion Policy의 `drop_n_last_frames=7`을 그대로 쓸 것인가, 31로 재계산할 것인가?** `[결정필요]`
- 소스 주석의 공식(`horizon − n_action_steps − n_obs_steps + 1`)을 현 기본값(64/32/2)으로 계산하면 **31**이다. 값 7은 구 기본값(16/8/2)의 잔재로 보인다.
- **정하려면**: 두 값으로 각각 학습해 실기 성공률을 비교한다. 확정 전까지 GUI에 노출하고 **기본값이 소스 그대로(7)임을 명시**한다.

**Q5. `pi0_fast`를 지원 목록에 넣을 것인가?** `[결정필요]`
- `get_policy_class`(`:131`)에는 있지만 **`make_policy_config`(`:187-247`)에는 없다.** openpi 공식 README는 PyTorch 구현 미지원이라 한다.
- **정하려면**: `--policy.type=pi0_fast`로 100 step 학습을 실제로 돌린다. 예외 없이 손실이 내려가면 지원, 아니면 목록에서 제거.

**Q6. GR00T N1.7은 LoRA를 지원하는가?** `[미확인]`
- N1.7 `gr00t/configs/finetune_config.py`에 LoRA 필드가 없다(grep 0건). N1.5 브랜치에는 `lora_rank`/`lora_alpha`/`lora_dropout`이 있었다.
- LoRA가 없으면 GR00T는 **기본 FT(피크 ~35 GB/GPU)만** 가능하며, 이것이 VRAM 사전검증(FR-TRN-018)의 하한을 결정한다.

**Q7. 세션 병합 도구를 무엇으로 할 것인가?** `[미확인]`
- 다중 `repo_id` 학습이 `NotImplementedError`이므로 병합이 필수다. `lerobot-edit-dataset`이 in-tree에 존재하나(`pyproject.toml [project.scripts]`) **병합 기능 지원 여부는 확인하지 않았다.**
- **정하려면**: `lerobot-edit-dataset --help`를 덤프하고 `datasets/dataset_tools.py`의 연산 목록을 읽는다. 병합이 없으면 자체 구현(FR-TRN-071).

**Q8. 폐쇄망 로컬 로거를 무엇으로 만들 것인가?** `[미확인]`
- LeRobot에서 확인된 로거는 **wandb + 콘솔**뿐이다. `mode=offline`으로도 UI가 없다.
- **정하려면**: LeRobot의 로거 팩토리에 훅 포인트가 있는지 확인한다. 없으면 `lerobot-train`의 stdout(`MetricsTracker` 포맷)을 파싱하는 자체 로거를 만든다(FR-TRN-036).

**Q9. 학습 잡 큐를 어느 계층에 둘 것인가?** `[결정필요]`
- 선택지: (a) 자체 스케줄러(SQLite + 프로세스 매니저), (b) HF Jobs에 위임.
- **정하려면**: 원격 GPU 서버 보유 여부와 폐쇄망 요건을 확정한다. 폐쇄망이면 (b)는 탈락한다.
