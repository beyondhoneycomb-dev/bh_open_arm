# 07. 텔레오퍼레이션 데이터 수집

## 1. 범위

텔레오퍼레이션(VR → `05`, KER → `05`) 중에 **에피소드를 녹화해 LeRobot 데이터셋으로 디스크에 확정**하기까지를 다룬다. 에피소드 라이프사이클 상태머신, 태스크 정의, 성공/실패 라벨링, 실시간 품질 관리, 카메라 등록, 세션 중단 복구, 녹화 화면 요구사항이 이 문서에 있다.

**정본 포맷은 LeRobot v3.0이며, 변환 단계가 존재하지 않는다.** 우리는 LeRobot으로 **기록**하므로 LeRobot 포맷이 처음부터 원본이다(§2.1).

다루지 않는 것: VR 입력·IK·클러치(→ `05-텔레오퍼레이션.md`), 카메라 물리 배치·캘리브레이션(→ `06-카메라-퍼셉션.md`), 수집 **이후**의 데이터셋 브라우징·편집·통계·계보(→ `08-데이터셋-관리.md`), CAN 소유권 중재의 일반 규칙(→ `01-시스템-아키텍처.md`), 안전 계층(→ `12-충돌감지-및-안전.md`).

요구사항 ID 영역 코드: **`REC`**.

---

## 2. 확정된 사실 기반

인용은 전부 `huggingface/lerobot` **v0.6.1**(`pyproject.toml:28`) 로컬 스냅샷과 `enactic/openarm_dataset` 소스를 **직접 열람**한 결과다. 공식 문서(`docs/source/openarm.mdx`)는 **낡았으므로 인용하지 않는다**(§2.8).

### 2.1 정본 포맷은 LeRobot v3.0이다 — 변환 단계는 없다

```python
# src/lerobot/datasets/dataset_metadata.py:58
CODEBASE_VERSION = "v3.0"
```

`LeRobotDataset.create()`에는 **버전 인자가 없다**(`lerobot_dataset.py:663-684`). `LeRobotDatasetMetadata.create()`가 `CODEBASE_VERSION`을 하드코딩해 `info.json`에 박는다. **v2.1로 기록하는 옵션은 존재하지 않는다.**

**`openarm_dataset` pip 패키지에는 라이터가 없다.** 소스 트리 전수 확인:

```python
# openarm_dataset/dataset.py:32-50
class Dataset:
    def __init__(self, path, meta=None, camera_names=None):   # ← 기존 데이터셋을 '연다'
# openarm_dataset/dataset.py:435-451
    def write(self, output, format=None, **options):          # ← 열어둔 것을 '내보낸다'
        if format is None or format == "openarm": ...
        elif format == "lerobot_v2.1": ... / "lerobot_v3.0" / "gr00t" / "rrd"
# openarm_dataset/convert.py (말미)
    old_dataset = openarm_dataset.Dataset(args.input)         # ← 입력은 언제나 OpenArm 포맷
    old_dataset.write(args.output, **write_kwargs)
```

즉 이 패키지는 **리더 + 컨버터**이며, **라이브 스트림을 받아 기록하는 경로가 없다.** 유일한 OpenArm-포맷 라이터는 `dora-openarm-dataset-recorder`이고 **우리는 dora를 쓰지 않는다**(→ `GROUND-v2 §0`).

> **폐기**: v1 명세의 "OpenArm Dataset v0.3.0이 1차 포맷이고 LeRobot은 파생물"은 **성립하지 않는다.** OpenArm 포맷으로 기록할 수단 자체가 우리 런타임에 없다. 그 전제 위에 서 있던 요구사항(무손실 확장 exporter, 변환 손실 리포트, 역변환 금지, 교집합 커버리지 검사, 벤더 스무딩 우회)은 **전부 삭제되었다** — 벤더 리샘플러·스무더·교집합 창은 **우리 경로를 통과하지 않기 때문이다.**

### 2.2 `lerobot-record`가 실제로 기록하는 feature 전수

하드웨어 feature → 데이터셋 스키마 변환의 정본은 `hw_to_dataset_features()`다:

```python
# src/lerobot/utils/feature_utils.py:68-100
joint_fts = {k: t for k, t in hw_features.items()
             if t is float or (isinstance(t, PolicyFeature) and t.type != FeatureType.VISUAL)}
cam_fts   = {k: s for k, s in hw_features.items() if isinstance(s, tuple)}

if joint_fts and prefix == ACTION:
    features[prefix]            = {"dtype": "float32", "shape": (len(joint_fts),), "names": list(joint_fts)}
if joint_fts and prefix == OBS_STR:
    features[f"{prefix}.state"] = {"dtype": "float32", "shape": (len(joint_fts),), "names": list(joint_fts)}

for key, shape in cam_fts.items():
    dtype = "video" if use_video else "image"
    features[f"{prefix}.images.{key}"] = {"dtype": dtype, "shape": shape,
                                          "names": ["height","width","channels"],
                                          "info": {"is_depth_map": shape[2] == 1}}
```

`ACTION = "action"`, `OBS_STR = "observation"` (`utils/constants.py:20,33`).

**기록되는 feature는 정확히 이것뿐이다:**

| 키 | dtype | shape | names | 출처 |
|---|---|---|---|---|
| `action` | `float32` | `(D,)` | `list(robot.action_features)` | `feature_utils.py:76-80` |
| `observation.state` | `float32` | `(D,)` | 동일 리스트 | `feature_utils.py:82-86` |
| `observation.images.<cam>` | `video`(기본) / `image` | `(H, W, 3)` | `["height","width","channels"]` | `feature_utils.py:88-100` |
| `observation.images.<cam>_depth` | `video`(기본) / `image` | `(H, W, 1)` | 동일, `info.is_depth_map=true` | RealSense `use_depth=True`가 자동 생성 (LeRobot **0.6.x** 필요 — §2.10) |
| `timestamp` | `float32` | `(1,)` | `None` | `constants.py:80-87` |
| `frame_index` | `int64` | `(1,)` | `None` | 동상 |
| `episode_index` | `int64` | `(1,)` | `None` | 동상 |
| `index` | `int64` | `(1,)` | `None` | 동상 |
| `task_index` | `int64` | `(1,)` | `None` | 동상 |

메타 5종은 `dataset_metadata.py:782`에서 `features = {**deepcopy(features), **DEFAULT_FEATURES}`로 병합된다.

**`timestamp`는 벽시계가 아니다:**

```python
# src/lerobot/datasets/dataset_writer.py:207-208
frame_index = self.episode_buffer["size"]
timestamp   = frame_index / self._meta.fps
```

→ **`timestamp = frame_index / fps`의 균등 격자**다. 캡처 시각·지터·스트림 오프셋은 **어디에도 남지 않는다**. 실측 시각이 필요하면 우리가 별도로 남겨야 한다(→ §5 Q1).

**`timestamp`는 학습에 영향을 주지 않는다** — `dataset_to_policy_features()`(`feature_utils.py:169-212`)가 feature를 VISUAL/ENV/STATE/ACTION으로만 분류하고 `observation`/`action` 접두가 아닌 키는 `else→continue`로 배제한다. `timestamp`·`index`·`episode_index`·`frame_index`·`task_index`는 정책 입력·출력에 **절대 들어가지 않는다**(→ v3-7). → **기록해 두어도 안전하며, 우리가 추가하는 벽시계 사이드카(FR-REC-045)의 커스텀 feature 방식도 같은 이유로 학습을 오염시키지 않는다.**

**`success` 필드는 기록되지 않는다.** `SUCCESS = "next.success"` 상수는 존재하나(`constants.py:40`), `lerobot_record.py` 전체에 `success` 문자열이 **0회 등장한다**(직접 grep). → **성공/실패 라벨은 [신규구현]**(§2.13).

### 2.3 🔴 토크는 `observation.state`로 평탄화된다 — `observation.effort`는 존재하지 않는다

전체 트리 grep 결과 **`observation.effort` / `OBS_EFFORT` 0건**(직접 재확인). §2.2의 `hw_to_dataset_features()`가 **모든 float feature를 하나의 벡터로 concat**하기 때문이다. 토크는 **별도 컬럼이 아니라 `observation.state`의 부분 인덱스**이며, `names` 리스트에 `joint_1.torque` 같은 **문자열만 남는다.**

팔로워가 내보내는 하드웨어 feature:

```python
# src/lerobot/robots/openarm_follower/openarm_follower.py:90-99
@property
def _motors_ft(self) -> dict[str, type]:
    features: dict[str, type] = {}
    for motor in self.bus.motors:
        features[f"{motor}.pos"] = float
        if self.config.use_velocity_and_torque:
            features[f"{motor}.vel"]    = float
            features[f"{motor}.torque"] = float
    return features
```

모터 순서 = `motor_config` dict 순서 (`config_openarm_follower.py:87-98`):
`joint_1, joint_2, joint_3, joint_4, joint_5, joint_6, joint_7, gripper` (8개).

값도 실제로 채워진다 (`openarm_follower.py:239-246`, `bus.sync_read_all_states()` → `state.get("torque", 0.0)`). 버스는 tau를 버리지 않는다 — `motors/damiao/damiao.py:68-73` `MotorState{position, velocity, torque, temp_mos, temp_rotor}`. 주석: *"Reads ALL motor states (position, velocity, torque) … in ONE refresh cycle"* → **추가 CAN 비용 없음.**

#### 2.3.1 차원

| 구성 | `use_velocity_and_torque` | `observation.state` / `action` shape |
|---|---|---|
| 단완 | `False` (기본) | **`(8,)`** — pos만 |
| 단완 | `True` | **`(24,)`** — 8 × (pos, vel, torque) |
| 양완 | `False` (기본) | **`(16,)`** |
| 양완 | `True` | **`(48,)`** |

양완 키는 `left_` / `right_` 접두, **left가 먼저**다:

```python
# src/lerobot/robots/bi_openarm_follower/bi_openarm_follower.py:100-104
def _motors_ft(self) -> dict[str, type]:
    return {**{f"left_{k}":  v for k, v in self.left_arm._motors_ft.items()},
            **{f"right_{k}": v for k, v in self.right_arm._motors_ft.items()}}
```

#### 2.3.2 🔴 OpenArm `qtorque` ↔ LeRobot 매핑 규칙 (정본)

`names` 리스트는 모터별로 `(pos, vel, torque)`가 **인터리브**된다. 모터 인덱스 `m ∈ [0, 8)`(`joint_1`…`joint_7`, `gripper`)에 대해:

**단완 (24차원):**

| OpenArm 개념 | LeRobot 위치 | 인덱스 |
|---|---|---|
| `qpos[m]` | `observation.state[3m + 0]` | `names[3m+0] == f"joint_{m+1}.pos"` (m=7이면 `gripper.pos`) |
| `qvel[m]` | `observation.state[3m + 1]` | `names[3m+1] == …".vel"` |
| **`qtorque[m]`** | **`observation.state[3m + 2]`** | `names[3m+2] == …".torque"` |

**양완 (48차원):** left 블록이 `[0, 24)`, right 블록이 `[24, 48)`.

| OpenArm 개념 | LeRobot 인덱스 |
|---|---|
| `arms/left/qtorque[m]` | `observation.state[3m + 2]` |
| `arms/right/qtorque[m]` | `observation.state[24 + 3m + 2]` |

**동일한 규칙이 `action` 벡터에도 그대로 적용된다** — `action`의 `names`는 `robot.action_features`이고 그것은 `_motors_ft`와 같은 dict이기 때문이다(`openarm_follower.py:101-107`).

> ⚠️ **인덱스를 코드에 하드코딩해서는 안 된다.** `use_velocity_and_torque=False`이면 차원이 8/16으로 줄고 위 산식이 **전부 어긋난다**. 소비자는 반드시 **`meta/info.json`의 `features["observation.state"]["names"]`에서 문자열로 조회**해야 한다(FR-REC-006).

#### 2.3.3 토크 **액션**은 기록되지만 실행되지 않는다

```python
# src/lerobot/robots/openarm_follower/openarm_follower.py:340
commands[motor_name] = (kp, kd, position_degrees, 0.0, 0.0)   # ← vel=0, tau=0 하드코딩
# :288
goal_pos = {k.removesuffix(".pos"): v for k, v in action.items() if k.endswith(".pos")}
```

`send_action()`은 `.pos`만 읽는다. `action_features`가 `.torque`를 선언해도 **로봇에는 전달되지 않는다** — 데이터셋에만 남는 장식용 액션 차원이다. (버스 레벨 `_mit_control_batch(commands: {motor: (kp, kd, pos_deg, vel_deg_s, torque)})`(`damiao.py:492-502`)는 이미 tau 인자를 받으므로, 막혀 있는 것은 상위 함수 하나뿐이다 → 바이래터럴은 `05` 소관.)

수집 관점의 귀결: **`action`의 `.torque` 차원은 "리더가 측정한 토크"이지 "팔로워에 가해진 토크"가 아니다.** 학습 시 이를 지령으로 해석하면 의미가 틀어진다.

### 2.4 🔴 단위 혼재 — 한 `observation.state` 벡터에 deg와 Nm이 섞인다

```python
# src/lerobot/motors/damiao/damiao.py:563-568
position_rad         = self._uint_to_float(q_uint,   -pmax, pmax, 16)
velocity_rad_per_sec = self._uint_to_float(dq_uint,  -vmax, vmax, 12)
torque               = self._uint_to_float(tau_uint, -tmax, tmax, 12)
return np.degrees(position_rad), np.degrees(velocity_rad_per_sec), torque, t_mos, t_rotor
```

**위치·속도만 `np.degrees()`를 통과하고 토크는 변환 없이 원단위로 나간다.**

| 접미사 | 단위 | 대략적 동적 범위 |
|---|---|---|
| `.pos` | **deg** | 관절 리밋 폭 (수십~수백) |
| `.vel` | **deg/s** | 정지 시 ≈0, 이동 시 수백 |
| `.torque` | **Nm** | 모터별 `tmax`: DM4310 ±10 / DM4340 ±28 / DM8009 ±54 (`motors/damiao/tables.py`) |

모터 배치: J1·J2 = DM8009, J3·J4 = DM4340, J5·J6·J7·gripper = DM4310 (`config_openarm_follower.py:87-98`).

**정규화 통계에 미치는 영향 (LeRobot 정책별 정본, 소스 확인):**

| 정책 | STATE / ACTION 정규화 | 필요한 통계 키 |
|---|---|---|
| ACT | `MEAN_STD` (`configuration_act.py:90-92`) | `mean`, `std` |
| Diffusion | **`MIN_MAX`** (`configuration_diffusion.py:109-110`) | `min`, `max` |
| pi0 | `MEAN_STD` (`configuration_pi0.py:74-75`) | `mean`, `std` |
| SmolVLA | `MEAN_STD` (`configuration_smolvla.py:35-36`) | `mean`, `std` |
| **pi0.5** | **`QUANTILES`** (`configuration_pi05.py:75-77`, 주석 *"Pi0.5 uses quantiles for state/action"*) | **`q01`, `q99`** |
| **GR00T** | **`IDENTITY`** (`configuration_groot.py:263-268`) | — (자체 modality stats 사용) |

`MEAN_STD` / `MIN_MAX` / `QUANTILES`는 **모두 채널별(per-dimension)** 이므로 deg와 Nm의 스케일 차이 자체는 수학적으로 흡수된다. **실제 위험은 스케일이 아니라 분산이다:**

- 정지 상태가 지배적인 관절의 `.vel` 채널은 **`std ≈ 0`** → `MEAN_STD`에서 `(x-mean)/std`가 폭발한다.
- 접촉이 없는 구간이 지배적인 `.torque` 채널도 같은 위험을 갖는다.
- → **채널별 `std`(또는 `q99-q01`)의 하한 검사가 필수**다(FR-REC-007).

### 2.5 🔴 `use_velocity_and_torque` — 기본값 `False`, 그리고 리더·팔로워 **동시** 설정

```python
# src/lerobot/robots/openarm_follower/config_openarm_follower.py:69-71
# When True, expose `.vel` and `.torque` per motor in observation features.
# Default False for compatibility with the position-only openarm_mini teleoperator.
use_velocity_and_torque: bool = False
```

리더도 동일 플래그를 갖는다 (`teleoperators/openarm_leader/config_openarm_leader.py:65`).

**끄면 조용히 위치-only 데이터셋이 만들어진다.** 예외도, 경고도 없다. 힘/컴플라이언스 데이터 수집이라는 플랫폼 정체성이 **기본값 하나로 소실된다.**

**불일치는 런타임 사망이다.** `lerobot_record.py:394-401`이 데이터셋 `action` feature를 **팔로워의** `robot.action_features`로 만들고, 값은 **리더(teleop)의 `get_action()` dict**에서 뽑는다:

```python
# src/lerobot/utils/feature_utils.py:131-132
elif ft["dtype"] == "float32" and len(ft["shape"]) == 1:
    frame[key] = np.array([values[name] for name in ft["names"]], dtype=np.float32)
```

→ 팔로워만 `True`면 `values["joint_1.torque"]`에서 **`KeyError`**로 record가 죽는다.

→ **GUI는 이 값을 팔로워·리더에 동시에 적용하는 단일 스위치로 묶고, 기본을 `True`로 강제해야 한다**(FR-REC-004, FR-REC-005).

### 2.6 🔴 `--robot.side=left|right` 필수 — 미지정 시 관절 리밋이 ±5°

```python
# src/lerobot/robots/openarm_follower/config_openarm_follower.py:55-56

# :107-120  — side 미지정 시 실제로 쓰이는 기본 joint_limits
joint_limits: dict[str, tuple[float, float]] = field(default_factory=lambda: {
    "joint_1": (-5.0, 5.0), "joint_2": (-5.0, 5.0), "joint_3": (-5.0, 5.0),
    "joint_4": ( 0.0, 5.0), "joint_5": (-5.0, 5.0), "joint_6": (-5.0, 5.0),
    "joint_7": (-5.0, 5.0), "gripper": (-5.0, 0.0),
```

### 2.9 에피소드 제어 — 키보드 이벤트와 record 루프 구조

```python
# src/lerobot/utils/keyboard_input.py:152-171, 420-423
events = {"exit_early": False, "rerecord_episode": False, "stop_recording": False}

def apply_recording_control(control: str, events: dict) -> None:
    if   control == "right": events["exit_early"] = True                                    # 종료 후 저장
    elif control == "left":  events["rerecord_episode"] = True; events["exit_early"] = True # 재기록
    elif control == "esc":   events["stop_recording"]  = True; events["exit_early"] = True  # 전체 중단
```

백엔드는 `pynput` → 실패 시 TTY 리스너 → 그것도 안 되면 리스너 없음(타이머/Ctrl+C만). **환경 의존적이다.**

**루프 구조** (`lerobot_record.py:467-516`):

```python
with VideoEncodingManager(dataset):
    recorded_episodes = 0
    while recorded_episodes < cfg.dataset.num_episodes and not events["stop_recording"]:
        record_loop(..., dataset=dataset, control_time_s=cfg.dataset.episode_time_s, ...)   # 녹화
        if not events["stop_recording"] and ((recorded_episodes < num_episodes - 1) or events["rerecord_episode"]):
            record_loop(..., control_time_s=cfg.dataset.reset_time_s, ...)                  # 리셋(비기록: dataset 인자 없음)
        if events["rerecord_episode"]:
            events["rerecord_episode"] = False; events["exit_early"] = False
            dataset.clear_episode_buffer()                                                   # 버퍼 폐기
            continue                                                                          # ← 카운터 증가 없음
        dataset.save_episode()
        recorded_episodes += 1
```

세 가지가 명세에 직결된다:

1. **재기록은 `clear_episode_buffer()` + `continue`** — `recorded_episodes`가 증가하지 않는다. 우리 상태머신의 `RE_RECORD`와 의미가 정확히 일치한다.
2. **리셋 구간은 `dataset` 인자 없이 `record_loop`을 도는 것**이다 — 텔레옵은 살아 있고 기록만 안 된다.
3. **`record_loop`은 `events` dict를 인자로 받는다**(`lerobot_record.py:228-248`). → **GUI가 `events`를 소유하면 키보드 백엔드에 의존하지 않고 에피소드를 제어할 수 있다**(FR-REC-024).

**데이터셋 API** (`datasets/lerobot_dataset.py`): `add_frame(frame)`(:404), `save_episode(...)`(:420), `clear_episode_buffer(delete_images=True)`(:438), **`finalize()`**(:460).

> 🔴 **`finalize()` 미호출 = 데이터셋 무효.** parquet 푸터가 기록되지 않는다. 모든 종료 경로(예외·크래시·강제종료)에서 `finally`로 보장해야 한다(FR-REC-032).

### 2.10 카메라 등록 — 빈 dict 사용자 설정, RGB(OpenCV) + 뎁스(RealSense) 혼합

**openarm 로봇은 카메라를 고정하지 않는다.**


→ **고정 슬롯도 표준 카메라 셋도 소프트웨어 계약에 없다.** 카메라는 전량 `--robot.cameras`로 사용자가 등록하며, HF openarm docs의 record 예시에도 `--robot.cameras`가 없다(전량 사용자 지정). 물리 카메라 배치·선정은 → `06`. **이전 명세의 "고정 4슬롯(Arducam ×3 + ZED mini)"은 하드웨어 참조설계였을 뿐 LeRobot 소프트웨어 계약이 아니므로 폐기한다**(D-12, → v3-8).

**LeRobot 카메라 클래스는 4종이다** (`src/lerobot/cameras/`, `make_cameras_from_configs`): `opencv`(USB 웹캠 — 로지텍 등, **RGB 3채널 only**), `realsense`(타입명 `intelrealsense`, **뎁스 지원**), `reachy2_camera`, `zmq`(네트워크). **ZED 전용 클래스도 스테레오 SBS 분할 로직도 없다**(직접 확인) — 뎁스는 RealSense가 준다.
```python
# cameras/opencv/configuration_opencv.py   (L56: "Only 3-channel color output (RGB/BGR) is currently supported.")
#   index_or_path, fps, width, height, color_mode=RGB, rotation, warmup_s=1, fourcc, backend
# cameras/realsense/configuration_realsense.py
#   serial_number_or_name, fps, width, height, color_mode=RGB, use_depth: bool = False
#   뎁스 스트림은 컬러와 동일 fps 락스텝(같은 rs pipeline)
```

> ⚠️ **`width`/`height`/`fps`를 명시하지 않으면** `_cameras_ft`의 shape가 `(None, None, 3)`이 되어 `hw_to_dataset_features()`가 터진다. **항상 명시한다**(FR-REC-013).

**뎁스는 LeRobot 버전이 가른다.** 0.6.x부터 뎁스가 관측→데이터셋으로 엔드투엔드로 흐른다 — 0.5.1(참조 레포 핀)은 `async_read()`/`read_latest()`가 *"Missing implementation for depth for now"*라 뎁스가 기록 경로에 없다. **우리 정본은 0.6.x**(§2.2 뎁스 row · NFR-REC-007). 0.6.x openarm_follower의 뎁스 배선 관용구:

```python
# robots/openarm_follower/openarm_follower.py (0.6.x)
# _cameras_ft:      if use_rgb:   features[cam]            = (H, W, 3)
#                   if use_depth: features[f"{cam}_depth"] = (H, W, 1)
# get_observation:  obs[cam] = cam.read_latest()
#                   if cam.use_depth: obs[f"{cam}_depth"] = cam.read_latest_depth()  # (H,W,1) uint16 mm, 0=측정불가
```

→ 뎁스 카메라 하나가 RGB 키 `observation.images.{cam}`와 뎁스 키 `observation.images.{cam}_depth` **두 스트림**을 낸다. 같은 패턴이 `so_follower.py`/`omx_follower.py`에도 있어 **0.6.x의 표준 뎁스 관용구**다.

**양완 카메라 키 네이밍** (`bi_openarm_follower.py:112-118`):

#### 2.10.1 뎁스 인코딩과 대역폭 — RGB와 인코더가 다르다

뎁스는 커스텀 카메라 클래스가 필요 없다(RealSense 네이티브, §2.10). 대신 **인코더가 RGB와 다르다**(`configs/video.py`):

| | RGB (`RGBEncoderConfig`) | 뎁스 (`DepthEncoderConfig`) |
|---|---|---|
| vcodec | `libsvtav1` (AV1) | `hevc` (HEVC **Main 12**) |
| pix_fmt | `yuv420p` | `gray12le` (12-bit) |
| 손실성 | 손실 (`crf=30`) | **무손실** (x265 `lossless=1`) |
| 양자화 | — | 12-bit(`DEPTH_QMAX=4095`), depth 0.01–10.0 m, log |
| 디코딩 | (일반) | **pyav 백엔드만** |

> 🔴 **뎁스 비디오는 무손실 12-bit이라 손실 AV1 RGB보다 훨씬 크다.** 뎁스 카메라 수만큼 스토리지·인코딩 부하가 급증하므로 용량 모델은 "카메라 수 × (RGB 손실 + 뎁스 무손실)"으로 분리해야 한다(FR-REC-056). 학습 파이프라인은 `video_backend=pyav`로 고정해야 뎁스 비디오를 디코딩할 수 있다.

### 2.11 세션 기동 전제 — CAN FD와 소유권

**`bitrate`/`data_bitrate` 인자는 python-can socketcan 백엔드에서 무시된다.** `ip link`로 CAN FD를 켜는 단계가 **필수이고 코드가 대신할 수 없다.** LeRobot이 공식 도구를 제공한다:

```bash
lerobot-setup-can --mode=setup --interfaces=can0,can1,can2,can3   # scripts/lerobot_setup_can.py
lerobot-setup-can --mode=test  --interfaces=can0
lerobot-setup-can --mode=speed --interfaces=can0
```
```python
# src/lerobot/scripts/lerobot_setup_can.py:66-67, 96-103
bitrate: int = 1000000            # 1 Mbps nominal
data_bitrate: int = 5000000       # 5 Mbps data (CAN FD)
cmd = ["sudo", "ip", "link", "set", interface, "type", "can", "bitrate", str(bitrate)]
if use_fd:
    cmd.extend(["dbitrate", str(data_bitrate), "fd", "on"])
```

**커널은 배타성을 강제하지 않는다.** SocketCAN RAW는 배타 bind 플래그가 없어 `cansend`·`openarm-can-cli`·두 번째 LeRobot 인스턴스가 무방비로 끼어든다. → **`flock` 애플리케이션 락**은 여전히 필요하다. 다만 역할이 "모드 중재"에서 **"외부 침입자 방어"**로 축소된다(→ `01`, `GROUND-v2 §5`).

### 2.12 🔴 `connect()`가 영점을 파괴한다 — 세션당 1회

```python
# src/lerobot/robots/openarm_follower/openarm_follower.py:152-153
if self.is_calibrated:
    self.bus.set_zero_position()      # → CAN_CMD_SET_ZERO (damiao.py:335-340)
```

**`connect()`는 현재 물리 자세를 영점으로 확정한다.** 팔이 rest 자세가 아닌 순간 재연결하면 **그 자세가 영점이 되고**, 관절 리밋과 **이미 기록된 데이터셋의 좌표계가 통째로 어긋난다.**

→ **모드 전환(텔레옵 ↔ 기록 ↔ 추론 ↔ 수동)에 disconnect/reconnect를 절대 하지 않는다. 세션당 `connect()`는 1회.** 모드는 **같은 robot 객체 위의 상태 전이**다.

→ 귀결: **`lerobot-record` CLI를 모드마다 서브프로세스로 스폰하는 설계는 금지된다.** CLI 스크립트는 각각 독립 프로세스이고 시작 시 `connect()` / 종료 시 `disconnect()`를 하기 때문이다(`pyproject.toml [project.scripts]`). **GUI가 LeRobot Python API를 임베드하고 `record_loop()`을 직접 호출한다**(FR-REC-023).

**추가**: OpenArm에는 **홀딩 브레이크가 없다.** 인에이블 이탈 = 토크 0 = **중력 낙하**(→ `12`).

## 4. 상태 · 오류 처리

### 4.1 에피소드 라이프사이클 상태 전이표

**상태**

| 상태 | 의미 |
|---|---|
| `IDLE` | 세션 미시작 |
| `SESSION_READY` | 데이터셋 root·로봇·카메라·태스크 확정. **CAN lock 미획득, `connect()` 미수행** |
| `ARMED` | `flock` 획득 → **`robot.connect()` 1회 수행**(rest 자세 확인 후) → 스트림 워밍업 완료. **태스크 미선택** |
| `TASK_SELECTED` | 태스크 선택 완료. 녹화 가능 |
| `RECORDING` | `record_loop(dataset=…)` 진행 중 |
| `STOPPING` | `exit_early` 세팅 → 루프 이탈 → `save_episode()` 진행 중 |
| `REVIEW` | 판정 대기 (즉시 재생 가능) |
| `SAVED` | `save_episode()` 완료, `dataset.num_episodes` 증가 확인 |
| `DISCARDED` | `clear_episode_buffer()` 수행, 저장 안 함 |
| `RESET` | 환경 리셋 구간 — `record_loop(dataset=None)` (비기록, 텔레옵 활성) |
| `SESSION_ENDED` | **`finalize()` 완료**, `disconnect()`, CAN lock 해제 |
| `FAULT` | 오류 상태 (§4.2) |

### 4.3 세션 중단 복구 (크래시 시 불완전 데이터셋)

**불변식**: `finalize()`가 완료된 데이터셋은 유효하다. 그렇지 않으면 **parquet 푸터가 없어 로드 자체가 실패한다.**

크래시 후 상태는 두 가지다:

1. **`finalize()` 완료** → 정상. `dataset.num_episodes`까지는 완전하다.
2. **`finalize()` 미완료** → **불완전 데이터셋.** 아래 절차를 적용한다.

```
불완전 데이터셋 복구 절차
 ① 격리      : <root>/ → quarantine/<stamped_repo_id>/ 로 이동 (원본 훼손 금지)
 ② 판독      : data/chunk-*/file-*.parquet 를 순차 판독하여 마지막으로 푸터가 온전한 파일을 찾는다
 ③ 절단      : 그 이후의 chunk/file 및 대응하지 않는 videos/*/chunk-*/file-*.mp4 제거
 ④ 메타 재구성 : meta/episodes/*.parquet 와 meta/info.json 의 total_* 카운터를 살아남은 프레임에 맞게 재계산
 ⑤ 재검증     : LeRobotDataset 로드 1회 성공 + stats 재계산
 ⑥ 판정      : 사용자에게 리포트 제시 → [저장] 이면 복구본 채택 / [폐기] 면 삭제
```

- **자동 저장은 금지한다.** 마지막 에피소드는 정의상 라벨이 없고 태스크가 완결되지 않았을 가능성이 높다.
- 복구된 데이터셋에는 `recovered` 플래그를 사이드카에 남겨 학습 데이터 선별 시 필터링할 수 있게 해야 한다.
- 저널(FR-REC-060)이 있으면 ⑥의 판정에 태스크·설정 컨텍스트를 함께 제시할 수 있다.

## 5. 미해결 · 결정필요

### Q1. 벽시계 타임스탬프를 어떻게 보존할 것인가? [결정필요]

**사실**: LeRobot `timestamp = frame_index / fps`(`dataset_writer.py:207-208`)는 **격자 좌표**이지 관측 시각이 아니다. 캡처 지터·카메라 도착 오프셋·CAN 지연은 **어디에도 기록되지 않는다.** 이것들 없이는 VR 지연이나 드롭이 데이터 품질에 미친 영향을 사후에 분석할 방법이 없다.

선택지:
- **(A) 별도 사이드카 파일**(에피소드별 `.parquet`/`.jsonl`). LeRobot 스키마를 건드리지 않는다. 데이터셋을 HF Hub에 올리면 사이드카가 따라가지 않을 수 있다.
- **(B) LeRobot 커스텀 feature 추가**(`capture_ts` 등). `DatasetInfo`가 feature 이름·dtype을 검증하지 않으므로 구조적으로 허용된다. **단 dtype이 자명하지 않다** — 유닉스 시각 ≈1.772×10⁹ s는 **float32로 표현 불가**(ULP = 128 s). float64(초, ULP ≈ 238 ns) 또는 int64(ns, 정확) 중 선택해야 하며, 후자는 LeRobot의 부동소수 feature 관례에서 벗어난다.

**결정하려면 확인할 것**: (a) 커스텀 feature를 실은 데이터셋이 `LeRobotDataset` 로드를 통과하는지, (b) [확인됨] `dataset_to_policy_features`(`feature_utils.py:169-212`)가 `observation`/`action` 접두가 아닌 feature를 `else→continue`로 배제하므로 `lerobot-train`이 이를 학습에서 자동 제외한다(→ v3-7) — 커스텀 시각 feature는 학습을 오염시키지 않는다, (c) 파생물 단독으로 타이밍 분석을 해야 하는 소비자가 실제로 있는지. **남은 것은 사이드카 vs 커스텀 feature의 운반성 선택이며, 실제 기록 1회 + 로드 1회로 확인한다.**

### Q2. 품질 게이트 임계치는 얼마인가? [결정필요]

인용 가능한 선례는 ALOHA의 **"목표 50 Hz 대비 42 Hz(84%) 미만이면 에피소드 저장 거부"** 하나뿐이며 다른 하드웨어의 값이다. 우리 목표는 30 fps이고 스택도 다르다(Python LeRobot 루프 + CAN FD).

**결정하려면 측정할 것**: 정상 세션 20~30 에피소드의 **실측 레이트 분포와 지터 분포**. 하위 백분위로 임계치를 잡는 것이 유일하게 방어 가능한 방법이다. 대상 지표: 루프 레이트, 카메라 fps, 사이클 타임 지터 최대, CAN 드롭률, 채널별 std 하한.

**선행 의존**: NFR-REC-002(LeRobot Python 루프의 실측 사이클 타임)가 확정되지 않으면 이 항목은 정할 수 없다.

### Q3. 성공/실패 라벨을 어디에 저장할 것인가? [결정필요]

LeRobot 네이티브 record는 성공 라벨을 기록하지 않는다(§2.13). 선택지:

- **(A) 사이드카 메타 파일** (`meta/` 아래 우리 파일). 파괴적이지 않고 `lerobot-edit-dataset`과 충돌하지 않는다. HF Hub 업로드 시 따라가는지 확인 필요.
- **(B) 태스크 문자열에 인코딩** (예: 접미사). `meta/tasks.parquet`이 자동으로 실어 나르지만 태스크 공간을 오염시키고 `task_index`가 2배로 늘어난다.
- **(C) LeRobot 커스텀 프레임 feature `next.success`** (상수는 이미 존재). 프레임마다 상수값이 반복되어 낭비이고, 라벨 수정이 parquet 재기록을 유발한다 → **FR-REC-040 위반**.

**결정하려면 확인할 것**: `lerobot-edit-dataset`의 `delete_episodes` / `split` / `merge`가 우리 사이드카를 어떻게 취급하는지(무시할 것으로 **추정**). 무시한다면 **에피소드 인덱스 재번호 시 라벨이 어긋난다** → 매핑 유지 책임이 우리에게 온다.

### Q4. 뎁스 스트림을 정책 학습 입력으로 쓸 것인가? [미확인]

0.6.x는 뎁스를 `observation.images.{cam}_depth` `(H,W,1)`로 **데이터셋에 기록**하는 것까지는 소스로 확인됐다(§2.10). 그러나 표준 정책(ACT/SmolVLA/pi0)이 이 뎁스 키를 **실제 학습 입력으로 소비**하는지(전처리·정규화 경로)는 소스로 끝까지 확인하지 못했다(→ v3-8 §5).

**결정하려면 확인할 것**: (a) 정책 config의 image key 매핑이 `_depth` 키를 VISUAL input으로 잡는지, (b) 잡는다면 12-bit 뎁스가 RGB 전처리(255 정규화·리사이즈)와 호환되는지. **기록 1회 + 학습 1스텝을 돌려봐야 안다.** 소비되지 않는다면 뎁스는 기록·시각화 용도로만 남고, 학습 입력 포함 여부는 FR-REC-070(학습 입력 토글)이 관장한다.

### Q5. 정지 채널의 `std ≈ 0` 문제를 어떻게 다룰 것인가? [결정필요]

`use_velocity_and_torque=True`로 켜면 `observation.state`에 `.vel`·`.torque` 채널이 들어오는데, 정지가 지배적인 관절은 `std ≈ 0`이 되어 `MEAN_STD` 정규화(`ACT`, `pi0`, `SmolVLA`)에서 `(x-mean)/std`가 폭발한다.

**결정하려면 확인할 것**: LeRobot `Normalize` 모듈이 `std`에 epsilon floor를 적용하는지(적용한다면 문제가 아니다). 소스로 확인하지 못했다 → **[미확인]**. 적용하지 않는다면 (a) 채널별 std floor를 우리가 stats에 주입하거나, (b) `QUANTILES` 정규화 정책(pi0.5)만 쓰거나, (c) 문제 채널을 학습 입력에서 제외하는 선택이 필요하다.

### Q6. 비디오 인코더 기본값을 그대로 쓸 것인가? [결정필요]

LeRobot 기본은 `vcodec=libsvtav1`, **`g=2`**, `crf=30`이다(`configs/video.py:86-94`). **GOP가 2프레임**이라 랜덤 액세스는 빨라지지만 **비트레이트가 크게 부풀고** 인코딩 CPU 부하가 커진다 — 이는 FR-REC-056(용량 모델)과 NFR-REC-005(저장 블로킹)를 동시에 지배하는 항이다.

**결정하려면 측정할 것**: 우리 카메라 구성(RGB 웹캠 + RealSense 뎁스 혼합)에서 `g=2` vs 더 큰 GOP의 (a) 에피소드당 바이트, (b) 인코딩 wall-clock, (c) 학습 시 랜덤 프레임 액세스 지연. 하드웨어 인코더(`vcodec=auto`) 가용성도 함께 확인한다. (뎁스는 무손실 HEVC 고정이므로 이 GOP 결정과 무관하다 — §2.10.1.)

### Q7. `openarm_control` IK 결과의 기록 좌표계 [결정필요]

`openarm_control`은 **rad**, LeRobot은 **deg**다. 그리고 **기계 상한(정본) ≠ LeRobot 소프트 클램프** — v2 URDF/MJCF right joint_1 = **−80°…+200°(정본)**, LeRobot deg 리밋 = **−75°…+75°(v1-era 소프트 클램프)**(→ `08 §5-5`, v3-6). IK가 LeRobot이 클램프할 해를 낼 수 있고, `kinematics.py:220-231`의 폴백은 `limits=[]`로 재시도하여 **관절 리밋을 포기한 해**를 낼 수 있다.

수집 관점의 함의: **기록되는 `action`은 클램프 전인가 후인가.** 클램프 전을 기록하면 로봇이 실행하지 않은 값이 지령으로 남고, 클램프 후를 기록하면 텔레옵 의도가 소실된다.

→ **정하려면**: `05`의 어댑터 클램프 위치가 확정되어야 한다. 그 전까지 두 값을 섞어 쓰지 말 것. (상세 → `05-텔레오퍼레이션.md`)
