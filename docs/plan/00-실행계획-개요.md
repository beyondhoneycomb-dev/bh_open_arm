# 00. 실행 계획 개요 — 읽는 프레임과 전역 규칙

이 문서는 `docs/plan/` 10문서를 읽고 실행하기 위한 **선언적 프레임**이다 — 무엇이 고정되어 있고(전제·확정), 무엇이 모든 작업 단위에 무조건 적용되며(불변식·명명 규약·게이트 상태머신), 어떤 규율이 수치를 못 박는 것을 금지하는가. **절차는 여기 없다**(→ `05-워크플로우-오케스트레이션.md`).

---

## 1. 이 묶음이 무엇인가

### 1.1 명세와의 관계 — 배제된 절반

`docs/spec/`(18문서)는 **기능 명세서**이고, 스스로 무엇을 배제했는지 선언했다:

> *"이 문서는 기획서가 아니다. 로드맵, 일정, 마일스톤, 인력, 비용, 시장 분석, 경쟁 우위는 이 묶음에 없다. 오직 '시스템이 무엇을 할 수 있어야 하는가', 그 입출력과 파라미터와 제약과 인터페이스 계약만 기술한다."* (`00-개요-및-문서규약.md` §1)

`docs/plan/`은 그 배제된 부분 중 **엔지니어링 실행 기획만** 담는다. 명세가 배제한 것 중에서도 **다시 배제하는 것**이 있다:

| 명세가 배제한 것 | 계획이 다루는가 | 근거 |
|---|---|---|
| 인터페이스 계약·파라미터·제약 | ❌ (명세 소관 — 재서술 금지) | `spec/`이 정본 |
| **의존성 순서·병렬화·배리어** | ✅ **이 묶음의 척추** | `01-의존성-DAG-및-병렬화.md` |
| **측정 게이트·음성 분기·후손 무효화** | ✅ | `03-측정-게이트.md` |
| **물리 안전 브링업 순서** | ✅ | `04-리스크-및-안전-브링업-게이트.md` |
| **워크플로우 fan-out 형상·격리** | ✅ | `05-워크플로우-오케스트레이션.md` |
| 로드맵·일정·마일스톤·기간·날짜 | ❌ **금지** | §1.3 |
| 인력·비용·예산·시장·경쟁 | ❌ **금지** | §1.3 |

### 1.2 방향 규칙 — 충돌 시 누가 이기는가

```
docs/spec/ (요구사항 정본)  ──정규화──>  Wave −1 원장  ──소비──>  docs/plan/ (실행 순서)
                                              ▲
                                     명세 내부 모순은 여기서만 해소된다
```

- **계획은 요구사항을 만들지 않는다.** 계획 문서에 새 FR/NFR이 등장하면 그것은 결함이다 — 명세에 올려야 한다.
- **계획은 요구사항을 고르지 않는다.** 명세 내부 모순(`01` §4.2 T1 `robot.connect(calibrate=True)` vs `02` §2.0.4 + `12` `FR-SAF-075` `connect_readonly()`)은 **Wave −1 정규화 원장**이 {승리 요구사항, 폐기 텍스트, 소유 계약버전}으로 해소한다. 개별 WP가 국지적으로 판단하는 것은 금지다.
- **계획은 게이트 ID를 소유한다.** 명세 측정 ID(`M-n`)와 계획 게이트 ID(`PG-*`)는 다른 네임스페이스다(§7.2).

### 1.3 🔴 이 묶음에 절대 들어가지 않는 것

| 금지 | 왜 |
|---|---|
| 일정·기간·날짜·주차·마일스톤 날짜 | 구현자가 사람 팀이 아니므로 인시(人時) 기반 추정이 정의되지 않는다. 순서는 **의존성**으로만 표현한다 |
| 비용·인력·예산 | 위와 동일. 조달 트랙 자체가 없다(§2.1) |
| 시장·경쟁·우선순위 논거 | 명세 §1이 배제했고 계획도 승계한다 |
| "개발자가", "팀이", "스프린트", "담당자" | 구현자 = **Claude dynamic workflow 병렬 fan-out**. 사람은 §4의 두 클래스에서만, **역할이 아니라 물리 행위**로 등장한다 |
| 측정 전 수치 목표 | §7.1 |

**웨이브는 일정이 아니다.** 웨이브는 **배리어로 구분된 의존성 대역**이며, 같은 대역 안의 WP는 동시에 fan-out된다. "Wave 2가 Wave 1보다 나중"은 시간이 아니라 **선행 게이트 통과**를 뜻한다.

### 2.3 🔴 U-1 종결 근거 — 선택이 아니라 물리

바이래터럴 힘반사는 "하지 않기로 했다"가 아니라 **보유 리그에서 성립할 수 없다**. 이유는 하나의 물리적 사실이다:

> **동력 리더암이 존재하지 않는다.**

| 후보 리더 | 힘반사 가능? | 1차 근거 |
|---|---|---|
| **OpenArm KER** | ❌ **원천 불가** | *"OpenArm KER (Kinematic Equivalent Replica) is a **motorless leader arm** … joints incorporate magnetic encoders and bearings."* → **"모터가 없으므로 KER로는 바이래터럴이 원천 불가능하다"** (`01`:264). 동일 확정: `16`:145 |
| **Quest 3S** | ❌ **진동만** | HMD 컨트롤러의 햅틱은 **진동 액추에이터**다. 관절 토크를 오퍼레이터 손에 반사할 자유도가 없다 |

**힘반사는 리더 관절에 토크를 걸어야 성립한다. 토크를 걸 모터가 리그에 없다.** 소프트웨어를 아무리 잘 써도 없는 액추에이터를 만들 수 없다.

**따라서 소멸하는 것:**

| 소멸한 것 | 왜 살아 있었나 | 왜 죽었나 |
|---|---|---|
| **500 Hz 제어 분기** | `openarm_bilateral_control.cpp:42,72,103` = 500 Hz 스레드 | `16` §5 **M-21: "500 Hz = 바이래터럴 힘반사 전용"**. 주경로(record 30–60 Hz / 추론 / 평가 250 Hz)엔 불필요 |
| **분리 RT 프로세스 경로** | Python이 500 Hz를 못 낼 경우의 대비 | 500 Hz 요건 자체가 없어짐 |
| **CAN 배타소유 이양 (T-BI 전이)** | `01` §4.2 (T-BI) = **유일하게 버스를 놓는 전이**, `flock`을 자식 프로세스에 이양 | `01` §4.2 스스로 인정: *"이 전이가 존재하는 것 자체가 아키텍처 부채다."* U-1이 그 부채의 트리거를 제거한다 → **T-BI는 계획 DAG에 없다** |

**남는 것과 남지 않는 것 (경계를 흐리지 마라):**

- ✅ **소프트웨어 실험 어댑터는 선택으로 남길 수 있다.** 버스 레벨 토크 채널은 이미 뚫려 있다(`damiao.py:492-502` `_mit_control_batch`가 tau 인자 수용, `16` §10.1). 막힌 것은 `send_action`의 tau=0 하드코딩 하나뿐이고, 그 확장은 어차피 `FR-SAF-069`(τ_grav 전달)가 **안전 목적으로 요구**한다.
- ❌ **하드웨어 수용 주장 불가.** "바이래터럴 동작 확인"이라는 수용 기준을 리그에서 세울 수 없다. 어댑터를 만들더라도 **게이트를 걸 수 없고**, 게이트 없는 산출물은 계획의 수용 대상이 아니다.

> **이것은 무엇을 나쁘게 만드는가.** U-1은 **플랫폼의 정체성 일부를 포기한다** — OpenArm의 높은 백드라이버빌리티는 애초에 바이래터럴을 위해 설계된 특성이고(`00` §3.3), 우리는 그 대가(홀딩 브레이크 부재, §6)는 전부 지불하면서 이득은 받지 않는다. 또한 동력 리더암이 나중에 리그에 들어오면 **500 Hz 판정·RT 분리·CAN 소유권이 전부 되살아나며**, 그때 §5 불변식 1(상시 ActuationScheduler)과 배타 버스 소유가 정면 충돌한다. U-1은 그 충돌을 **해결한 것이 아니라 연기한 것**이다.

### 2.4 🔴 U-4 데드맨 리스 의미론

**문제**: `04` `FR-MAN-050`은 데드맨 릴리즈 시 즉시 Cat-2 홀드를 요구한다. 그런데 D-2가 확정한 단일 WebSocket에는 **HOL(head-of-line) 블로킹이 불가피**하다(`16` §5 D-2: RFC6455 비인터리브 + 수신 backpressure 부재). 카메라 프레임이 앞을 막으면 "릴리즈" 이벤트가 지연된다.

**폐기한 해법**: D-2 재개(채널 분리) — 하지 않는다.

**채택한 의미론 — 리스(lease)**: 데드맨을 **엣지 이벤트가 아니라 계속 갱신해야 하는 임차권**으로 뒤집는다.

| 폴링/엣지 데드맨 (❌) | 리스 데드맨 (✅) |
|---|---|
| 눌림 = 이벤트, 릴리즈 = **이벤트** | 눌림 = **주기적 갱신 스트림**, 릴리즈 = **갱신 중단** |
| WS 지연 → 릴리즈 이벤트 지연 → **정지가 늦어진다** | WS 지연 → 갱신 누락 → 리스 만료 → **스케줄러 자동 홀드** |
| 지연이 안전을 **깎는다** | 지연이 정지를 **앞당긴다** |

> **핵심 성질**: 실패 모드가 뒤집힌다. 리스 만료 판정은 **백엔드 스케줄러 틱 안**에서 일어나므로 브라우저·네트워크 상태와 무관하게 성립한다. 전송로가 나빠질수록 시스템은 **더 빨리** 홀드한다.

**동반 규칙:**

| 항목 | 규칙 |
|---|---|
| 프리뷰(카메라) | **lossy · latest-wins · FPS 저하 허용.** 프리뷰 품질은 안전 예산을 사지 못한다(`NFR-PRF-041` latest-wins + 드롭 카운터 노출) |
| 리스 갱신 주기 | **못 박지 않는다** — `PG-STOP-001` 측정 후 도출(§7.1) |
| 정지 경로 | `bus.disable_torque()` **금지** — 모터마다 순차 송신 후 최대 1 ms 블로킹(`03` `NFR-MOT-005`, `04` `NFR-MAN-002`) |

**🔴 잔여 위험 (명시한다):**

브라우저 소프트 E-Stop은 **HOL만큼 늦을 수 있다.** 리스는 *데드맨*을 고칠 뿐 *E-Stop 버튼*을 고치지 않는다 — 버튼은 여전히 브라우저→WS→백엔드 홉을 탄다(`13` `NFR-GUI-005`, Q-5 = **[결정필요]**, 수치는 자체 벤치로 확정).

> **진짜 안전장치는 전원 라인의 물리 버튼이다** (`16` §4 **M-2**: *"E-Stop = 모터 전원 자체 차단 → CAN도 끊긴다"*). 소프트 E-Stop은 편의 기능이고, 하드 E-Stop은 안전 기능이다. GUI가 이 둘을 같은 위젯처럼 보이게 만들면 그 자체가 결함이다(`00` §3.3: 소프트 스톱과 하드 E-Stop은 **분리되어야 한다**).

### 2.5 🔴 U-6 — 부트스트랩 먼저, 접기는 보류

**진단은 둘이었고, 둘 다 사실이다. 순서가 문제였다.**

| 진단 | 사실인가 | 왜 이것이 먼저가 아닌가 |
|---|---|---|
| **다수 WP가 독립 병합 단위가 아니다** → 접어야 한다 | ✅ 타당하다 | 접기의 이득은 **오버헤드 절감**인데, 우리는 그 오버헤드를 **한 번도 측정한 적이 없다.** 측정 없이 접으면 그것은 최적화가 아니라 추측이다 |
| **레지스트리를 돌릴 기계가 없다** | ✅ **이쪽이 진짜 병목** | `06`은 스스로를 "기계 정본"이라 부르지만 **존재하는 것은 예시 레코드 4건뿐**이다. 질의·락·스폰·병합·무효화가 전부 산문 위에 서 있다 |

**왜 부트스트랩이 이긴다.** 병렬 fan-out된 워크플로우는 산문을 읽지 않는다 — **질의한다.** 그런데 질의할 대상이 없다. §3.3이 규정한 후손 무효화 4종, §3.4의 파일 소유권, §8의 ID 실재 검사 — 이 문서가 "CI가 검증한다"고 쓴 모든 문장이 **현재는 실행체 없는 규칙**이다. 규칙만 있고 검사기가 없으면 그것은 규율이 아니라 **희망**이다.

**그리고 접기를 지금 하면 안 되는 진짜 이유**: 접기는 **ID 공간을 다시 연다.** 감사가 닫은 ID 공간(§8)을 재개하는 대가는 확실하고, 접기의 이득은 아직 추측이다. **확실한 대가를 추측된 이득과 바꾸지 않는다.**

| 결정 | 내용 |
|---|---|
| ✅ **한다** | Wave −2 `BOOT` 대역 신설(§3.5) — 부트스트랩이 착지해야 어떤 WP도 착수 가능 |
| ✅ **한다** | D2 파일럿(§3.5b) — WP 포맷의 실행 가능성을 **1개로 증명**한 뒤에 나머지를 돌린다 |
| ⏸️ **보류** | WP 접기. 재검토 입력 = **파일럿이 측정한 실제 오버헤드**. 그전까지 감사가 닫은 ID 공간은 **재개하지 않는다** |

> **이것은 무엇을 나쁘게 만드는가.** U-6은 **가장 지루한 것을 크리티컬 패스 맨 앞에 놓는다** — 스키마·검사기·상태 저장소는 로봇을 1 mm도 움직이지 않으면서 모든 것을 막는다. 그리고 부트스트랩이 틀리면 **그 위에 세운 171개가 전부 틀린다.** 그 위험은 D2 파일럿(§3.5b)이 1개 표본으로 줄일 뿐 없애지 못한다. 대안은 검사기 없이 fan-out을 시작하는 것인데, 그건 **아무도 안 보는 규칙을 171번 위반하는 것**이다.

### 3.1 세 가지 객체

| 객체 | 정의 | 무엇이 아닌가 |
|---|---|---|
| **WP (작업 패키지)** | **단일 워크플로우가 소유하고 끝까지 책임지는 최소 단위.** {입력 · 산출 · 인터페이스 계약 · 수용 게이트 · 실행 클래스 · 워크플로우 형상}이 전부 명시된 것 | 태스크 목록이 아니다. **파일/모듈 소유권을 동반**하지 않으면 WP가 아니다(§3.4) |
| **게이트** | **통과 여부가 하류 WP의 착수 가능성을 바꾸는 판정점.** 상태머신(§3.3)을 갖고, 음성 분기가 명명되어 있으며, 후손 무효화 규칙이 있다 | 체크리스트가 아니다. "확인한다"로 끝나면 게이트가 아니다 |
| **웨이브** | **배리어로 구분된 의존성 대역.** 같은 대역 = 동시 fan-out 가능 | **일정이 아니다**(§1.3) |

### 3.2 WP 성립 조건 (6요소 전부 없으면 WP가 아니다)

| 요소 | 없으면 생기는 실패 |
|---|---|
| **입력** (선행 산출 + 정규화 해시) | Wave −1 정규화 없이 착수 → 폐기된 명세 텍스트 구현 |
| **산출** (파일 경로 수준) | 후손 stale 전파가 불가능 |
| **인터페이스 계약 + 버전** | 병렬 빌더가 서로의 시그니처를 깬다 |
| **수용 게이트** | "동작한다"로 종료 → 검증 불가 |
| **실행 클래스** (§4) | 사람이 필요한 시점에 워크플로우가 벽에 부딪힌다 |
| **워크플로우 형상** (fan-out 폭·worktree 격리·verify 루프) | `05` 소관. 여기서 정의하지 않는다 |

### 3.3 게이트 상태머신 (전 게이트 공통, 예외 없음)

```
                    ┌──────────────────────────────────────┐
                    │                                      │
   [측정 실행] ──> 판정 ──> PASS ─────────────────> 하류 착수 허가
                     │
                     ├──> RETRY_WITH_VARIANT ──> [명명된 변형으로 재측정] ──┘
                     │
                     ├──> DEGRADED_ACCEPTED ──> 하류 착수 허가 (축소 스코프 + 표시 의무)
                     │
                     ├──> FAIL_BLOCKING ──> 🔴 후손 전면 차단 + 대체 WP 생성
                     │
                     └──> SUPERSEDED ──> 상위 게이트/정규화가 이 판정을 무효화
```

| 상태 | 뜻 | 필수 동반 산출 |
|---|---|---|
| `PASS` | 수용 기준 충족 | 증거 아티팩트 + 측정 조건 |
| `RETRY_WITH_VARIANT` | 실패했으나 **변형이 명명되어 있고** 그 변형이 계약을 바꾸지 않는다 | 시도한 변형 목록 + 각 결과 |
| `DEGRADED_ACCEPTED` | 축소된 능력으로 수용 | **무엇이 축소됐는지 + 어디에 표시되는지**(예: "이 타깃 미지원") |
| `FAIL_BLOCKING` | 하류가 물리적으로 성립 불가 | 차단된 후손 목록 + 대체 WP ID |
| `SUPERSEDED` | 이 게이트의 전제가 사라짐 | 무효화한 상위 판정 참조 |

**🔴 상태 변경 시 자동으로 일어나야 하는 것 (넷 다 필수):**

1. **후손 아티팩트 자동 stale 표시** — 이 게이트 산출을 입력으로 받은 모든 아티팩트
2. **미통합 작업 자동 취소** — stale 입력 위에서 진행 중이던 fan-out 브랜치
3. **명명된 대체 WP 생성** — `FAIL_BLOCKING` / `RETRY_WITH_VARIANT`
4. **추적성 레지스트리 갱신** — `06-추적성-레지스트리.md`

> 🔴 **리스크 등록부에 한 줄 적고 넘어가는 것은 금지다.** 상태 변경이 후손을 건드리지 않으면 그것은 상태 변경이 아니라 메모다. 게이트가 부정 판정을 냈는데 하류가 계속 초록불이면 **게이트가 없는 것보다 나쁘다** — 없으면 최소한 아무도 믿지 않는다.

> **이것은 무엇을 나쁘게 만드는가.** 자동 stale 전파는 **재작업 폭을 키운다** — 깊은 게이트 하나가 뒤집히면 넓은 후손 트리가 통째로 무효화되고, 그중 일부는 실제로는 영향받지 않았을 수 있다. 우리는 **정밀한 영향 분석보다 과잉 무효화를 택했다**. 이유는 비대칭이다: 과잉 무효화의 대가는 재실행이고, 과소 무효화의 대가는 **잘못된 전제 위에 세워진 것을 모른 채 토크를 인가하는 것**이다.

### 3.4 파일/모듈 소유권 — 병렬화의 진짜 제약

## 4. 실행 클래스 4종

**AI/사람 2분법은 폐기한다.** "사람이 필요한가"는 이진 질문이 아니라 **사람이 무엇을 하는가**의 질문이다 — 잡아주는 것과 판정하는 것은 워크플로우 형상이 전혀 다르다.

| 클래스 | 뜻 | 사람의 역할 | 예시 |
|---|---|---|---|
| **`AI-offline`** | 코드·시뮬·더미·결함주입. **사람 불필요** | 없음 | MuJoCo 백엔드, IK 어댑터, FK↔IK 회귀, 합성 GIL 부하 하네스, 정책 호환 매트릭스(`FR-TRN-064`), 학습/평가 통계 |
| **`AI-on-HW`** | 스크립트 읽기/측정. **물리 조작 없음** | 없음 (리그가 안전 초기상태에 있음을 전제) | RID 리드(`openarm-can-cli show_param`), `ip -details link show`, udev 열거, `candump`, 카메라 열거, `PG-RT-001` 읽기전용 측정 |
| **`Human-assisted-HW`** | 워크플로우가 하네스를 돌리고 **사람이 물리적으로 개입** | **잡아주기 / 정렬 / 누르기 / 리셋 / 시연** | 명시 영점(rest 정렬 → 버튼), 마찰 여기 궤적, E-Stop 시험, APK 사이드로드, 핸드아이 캘리브, 데이터 수집 |
| **`Human-judgment`** | 워크플로우가 판정할 수 없는 것 | **성공 라벨링 / 물리 정렬 수용 / 충돌없음 판정** | 에피소드 성공 라벨, 영점 자세 수용, 잔차 임계 수용 |

> **왜 `AI-on-HW`와 `Human-assisted-HW`를 가르는가.** `AI-on-HW`는 **블로킹 없이 fan-out된다** — 사람의 가용성이 DAG에 들어오지 않는다. `Human-assisted-HW`는 **워크플로우가 멈추고 재개 신호를 기다린다**. 이 둘을 뭉치면 계획의 절반이 근거 없이 직렬화된다. Wave 0-B가 존재하는 이유가 정확히 이것이다 — CAN·udev·RID·카메라 검증의 **대부분은 vcan으로 AI 단독 검증**되고, 실물이 필요한 것은 `ethtool -i` / `dev_id` / USB RTT뿐이다.

### 4.1 🔴 모든 물리 게이트가 명시해야 할 6요소

`Human-assisted-HW` 또는 `Human-judgment`를 포함하는 **모든 게이트**는 아래 6칸을 채운다. 하나라도 비면 게이트로 인정하지 않는다.

| # | 요소 | 비었을 때의 실패 |
|---|---|---|
| 1 | **에이전트가 무엇을 하는가** | 워크플로우가 사람 행위를 대신 시도하거나, 반대로 자기 몫을 사람에게 미룬다 |
| 2 | **사람이 무엇을 하는가** | 물리 행위가 "알아서" 일어날 것으로 가정된다 |
| 3 | **중단 조건** | 사람이 언제 멈춰야 하는지 모른 채 계속한다 — **홀딩 브레이크가 없는 팔 옆에서**(§6) |
| 4 | **안전 초기상태** | 토크 ON 상태에서 사람이 팔을 잡는다 / 지지 없이 인에이블한다 |
| 5 | **산출 증거** | 통과 주장이 검증 불가 → 후손 stale 전파가 불가능(§3.3) |
| 6 | **재개 지점** | 사람 개입 후 워크플로우가 어디로 돌아갈지 몰라 **처음부터 다시 = 물리 리그 재세팅** |

**전수 기입은 `04-리스크-및-안전-브링업-게이트.md`가 게이트별로 한다.** 여기서는 계약만 선언한다.

### 5.1 🔴 I-1 — 왜 뮤텍스로 대체할 수 없는가

**틱 규칙 — 이 그림이 규범이다. 표와 어긋나면 이 그림이 이긴다.**

```
                    ActuationScheduler  (유일 CAN writer)
                    토크-ON ─────────────────────────► 토크-OFF
                              틱은 이 구간에서 결코 멈추지 않는다

  생산자 (CAN 틱을 소유하지 않는다)          매 틱 (주기 = PG-RT-001 실측)
  ┌──────────────┐                          ┌──────────────────────────┐
  │ MANUAL 조그   │──┐                      │ 1. 메일박스 read         │
  │ TELEOP VR/KER │──┤   timestamped        │ 2. freshness 판정        │
  │ RECORD        │──┼──► [ 메일박스 ]────► │ 3. 안전 게이트웨이 통과  │
  │ INFERENCE     │──┤    (atomic,          │ 4. ─► CAN 프레임 1개     │
  │ REPLAY        │──┘     latest-wins)     └──────────┬───────────────┘
  └──────────────┘                                     │
        ▲                                              ▼
        │ 모드 전환 = 생산자만 원자 교체        방출 4종 (반드시 정확히 하나)
        │ ❌ 스케줄러 stop()/join() 없음        ┌────────────────────────┐
        └───────────────────────────────────── │ ① 수용 타깃            │
                                               │ ② stale-source 홀드    │◄ 생산자 침묵
                                               │ ③ 모드전환 홀드        │◄ 교체 중
                                               │ ④ 안전래치 홀드        │◄ 래치 발동
                                               └────────────────────────┘
                                                          │
                     ┌────────────────────────────────────┴──────────┐
                     │ 방출 0회 = RID 9 comm-loss → 인에이블 이탈    │
                     │          → 토크 0 → 🔴 중력 낙하 (브레이크 無) │
                     └───────────────────────────────────────────────┘
```

> 🔴 **읽는 법**: 4종 방출은 배타적이고 **전면적**이다 — 틱마다 정확히 하나가 나간다. "보낼 게 없으면 안 보낸다"는 경로가 **존재하지 않는다.** 생산자가 죽든, 모드가 바뀌든, 래치가 걸리든 프레임은 나간다. 이 그림에 없는 화살표(생산자 → CAN 직행, 스케줄러 정지)를 구현하면 팔이 떨어진다.

명세는 모드 전환을 뮤텍스로 조율한다:

> `01` §4.2 (T2) 2단계: *"`send_action()` 뮤텍스가 해제되었는지 확인"*

**뮤텍스는 옳은 도구이지만 다른 질문에 답한다.**

| | 뮤텍스가 보장하는 것 | I-1이 요구하는 것 |
|---|---|---|
| 명제 | **"둘 이상이 동시에 부르지 않는다"** (배타) | **"매 틱 누군가 반드시 부른다"** (전면성) |
| 실패 모드 | 중복 호출 | **침묵** |
| 침묵을 막는가 | ❌ **아니다** — 아무도 락을 잡지 않는 상태가 완벽히 합법이다 | ✅ 그것이 유일한 존재 이유 |

**그리고 이 시스템에는 침묵이 설계상 존재한다.**

`01` §4.1의 모드 표를 그대로 읽으면:

| 모드 | `send_action()` 호출권 | 비고 (원문) |
|---|---|---|
| **`CONNECTED`** | 🔴 **없음 (유휴)** | **"관측만 폴링. 모든 모드의 허브"** |
| `SAFE_HOLD` | 없음 | (중력보상 홀드) |
| `FAULT` | 없음 | 사람 개입 전까지 자동 복구 금지 |
| `CALIBRATE` | 없음 (토크 OFF) | — |

**`CONNECTED`가 문제다. 유휴 상태여서가 아니라 허브여서다.**

`01` §4.1 다이어그램상 `MANUAL`·`TELEOP`·`RECORD`·`INFERENCE`·`CALIBRATE`는 전부 `CONNECTED`에서 뻗어 나온다. 그리고 (T2) 전이는 **① 현재 루프 정지 → `join()` → ② 뮤텍스 해제 확인 → ③ 새 루프 시작**이다. 즉 **모든 모드 전환은 호출권이 "없음"인 상태를 통과한다.**

**토크는 그동안 켜져 있다** — `01` §4.2 (T1) 6단계가 `enable_torque()`를 부르고, (T2)는 **버스를 놓지 않으며**, 토크-OFF도 하지 않는다.

**연쇄:**

```
CONNECTED 통과 (토크 ON, 호출권 없음)
        ↓
  CAN 명령 스트림 침묵
        ↓
  RID 9 TIMEOUT 경과            ← 通讯丢失防护 (12 §2.3)
        ↓
  모터 자동 인에이블 이탈         ← "모터의 자체 워치독은 안전 홀드가 아니라 안전 낙하다"
        ↓
  토크 0
        ↓
  🔴 낙하 (홀딩 브레이크 없음 — §6)
```

**뮤텍스는 이 경로의 어느 화살표도 끊지 못한다.** 뮤텍스는 완벽하게 정상 동작 중이다 — 아무도 락을 원하지 않을 뿐이다. `join()`과 새 루프 `start()` 사이의 간격, GIL 경합으로 늘어난 스레드 기동 지연, `FAULT` 진입 후 사람을 기다리는 시간 — 전부 **뮤텍스 관점에서 아무 문제 없는 상태**이고, **모터 관점에서 낙하 카운트다운**이다.

`12` `FR-SAF-073`이 같은 결론에 이미 도달해 있다:

> *"시스템은 **안전 정지를 '루프 중단'으로 구현해서는 안 된다.** 정지는 **루프가 계속 돌면서 보내는 명령을 `STOP_HOLD`로 바꾸는 것**이어야 한다."*

**I-1은 이 요구를 `FAULT`·`SAFE_HOLD`에서 `CONNECTED`와 모든 전이 구간으로 확장한 것이다.** 안전 정지에서 루프를 끊으면 안 된다면, **모드를 바꿀 때도 끊으면 안 된다** — 모터는 그 둘을 구분하지 못한다. 모터가 아는 것은 프레임이 왔는가 안 왔는가뿐이다.

**따라서 소유권이 뒤집힌다:**

| | 명세의 기본형 | I-1 |
|---|---|---|
| CAN 틱 소유 | **생산자**(각 모드의 제어 루프) | **`ActuationScheduler`**(단일, 상시) |
| 모드 전환 | 루프 **정지 → 시작** | 생산자 **원자 교체** (스케줄러는 안 멈춤) |
| 생산자 역할 | `send_action()` 호출 | **메일박스에 타임스탬프 타깃 publish만** |
| 생산자 없음 | 침묵 | **stale-source 홀드 방출** |
| 전환 중 | 침묵 | **모드전환 홀드 방출** |
| 안전 래치 | 루프 중단 | **안전래치 홀드 방출** |

> ⚠️ **`01` §4.2 (T2)의 뮤텍스 문구는 Wave −1 정규화 대상이다** — `01` §4.1의 `CONNECTED` 호출권 "없음"이 `12` `FR-SAF-073`의 상시 송신 의무와 정면 충돌한다. 계획이 국지적으로 판단하지 않는다(§1.2). 정규화 원장이 승리 요구사항을 확정하기 전까지 이 지점의 WP는 착수 금지다.

> **이것은 무엇을 나쁘게 만드는가.** I-1의 대가는 셋이다. **(1) 디버깅이 나빠진다** — 생산자를 멈춰서 격리하는 기법이 사라진다. 항상 CAN 라이터가 있으므로 "누가 이 프레임을 보냈나"는 메일박스와 방출 사유 태그를 통해서만 답할 수 있고, 그래서 **방출 4종에 사유 태그를 의무화**했다(§8 계약). **(2) 지연 회계가 나빠진다** — 생산자와 CAN 사이에 메일박스 홉이 하나 늘고, 이 홉의 지연이 `PG-STOP-001` 예산에 들어간다. **(3) "중단"이 원시 연산에서 사라진다** — 진짜 침묵이 필요한 시나리오(버스 진단, `candump` 단독 관찰)는 반드시 **토크-OFF를 선행**해야 하며, 워크플로우가 이를 잊으면 스케줄러가 진단을 오염시킨다. 우리는 이 셋을 지불하고 **"낙하하지 않음"을 타입 수준 성질로** 산다.

### 5.2 🔴 I-2 — 논리 세션 불변식이지 소켓 수명 불변식이 아니다

**I-2를 "소켓을 두 번 열지 마라"로 읽으면 하드 E-Stop 이후 복구가 원리적으로 불가능해진다.**

연쇄는 이렇다. 하드 E-Stop은 **전원과 CAN을 함께 끊는다**(I-3 예외, `16` §4 M-2). 전원이 돌아오면 CAN 인터페이스는 사라졌다 다시 나타나고, **기존 소켓은 죽어 있다.** 이때 I-2를 소켓 수명으로 읽으면 두 갈래뿐이고 둘 다 틀렸다:

| 오독의 귀결 | 무엇이 잘못되나 |
|---|---|
| 소켓을 다시 열지 않는다 | **복구 불가.** E-Stop을 한 번 누르면 프로세스를 죽이는 것 외에 방법이 없다 — E-Stop이 편의가 아니라 **일회용 폭탄**이 된다 |
| `connect()`를 다시 부른다 | 🔴 **I-2 위반이자 영점 파괴.** `connect()`는 auto set-zero를 동반하고(`FR-CON-061`), 현재 자세를 영점으로 다시 찍는다 — **E-Stop으로 낙하한 자세가 새 원점이 된다** |

**따라서 I-2가 무엇을 세는지 정확히 못 박는다.**

| | **논리 세션** (I-2가 세는 것) | **소켓/전송로** (I-2가 세지 않는 것) |
|---|---|---|
| 무엇인가 | 캘리브레이션 원점 · 모터 인에이블 상태 · 스케줄러 생애 · 제어권 세대 | SocketCAN fd, `ip link` 인터페이스 상태 |
| 몇 번 성립하는가 | **세션당 정확히 1회** | 필요한 만큼 (전원 사이클마다 죽는다) |
| 다시 만들면 | 🔴 **영점이 파괴된다 — 금지** | ✅ 아무것도 파괴되지 않는다 |

**전송 전용 재열기 (transport-only reopen) — 신설 연산:**

| 이 연산이 하는 것 | 이 연산이 **하지 않는 것** |
|---|---|
| ✅ SocketCAN 소켓 재생성 | ❌ **영점을 찍지 않는다** (`set_zero` 호출 0회) |
| ✅ 인터페이스 재바인드 + `flock` 재획득 | ❌ **모터를 인에이블하지 않는다** |
| ✅ 이중바인드 능동검사 재실행(I-7) | ❌ 캘리브레이션을 다시 읽거나 쓰지 않는다 |
| ✅ 스케줄러에 전송로 복구를 통지 | ❌ 토크를 켜지 않는다 — 재무장은 **명시적 핸드셰이크**의 몫 |

> 🔴 **판정 기준 하나**: *이 연산 이후 디스크 캘리브레이션과 논리 세션 상태가 **비트 단위로 동일한가**.* 동일하면 전송 전용 재열기이고 I-2를 위반하지 않는다. 하나라도 달라지면 그것은 재열기가 아니라 **재연결이고 I-2 위반**이다.

**계약상 귀결:**

| 항목 | 규칙 |
|---|---|
| 계약 명시 위치 | `CTR-CAL@v1` + 연결 계약(`CTR-PLUG@v1`)이 이 연산을 **명시적으로 선언**한다. 선언 없는 소켓 재생성은 I-2 위반으로 취급 |
| 재열기 후 상태 | **토크 OFF · 안전래치 홀드 유지.** 전송로가 살아났다는 사실은 **동작 재개 권한이 아니다** |
| 동작 재개 | 전송 전용 재열기 ≠ 재무장. 재개는 **명시적 재무장 핸드셰이크**(새 세대 ID + 오퍼레이터 의도 확인)로만 |
| GUI | **여전히 재연결 버튼 없음.** 전송 전용 재열기는 E-Stop 복구 절차의 내부 단계이지 사용자가 누르는 것이 아니다 |

> **이것은 무엇을 나쁘게 만드는가.** 이 예외는 **I-2의 정적 검사 가능성을 깎는다** — "`connect()` 호출 1회"는 grep으로 세지만 "논리 세션 1회"는 세지 못한다. 재열기 경로가 조용히 영점을 건드리는지는 **호출 횟수가 아니라 캘리브레이션 해시 불변성**으로만 검증되고, 그래서 재열기 전후 해시 대조가 계약의 일부다. 대안은 없다 — 예외를 지우면 E-Stop이 복구 불가능해지고, 그건 **사람이 E-Stop을 누르기 싫어하게 만든다.** 누르기 싫은 E-Stop은 없는 E-Stop보다 나쁘다.

## 6. 🔴 홀딩 브레이크 부재 = 영구 물리 위험 (결함이 아니다)

**이것은 고칠 수 있는 것이 아니다. 계획 전체가 이 사실 위에 서 있다.**

> 공식 Safety Guide: *"OpenArm has high backdrivability suitable for bilateral control. **Be aware that if power is lost due to an emergency stop, the load being held will fall rapidly.**"* (`00` §3.3)

**왜 결함이 아닌가.** 백드라이버빌리티는 **버그가 아니라 설계 목표**다 — QDD 관절 + 브레이크 없음이 바이래터럴의 전제조건이다. 우리는 U-1로 그 이득을 포기했지만(§2.3), **비용은 하드웨어에 내장되어 있어 포기할 수 없다.**

### 6.1 소프트웨어가 막을 수 없는 것

`12`의 실패 모드 표(§2.x)가 확정한 경계:

| 사건 | 소프트 홀드 가능? | 결과 | 유일한 대응 |
|---|---|---|---|
| 안전 래치 / 데드맨 릴리즈 / 충돌 감지 | ✅ | `STOP_HOLD` = `MIT(kp, kd, q_hold, 0, τ_grav)` **지속 송신** | I-1 |
| **프로세스 사망**(SIGKILL/OOM/deadlock) | ❌ | 🔴 **낙하** — `STOP_HOLD`를 보낼 주체가 사라진다 | **기계·전기적 수단만** |
| **CAN bus-off** | ❌ | 🔴 **낙하** — 홀드 명령을 전송할 수 없다 | **기계·전기적 수단만** |
| **전원 상실 / 하드 E-Stop** | ❌ | 🔴 **낙하** (설계된 동작) | **기계·전기적 수단만** |

> `12` 원문: *"**③ 범주(전원 차단·프로세스 사망·bus-off)는 소프트웨어로 홀드할 수 없다.** 소프트 홀드(②)는 제어 프로세스가 살아 CAN 명령을 계속 보낼 때만 성립한다"* (`NFR-SAF-009`)

**SW 워치독은 낙하를 늦출 뿐 못 막는다.** 크래시 워치독은 프로세스 사망을 **감지**하고 재기동할 뿐이고, 그 사이 팔은 이미 떨어져 있다. RID 9 타임아웃은 **낙하 시점을 정할 뿐**이다.

### 6.2 성립 조건 — 셋 다 필요하다 (`NFR-SAF-009`)

| 조건 | 뜻 | 게이트 |
|---|---|---|
| **기계적 지지** | 인에이블 이탈 시 팔이 도달 가능한 자세를 물리적으로 제한 | `PG-SAFE-001` |
| **낙하영역 격리** | 낙하 궤적 안에 사람·물체·카메라가 없음 | `PG-SAFE-001` |
| **독립 전원차단** | 제어 프로세스와 무관하게 동작하는 전원라인 물리 버튼(`16` §4 M-2) | `PG-SAFE-001` |

### 6.3 계획상 귀결 (타협 불가)

| 규칙 | 내용 |
|---|---|
| **`PG-SAFE-001`이 모든 토크-ON의 선행** | 불일치 = **`FAIL_BLOCKING`** → 토크-ON 후손 **전면 차단** |
| **`FAIL_BLOCKING`의 해소는 조달이 아니라 통합 교정** | 리그 교정(지지 추가·낙하영역 정리·전원도메인 분리) 후 재실행. §1.3에 따라 조달은 계획 밖 |
| **모든 `Human-assisted-HW` 게이트의 4번 칸(안전 초기상태)이 이 사실을 참조** | 사람이 팔 옆에 있는 동안 인에이블 이탈은 **언제든** 가능하다 |
| **`SAFE_HOLD`는 토크 0이 아니다** | `01` §4.1 원문: *"정지는 중력보상 홀드를 거쳐 감속 하강해야 한다"* |
| **`M-3`(낙하 속도 게이트)은 삭제됐으나 낙하 자체는 유지** | `16` §4 M-3. 속도를 안 재는 것이지 낙하가 없다는 뜻이 아니다 |

> **이것은 무엇을 나쁘게 만드는가.** §6은 **리그를 무겁고 굼뜨게 만든다** — 지지 구조와 낙하영역 격리는 작업 공간을 좁히고, 모든 물리 게이트에 세팅 오버헤드를 얹으며, `Human-assisted-HW` 클래스를 실제보다 크게 만든다. 대안은 없다. 브레이크를 소프트웨어로 합성할 방법은 존재하지 않는다.

## 7. 측정-선행 규율

### 7.1 🔴 측정 전 목표 금지 (I-6)

**게이트를 통과하기 전에 하류 주파수·지연 수치를 못 박지 않는다.**

> 🔴 **단서 — 측정-선행은 *성능 목표*에 적용되지 *안전 한계*에는 적용되지 않는다.**
>
> **측정은 실현가능성을 정하지 안전을 정의하지 않는다.** 리그가 무엇을 할 수 있는가는 관측이 답하지만, 리그가 무엇을 해도 되는가는 관측이 답하지 않는다. 안전 한계를 실측에서 유도하면 **리그의 능력이 곧 허용치가 되고**, 그것은 "빠르게 움직일 수 있으니 빠르게 움직여도 된다"와 같은 문장이다. 이 추론에는 사람도 낙하영역도 등장하지 않는다.
>
> **따라서 방향이 반대다:**
>
> | | 성능 목표 (I-6 적용) | 안전 한계 (I-6 **비적용**) |
> |---|---|---|
> | 어디서 오는가 | **측정이 낳는다** — 실측 전에는 값이 없다 | **1차 원리에서 도출한다** — 측정 전에 이미 값이 있다 |
> | 도출원 | 게이트 실측(`PG-RT-001` 등) | **정지거리 · 에너지 · 도달 워크스페이스 · 수동 구속** |
> | 측정의 역할 | 값을 **만든다** | 도출된 기준에 리그를 **대고 잰다**(충족/미달 판정) |
> | 미달 시 | 목표를 실측에 맞춘다 | 🔴 **기준을 낮추지 않는다** — 리그를 고치거나 그 동작을 하지 않는다 |
>
> **구체적으로 이 규율의 적용을 받지 않는 것:** 허용 정지시간, 운영 속도 상한, 지지 구조 커버리지, 낙하영역 경계. 이것들은 `PG-STOP-001`·`PG-VEL-001`·`PG-SAFE-001`의 **입력이지 산출이 아니다.** 게이트는 "얼마인가"가 아니라 **"도출된 값을 만족하는가"**를 묻는다.
>
> **왜 이 단서가 필요한가.** I-6을 무단서로 읽으면 `PG-VEL-001`이 "속도 한계를 **찾는** 게이트"가 되고, 그 순간 한계는 모터가 낼 수 있는 최대 속도로 수렴한다 — 물리적 상한을 안전 상한으로 오해한 것이다. 같은 오독이 `PG-STOP-001`에서는 "측정된 정지 지연이 곧 허용 정지 지연"이 된다. **`NFR-SAF-009`가 성립하는 근거는 리그의 실측이 아니라 낙하가 사람에게 닿지 않는다는 기하학**이며, 기하학은 벤치마크를 기다리지 않는다.
>
> **경계가 헷갈릴 때의 판정법**: *"이 수치가 두 배 느슨해지면 무엇이 나빠지는가?"* — 답이 **"처리량·품질"**이면 성능 목표(I-6 적용). 답이 **"사람이 다치거나 팔이 사람 위로 떨어진다"**면 안전 한계(I-6 비적용, 도출 우선).

이것은 명세의 규율을 계승한 것이다:

> `15` `NFR-PRF-053`: *"시스템은 **벤치마크 결과가 확정되기 전에는 어떤 주파수·지연 목표도 릴리스 합격 기준으로 사용해서는 안 된다.** 미확정 NFR은 진단·리포트에서 '측정 대기'로 명시적으로 표시되어야 한다."*
>
> `14` `NFR-OPS-002`: *"제어 루프 주파수는 1000 Hz를 초과해서는 안 된다. 🔴 **기본 목표값은 LeRobot Python 루프의 실측 사이클 타임이 나오기 전까지 못 박지 않는다.**"*

**왜.** 근거 없는 수치는 두 방향으로 틀린다 — 너무 빡빡하면 **멀쩡한 구성이 실패로 판정**되어 재작업 트리가 통째로 켜지고, 너무 느슨하면 **게이트가 아무것도 거르지 않는다**. 어느 쪽이든 게이트 상태머신(§3.3)의 후손 무효화가 잘못된 신호로 발화한다.

**목표치가 아직 존재하지 않는 대표 항목** (전수는 `03-측정-게이트.md`):

| 항목 | 상태 | 근거 |
|---|---|---|
| 제어 루프 사이클 타임 | **[미측정]** — 모든 주파수 NFR의 뿌리 | `15` `NFR-PRF-054`, `15` §2.10 |
| 주경로 제어 주파수(record 30–60 Hz / 추론 / 평가 250 Hz) | **목표값 자체가 없다** | `15` `NFR-PRF-053` |
| 데드맨 릴리즈 → 정지 도달 지연 | `04` `NFR-MAN-002` ≤20 ms P99 = **[미확인] 목표치이지 실측이 아니다**. ⚠️ **허용치는 안전 한계 → 정지거리·낙하영역에서 도출**(위 단서). 측정은 도출값 **충족 여부**를 판정한다 | `04` §5-7 |
| 비상정지 왕복 지연 상한 | **[결정필요]** — 자체 벤치로 확정 | `13` `NFR-GUI-005`, Q-5 |
| ARM/Jetson IK solve time | 모든 실측이 x86 | `05` U-9, `NFR-TEL-004`, `15` `NFR-PRF-048` |
| 관절 속도 한계 정본 | **학습 전 확정 필요.** ⚠️ **안전 한계 — I-6 비적용.** 모터 vMax·감속비에서 **도출**하고, 스윕은 그 도출값을 **검증**한다(발견하지 않는다) | `16` §3.2 M-16 → `PG-VEL-001` |
| bytes/episode · 트랜스코딩 RTF | PNG/tiff 중간산물 예산 재산정 | `06` §5 Q7, `NFR-CAM-005` |

**규율 위반의 형태 (전부 금지):**

- ❌ 게이트 문서에 "목표: 250 Hz"라고 적고 측정을 나중으로 미룬다
- ❌ 명세의 `[미확인]` 목표치를 계획에서 `[확정]`으로 승격한다
- ❌ 상류 실측치(x86 IK 0.345 ms)를 타깃 하드웨어 목표로 이식한다 — `NFR-TEL-004`가 명시 금지
- ✅ **허용**: "측정 대기" 표시 + 측정 프로토콜 + **음성 분기의 명명**(수치 없이도 분기는 미리 정할 수 있다)

> **이것은 무엇을 나쁘게 만드는가.** I-6은 **계획을 덜 확정적으로 보이게 만든다** — "얼마나 빠른가"에 답하지 못한 채 넓은 범위의 WP가 진행되고, 하류 설계(예: 리스 갱신 주기, GMO 임계)는 **분기 구조만 잡아둔 채 값을 비워둬야** 한다. 그 대가로 사는 것은, 뒤집힌 수치 하나가 후손 트리를 태우는 사태를 **측정 시점으로 앞당기는 것**이다.

### 7.2 🔴 M-8 ID 충돌 경고 — 계획은 `M-8`을 쓰지 않는다

**명세의 `M-8`은 두 문서에서 서로 다른 뜻으로 쓰인다. 둘 다 실재한다.**

| 출처 | `M-8`의 뜻 | 상태 |
|---|---|---|
| `16` §8 | **"Quest 추적 정확도 — 무조사 종결"** (*"알려진 대로, 신경 쓰지 않는다"*) | **종결됨** |
| `15` §2.10 (측정 프로토콜 규격 M-1~M-8) · `15` `NFR-PRF-054` · `NFR-PRF-053` · Q-0 · U-0 · `14` `NFR-OPS-010` | **"LeRobot 제어 루프 사이클 타임 — 이 문서의 최우선 측정 항목"** | **[미측정], 모든 주파수 NFR의 뿌리** |

**충돌의 위험이 정확히 무엇인가:** 워크플로우가 `16`을 먼저 읽으면 `M-8`을 **"종결된 항목"**으로 파싱한다. 그러면 `15` `NFR-PRF-054`가 *"M-8 프로토콜로 측정·기록해야 한다"*고 지시하는 **최우선 측정이 조용히 스킵된다** — 그리고 그 결과가 없으면 **어떤 제어 주파수도 합격 기준이 될 수 없으므로**(`NFR-PRF-053`), 하류 전체가 근거 없는 수치 위에 서게 된다. 이것은 예외도 로그도 없는 조용한 실패이며, `00` §4가 열거한 함정과 같은 종류다.

**계획의 규칙 (예외 없음):**

| 규칙 | |
|---|---|
| 🔴 계획 문서 어디에도 **`M-8`을 쓰지 않는다** | 인용문 안에 나타나는 경우, 반드시 어느 뜻인지 병기한다 |
| 제어루프 측정은 **반드시 `PG-RT-001`** | 명세 매핑 = **`NFR-PRF-054`**, `15` §2.10 — **`16` M-8이 아니다** |
| Quest 추적 정확도는 게이트가 없다 | `16` §8 M-8이 무조사 종결. 되살리지 않는다 |
| Wave −1 정규화 원장이 이 충돌을 (c)항으로 기록 | 명세 측 텍스트 수정은 명세 소관 |

### 7.3 `PG-*` 규약 — 왜 게이트 ID를 새로 판을 짜는가

**명세의 `M-n` / `D-n`을 게이트 ID로 재사용하지 않는다.** 이유는 셋:

| 이유 | 설명 |
|---|---|
| **깨진 ID가 실재한다** | `M-8`(§7.2). 재사용하면 충돌을 계획으로 수입한다 |
| **의미론이 다르다** | `M-n` = *"측정하라"*(질문). `PG-*` = *"통과 여부가 하류 착수를 바꾸는 판정점"*(상태머신 + 음성 분기 + 후손 무효화). `M-n`에는 상태머신이 없다 |
| **소유권이 다르다** | `M-n`/`D-n`은 명세(`16` = 유일한 결정 등록부)가 소유한다. `PG-*`는 **계획이 소유**하고, 명세를 참조하되 명세를 수정하지 않는다 |

**따라서 매핑은 단방향이다:**

```
PG-*  ──"명세 매핑"──>  FR/NFR/M-n/D-n   (참조)
PG-*  <──────────────   FR/NFR           (명세는 PG-*를 모른다)
```

| 명세 객체 | 계획에서의 취급 |
|---|---|
| `D-n` (사람이 골라야 함) | **이미 골라진 것**은 전제로 소비(`16` §4~§10). U-1~U-5가 나머지를 종결 |
| `M-n` (측정하라) | **`PG-*`의 입력**. `PG-*`가 그 측정에 상태머신·분기·무효화를 얹는다 |
| `[결정필요]` (지역) | Wave −1 정규화 원장 (d)항 — D-3/D-4와 충돌하는 지역 태그(예: `12` `FR-SAF-045`) |

## 8. ID 네임스페이스 · 명명 규약

**명세의 표기 규약(`00` §6)을 계승한다.** 우선순위 코드(M/S/C)와 비고 태그(`[확정]`/`[미확인]`/`[결정필요]`/`[신규구현]`)는 그대로 쓴다.

### 8.0 🔴 네임스페이스 전수 (계획이 쓰는 접두사는 이게 전부다)

**여기 없는 접두사를 발명하지 마라.** 같은 접두사가 두 문서에서 다른 뜻이면 그것은 결함이며, 아래 표가 이긴다(§9.2).

| 접두사 | 형식 | 뜻 | 정본 소유 문서 |
|---|---|---|---|
| **`PG-*`** | `PG-<축>-<3자리>` | **측정 게이트.** 통과 여부가 하류 착수를 바꾼다. 상태머신(§3.3) + 음성 분기 + 후손 무효화를 갖는다 | `03` (§8.1에 축 고정) |
| **`CG-*`** | `CG-<대역>-<번호><영문자>` | 🔴 **WP 수용 검사 — 게이트가 아니다.** 해당 WP 하나의 종료 판정일 뿐. **상태머신을 갖지 않는다 — `PASS`/`FAIL` 이진**(§8.0a). 후손 무효화도 없다. `PG-*`에 대응되지 않으면 그것은 수용 기준이지 게이트가 아니다 | `02a`/`02b`/`02c`/`02d` |
| **`WP-*`** | `WP-<대역>-<2자리>` | 작업 패키지(§3.1) | `02a`/`02b`/`02c` (§8.2) |
| **`WP-G-*`** | `WP-G-<2자리>` · `WP-G-S<2자리>` | **GUI WP 전용 네임스페이스.** 교차 스트림이라 웨이브 대역에 고정되지 않으므로 대역 필드를 쓰지 않는다 | `02d` (배치는 `01`) |
| **`CTR-*`** | `CTR-<NAME>@v<n>` | **인터페이스 계약.** 13종: OWN·UNIT·ACT·PLUG·ERR·GW·CAL·PRIM·CAM·CAP·TEL·WS·REC | `01` §6.2 |
| **`SHAPE-*`** | `SHAPE-CF` / `SHAPE-IM` / `SHAPE-IG` / `SHAPE-MS` / `SHAPE-HG` | **워크플로우 형상** 5종(계약동결 / 구현 / 통합배리어 / 측정 / Human 게이트) | `05` |
| **`R-nn`** | `R-<2자리>` | **리스크 등록부 항목.** 🔴 `04` **전용**. 다른 문서에서 `R-*`를 규칙 번호로 쓰지 않는다 | `04` |
| **`CR-1..6`** | `CR-<번호>` | **계약 동결·해동 규칙** | `01` |
| **`CI-nn`** | `CI-<번호>` | **CI 검사 규칙** | `06` |

> **왜 `CG-*`와 `PG-*`를 가르는가.** `CG-*`가 게이트로 오독되면 §3.3의 후손 무효화가 **수백 개 지점에서 발화 가능한 것처럼** 보이고, 반대로 `PG-*`가 수용 기준으로 오독되면 부정 판정이 후손을 건드리지 않고 지나간다. 둘 다 §3.1 정의("통과 여부가 하류 WP의 착수 가능성을 바꾸는가")를 깨는 방향이다.
>
> **왜 `R-*`를 `04`에 가뒀는가.** `R-nn`은 세 문서에서 세 뜻(리스크·계약규칙·CI규칙)으로 쓰인 실적이 있다. 접두사 하나가 세 뜻이면 CI가 검증할 대상을 특정할 수 없다.

### 8.0a 🔴 `CG-*`는 상태머신을 갖지 않는다 — 이 절이 정본이다

**5상태 FSM(§3.3)은 `PG-*` 게이트만의 것이다. `CG-*`는 `PASS`/`FAIL` 이진이다.**

| | `PG-*` (측정 게이트) | `CG-*` (WP 수용 검사) |
|---|---|---|
| 상태 | **5상태 전부** — `PASS` / `RETRY_WITH_VARIANT` / `DEGRADED_ACCEPTED` / `FAIL_BLOCKING` / `SUPERSEDED` | 🔴 **`PASS` / `FAIL` — 둘뿐** |
| 후손 무효화 | ✅ §3.3의 4종 자동 동작 | ❌ **없음** — 그 WP가 안 끝났을 뿐 |
| `FAIL`이 뜻하는 것 | 하류 착수 가능성이 바뀐다 | **그 WP를 아직 못 끝냈다.** 고쳐서 다시 돌린다 |

**왜 `CG-*`에 5상태를 주면 안 되는가 — 셋 다 파괴적이다:**

| 상태 | `CG-*`에 부여했을 때 벌어지는 일 |
|---|---|
| `DEGRADED_ACCEPTED` | 🔴 **가장 위험하다.** WP가 **자기 수용선을 자기가 낮추고 통과를 선언**한다. `CG-*`는 그 WP가 정의했으므로 심판과 선수가 같다 — `PG-*`의 축소 수용은 **계획이 명명한 대안 경로**를 뜻하지만, `CG-*`의 축소 수용은 **그냥 미완성**이다 |
| `FAIL_BLOCKING` | 후손 무효화 기계(§3.3)가 **수백 개 지점에서 발화 가능한 것처럼** 보인다. 실제로는 `CG-*`에 후손 인덱스가 없으므로 **발화하지 않는다** — 있다고 믿게 만드는 것이 실제 위험이다 |
| `RETRY_WITH_VARIANT` / `SUPERSEDED` | 재실행은 `CG-*`의 **기본 동작**이다(안 끝났으면 다시 한다). 별도 상태로 이름 붙이면 `PG-*`의 변형 재측정과 혼동되고, 그쪽은 **명명된 변형 + 계약 불변**이라는 조건이 붙는다 |

> 🔴 **다른 계획 문서가 `CG-*`에 5상태를 부여하면 그 문서가 틀렸다** — §9.2에 따라 `00`이 이긴다. `CG-*`의 종료 판정은 **관측 가능한 산출 증거의 유무**이지 협상 가능한 등급이 아니다.
>
> **경계**: `CG-*`가 `FAIL`인데 그 원인이 "다른 접근이 필요하다"면 그것은 `CG-*`의 상태 문제가 아니라 **WP 설계가 틀린 것**이며, 대체 WP는 `PG-*`의 `negative_branch`나 웨이브 재배치(`01`)가 낳는다. 수용 검사에 상태를 늘려서 해결할 문제가 아니다.

### 8.1 게이트 ID

```
PG-<축>-<3자리>
 │    │      └── 축 내 일련번호. 000 금지. 결번 허용(삭제된 게이트의 번호는 재사용 금지)
 │    └── 측정 축 (아래 표에 고정)
 └── Plan Gate — 계획 소유임을 표시. 명세 M-n/D-n과 네임스페이스 분리
```

| 축 | 뜻 | 게이트 |
|---|---|---|
| `RT` | 실시간 루프 | `PG-RT-001` |
| `CAN` | CAN 프레임/버스 | `PG-CAN-001` |
| `RID` | 모터 레지스터 | `PG-RID-001` |
| `J7` | J7 모터 타입 정합 | `PG-J7-001` |
| `SAFE` | 물리 안전 설비 | `PG-SAFE-001` |
| `VEL` | 속도 한계 정본 | `PG-VEL-001` |
| `STOP` | 정지 경로 지연 | `PG-STOP-001` |
| `FRIC` | 마찰 적합 품질 | `PG-FRIC-001` |
| `CAM` | 카메라 능력 | `PG-CAM-001` |
| `STO` | 스토리지/트랜스코딩 | `PG-STO-001` |
| `IK` | IK 성능 (**타깃별**) | `PG-IK-001` |

**규칙:**
- 🔴 **`PG-*` ID는 불변이다.** 게이트 내용이 바뀌면 새 ID를 발급하고 구 ID는 `SUPERSEDED`로 남긴다. **재번호 금지** — 후손 stale 전파(§3.3)가 ID를 키로 쓴다.
- **타깃별 게이트는 ID를 나누지 않고 판정을 나눈다.** `PG-IK-001`은 하나이며, 판정은 {Jetson Nano, Jetson Orin, RTX 5090, RTX A6000}별로 독립적이다. `DEGRADED_ACCEPTED`(그 타깃 미지원 표시)도 타깃별이다.
- **모든 `PG-*`는 "명세 매핑" 칸이 필수**다. 매핑 없는 게이트는 계획이 발명한 요구사항이며 §1.2 위반이다.

### 8.2 WP ID

```
WP-<대역>-<2자리>
 │    │      └── 대역 내 일련번호. 2자리 고정(01, 02, … ). 영역 코드 없음
 │    └── 웨이브 대역: BOOT(=−2) / N1(=−1) / ENV / 0A / 0B / 0C / OPS / 1 / 2A~2D / 3A~3D / 4A~4C / 5
 └── Work Package
```

**예시**: `WP-0B-04` = Wave 0-B의 4번 WP. `WP-3B-10` = Wave 3-B의 10번 WP. `WP-BOOT-01` = Wave −2 부트스트랩의 1번 WP(§3.5).

**GUI WP만 대역 대신 `G`를 쓴다** — `WP-G-00`~`WP-G-04`(공유 기반) + `WP-G-S01`~`WP-G-S13`(화면별). GUI는 종단이 아니라 **교차 스트림**이라 하나의 웨이브 대역에 속하지 않기 때문이다(카탈로그 = `02d`, 배치 = `01`).

**🔴 영역 코드는 존재하지 않는다.** WP ID는 **어느 웨이브 대역인가**만 인코딩하며, 어느 명세 영역을 건드리는가는 ID가 아니라 **`06-추적성-레지스트리.md`의 매핑**이 답한다. 이유는 WP가 영역 경계를 정상적으로 가로지르기 때문이다 — 예컨대 스케줄러+통합 게이트웨이 WP는 `SAF`·`CON`·`MOT`를 동시에 건드리며, 그것은 계약 경계가 잘못 그어진 것이 아니라 **불변식 I-1이 요구하는 단일 집행점**(§5.1)이다. 영역 코드를 ID에 박으면 이런 WP는 인코딩 자체가 불가능하다.

**규칙:**
- **WP ID의 대역 필드는 재배치 시 바뀔 수 있다.** 게이트 ID와 달리 불변이 아니다 — 다만 `06-추적성-레지스트리.md`가 이전 ID를 별칭으로 유지한다.
- **재시도는 새 WP ID를 만들지 않는다.** 게이트 상태머신의 `RETRY_WITH_VARIANT`(§3.3)가 재시도를 표현한다. `WP-1-02R` 같은 파생 접미사는 금지다 — 두 개의 ID가 하나의 소유권을 주장하게 된다.
- 🔴 **단일 소유자 WP는 명시 표시**(§3.4). 예: `OpenArmFollower` 서브클래스를 건드리는 WP는 **동시에 하나만** 활성화될 수 있다. **순차 인계는 허용된다** — 금지되는 것은 병렬 빌더이지 소유권 이전이 아니며, 레지스트리는 이를 "동시점 소유자 1명"으로 인코딩한다.

| **게이트가 부정 판정을 냈다** | `03`(상태머신·분기) → `06`(후손 무효화) → `01`(재배치) → `02a`~`02d`(대체 WP) |
| **CI가 매핑 누락을 보고했다** | `06` → `02a`~`02d` |

> **이 순서는 문서 의존성이지 작업 일정이 아니다.** 이 묶음은 일정을 다루지 않는다(§1.3).
