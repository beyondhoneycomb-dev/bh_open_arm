## 1. 등급 규약과 게이트 상태머신

### 1.1 심각도 등급 — "무는 방식"으로 정의한다

| 등급 | 정의 | 판별 질문 |
|---|---|---|
| **BLOCKER** | 실현되면 **토크-ON 후손 전체 또는 안전 계층 전체가 무효**가 된다. 조용히 무는 것을 포함한다 | "이게 틀린 채로 진행하면 나중에 뭘 재해도 그 값이 오염되는가?" |
| **HIGH** | 실현되면 **한 웨이브 대역이 재작업**되거나 특정 배포 타깃이 미지원으로 떨어진다 | "우회 경로가 실재하지만 대가가 큰가?" |
| **MED** | 실현되면 **국소 재작업 + 명명된 대체 경로**로 흡수된다 | "분기가 이미 문서에 있고 후손이 좁은가?" |
| **LOW** | 표시·표기·문서 정합 수준. 코드 경로를 바꾸지 않는다 | "라벨을 고치면 끝나는가?" |

**등급은 확률이 아니라 파급으로 매긴다.** 발생확률 컬럼을 두지 않는다 — 측정 전 확률 추정은 근거 없는 숫자이며, SPINE §2-6(측정 전 목표 금지)의 같은 규율이 여기에도 적용된다.

### 1.2 게이트 상태머신 (SPINE §3, 전 게이트 공통)

| 상태 | 뜻 | 후손 처리 |
|---|---|---|
| `PASS` | 수용 기준 전부 충족 | 후손 해제 |
| `RETRY_WITH_VARIANT` | 명명된 변형으로 재실행. 변형은 **게이트 정의가 미리 열거한 것만** | 후손 보류. 변형 미열거 시 `FAIL_BLOCKING`으로 강등 |
| `DEGRADED_ACCEPTED` | 축소 범위로 수용. **무엇을 포기했는지 명시 기록 필수** | 포기된 능력에 의존하는 후손만 stale |
| `FAIL_BLOCKING` | 후손 전부 차단 | **후손 아티팩트 자동 stale + 미통합 작업 취소 + 명명된 대체 WP 생성** |
| `SUPERSEDED` | 상위 사실이 바뀌어 게이트 자체가 무의미 | 게이트와 후손 동시 재평가 |

> **상태 변경 시 리스크 등록부에 한 줄 적고 넘어가는 것은 금지다.** 상태 전이는 반드시 {후손 stale 표시, 취소된 미통합 작업 목록, 생성된 대체 WP 이름} 3종을 산출해야 한다. 이 3종이 없는 전이는 전이가 아니다.

## 2. U-1로 **삭제**된 리스크 — 판단이 아니라 물리다

**U-1: 바이래터럴 힘반사 = 계획에서 종결(OUT).** 근거는 설계 판단이 아니라 **리그의 물리적 사실**이다: **동력 리더암이 존재하지 않는다.**

| 사실 | 근거 |
|---|---|
| KER에는 모터가 없다 | `16` §10.1 — `openarm_ker/ker_stream.py` `transport="usb"`, `vid=0x303A`(ESP32-S3), `pid=0x4002`. **CAN 채널 0 소비.** 명세 원문: *"모터가 없으므로 KER로는 바이래터럴 원천 불가"* |
| Quest 3S는 진동만 준다 | `16` §5 M-22 — VR 입력은 UDP:5006 평문 JSON 포즈 스트림. 힘 출력 액추에이터 없음 |
| 500 Hz는 바이래터럴 전용이다 | `16` §5 M-21 / §9-11 — `openarm_bilateral_control.cpp:42,72,103`. *"유니래터럴 / LeRobot 위치 스트리밍 / dora eval(250 Hz) / LeRobot record(30~60 Hz)엔 불필요"* |

**힘을 되돌려줄 하드웨어가 없으면 힘반사 루프는 닫히지 않는다.** 이는 우선순위 결정이 아니라 폐루프의 부재이므로, 아래 3건은 **연기가 아니라 삭제**다.

| ID | 삭제된 리스크 | 원래 무는 방식 | 삭제 근거 (물리) | 되살아나는 유일한 조건 |
|---|---|---|---|---|
| **R-X01** | **500 Hz 미달 분기** — Python 루프가 500 Hz를 못 내면 릴리스 게이트 실패 | `NFR-PRF-054`가 힘반사 경로에 한해 판정선(GIL 부하 p99 ≤ 2.0 ms, 오버런 ≤ 0.1%)을 건다. 주경로는 **이 판정선의 대상이 아니다** | 힘반사 경로가 계획에 없으므로 **판정 대상 자체가 없다.** 주경로(record 30–60 Hz · 추론 · 평가 250 Hz)는 `NFR-PRF-054` 근거란이 명시하듯 예산 여유가 크다 | **동력 리더암이 리그에 실재**할 때만. 소프트웨어 어댑터로는 되살아나지 않는다 |
| **R-X02** | **바이래터럴 분리 RT 프로세스** — 500 Hz 미달 시 힘반사만 별도 실시간 프로세스로 분리 | `NFR-PRF-054` 본문의 미달 분기가 **아키텍처를 다시 열도록** 규정한다 | 분리 대상 루프가 존재하지 않는다. `PG-RT-001` 미달 분기는 **워커 분리**(SPINE §3)이지 RT 프로세스 신설이 아니다 | R-X01과 동일 |
| **R-X03** | **CAN 배타 소유권 이양 부활** — 분리된 RT 프로세스가 CAN에 쓰면서 이중 bind 유발 | `12` §2.9 — *"두 번째 소켓 = SocketCAN 이중 bind = 조용한 실패"*. `NFR-PRF-054`도 미달 시 *"CAN 배타 소유권 이양 프로토콜을 되살린다"*고 규정 | 두 번째 CAN 라이터가 생기지 않는다. `16` §5 M-21: *"단일 GUI 프로세스가 CAN을 소유하므로 CAN 배타 소유권 문제도 부활하지 않는다"* | R-X01과 동일 |

> ⚠️ **삭제의 경계 — 무엇이 삭제되지 **않았는가**.**
> - **`flock`은 살아 있다.** `16` §10.1 — SocketCAN RAW에 배타 플래그가 없다는 사실은 그대로이고, `flock`의 역할이 *"모드 중재"*에서 **"외부 침입자 방어"**(`cansend` / `openarm-can-cli` / 두 번째 LeRobot 인스턴스)로 **축소**됐을 뿐 사라지지 않았다. → R-08.
> - **`PG-RT-001`은 살아 있다.** 판정선이 사라진 것이지 측정이 사라진 게 아니다. `NFR-PRF-054`는 결과가 나오기 전 **주경로 주파수를 릴리스 합격 기준으로 못 박는 것 자체를 금지**한다(`NFR-PRF-053`과 동일 규율). → R-03.
> - **`NFR-SAF-001`(1 kHz 감지 루프)은 살아 있다.** 이것은 힘반사가 아니라 **충돌 감지 루프**이며 `12` §5-Q17의 최대 미해결로 그대로 남는다. → R-09.
> - **소프트웨어 실험 어댑터는 선택으로 남길 수 있다.** 단 **하드웨어 수용 주장은 불가**하다(U-1). 어댑터를 만들더라도 그것은 시뮬/합성 검증물이며 실기 힘반사 능력의 증거로 인용할 수 없다.

**대가.** 이 삭제는 **힘반사 텔레옵 능력을 계획 밖으로 내보낸다.** 나중에 동력 리더암이 실재하게 되면 R-X01~X03이 `SUPERSEDED` 경로로 한꺼번에 되살아나고, 그 시점의 `ActuationScheduler`·게이트웨이 계약은 500 Hz 급 생산자를 전제하지 않은 채 동결돼 있으므로 **계약 버전 상향 + 스케줄러 재검증**이 필요하다. 이 대가를 지금 지불하는 이유는, 존재하지 않는 하드웨어를 위해 아키텍처 전체를 인질로 잡는 대가가 더 크기 때문이다.

## 3. 리스크 등록부

> **「소유 앵커」 열의 뜻 = 완화(구현)를 소유하는 웨이브다. 게이트를 *측정*하는 웨이브가 아니다.**
> 이 둘은 자주 다르다. 게이트는 측정 가능해지는 시점에 찍히고, 그 값을 쓰는 구현은 그 뒤에 온다.
> 측정 웨이브가 완화 웨이브와 다를 때는 해당 칸에 **「게이트 측정 = Wave n (`WP-…`)」**를 함께 적는다(SPINE v2 B-4).

> 🔴 **안전 한계를 관측 능력에서 도출하지 마라 — 방향이 반대다.**
> **측정은 실현가능성을 정하지, 안전을 정의하지 않는다.** 허용 정지시간·운영 속도·감지 지연 예산은 **정지거리 · 낙하/충돌 에너지 · 도달 워크스페이스 · 수동 구속의 물리적 한계**에서 **먼저 도출**한다. 리그는 그 뒤에 **그 기준에 대고 재는** 대상이다.
> - ✅ "정지거리 X를 넘으면 안 되므로 정지시간 예산은 T다. 리그가 T를 내는가?" — 도출이 먼저, 측정이 뒤.
> - ❌ "리그가 T'를 낸다. 그러니 정지시간 기준은 T'다." — 관측 능력이 안전을 정의해 버린 것. **이 형태가 등록부에 나타나면 결함이다.**
> - 리그가 도출값에 미달하면 답은 **기준을 리그에 맞춰 늘리는 것이 아니라, 운영 속도·도달 집합을 줄여 도출값을 리그가 낼 수 있는 범위로 되돌리는 것**이다.
>
> **SPINE §2-6(측정 전 목표 금지)은 *성능 목표*에 적용되지 *안전 한계*에는 적용되지 않는다.** 성능 수치는 재기 전에 못 박으면 안 되지만, 안전 한계는 **재기 전에 도출돼 있어야** 한다 — 그것이 측정의 합격선이기 때문이다. 이 단서 없이 §2-6을 안전 한계에 적용하면 "아직 못 재서 한계가 없다"는 상태로 토크가 켜진다.
> 적용 대상: **R-09 · R-15 · R-25 · R-26 · R-36**.

### 3.1 BLOCKER (6건)

#### R-01 · 홀딩 브레이크 부재 — 영구 물리 위험

| 항목 | 내용 |
|---|---|
| **왜 무는가** | **결함이 아니라 하드웨어의 성질이다.** DM 모터는 QDD 백드라이버블이고 기계식 브레이크가 없다(`12` §2.10). 인에이블 이탈 = 토크 0 = 중력 낙하. 공식 안전 가이드 원문: *"if power is lost due to an emergency stop, the load being held will fall rapidly."* 프로세스 사망(SIGKILL/OOM/deadlock)·CAN bus-off·전원 상실에서 **소프트 홀드는 원천 불가능**하다 — 홀드 명령을 보낼 주체가 없거나 전송 경로가 없다 |
| **결정적 성질** | **`PG-RT-001` 결과와 무관하게 유효하다.** `12` §5-Q17 원문: *"이 루프가 1 kHz를 내든 못 내든 … 소프트웨어 워치독은 낙하를 늦출 뿐 막지 못한다."* 즉 **어떤 소프트웨어 개선으로도 등급이 내려가지 않는 유일한 리스크**다 |
| **근거** | `NFR-SAF-009`(정본) · `12` §2.3 모터 워치독 · `12` §2.10 정지 matrix 마지막 2행 · `13` `FR-GUI-093` · `14` `FR-OPS-024` |
| **완화 (성립하는 유일한 3종)** | ① 기계적 지지 ② 낙하영역 격리 ③ 독립 전원차단 회로 — `NFR-SAF-009`가 열거한 그대로. **소프트웨어 항목이 이 목록에 없다는 것이 요점이다** |
| **집행 지점** | **`PG-SAFE-001`**(§4). 3종의 실재를 첫 토크-ON 이전에 물리적으로 검증한다 |
| **소유 앵커** | `PG-SAFE-001` 게이트 WP + Wave 0-Ops(크래시 워치독·외부 수집기, `FR-OPS-024`) |
| **금지 완화 (안티패턴)** | ❌ "프로세스 사망 시 STOP_HOLD로 규정" — `FR-OPS-024`가 **명시적으로 금지**한다. ❌ "마지막 셋포인트가 MIT 게인으로 유지되어 프로세스가 죽어도 안 멈춘다" — `FR-GUI-093`이 **틀린 전제로 정정**했다. ❌ `disable_all()`을 "soft stop"이라 부르기(`12` §2.10 용어 정정 강제) |
| **대가** | 이 리스크는 **닫히지 않는다.** 등록부에서 영구히 열린 채 남으며, 이것은 **완화가 실패했다는 뜻이 아니라 완화의 성질이 물리적 격리라서 소프트웨어 상태로 표현되지 않는다는 뜻**이다. 대가: 리그를 옮기거나 페이로드를 바꿀 때마다 `PG-SAFE-001`을 **재실행**해야 하며, 이는 물리 구성 변경을 무겁게 만든다 — 구속물 하나를 옮기는 것이 S-2부터의 전 단계 재실행을 부른다 |

#### R-02 · `connect()`의 조용한 영점 파괴

#### R-34 · 마찰 로깅이 두 번째 CAN 라이터를 만든다 — 식별 하네스가 낙하를 부른다

| 항목 | 내용 |
|---|---|
| **왜 무는가** | R-05의 완화(경로 A)는 **1 kHz `pos/vel/tau` 로깅**을 요구하는데, 그 대역은 **패턴 A(16프레임/사이클)**에서만 나오고 패턴 A는 **`robot.bus` 직접 접근**을 요구한다(R-03 동반 사실). 직접 접근이 **송신을 겸하는 순간 `ActuationScheduler` 외의 두 번째 CAN 라이터**가 생긴다 — 그리고 SocketCAN RAW에는 배타 bind가 없으므로 **둘 다 성공하고 에러가 나지 않는다**(`12` §2.9, R-08) |

### 4.2 🔴 진입 조건 — **전원상실 복구 시퀀스가 먼저 존재해야 한다**

**이것은 권고가 아니라 게이트의 진입 조건이다.** 복구 시퀀스가 실재하고 검증되기 전에는 `PG-SAFE-001`의 어떤 단계도 시작하지 않는다.

#### 4.2.1 왜 이것이 먼저인가 — 오염 연쇄

E-Stop은 **모터 전원 자체를 차단하며 CAN도 함께 끊긴다**(`16` §4 M-2: *"정지 후 상태를 읽어 복구는 불가능하다는 전제로 설계"*). 브레이크가 없으므로 **팔은 rest가 아닌 자세로 눕는다**(`NFR-SAF-009`, `12` §2.10). 이 상태에서 복구 경로가 정의돼 있지 않으면 즉석 재연결이 일어난다:

| 단계 | 일어나는 일 | 근거 |
|---|---|---|
| 1 | E-Stop 차단 → 전원·CAN 동시 소실 → **낙하** | `16` M-2 · `NFR-SAF-009` |
| 2 | 팔이 **rest 아닌 임의 자세**로 정지 | 브레이크 부재 |
| 3 | 복구 경로 부재 → **즉석 재연결**(`connect()`) | — |
| 4 | `connect()`가 `set_zero_position()` 호출 → **그 임의 자세가 영점이 된다** | `openarm_follower.py:152-153` · `12` §2.14 |
| 5 | **아무 에러도 나지 않는다** | `12` §2.14 |
| 6 | `joint_limits` · 가상벽 · TCP 속도 상한 · 자기충돌 검사 · `τ_grav(q)` **동시 무효** | `12` §2.14 · `FR-SAF-071` |
| 7 | 틀린 `q` → **틀린 중력항** → 이후 마찰동정이 **조용히 오염**된다 | `12` §2.6(*"J2 영점 +π/2 미반영 시 중력항이 처음부터 틀려 마찰 식별이 오염된다"*) · `12` §5-Q1 |
| 8 | 오염된 `friction.yaml` → GMO 잔차 상시 오프셋 → **상시 오탐/미탐** | `12` §2.14 · `FR-SAF-030` |

> **연쇄의 성질**: 1단계는 시끄럽고(팔이 떨어진다), **4~8단계는 전부 조용하다.** E-Stop 시험이 시끄러운 사고가 아니라 **조용한 데이터 오염**으로 끝나는 것이 이 연쇄의 위험이다. 그리고 오염은 마찰동정 시점이 아니라 **그 결과를 쓰는 시점**에 드러난다 — 즉 몇 웨이브 뒤에.

| **RC-5** | **논리 세션 재개 금지 · 전송 전용 재열기만 허용** | **E-Stop 이후에도 논리 세션을 다시 열지 않는다** — `connect()`(영점·인에이블을 수반하는 그 연산)는 **세션당 1회**이며 E-Stop이 그 카운터를 리셋하지 않는다. 홀드는 연결을 유지한 채 명령을 바꾸는 것이다. 🔴 **그러나 E-Stop은 전원과 CAN을 함께 끊으므로 소켓이 살아남는다는 보장이 없다**(§4.2.2 아래 주석) → **소켓이 죽었을 때 쓰는 연산은 `connect()`가 아니라 `reopen_transport()`다**(아래 정의) | `12` §2.9 · `12` §2.14 · `FR-SAF-071` · `14` `FR-OPS-083` · `16` §4 M-2 | 세션 전체 **`connect()` 호출 횟수 = 1** (감사 로그). `reopen_transport()` 호출은 **별도 카운터**로 기록되며 이 불변식에 계상되지 않는다. 각 `reopen_transport()` 구간의 candump에 **0xFC(enable)·0xFE(set-zero) 프레임 부재** 증명 |

> 🔴 **RC-5 개정 — I-2("connect 세션당 1회")는 *논리 세션* 불변식이지 *소켓 수명* 불변식이 아니다.**
> **왜 개정이 필요했나**: RC-5의 원문("`disconnect()`하지 않는다")은 **E-Stop 이후에도 소켓이 살아 있다**고 암묵 가정했다. 그 가정은 틀렸다 — E-Stop은 **모터 전원과 CAN을 함께 끊고**(`16` §4 M-2), 그 과정에서 **USB 어댑터 리셋 · CAN bus-off 진입 · 인터페이스 재생성(`ip link` down/up)** 중 어느 것이든 **소켓/인터페이스를 무효화**할 수 있다. 무효화되면 RC-1(연결 유지한 채 위치 읽기)이 **물리적으로 불가능**해지고, 남는 선택지는 두 개뿐이었다: **복구를 포기**하거나 **`connect()`를 다시 불러 I-2를 위반**하고 §4.2.1 오염 연쇄(4단계 auto-set-zero)로 들어가거나. **둘 다 틀렸다. 세 번째 연산이 없었던 것이 결함이다.**
>
> **`reopen_transport()` — 전송 전용 재열기**
>
> | 항목 | 내용 |
> |---|---|
> | **하는 일** | **전송 계층만** 재구성한다: SocketCAN 소켓 재생성 · 인터페이스 상태 복구(`ip link` 재설정 + `FR-OPS-084` 검증 재실행) · bus-off 복구 · `flock` 재획득(R-08) · 링크 파라미터(`fd on`/`bitrate`/`dbitrate`) 재검증 |
> | **절대 하지 않는 일 (이것이 이 연산의 정의다)** | ❌ **영점을 찍지 않는다** — `set_zero_position()`/0xFE **미호출**. ❌ **모터를 인에이블하지 않는다** — `enable_torque()`/0xFC **미호출**. ❌ **캘리브레이션 상태를 건드리지 않는다** — 디스크 JSON SoT(`FR-CON-064`)를 읽지도 쓰지도 않는다. ❌ **논리 세션을 새로 만들지 않는다** — 세션 ID·모드·캘리브 컨텍스트는 **전부 그대로 유지**된다 |
> | **성립 조건** | **torque-OFF 상태에서만** 호출 가능하다. 그리고 재열기 후 진입점은 **RC-1(읽기)**이지 RC-4(재개)가 아니다 — 재열기는 **관측 능력을 되찾는 것**이지 제어를 되찾는 것이 아니다 |
> | **왜 이것이 R-02를 우회하지 않는가** | R-02가 무는 지점은 **`connect()` 안의 무조건 auto-set-zero**(`openarm_follower.py:152-153`)다. `reopen_transport()`는 **그 코드 경로를 아예 지나지 않는다** — 소켓만 다시 여는 연산이므로 영점을 확정할 기회 자체가 없다. 즉 이것은 R-02 방어의 **예외가 아니라, 방어를 유지한 채 소켓을 되찾는 유일한 문**이다 |
> | **계약면** | **`CTR-CAL@v1` + 연결 계약**에 `reopen_transport()`를 **명시 연산으로 등재**한다. 등재되지 않으면 구현자가 "`connect()`를 한 번 더 부르되 조심해서"라는 형태로 **같은 구멍을 다시 판다** |
> | **감사** | `connect()` 카운터와 `reopen_transport()` 카운터는 **분리 기록**한다. 전자가 2 이상이면 **I-2 위반**이다. 후자는 몇 번이든 정상이며, 다만 **각 호출 구간에 0xFC·0xFE가 부재**함이 candump로 증명돼야 한다 |
>
> **대가.** 연산을 하나 더 만들면 **"둘 중 무엇을 부를 것인가"라는 판단 지점**이 생기고, 잘못 고르면 조용히 R-02다. 그래서 `connect()`는 **세션 진입점에서만 도달 가능**하도록 좁히고(그 밖의 호출 사이트는 CI 정적검사로 금지), 복구 경로에서 도달 가능한 것은 `reopen_transport()`뿐이도록 만든다. **판단 지점을 만들되, 판단을 코드 구조로 대신한다.**

> ⚠️ **RC-1의 성립 조건이 §4.3 S-1에 종속된다.** `12` §2.10 마지막 문단: *"전원 상실 후 복구 절차(enable 전에 위치를 읽는다)는 **로직/CAN 전원이 살아 있어야 성립**하므로, 모터 전원과 로직 전원의 분리가 전제된다. 분리가 안 되면 우리가 직접 분리 배선을 설계해야 한다."*
> → **S-1이 전원 도메인의 실제 거동을 확정하기 전까지 RC-1이 "차단 중 읽기"인지 "재인가 후 읽기"인지 결정되지 않는다.** M-2는 후자를 설계 전제로 못 박았고, S-1은 **그 전제가 이 리그에서 참인지를 확인**한다. 둘이 어긋나면 `FAIL_BLOCKING`이다.

**대가.** 복구 시퀀스를 진입 조건으로 박으면 **E-Stop 시험이 늦춰진다** — 회로를 만지기 전에 소프트웨어 경로(`connect_readonly` + 영점 검사 + 3D 판정)가 먼저 실재해야 하므로, Wave 1-② 산출물이 `PG-SAFE-001`의 선행이 된다. 즉 **이 게이트는 순수 하드웨어 게이트가 아니라 소프트웨어에 종속된 게이트**다. 대안(시험 먼저, 복구 나중)의 대가는 §4.2.1 연쇄이며, 그것은 되돌릴 수 없다.

### 4.3 단계별 절차

**순서는 불변이다.** 각 단계는 6요소를 갖는다: **에이전트가 하는 일 / 사람이 하는 일 / 중단조건 / 안전 초기상태 / 산출증거 / 재개지점.**

> **순서의 근거**: S-5(E-Stop 리셋 거동)는 **차단 회로를 실제로 작동**시킨다. 그 시점에 팔이 지지 없이 서 있으면 안 되므로 **S-2(지지) · S-3(낙하영역) · S-4(무부하)가 반드시 S-5보다 앞선다.** 그리고 S-1(전원 도메인)은 S-5의 관측 대상이 무엇인지를 정의하므로 맨 앞이다.

#### S-1 · 전원 도메인 거동 — **Cat-0가 모터 전원과 CAN을 함께 끊는가**

| 요소 | 내용 |
|---|---|
| **왜 이 단계인가** | `16` §4 **M-2**는 *"E-Stop = 모터 전원 자체 차단 → CAN도 끊긴다"*를 **설계 전제**로 확정했다. 그러나 `12` §2.10 마지막 문단은 **"E-Stop 하드웨어 회로는 [미확인]"**이며 안전 가이드가 *"회로도·인터록 사양·차단 범위(모터 전원만인지 전체인지)를 제시하지 않는다"*고 명시한다. **전제와 리그가 일치하는지는 확인된 적이 없다.** |
| **에이전트가 하는 일** | ① `connect_readonly()`로 버스 오픈(torque-OFF) ② `ip -details link show` 상태 스냅샷 + `candump` 연속 캡처(하드웨어 타임스탬프) 시작 ③ 차단 전/중/후 3구간의 **CAN 링크 상태 전이**와 **프레임 수신 단절 시각**을 기록 ④ 로직 전원 도메인(호스트·어댑터)의 생존 여부를 `ip link` 반응성으로 판정 ⑤ 두 토폴로지 중 어느 것인지 판정: **T-A**(모터 전원만 차단, 로직/CAN 생존) vs **T-B**(모터 전원 + CAN 동시 차단 = M-2 전제) |
| **사람이 하는 일** | 팔을 **무전원 상태에서 손으로 rest 부근에 두고 지지**한 뒤, 워크플로우 지시에 따라 **물리 E-Stop 버튼을 누른다.** 이후 지시에 따라 리셋한다. **팔에서 손을 떼지 않는다** |
| **중단조건** | ① 차단 중 **CAN 프레임이 계속 관측**된다(= 모터 전원이 안 끊겼거나 어댑터가 별도 도메인) → 차단 범위 미상 → 중단 ② 차단 후 **로직 전원까지 죽어 호스트가 재부팅**된다 → 복구 시퀀스 RC-1이 성립 불가 → 중단 ③ 버튼 조작에 **CAN·전원 어느 것도 반응하지 않는다** → 회로 미연결 의심 → 즉시 중단 |
| **안전 초기상태** | **torque-OFF**(`connect_readonly`, `enable_torque` 미호출). 팔은 **사람이 지지**. 페이로드 **없음**. 도달 포락 내 인체·물체 **없음** |
| **산출증거** | ① 판정된 토폴로지(T-A/T-B) + 근거 로그 ② `ip link` 상태 전이 타임라인 ③ candump 프레임 단절 시각(하드웨어 타임스탬프) ④ **M-2 전제와의 일치/불일치 판정문** ⑤ 리그 전원 도메인 결선도(관측 기반) |
| **재개지점** | T-B 확정(M-2 전제 일치) → **S-2**. T-A 확정 → **`RETRY_WITH_VARIANT`**(변형: 복구 시퀀스 RC-1을 "차단 중 읽기"로 재정의 + `NFR-SAF-007` 송신 주기 설계 재검토) → 재판정 후 S-2. **판정 불가 → `FAIL_BLOCKING`** |
| **불일치의 뜻** | T-A도 T-B도 아닌 제3의 거동(예: CAN만 끊기고 모터 전원 생존, 또는 부분 차단)은 **`FAIL_BLOCKING`**이다. `12` §2.10: *"분리가 안 되면 우리가 직접 분리 배선을 설계해야 한다"* — 이것이 **통합 교정 경로**이며 조달이 아니다 |

#### S-2 · 기계적 구속 — **도달 포락을 경계하는 하중등급 물리 구속**

> 🔴 **이 단계는 시뮬레이션으로 통과할 수 없다. 어떤 시뮬레이션 결과도 이 단계의 증거가 아니다.**
>
> **왜 시뮬 증명을 삭제했나 — 두 가지 독립적인 이유. 각각 단독으로 치명적이다.**
>
> **① 물리 모델이 아직 없다.** "각 자세에서 중력 낙하 종착 자세를 시뮬레이션한다"는 **마찰 · 접촉 거동 · 페이로드 동역학**을 요구한다. 그런데 이 게이트 시점에 **마찰은 존재하지 않고**(`friction.yaml` = **0바이트**, R-05), 접촉 파라미터는 검증된 적이 없으며, 케이블의 강성·구속은 모델에 **아예 없다**. J7 모터 타입조차 미확정이다(`PG-J7-001`, R-19 — MJCF `motor_DM3507`은 **자산 버그**). **그 모델로 낸 종착 자세는 예측이 아니라 미검증 코드의 출력**이다. 순서도 뒤집혀 있다: 마찰 식별(Wave 2B)은 **이 게이트의 후손**이고, 후손의 산출물로 조상 게이트를 통과시키는 것은 순환이다.
>
> **② 샘플링은 연속 배위공간의 커버리지를 증명할 수 없다.** 후보 자세를 유한 개 골라 "각각에서 지지물이 닿았다"를 확인하는 것은 **그 유한 개에 대한 진술**이다. 배위공간은 **연속**(16 자유도)이고, 커버리지 실패는 **샘플 사이에서** 일어난다. 100개를 확인하고 101번째에서 떨어지는 것은 반례가 아니라 **샘플링이 원래 증명하지 못하는 것**이다. **"미커버 영역 = ∅"은 샘플링으로 도달할 수 없는 판정문이다.**
>
> **무엇으로 대체하는가 — 모델 독립적 경계 논증.**
> 낙하가 **어디서 멈추는지**는 마찰·접촉에 달렸고 우리는 그것을 모른다. **그러나 팔이 어디에 도달할 수 있는지는 모델 없이 안다** — 베이스는 고정돼 있고 링크는 강체이며 관절 리밋은 정본이 있다. 낙하 종착 자세가 무엇이든 그것은 **반드시 도달 포락 안**이다. 마찰이 얼마든, 접촉이 어떻든, 페이로드 동역학이 어떻든 **이 명제는 참으로 남는다.**
> → 그러므로 구속을 **낙하 종착점에 맞추지 말고 도달 포락 전체의 경계에 세운다.** 이것은 샘플링이 아니라 **기하학적 포함 관계**이며, **미검증 물리 모델을 단 하나도 소비하지 않는다.**
> **시뮬레이션은 이 단계에서 시각화 보조로만 쓴다** — 사람이 포락과 구속을 눈으로 보게 하는 그림이다. **판정 근거가 아니다.** 이 단계의 어떤 산출물도 "시뮬로 검증/증명/인증했다"고 적지 마라.

| 요소 | 내용 |
|---|---|
| **왜 이 단계인가** | `NFR-SAF-009`가 열거한 완화 3종 중 **①**. 요점은 **"rest 자세를 받친다"가 아니라 "팔이 갈 수 있는 어디에서 떨어져도 구속이 경계에 이미 있다"**이다. E-Stop은 임의 시점에 눌리므로 낙하 시작 자세가 rest라는 보장이 없고, **낙하 종착 자세는 알 수 없다**(위 ① 참조). 알 수 없는 것을 예측하는 대신 **알 수 있는 것(도달 포락)을 경계한다** |
| **판정 기준 (이 단계의 유일한 합격선)** | **모든 링크와 페이로드의 도달 포락 전체가 하중등급 물리 구속의 내부에 포함된다.** 두 낱말이 전부 요구사항이다: **「전체」** = 유한 표본이 아니라 포락의 **닫힌 경계**. **「하중등급」** = 구속이 최악 조건의 낙하 하중을 **받도록 정격된 것**임이 물리적 근거(구조·고정·재료·정격)로 진술된 것 |
| **에이전트가 하는 일** | ① 정본 관절 리밋(`PG-VEL-001` 이전이므로 **위치 리밋만**, 정본은 `FR-SAF-045` 선택값)과 **URDF 링크 지오메트리**로 **도달 포락**을 산출 — **순수 기하/FK. 마찰·접촉·동역학 파라미터를 입력으로 받지 않는다**(받으면 그 순간 이 논증이 무효다) ② 포락을 **모든 링크에 대해** 산출한다 — TCP 궤적이 아니다. 팔꿈치·상완·손목 케이싱이 **각각** 포락을 갖는다 ③ **페이로드 여유**를 포락에 더한다(무부하 게이트라도 그리퍼 케이싱·인핸드 카메라는 링크다 — `16` §10.2 M-11) ④ 포락의 **닫힌 경계면**을 산출하고 **제안된 구속 지오메트리와의 포함 관계**를 판정 — 포함이 **깨지는 영역이 있으면 그 영역을 경계면 위에서 연속적으로 표시**(표본 목록이 아니라 면적/체적으로) ⑤ 포락·구속·미포함 영역을 3D로 렌더 — **시각화 보조**(사람의 판정 입력이지 판정 자체가 아니다) ⑥ 셀 씬 충돌 geom 활성 상태를 확인 — `FR-SAF-013` 근거: `cell_left_wall_col`·`cell_right_wall_col`·`cell_front_wall_col`·`cell_roof_col`·`cell_rail_col1/2`가 런타임에 **`contype=0; conaffinity=0`으로 비활성화**돼 있고 **테이블(`cell_table_col`)만 항상 활성**이다 |
| **사람이 하는 일** | ① 구속을 물리적으로 설치 ② **구속의 기하가 포락을 경계하는지**를 실물에서 확인 — **자세를 하나씩 시험하는 것이 아니라, 구속이 포락 경계를 따라 연속적으로 존재하는지**(틈·개구부·미연장 구간이 없는지)를 본다 ③ **틈이 있으면 그 틈을 보고**한다 — 틈은 "그 자세가 안 나오길 기대하는 곳"이며, 기대는 증거가 아니다 ④ **구속의 하중 정격 근거**를 진술 — 고정 방식(볼트/클램프/자중), 하중 방향, 정격 근거. 최악 조건 = **최대 도달 높이에서의 자유낙하 + 최대 관절 속도로 밀어붙이는 경우** ⑤ **`Human-judgment`**: 구속이 **낙하 하중을 받는 방향**인지(스치는 접촉·굽힘 파손이 아닌지) 판정 ⑥ torque-OFF에서 손으로 팔을 움직여 확인하는 것은 **보조 확인이며 커버리지 증거가 아니다** — 이 행위로 얻는 것은 "닿더라" 몇 건이지 "전부 닿는다"가 아니다 |
| **중단조건** | ① 도달 포락 경계에 **구속이 없는 구간이 존재**한다(면적 > 0) ② 구속이 **팔의 정상 작업 궤적을 간섭**한다(= 구속이 충돌원이 된다) ③ 구속의 **하중 정격 근거를 진술할 수 없다**(밀린다/넘어진다/굽는다/근거 미상) ④ 손으로 움직이는 중 **하드스톱 충돌** 발생 → 리밋 정본 의심 → 중단(R-16으로 회부) ⑤ 🔴 **커버리지 판정의 근거로 시뮬레이션 낙하 결과가 제출된다** → **즉시 중단.** 그것은 이 단계가 요구하는 종류의 증거가 아니다 |
| **안전 초기상태** | **torque-OFF.** 페이로드 **없음**. 사람이 팔을 지지한 채로만 이동. 도달 포락 내 다른 인체 **없음** |
| **산출증거** | ① **도달 포락 정의** + 산출에 쓴 입력 목록(관절 리밋 정본 + 링크 지오메트리 **뿐**임을 명시 — **동역학 파라미터 미사용 진술**) ② 포락 경계 vs 구속 기하의 **포함 관계 판정문** ③ **미포함 구간 = ∅**(면적 기준) 판정문 — **표본 목록이 아니다** ④ 구속 **하중 정격 근거 기술서**(고정 방식·하중 방향·정격 근거·최악 조건 산정) ⑤ 구속 설치 실물 사진(경계 연속성이 보이는 각도) ⑥ 셀 충돌 geom 활성/비활성 상태 스냅샷 ⑦ (선택) 3D 렌더 — **"시각화 보조" 라벨 필수** |
| **재개지점** | 미포함 = ∅ **및** 하중 정격 근거 진술 완료 → **S-3**. 둘 중 하나라도 미달 → **`FAIL_BLOCKING`** → 리그 교정(구속 연장/보강 **또는 도달 포락 축소** = 리밋 정본을 좁히는 것) → **S-2부터 재실행** |
| **명시적 선택지** | 포락을 구속으로 못 덮으면 **리밋을 좁혀 도달 포락을 줄이는 것**이 정당한 교정이다 — 그리고 이것은 **패배가 아니라 이 논증의 설계된 출구**다. 포락은 리밋의 함수이므로 **리밋을 좁히면 포락이 확실하게 줄어든다**(모델 없이 참). **대가**: 작업 공간이 줄고, 좁힌 리밋이 `FR-SAF-045` 정본 선택(`CUSTOM`)으로 굳어 이후 IK·텔레옵·학습 데이터 전부가 그 공간 안에서만 산다 |
| **대가** | 도달 포락은 **낙하 종착 포락보다 반드시 크다** — 대개 훨씬 크다. 즉 이 논증은 **필요 이상으로 큰 구속을 요구**하며, 우리는 **모르는 것을 안다고 가정하지 않는 대가로 재료와 작업 공간을 지불**한다. 언젠가 마찰·접촉이 검증되면(Wave 2B) 포락을 조일 수 있다는 유혹이 생길 것이다 — **그때도 이 게이트의 판정 기준은 바뀌지 않는다.** 검증된 마찰 모델은 **낙하 종착점을 예측하지만 그 예측이 틀렸을 때의 결과는 여전히 떨어진 팔**이고, 이 게이트는 예측이 틀린 경우를 위해 존재한다 |

#### S-3 · 낙하영역 비움

| 요소 | 내용 |
|---|---|
| **왜 이 단계인가** | `NFR-SAF-009` 완화 **②**. S-2가 "무엇이 경계에 서 있는가"라면 S-3은 **"경계에 닿기 전에 무엇을 지나가는가"**다. 구속이 포락을 완전히 경계해도 **구속에 닿기까지의 통과 공간**은 비어 있어야 한다 |
| **에이전트가 하는 일** | ① **스윕 볼륨 = S-2의 도달 포락 그 자체**로 잡는다. 🔴 **낙하 궤적을 시뮬레이션해 통과 공간을 좁히지 마라** — 궤적은 마찰·접촉의 함수이고 그 모델은 존재하지 않는다(S-2 서두 ①). **팔이 지나갈 수 있는 공간은 팔이 도달할 수 있는 공간과 같으며, 이 명제는 모델 없이 참이다.** 좁힌 볼륨은 미검증 모델을 믿는 것이고, 넓은 볼륨은 **믿지 않아도 되는 것**이다 ② 볼륨을 **바닥 투영 + 3D**로 렌더 — **시각화 보조** ③ 볼륨 내부와 교차하는 **셀 고정물**(테이블·레일·카메라 마운트·케이블 트레이) 열거 ④ 카메라 열거 결과(`PG-CAM-001`)의 물리 마운트 위치와 대조 |
| **사람이 하는 일** | ① 스윕 볼륨 내 **공구·부품·케이블·의자·모니터 등 이동 가능 물체 전부 제거** ② 제거 불가 고정물(테이블 등)에 대해 **접촉 시 손상/튕김 여부 판정** — `Human-judgment` ③ **인체 접근 경계 표시**(테이프/펜스) ④ 제거 후 상태 촬영 |
| **중단조건** | ① 스윕 볼륨 내 **제거 불가 물체가 남고 그것이 튕김/파손을 유발**할 수 있다 ② 볼륨이 **통행로를 침범**해 사람이 지나갈 수 있다 ③ **케이블(CAN/USB/전원)이 볼륨을 통과**한다 → 낙하가 케이블을 당겨 **2차 사고**(어댑터 탈락 → 이중 bind 재현 조건, R-08) |
| **안전 초기상태** | **torque-OFF.** 팔은 구속 위 또는 사람이 지지. 페이로드 **없음** |
| **산출증거** | ① 스윕 볼륨 3D + 바닥 투영 + **볼륨 = S-2 도달 포락임을 명시**(궤적 시뮬로 좁히지 않았다는 진술) ② 제거 대상 목록 + 제거 전/후 사진 ③ 잔존 고정물 목록 + 각각의 접촉 판정 ④ 인체 경계 표시 사진 ⑤ 케이블 라우팅이 볼륨 외부임을 보이는 사진 |
| **재개지점** | 볼륨 비움 확인 → **S-4**. 잔존 위험 → **`FAIL_BLOCKING`** → 리그 재배치 → **S-3부터 재실행**(S-2 포락이 안 바뀌었으면 S-2 재실행 불요; 마운트를 옮겼으면 **S-2부터**) |
| **대가** | 볼륨을 도달 포락으로 잡으면 **비워야 할 공간이 낙하 궤적보다 훨씬 넓다** — 셀 안의 물건 배치와 사람의 통행이 그만큼 제약된다. S-2와 같은 교환이다: **넓은 공간을 지불하고, 믿을 수 없는 모델을 믿지 않아도 되는 상태를 산다** |

#### S-4 · 무부하 확인

| 요소 | 내용 |
|---|---|
| **왜 이 단계인가** | §4.4 참조. 공식 안전 가이드가 낙하 대상으로 지목한 것은 정확히 ***"the load being held"***다. 부하는 낙하 에너지를 키우고 그리퍼에서 이탈해 **비행체**가 될 수 있다 |
| **에이전트가 하는 일** | ① 그리퍼 상태를 `connect_readonly` 관측으로 읽어 **open/close 엔드포인트 대비 현재 개도** 보고 ② 그리퍼 개도가 파지 상태를 시사하면 경고 ③ 페이로드 관련 후속 트랙(Wave 2B 페이로드·중력모델)이 **이 게이트의 후손임을 명시**해 순서 역전 방지 |
| **사람이 하는 일** | ① 그리퍼에서 **모든 물체 제거** ② 팔·손목·그리퍼 케이스에 부착된 **임시 부착물(지그·마커·테스트 웨이트) 전부 제거** ③ 무부하 상태 촬영 ④ **인핸드 카메라는 페이로드가 아님**을 확인 — `16` §10.2 M-11: v2 pinch 그리퍼 **케이스 내장** 카메라 = 손목 카메라, **같은 물건 하나**이며 별개 슬롯으로 세면 중복 계상 |
| **중단조건** | ① 그리퍼에 물체가 남아 있다 ② 제거 불가 부착물이 있다(→ 그것은 페이로드이며 이 게이트는 **무부하 구성에만** `PASS`를 준다) ③ 그리퍼 개도 관측이 **파지 상태를 시사**하는데 육안으로는 비어 있다 → 엔드포인트 캡처 미완(`FR-CON-064`, `capture_gripper_endpoint`) 의심 → 중단 |
| **안전 초기상태** | **torque-OFF.** 그리퍼 **비어 있음**. 팔 지지됨 |
| **산출증거** | ① 무부하 상태 사진(그리퍼 근접 + 전신) ② 그리퍼 개도 관측값 + 캡처된 open/close 엔드포인트 대비 판정 ③ **"이 `PASS`는 무부하 구성에만 유효"** 명시 문구 |
| **재개지점** | 무부하 확인 → **S-5**. 부하 존재 → 제거 후 **S-4 재실행**. 제거 불가 → **`FAIL_BLOCKING`**(그 구성은 이 게이트의 대상이 아니다) |
| **범위 못박기** | **`PASS`는 무부하 구성에 한정된다.** 페이로드를 매다는 순간 `PG-SAFE-001`은 **`SUPERSEDED`**가 되고 그 구성으로 **재실행**해야 한다. 이유는 **두 가지이며 둘 다 모델과 무관하다**: ① 페이로드는 그리퍼 너머로 **기하를 연장**하므로 **도달 포락이 커진다**(S-2 판정 기준의 「모든 링크와 **페이로드**」) ② 페이로드는 **낙하 하중을 키우므로 구속의 하중 정격 근거가 무효**가 된다(S-2 「하중등급」). 질량·관성이 종착 자세를 바꾸는지 여부는 **묻지 않는다** — 우리는 종착 자세를 애초에 주장하지 않기 때문이다 |

#### S-5 · E-Stop 리셋 거동

| 요소 | 내용 |
|---|---|
| **왜 이 단계인가** | `FR-SAF-057`(**E-Stop 해제 후 자동 추종 재개 금지, 홈잉/정렬 경유 필수**)이 회로 수준에서 성립하는지 확인한다. 리셋이 **자동 복전**이면 소프트웨어가 무슨 규율을 갖든 **회로가 먼저 재개**시킨다 |
| **범위 한정 (중요)** | 이 단계는 **torque-OFF에서 회로만** 시험한다. **토크가 걸린 상태의 E-Stop 낙하 실증은 이 게이트에 없다** — `PG-SAFE-001`은 첫 토크-ON을 막는 게이트이므로 자기가 토크를 켤 수 없다. 그 실증은 **Wave 1-⑤ 가드된 토크-ON의 명명된 후손 단계**이며 §4.4 무부하 제약을 그대로 상속한다 |
| **에이전트가 하는 일** | ① candump + `ip link` 연속 관측 ② 버튼 **누름 → 래치 유지 → 리셋 시도**의 각 구간에서 CAN/전원 도메인 상태 기록 ③ **리셋만으로 전원이 자동 복귀하는지**(자동 복전) vs **별도 의도적 인가 조작이 필요한지** 판정 ④ 리셋 직후 **`enable_torque`/0xFC 프레임이 자동 발생하지 않음**을 candump로 증명 ⑤ 리셋 후 **RC-1~RC-5 복구 시퀀스를 실제로 실행**해 경로가 실재함을 증명(torque-OFF 범위 내: RC-1·RC-2·RC-3·RC-5까지. **RC-4의 첫 MIT 프레임 검증은 토크-ON 후손으로 이월**) ⑥ 🔴 **소켓 생존 여부를 판정**한다 — 차단 구간 이후 기존 소켓이 살아 있는가, 아니면 무효화됐는가(USB 어댑터 리셋 · bus-off · 인터페이스 재생성 중 무엇이 일어났는지 `ip -details link show` + `dmesg`로 특정). **이 단계가 그 답을 알아내는 유일한 지점이다** — S-1은 차단만 관측했고 복구를 실행하지 않았다 ⑦ 소켓이 무효화됐으면 **`reopen_transport()`로 복구**하고, 그 구간 candump에 **0xFC·0xFE 프레임이 부재**함을 증명한다. **`connect()` 재호출로 복구가 시도되면 즉시 중단**(RC-5 위반 = §4.2.1 오염 연쇄 진입) |
| **사람이 하는 일** | ① 버튼 **누름** ② 래치 상태에서 **버튼이 물리적으로 걸려 있는지**(스스로 복귀하지 않는지) 확인 ③ 지시에 따라 **리셋** ④ 리셋 후 **팔에 손을 대지 않은 채** 워크플로우의 복구 시퀀스 실행을 기다린다 ⑤ 3D 자세 확인(RC-3) **판정** — `Human-judgment` |
| **중단조건** | ① **리셋만으로 전원이 자동 복귀**한다(자동 복전) → `FR-SAF-057`이 회로 수준에서 깨진다 → 중단 ② 버튼이 **래치되지 않는다**(모멘터리) → 손을 떼면 복전 → 중단 ③ 리셋 직후 candump에 **0xFC(enable) 프레임이 관측**된다 → 자동 인에이블 경로 존재 → **즉시 중단** ④ 복구 시퀀스 RC-1~RC-3 중 **어느 하나라도 실행 불가** → 중단(§4.2 진입조건 위반이 뒤늦게 드러난 것) ⑤ RC-2 **영점 잔차가 허용오차 초과** → 재영점 요구(`FR-CON-065` ②) |
| **안전 초기상태** | **torque-OFF.** 팔은 **하중등급 구속 안**(S-2 통과 상태). 스윕 볼륨 **비어 있음**(S-3 통과). **무부하**(S-4 통과). 사람은 **인체 경계 밖** |
| **산출증거** | ① 래치/리셋 거동 판정문(자동 복전 여부) ② 각 구간 CAN/전원 상태 타임라인 ③ **리셋 직후 0xFC 프레임 부재** candump 증명 ④ RC-1~RC-3·RC-5 실행 로그(관절 실측 위치·영점 잔차·3D 충돌 판정·**`connect()` 호출 횟수 = 1**) ⑤ **소켓 생존 판정문** + (무효화된 경우) **`reopen_transport()` 실행 로그 + 그 구간 0xFC·0xFE 부재 candump 증명 + `reopen_transport()` 호출 횟수**(별도 카운터) ⑥ 사람 판정 서명(3D 자세 수용) |
| **재개지점** | 전 항목 통과 → **게이트 판정(§4.5)**. 자동 복전 확인 → **`FAIL_BLOCKING`** → 회로 교정(래치형 + 의도적 인가 조작 요구) → **S-1부터 재실행**(전원 도메인 거동이 바뀌었으므로) |

### 4.4 🔴 부하를 매단 채 E-Stop을 시험하지 마라

**이것은 전역 규율이다** — `PG-SAFE-001` S-5뿐 아니라, Wave 1-⑤ 이후의 **토크-ON E-Stop 실증**과 모든 재실행에 상속된다.

| 항목 | 내용 |
|---|---|
| **규율** | E-Stop 시험(회로 시험이든 토크-ON 낙하 실증이든)은 **무부하 구성에서만** 수행한다. 그리퍼에 물체가 있거나 팔에 테스트 웨이트가 부착된 상태의 E-Stop 시험을 **금지**한다 |
| **근거** | 공식 안전 가이드 원문이 낙하 대상으로 지목한 것이 정확히 **"the load being held"**다(`16` §4 M-2 인용, `12` §2.10, `FR-SAF-042`). 부하는 **① 낙하 에너지를 키워 S-2 구속의 하중 정격 근거를 무효화하고 ② 낙하 중 그리퍼에서 이탈해 비행체가 되며 — 이탈한 물체는 팔의 도달 포락에 구속되지 않으므로 S-2·S-3의 경계 논증 자체가 그것에는 적용되지 않는다 ③ 그리퍼 너머로 기하를 연장해 S-2 도달 포락과 S-3 스윕 볼륨을 넓힌다** |
| **그리퍼 고유 사정** | 파지 유지에는 토크가 필요하다(v2 pinch 네이티브 제어 = **POS_FORCE**, `16` §6 D-5). E-Stop = 전원 차단 = **그리퍼 토크도 0** → **파지가 풀린다.** 즉 부하 E-Stop 시험은 "부하를 든 채 떨어지는" 시험이 아니라 **"떨어지면서 부하를 놓는"** 시험이며, 두 물체가 서로 다른 궤적으로 날아간다 |
| **관측 불가** | 부하 상태에서 낙하 거동을 재도 **얻는 것이 없다.** `16` §4 **M-3**이 **낙하 속도 게이트를 제거**했다 — *"낙하 속도는 측정하지 않는다. 손상될 수준이 아니며 낙하 속도 게이트/측정 FR을 제거한다."* 즉 **부하 낙하로 얻으려는 정량 산출물이 계획에 존재하지 않는다.** 위험만 있고 산출이 없다 |
| **`PASS` 범위** | `PG-SAFE-001`의 `PASS`는 **무부하 구성에만** 유효하다(S-4). 페이로드가 붙으면 **`SUPERSEDED` → 그 구성으로 재실행** |
| **페이로드는 어디서 다루는가** | **Wave 2B**(v2 중력모델 검증 · 페이로드 · 마찰동정). 페이로드 트랙은 `PG-SAFE-001`의 **후손**이며, 페이로드 상태의 안전 검증은 **재실행된 `PG-SAFE-001`**을 통과한 뒤에만 열린다 |
| **대가** | **이 규율은 페이로드 상태의 E-Stop 낙하 거동을 미검증으로 남긴다.** 실제 작업은 물체를 들고 하며, 그 상태의 낙하는 무부하 낙하와 다르다. 우리는 그 지식을 포기하고, 대신 **그 지식을 얻는 과정에서 발생할 사고**를 피한다. 이 교환이 성립하는 이유는 M-3이 낙하 정량 측정을 이미 계획 밖으로 내보냈기 때문이다 — **측정하지 않을 것을 위해 위험한 시험을 하는 것은 순손실**이다 |

## 6. 이 문서를 읽는 워크플로우를 위한 규율

| # | 규율 |
|---|---|
| 1 | **`M-8`을 인용하지 마라.** 제어루프 측정은 `PG-RT-001`, 명세 매핑은 `NFR-PRF-054`다 |
| 2 | **R-X01~X03을 되살리지 마라.** 되살아나는 유일한 조건은 **동력 리더암의 물리적 실재**다. 소프트웨어 어댑터는 조건이 아니다 |
| 3 | **`PG-SAFE-001` `PASS`를 무부하 밖 구성에 인용하지 마라.** 페이로드가 붙으면 `SUPERSEDED`다 |
| 4 | **복구 시퀀스 없이 E-Stop을 시험하지 마라.** §4.2는 권고가 아니라 진입 조건이다 |
| 5 | **부하를 매단 채 E-Stop을 시험하지 마라.** §4.4 |
| 6 | **`FAIL_BLOCKING`을 "대기"로 기록하지 마라.** 그것은 **리그 교정 WP를 생성하라는 지시**다 |
| 7 | **게이트 상태 전이에 후손 stale 처리 없이 넘어가지 마라.** §1.2 |
| 8 | **측정 전에 수치 목표를 못 박지 마라.** `NFR-PRF-053`/`NFR-PRF-054`, SPINE §2-6. **단, 이것은 *성능 목표*에만 적용된다** — **안전 한계는 재기 전에 도출돼 있어야 한다**(§3 도출 원칙). 두 규율을 뒤섞지 마라 |
| 9 | **R-02·R-04·R-12는 동일 `OpenArmFollower` 서브클래스다.** 병렬 빌더 금지 — 단일 소유자 WP |
| 10 | 🔴 **시뮬레이션으로 지지 커버리지를 증명하지 마라.** §4.3 S-2는 **모델 독립적 경계 논증**(도달 포락 ⊂ 하중등급 구속)으로만 통과한다. 시뮬은 **시각화 보조**이며, 어떤 산출물에도 **"시뮬로 검증/증명/인증했다"고 쓰지 마라.** 마찰(`friction.yaml` = 0바이트)·접촉·페이로드 동역학은 이 게이트 시점에 **존재하지 않고**, 그 식별은 이 게이트의 **후손**이다 |
| 11 | 🔴 **후보 자세 샘플링을 커버리지 증거로 인용하지 마라.** 배위공간은 연속이고 실패는 **샘플 사이에서** 일어난다. "N개 자세에서 닿았다"는 **"전부 닿는다"가 아니다** |
| 12 | 🔴 **E-Stop 복구에 `connect()`를 부르지 마라.** 소켓이 죽었으면 **`reopen_transport()`**다(RC-5) — 소켓/인터페이스만 재생성하고 **영점을 찍지 않고 모터를 인에이블하지 않는다.** `connect()` 카운터가 2 이상이면 **I-2 위반**이다. **I-2는 논리 세션 불변식이지 소켓 수명 불변식이 아니다** |
| 13 | 🔴 **안전 한계를 관측 능력에서 도출하지 마라.** §3. "리그가 T'를 내니 기준은 T'다"는 **결함이다.** 리그가 도출값에 미달하면 **기준이 아니라 운영 속도·도달 집합을 조정한다** |
| 14 | 🔴 **두 번째 CAN 라이터를 만들지 마라.** 마찰 로깅 포함 어떤 이유로도(R-34). 허용 경로는 **스케줄러 내부 탭** 또는 **송신 0의 read-only RX 탭** 둘뿐이다. **"식별 중에만 스케줄러를 잠시 내린다"는 낙하다** |
| 15 | 🔴 **리스 만료 후 도착한 갱신으로 동작을 재개시키지 마라.** 만료 = **홀드 래치**. 재개는 **새 세대 ID + 오퍼레이터 의도 확인**의 명시적 재무장뿐이다(R-35) |
| 16 | 🔴 **게이트가 자기 후손을 선행조건으로 갖게 하지 마라.** 순환이면 데드락이다. 해법은 **게이트 분할**(`PG-RT-001a`/`b`)이지 **새 FSM 상태가 아니다**(R-36) |
