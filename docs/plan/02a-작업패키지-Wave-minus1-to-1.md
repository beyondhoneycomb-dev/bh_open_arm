# 02a. 작업 패키지 — Wave −2 `BOOT` / −1 / 0-Env / 0-A / 0-B / 0-C / 0-Ops / 1

계약이 동결되기 전까지의 모든 작업 패키지(WP)를 {입력·산출물·인터페이스 계약·수용 게이트·실행 클래스·워크플로우 형상}으로 완전 명세한 실행 문서다 — 명세(`docs/spec/`)가 "무엇을"을 소유하고, 이 문서는 "어떤 순서로, 무엇이 통과해야, 무엇을 fan-out 하는가"를 소유한다.

> 🔴 **이 문서는 자기 대역(`BOOT`·`N1`·`ENV`·`0A`·`0B`·`0C`·`OPS`·`1`)의 `WP-*` ID 발급처다.** 발급처는 대역별로 나뉜다 — Wave 2~3 = `02b`, Wave 4~5 = `02c`, GUI(`WP-G-*`) = `02d`. `00` §8.0(네임스페이스)·`06` §2.2(대역 토큰 표)가 정본 경계다. `WP-BOOT-01`은 **`02a`~`02d` 전부**를 등재 입력으로 읽는다(이 문서만이 아니다 — 그렇지 않으면 §−2.2 차집합 ∅ 조건이 미달). 다른 문서는 여기서 발급된 이 대역 ID를 **인용**할 뿐이다. 따라서 **Wave −2 `BOOT` 대역(`WP-BOOT-01`~`05`)의 발급도 여기서 일어난다**(§−2). `06` §1.1의 `BOOT` 표는 그 인용이며, 발급이 이 문서에 없으면 `06`의 인용은 **`CI-05b` 유령 ID**로 잡힌다.

| 코드 | 형상 | 동시 워크플로우 | 격리 |
|---|---|---|---|

### 0.4 계약 ID 규약

계약 ID = **`CTR-<NAME>@v<n>`** 단일 네임스페이스. 이 문서 대역이 동결하는 계약: `CTR-ACT`(액션/관측 스키마) · `CTR-UNIT`(deg/rad/Nm 태그 타입) · `CTR-PLUG`(플러그인 API) · `CTR-OWN`(파일/모듈 소유권) · `CTR-ERR`(`OA-*` 에러코드 레지스트리) · `CTR-CAL`(영점·캘리브) · `CTR-GW`(통합 게이트웨이).

### 0.5 게이트 상태머신 (전 게이트 공통 — SPINE §3)

| 상태 | 뜻 | 후손 처리 |
|---|---|---|
| `PASS` | 수용 기준 충족 | 후손 fan-out 허가 |
| `RETRY_WITH_VARIANT` | 명명된 변형으로 재실행 | 후손 착수 보류. 기존 아티팩트 유지 |
| `DEGRADED_ACCEPTED` | 축소 프로파일로 수용 | **축소 범위 밖 후손 자동 stale + 미통합 작업 취소** |
| `FAIL_BLOCKING` | 후손 전면 차단 | 후손 아티팩트 stale + **명명된 대체 WP 생성 강제** |
| `SUPERSEDED` | 상위 결정이 게이트를 무효화 | 게이트·후손 동시 stale, 계약버전 bump |

**리스크 등록부에 한 줄 적고 넘어가는 것은 상태 전이가 아니다.** 상태 전이는 반드시 {증거 아티팩트, 후손 stale 목록, 대체 WP 이름}을 동반한다.

## −2. Wave −2 `BOOT` — 실행 부트스트랩 (최상위 선행 배리어 · WP 발급)

> **절 번호가 `−2`인 이유**: 이 문서의 §1~§9는 이미 다른 문서가 절 번호로 인용하고 있다(`01` §6.2 등). 새 대역을 §1로 밀어 넣으면 그 인용이 전부 어긋난다. 대역 이름(`Wave −2`)을 절 번호로 그대로 쓰는 것이 **인용 무결성을 깨지 않으면서 순서를 정직하게 표현**하는 유일한 방법이다 — 이 절은 §1보다 **앞에서 실행된다**.

### −2.1 왜 이 대역이 모든 것보다 앞서는가

`06`이 **"기계 정본"**이라 부르는 것 — 질의 가능한 레지스트리, 매니페스트 스키마, CI 검사기, 상태 저장소, 동결 잠금 — 이 **실재하지 않는다**(예시 레코드 몇 개뿐). 산문 표는 질의·락·스폰·병합·무효화의 대상이 될 수 없다. 즉 이 계획이 171번 말하는 "CI가 검증한다"·"후손이 stale 된다"·"계약이 동결된다"는 전부 **집행체 없는 선언문**이다.

`BOOT`은 그 집행체를 짓는다. 그리고 `BOOT` 없이 발행한 fan-out은 §9의 무효화 전파가 **시작점을 특정하지 못하므로 되돌릴 수 없다** — 그래서 이 대역은 Wave −1(명세 정규화)보다도 앞이다.

> **대가**: `BOOT`은 **로봇을 1 mm도 움직이지 않으면서 모든 것을 막는다.** 크리티컬 패스 맨 앞에 스키마·검사기·상태 저장소가 서고, Wave −1의 병렬도 0 배리어 **앞에 또 하나의 배리어**가 붙는다. 이것은 착수 반응성을 나쁘게 만든다. 그리고 부트스트랩이 틀리면 **그 위에 세운 전량이 틀린다** — 이 위험은 D2 파일럿(`00` §3.5b — `WP-0C-02` IK 어댑터 1개를 부트스트랩 직후 끝까지)이 **표본 1개로** 줄일 뿐 없애지 못한다. 이를 감수하는 이유는 대안이 **아무도 보지 않는 규칙을 전 WP에 걸쳐 위반하는 것**이기 때문이다. 접기(WP 통합) 재검토는 이 파일럿이 오버헤드를 **측정한 뒤**에만 열린다(`00` §2.5 U-6).

### −2.2 대역 수용 게이트 (5개 WP 공통 · 대역 단위 판정)

| 항목 | 값 |
|---|---|
| **수용 조건 ①** | **이 문서 대역과 `02b`·`02c`·`02d`가 발급한 WP 전량**이 `registry/traceability.yaml`에 등재되고 스키마 검증 통과 — **미등재 0건** |
| **수용 조건 ②** | **`CI-01`~`CI-17`이 자기 코퍼스(계획 문서 + 레지스트리 + 매니페스트) 위에서 green — 단 `CI-07` 제외**(아래 판정 주체) |
| **미달 시** | 🔴 **`FAIL_BLOCKING` — 어떤 WP도 착수 불가.** Wave −1조차 열리지 않는다(`00` §9.2, `06` CI-18) |
| **판정 주체** | `CI-01`~`CI-17` 자신, **`CI-07` 제외.** ⚠️ **`CI-18`(부트스트랩 선행)은 `BOOT`의 자기 검사가 아니다** — 착지 후 나머지를 지키는 규칙이다(`06` §5). 🔴 **그러나 `CI-18`의 실행체는 `WP-BOOT-03`이 짓는다**(§−2 계약 칸 · 수용 ⑫) — **짓는 범위 `CI-01`~`CI-18` ≠ 판정 범위이고 이 차이는 의도다.** 두 수를 같게 "정정"하는 것이 결함이다. 🔴 **`CI-07`(정규화 해시)도 판정에서 뺀다 — 이유는 `CI-18`과 다르다**: `CI-07`은 `WP-N1-04`가 Wave −1에서 발행하는 정규화 해시를 검사하는데, 그 해시는 BOOT 착지 시점(Wave −2)엔 존재할 수 없다. `CI-18`과 **똑같은 순환**(게이트→Wave −1→게이트)이나 술어가 아니라 **제외로** 푼다. **`CI-18`은 영구 제외, `CI-07`은 한시적** — Wave −1이 해시를 발행하고 레지스트리를 재파종하면 `CI-07`은 스스로 green이 되므로 그때 판정 범위로 되돌린다(`registry/checks/__init__.py` `JUDGE_EXCLUDED`) |
| **실행 클래스** | 5개 전부 `AI-offline` — **로봇도 사람도 불필요** |
| **워크플로우 형상** | 5개 전부 `SHAPE-CF` — 각 WP가 배타 소유 경로를 갖는다. `SHAPE-IM` fan-out은 **불가**: 폭 `n`을 계산할 소유자 레지스트리 자신이 아직 없다(`WP-0A-03`은 이 대역의 **하류**다) |

> **왜 5개가 전부 `SHAPE-CF`인가**: fan-out 폭을 확정하려면 소유권 겹침 증명이 필요하고(§0.3), 그 증명기는 `WP-0A-03`이 짓는다. `BOOT`은 그보다 앞에 있으므로 **자기 자신을 병렬화할 근거를 만들 수 없다.** 5개는 서로 다른 경로를 배타 소유하며 **선언된 순서로 직렬**이다 — `01`이 `02` 스키마를 필요로 하고, `03`이 `01`·`02` 산출을 검사 대상으로 삼고, `04`·`05`가 `01`의 축을 읽는다.

## 1. Wave −1 — 명세 정규화 (전역 블로킹 배리어)

### 1.1 왜 배리어인가

명세는 ~1100 FR을 18문서에 분산 소유한다. 같은 사실을 두 문서가 **서로 다른 계약으로** 진술한 지점이 실재하고, 그 위에서 병렬 fan-out을 돌리면 **두 워커가 각자 다른 정본을 구현한 뒤 통합 시점에 충돌**한다. 통합 시점의 충돌은 코드 두 벌을 버리는 비용이다.

> **대가**: 이 배리어는 **초기 병렬도를 0으로 만든다.** Wave −1 동안 fan-out 가능한 WP는 없다. 이것은 착수 반응성을 나쁘게 만든다. 이를 감수하는 이유는 정규화가 사후에 불가능하기 때문이다 — 이미 구현된 두 정본 중 하나를 죽이는 것은 문서 한 줄을 죽이는 것보다 비싸다.

### 1.2 정규화 원장 스키마

원장 1행 = **{승리 요구사항, 폐기 텍스트, 소유 계약버전}** + 판정 근거 + 후속 강제.

## 2. Wave 0-Env — 단독 배리어

### 2.1 왜 단독인가

`WP-ENV-*`는 전부 `pyproject.toml` / 락파일 / CI 설정을 건드린다. **두 트랙이 같은 파일을 동시에 편집하면 worktree 격리로도 해결되지 않는다** — worktree는 병합 충돌을 지연시킬 뿐 없애지 못하고, 락파일은 병합이 의미론적으로 성립하지 않는 아티팩트다(해시 트리를 손으로 합칠 수 없다).

> **대가**: Wave 0-Env는 **Wave −1이 끝난 직후 다시 병렬도를 1로 되돌린다.** 즉 배리어가 연속 2개다. 이것은 착수 지연을 나쁘게 만든다. 대안(0-Env를 0-B/0-C와 병렬)은 락파일 재생성 경합을 만들고, 그 경합은 "누구의 환경에서 통과했는가"를 불명확하게 만들어 **이후 모든 측정의 재현성을 무효화**한다. 측정 선행 규율(SPINE §2-6)이 성립하려면 환경이 먼저 결정적이어야 한다.

## 4. Wave 0-B — AI-on-HW (읽기/측정, 물리 조작 없음)

### 4.1 vcan 단독 검증 가능 vs 실물 필수 — 분리 원칙

**분리선은 "커널이 무엇을 시뮬레이션하는가"다.** SocketCAN vcan은 프로토콜 스택·소켓 의미론·bind 동작·`/proc/net/can/rcvlist_all` 엔트리·flock 파일 잠금까지 **전부 실재**한다. 재현하지 못하는 것은 **물리 어댑터의 sysfs 속성과 USB 전송 특성**뿐이다.

| 능력 | vcan 재현 | 사유 |
|---|---|---|
| `flock(LOCK_EX\|LOCK_NB)` 배타·PID 표시 | ✅ 완전 | VFS 파일 잠금, CAN 무관 |
| `/proc/net/can/rcvlist_all` RX 리스너 카운트 | ✅ 완전 | 커널 CAN core가 vcan에도 동일 적용 |
| RAW fan-out(매칭 프레임 → 전 소켓 복제) | ✅ 완전 | CAN core 동작. `01` FR-SYS-007의 "응답 탈취 아님" 모델을 그대로 검증 |
| 두 번째 **writer** 주입 → 상충 명령 | ✅ 완전 | 진짜 위험의 정확한 재현 |
| `ip -s link show` TX 카운터 대조 워치독 | ✅ 완전 | vcan도 TX 통계 유지 |
| `ip -details link show` **파싱기** | ✅ 완전 (합성 출력 + 실측 출력 픽스처) | 파싱은 문자열 처리. **값의 진위는 실물 소관** |
| `fd on` / `bitrate 1000000` / `dbitrate 5000000` **실측값** | ❌ | vcan에 비트레이트 개념 없음 |
| `ERROR-ACTIVE` / `BUS-OFF` 상태 전이 | ❌ | 물리 버스 오류 카운터 필요 |
| `ethtool -i` 드라이버 식별 | ❌ | 물리 드라이버 바인딩 |
| `ATTR{dev_id}` / `ATTRS{serial}`(iSerial) | ❌ | 물리 USB 디스크립터 |
| USB RTT · HOL · 대역폭 | ❌ | 물리 전송 |

> **대가**: vcan 검증은 **파싱기와 상태머신이 옳음**을 증명하지 코드가 **실물에서 옳음**을 증명하지 않는다. `ip -details link show` 파서가 vcan 픽스처 100% 통과해도 실물 어댑터의 출력 포맷이 다르면 조용히 오작동한다. 이것은 실물 브링업의 위험을 **줄이지 않고 뒤로 미룬다**. 이를 감수하는 이유는 대안 — 실물 없이는 CAN 코드 착수 자체를 금지 — 이 Wave 0-B 전체를 사람 가용성에 종속시키고 병렬도를 0으로 만들기 때문이다. 완화책: 모든 vcan WP는 **실물 출력 픽스처 획득 즉시 재실행하는 재검증 훅**을 산출물에 포함한다(아래 각 WP의 수용 게이트 마지막 항).

> **`WP-0C-08` 대가**: 정책 호환 매트릭스를 **차단**으로 집행하면 양완 + 토크 기록 구성에서 SmolVLA/pi0/pi05 계열이 통째로 사라진다. 이것은 정책 선택지를 나쁘게 만든다 — GR00T 계열에 사실상 종속된다. 회피로는 `use_velocity_and_torque=False`(24차원)로 내려가는 경로가 있으나, 그러면 `16` D-8이 "공짜"라 규정한 토크 채널을 버리는 것이고 `12`의 잔차 기반 감지 입력이 데이터셋에서 사라진다. **두 손실은 교환 관계이며 이 WP는 그 교환을 계산 가능하게 만들 뿐 해소하지 않는다.**

## 7. Wave 1 — 순차 서브-DAG (병렬 아님)

### 7.1 왜 순차인가

**`02` §2.0.4상 `connect()` 오버라이드와 `send_action()` 확장은 동일 `OpenArmFollower` 서브클래스에 산다.** 두 워커가 같은 클래스를 **동시에** 빌드하면 병합 충돌이 아니라 **의미론 충돌**이 난다 — 한쪽은 torque-OFF 브링업을 전제하고 다른 쪽은 토크-ON 게이트웨이를 전제한다. `WP-0A-03` 소유권 레지스트리가 이 경로를 **배타=true**로 고정한다.

> **정확히 무엇이 금지되는가**: 금지되는 것은 **동시 소유**이지 순차 인계가 아니다. `WP-1-02`(`connect_readonly` + 명시 영점)와 `WP-1-03`(스케줄러 + 통합 게이트웨이)은 **같은 파일을 순차로 인계**한다 — `WP-1-02`가 소유를 놓은 뒤에 `WP-1-03`이 잡는다. **어느 시점에도 소유자는 정확히 1명**이며, 레지스트리는 이것을 `{소유 WP, 소유 구간}`으로 인코딩한다(`WP-0A-03`). "이 파일을 여는 WP는 하나뿐"이 아니라 **"이 파일을 동시에 여는 WP는 없다"**가 정확한 진술이다 — 전자는 순차 인계마저 금지해 Wave 1을 표현 불가능하게 만든다.

> **대가**: Wave 1은 **이 계획에서 가장 긴 임계 경로**다. 6단계 전부가 직렬이고 병렬화 여지가 0이다. 이것은 총 소요 단계 수를 나쁘게 만든다. 대안(단계 ②와 ③을 병렬)은 서브클래스를 두 파일로 쪼개는 것을 뜻하는데, 그러면 `11` NFR-INF-008이 `[정본]`으로 못 박은 **"유일 집행점"**이 두 지점으로 갈라진다 — 그리고 `12` FR-SAF-074가 방어하려는 우회 경로가 정확히 그 틈에서 생긴다.

### 7.1a 🔴 Wave 1은 "한계를 찾는" 웨이브가 아니다 — 리미터가 스윕보다 먼저 선다

⑥의 속도 스윕(`PG-VEL-001`)에는 순환이 하나 있었다: **스윕이 속도 한계를 찾는데, 그 한계를 집행할 리미터(`WP-2A-04`, `02b`)는 스윕 결과를 기다린다.** 그리고 리미터가 없는 상태의 스윕은 **보호 없이 최대 속도를 찾는 것**이다 — 홀딩 브레이크가 없는 팔에서 그것은 측정이 아니라 사고다.

**해소는 리미터를 앞으로 옮기는 것이다**(정본 = `03` §5.6.0):

>
> **대가**: 부트스트랩 리미터는 **스윕의 정보량을 나쁘게 만든다** — 리미터 아래에서만 도므로 **관절의 진짜 최대 속도는 영원히 모른다.** 그 대가를 지불하는 이유는 진짜 최대 속도가 **알 필요가 없는 값**이기 때문이다: 운영 속도는 도출값이 정하고, 도출값 위쪽은 어차피 쓰지 않는다. **알 수 없어서 못 쓰는 게 아니라, 쓰지 않기로 먼저 정했으므로 알 필요가 없다.** 그리고 `WP-2A-04`의 "`PG-VEL-001` 미확정 시 착수 불가"가 걸리는 대상은 **정본 세트의 최종 봉인**이지 리미터의 존재 자체가 아니다.
