# 06. 추적성 레지스트리

명세 ~1100 요구사항을 `요구사항 → WP → 아티팩트 → 소유 워크플로우 → 파일/모듈 소유권 → 계약버전 → 게이트 → 음성분기 → 하류 소비자`로 잇는 **기계판독 정본**이며, CI가 이 파일을 검증해 누락·중복·고아를 빌드 실패로 만든다.

---

## 1. 이 레지스트리가 존재하는 이유

산문 DAG(`01-의존성-DAG-및-병렬화.md`)는 사람이 읽는 물건이다. 병렬 fan-out된 워크플로우는 산문을 읽지 않고 **질의**한다. 아래 질문들은 산문으로 답할 수 없다 — 답이 없어서가 아니라, **답이 틀렸을 때 아무도 모르기 때문**이다.

| # | 질문 | 이 레지스트리의 필드 | 답이 없으면 벌어지는 일 |
|---|---|---|---|
| Q1 | `PG-RID-001`이 `FAIL_BLOCKING`으로 뒤집혔다. **어떤 아티팩트가 지금 stale인가?** | `gate` ← 역인덱스 `downstream` | 죽은 게이트 위에 쌓인 작업이 통합까지 살아남는다 |
| Q2 | 이 워크플로우가 `openarm_follower_oa.py`를 열어도 되는가? | `owns[].glob` + `mode` | 두 fan-out이 같은 심볼을 동시 편집 → 머지가 아니라 **의미 충돌** |
| Q3 | `FR-SAF-045`(리밋 정본)는 **누가** 구현하는가? | `req → wp` | 아무도 안 하거나, 셋이 서로 다르게 한다 |
| Q4 | `CTR-WS`을 `@v1`에서 `@v2`로 올리면 **무엇이 깨지는가?** | `contract.consumes` 역인덱스 | 계약 변경이 조용히 하류를 부순다 |
| Q5 | 이 아티팩트는 **어느 요구사항 때문에 존재하는가?** | 역방향 `artifact → req` | 근거 없는 코드가 남아 영원히 유지된다 |
| Q6 | `PG-IK-001`이 Jetson Nano에서만 실패했다. **무엇을 미지원 표시해야 하는가?** | `gate.per_target` + `negative_branch` | 미달 타깃에 배포된다 |

> **대가**: 레지스트리는 **두 번째 진실**이다. 코드의 실제 import 그래프와 `downstream` 필드가 갈라질 수 있고, 갈라지면 stale 전파가 조용히 새 나간다. 이것은 **리팩터링을 나쁘게 만든다** — 파일을 옮기면 코드가 아니라 레지스트리가 먼저 깨진다. 이 대가는 §5.6(그래프 대조 검사)으로만 상쇄되며, 완전히 없앨 수는 없다.

**정본 경계** — 이 레지스트리는 **어느 ID도 새로 만들지 않는다.** 오직 잇고, 잇히지 않은 것을 고발한다. 발급처는 전부 바깥에 있다:

| ID 축 | 정본 문서 |
|---|---|
| 요구사항 `FR-*`/`NFR-*` | `docs/spec/` |
| WP `WP-<대역>-<2자리>` · GUI WP `WP-G-*` | `02a-작업패키지-Wave-minus1-to-1.md` · `02b-작업패키지-Wave-2-3.md` · `02c-작업패키지-Wave-4-5.md` · `02d-작업패키지-GUI.md` |
| 게이트 `PG-<축>-<3자리>` | `03-측정-게이트.md` (SPINE §3에서 동결) |
| WP 수용 검사 `CG-*` | 해당 WP를 정의한 `02a`~`02d` (선언 = `00` §8) |
| 계약 `CTR-<이름>@v<n>` | `01-의존성-DAG-및-병렬화.md` §6.2 (13종 — `CTR-PRIM` 포함, §4.1) |
| 워크플로우 형상 `SHAPE-*` | `05-워크플로우-오케스트레이션.md` §1 (5종) |

> 🔴 이 표 밖의 접두사를 이 파일이 발명하면 그 자체가 감사 대상이다. `AG-*`·`CT-*`·`WF-*`는 **폐기된 지역 네임스페이스**이며 각각 `CG-*`·`CTR-*`·`SHAPE-*`로 흡수됐다.

### 1.1 🔴 Wave −2 `BOOT` — 이 레지스트리를 실재하게 만드는 WP 대역 (U-6)

**결함이었다: 이 문서가 "기계 정본"이라 부르는 것을 실제로 존재하게 만드는 WP가 없었다.**

§1은 "병렬 fan-out된 워크플로우는 산문을 읽지 않고 **질의**한다"로 시작한다. 맞다. 그런데 **질의할 대상이 없었다** — 존재하는 것은 §2.5의 예시 레코드 4건뿐이고, 나머지는 전부 이 문서가 *"CI가 검증한다"*고 **선언한 규칙**이다. 규칙과 그 규칙의 실행체를 같은 것으로 취급한 것이 결함이다:

| 이 문서가 선언한 것 | 실제로 존재하는 것 | 그래서 벌어지는 일 |
|---|---|---|
| `registry/traceability.yaml` = **177 WP** 정본 | **예시 4건** | Q1~Q6(§1)이 **전부 답을 못 얻는다.** 질의는 산문 표를 락·스폰·병합·무효화할 수 없다 |
| `CI-01`~`CI-18` = 전부 빌드 실패 | **규칙 문장뿐, `registry/check.py` 없음** | 집행되지 않는 검사는 검사가 아니다 — **경고보다 나쁘다**(존재한다고 믿게 만든다) |
| §4.4 stale 전파 5단계 | **역인덱스 생성기 없음** | 게이트가 뒤집혀도 **후손 집합을 열거할 주체가 없다** |
| §4.2 동결 = 해시 잠금 | **해시 등재소 없음** | `CONTRACT_FROZEN`이 **선언일 뿐 잠금이 아니다** |

**따라서 `BOOT`이 소유·산출한다.** ID 발급처는 여전히 **`02a`**이며(위 정본 경계), 이 표는 인용이다 — 등재 전에는 CI-05b가 유령으로 잡는다:

| WP | 산출 | `owns` | 없으면 |
|---|---|---|---|
| `WP-BOOT-01` | 🔴 **`registry/traceability.yaml` 전량 — 177 WP × `{owns, consumes, produces, gate, stale_on, negative_branch}`** + `registry/schema/traceability.schema.json` | `registry/traceability.yaml`, `registry/schema/**` (`EXCLUSIVE`) | 이 문서 전체가 **산문** |
| `WP-BOOT-02` | WP 매니페스트 스키마 + **인덱스 생성기**(`req_index`/`wp_index`/`glob_index`/`contract_index`/`gate_index`) | `registry/generate/**` (`EXCLUSIVE`), `registry/build/**` (`GENERATED`) | 역인덱스 없음 → §4.4 stale 전파 **1단계에서 정지** |
| `WP-BOOT-03` | 🔴 **CI 검사기 실물 `registry/check.py` — `CI-01`~`CI-18` 전량** | `registry/check.py`, `registry/checks/**` (`EXCLUSIVE`) | 전 불변식이 **집행되지 않는 선언문** |
| `WP-BOOT-04` | **워크플로우 상태 저장소 + 취소 메커니즘.** 🔴 실행클래스별로 갈린다 — `AI-offline` = 현재 스텝 완주 허용 / `AI-on-HW`·`Human-assisted-HW` = **즉시 스케줄러 latch-to-hold 후 취소** | `registry/state/**`, `ops/cancel/**` (`EXCLUSIVE`) | §3.3-2 "미통합 작업 자동 취소"를 **실행할 주체가 없다.** 하드웨어 WP는 취소 중에도 팔이 움직인다 |
| `WP-BOOT-05` | **계약 해시 등재 + 동결 잠금**(`CONTRACT_FROZEN` 집행, `contract_index.json` 해시 등재) | `registry/contracts/**` (`EXCLUSIVE`) | §4.2 동결·§4.3 `@v(n+1)` 규약이 **검증 불가** |

**대역 수용 게이트 (둘 다, 예외 없음):**

| # | 조건 | 판정 |
|---|---|---|
| 1 | **177 WP 전량이 레지스트리에 등재** — `02a`~`02d`의 실재 ID와 1:1, 차집합 ∅ | CI-01·CI-01b·CI-05b가 자기 자신에게 적용 |
| 2 | 🔴 **`CI-01`~`CI-17`이 자기 코퍼스에서 green** | 검사기가 **자기가 검사할 레지스트리 위에서 실제로 통과** |

> 🔴 **미달 = `FAIL_BLOCKING` — 어떤 WP도 착수 불가.** Wave −1 정규화조차 열리지 않는다. 이것은 우선순위가 아니라 **선행 조건**이다: 정규화 원장의 해시를 **등재할 곳이 없으면** 원장은 파일 하나이고, §5의 CI-07(정규화 해시 검사)은 검사할 대상이 없다.
>
> **"자기 코퍼스에서 green"이 조건 2의 핵심이다.** 검사기가 통과한다는 주장은 **검사기를 돌려서만** 성립한다 — 177개 실 레코드 위에서 돌지 않은 검사기는 **문법적으로 존재할 뿐 작동이 증명되지 않았다.** §8의 F-5(ID 공간 드리프트)가 정확히 여기서 시작된다.
>
> **177의 산식 — `171 + 5(BOOT) + 1(WP-3A-00)`.** 감사가 닫은 ID 공간은 **171**이었다(`01` §3.2 대역별 노드 수 — DAG 노드 총계). 이후 두 번의 정식 발급이 더해졌다: `WP-BOOT-01`~`05`(위 표, 발급 = `02a`)와 `WP-3A-00`(`CTR-PRIM@v1` producer, 발급 = `02b` — 배리어 `B-3A.0a`). 🔴 **이 총계는 이 파일이 정하지 않는다** — `02a`~`02d`의 실재 ID를 센 결과이며, 어긋나면 어긋난 쪽이 아니라 **차집합 자체**가 CI-01·CI-01b로 잡힌다. 총계를 손으로 맞추는 것은 검사가 아니다.

> **이것은 무엇을 나쁘게 만드는가.** `BOOT`은 **로봇을 1 mm도 움직이지 않으면서 모든 것을 막는다** — 크리티컬 패스 맨 앞에 스키마·검사기·상태 저장소가 선다. 그리고 부트스트랩이 틀리면 **그 위에 세운 177개가 전부 틀린다.** 그 위험은 D2 파일럿(`00` §3.5b — `WP-0C-02` IK 어댑터를 부트스트랩 직후 1개만 끝까지)이 **표본 1개로** 줄일 뿐 없애지 못한다. 대안은 검사기 없이 fan-out을 시작하는 것이고, 그건 **아무도 안 보는 규칙을 177번 위반하는 것**이다.

## 2. 레지스트리 스키마

### 2.1 저장 위치와 형식

| 항목 | 값 |
|---|---|
| 파일 | `registry/traceability.yaml` (단일 파일, 정렬 강제) |
| 정렬 키 | `req` 오름차순 (`FR` → `NFR`, 영역코드, 3자리) |
| 스키마 검증 | `registry/schema/traceability.schema.json` (JSON Schema draft 2020-12) |
| 생성물 | `registry/build/{req_index,wp_index,glob_index,contract_index,gate_index}.json` — **전부 파생물. 손으로 편집 금지** |

### 2.2 필드 정의

| 필드 | 필수 | 타입 | 규칙 |
|---|---|---|---|
| `req` | ✅ | `FR-<영역>-nnn` \| `NFR-<영역>-nnn` | `docs/spec/`에 실재해야 한다 (§5.1) |
| `spec_ref` | ✅ | `"<문서번호>#<절>"` | 예: `"12#3.1"`. 절이 실재해야 한다 |
| `priority` | ✅ | `M` \| `S` \| `C` | 명세 원본과 일치. 계획이 우선순위를 바꾸지 않는다 |
| `tag` | ✅ | `확정` \| `미확인` \| `결정필요` \| `신규구현` | 명세 §6.4 계승. `결정필요`는 Wave −1 정규화 원장 해시 필수 |
| `normalization` | 조건부 | `sha256:<hex>` \| `null` | `tag ∈ {결정필요}` 또는 SPINE §5 모순 6건에 해당하면 **필수**. 없으면 착수 금지 |
| `wp` | ✅ | `WP-<대역>-nn` \| `WP-G-*` \| `OUT` \| `DEFERRED` | `OUT`은 사유 필수(예: U-1 바이래터럴) |
| `artifact[]` | ✅ | 객체 배열 | `{id, kind, path}`. `kind ∈ {code, schema, config, harness, dataset, report, doc}` |
| `workflow` | ✅ | `SHAPE-<형상>` | `05-워크플로우-오케스트레이션.md` 소유. 형상은 §2.4 |
| `exec_class` | ✅ | `AI-offline` \| `AI-on-HW` \| `Human-assisted-HW` \| `Human-judgment` | SPINE §4 |
| `owns[]` | 조건부 | 객체 배열 | `{glob, mode}`. `mode ∈ {EXCLUSIVE, CONTRACT_FROZEN, SHARED_APPEND, GENERATED}` (§3.1) |
| `contract.consumes[]` | ✅ | `CTR-<이름>@v<n>` | 빈 배열 허용. 버전 미명시 금지 |
| `contract.produces[]` | ✅ | `CTR-<이름>@v<n>` | 한 계약의 producer는 **정확히 1개 WP** (§5.3) |
| `gate[]` | ✅ | `PG-*` \| `CG-*` | **빈 배열 금지** (§5.4). 전역 측정 게이트=`PG-*`, WP 수용 검사=`CG-*`(§2.3) |
| `negative_branch[]` | ✅ | 객체 배열 | 각 `gate[]` 원소마다 1개 이상 (§5.5) |
| `downstream[]` | ✅ | `WP-*` \| `CTR-*` \| `ART-*` | 빈 배열이면 `terminal: true` 명시 필수 |
| `stale_on[]` | ✅ | `<게이트>:<상태>` \| `<계약>:MAJOR_BUMP` | stale 전파 트리거 (§4.4) |
| `targets[]` | 조건부 | 배포 타깃 배열 | SPINE §7에 해당하면 필수. `PG-IK-001`처럼 타깃별 판정 게이트는 반드시 |

**대역 토큰** (`wp` 필드의 `<대역>` 자리 — `02a`~`02d`가 실제로 발급한 것과 1:1. SPINE §5 웨이브명에 대응):

| 토큰 | `BOOT` | `N1` | `ENV` | `0A` | `0B` | `0C` | `OPS` | `1` | `2A`~`2D` | `3A`~`3D` | `4A`~`4C` | `5` | `G` |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
| 웨이브 | **−2 실행 부트스트랩** | −1 명세 정규화 | 0-Env 배리어 | 0-A 척추 | 0-B | 0-C | 0-Ops | 1 순차 서브-DAG | 2 | 3 | 4 | 5 | GUI 교차스트림 |
| 정본 | `02a` | `02a` | `02a` | `02a` | `02a` | `02a` | `02a` | `02a` | `02b` | `02b` | `02c` | `02c` | `02d` |

> `WP-*` ID를 이 레지스트리가 **발급하지 않는다.** 발급처는 `02a`~`02d`이고, CI-01~CI-06은 여기 적힌 ID가 거기 실재함을 전제로 돈다. 실재하지 않으면 CI-05b가 잡는다.
> GUI만 `WP-G-<nn>`(공유 인프라) / `WP-G-S<nn>`(화면별) 형식을 쓴다 — 대역이 웨이브가 아니라 **교차 스트림**이기 때문이며, 배치는 `02d`가 소유한다.

### 2.3 `PG-*`(측정 게이트) vs `CG-*`(WP 수용 검사)

"게이트 없는 WP = 빌드 실패"(§5.4)는 모든 WP를 전역 물리 게이트에 매달라는 뜻이 아니다. 그러면 `AI-offline` WP가 하드웨어 게이트를 기다리는 **가짜 의존**이 생긴다. 그래서 `gate[]`는 두 종류를 받는다 — 그리고 **둘은 같은 것이 아니다.**

| 종류 | 뜻 | 소유 | 상태머신 | 예 |
|---|---|---|---|---|
| `PG-<축>-<3자리>` | **측정 게이트** — 통과 여부가 하류 착수를 바꾸는 물리 판정점 | 계획(SPINE §3, 불변). 정본 = `03` | **5상태 전부** (`PASS` / `RETRY_WITH_VARIANT` / `DEGRADED_ACCEPTED` / `FAIL_BLOCKING` / `SUPERSEDED`) | `PG-RT-001a`/`001b`, `PG-SAFE-001` |
| `CG-<대역>-<nn><letter>` | **WP 수용 검사** — 아티팩트에서 결정론적으로 계산된다. 로봇도 GPU도 시행 횟수도 불필요 | 해당 WP를 정의한 `02a`~`02d` | 🔴 **없음 — `PASS`/`FAIL` 이진**(정본 = `00` §8.0a) | `CG-0C-03a` = "FK↔IK 왕복 잔차가 명시 허용치 이내" |

> 🔴 **`CG-*`는 게이트가 아니다.** `02b` §1의 규칙이 정본이다 — *"`PG-*`에 대응되지 않으면 그것은 수용 기준이지 게이트가 아니다."* 이 레지스트리는 `CG-*`를 **새로 발급하지 않으며**, `02a`~`02d`가 이미 선언한 것을 인용만 한다. `00` §8이 `CG-*` 네임스페이스를 정식 선언한다. 새 `PG-*`를 만드는 것도, 제3의 수용검사 네임스페이스를 만드는 것도 이 문서의 권한 밖이다.

> 🔴 **정정 — 전 회차에 이 파일은 `CG-*`에 5상태를 부여했다. 삭제한다. `00` §8.0a가 정본이다.**
>
> `CG-*`는 **`PASS`/`FAIL` 이진**이며 상태머신도 후손 무효화도 없다. 5상태는 **`PG-*` 게이트만의 것**이다. 왜 이것이 단순한 표기 문제가 아닌가:
>
> | 부여했던 상태 | 무엇이 깨지는가 |
> |---|---|
> | `DEGRADED_ACCEPTED` | 🔴 **WP가 자기 수용선을 자기가 낮추고 통과를 선언한다.** `CG-*`는 그 WP가 정의했으므로 심판과 선수가 같다 — §2.3의 "대가" 항목(느슨한 `CG-*`)이 **CI로 막을 수 없는 구멍에서 합법 경로로 승격**된다 |
> | `FAIL_BLOCKING` | §4.4의 stale 전파가 **발화할 것처럼 보이지만 발화하지 않는다** — `gate_index` 역인덱스는 `PG-*`로만 만들어지며 `CG-*`에는 후손 집합이 없다. 있다고 믿게 만드는 것이 실제 위험이다 |
> | `RETRY_WITH_VARIANT` / `SUPERSEDED` | 재실행은 `CG-*`의 **기본 동작**이다(안 끝났으면 다시 한다). 별도 상태로 이름 붙이면 `PG-*`의 변형 재측정(§2.6)과 혼동된다 |
>
> **CI-05c의 해석**: 상태 커버리지 검사는 **`gate[]`의 `PG-*` 원소에만** 적용된다. `CG-*`에 `negative_branch`를 다는 것은 여전히 필수이지만(CI-05), 그 `on:` 값은 **`FAIL` 하나**다. `CG-*`에 `DEGRADED_ACCEPTED`가 선언되면 CI-05e(§5)가 빌드를 실패시킨다.

> **대가**: `CG-*`를 허용하면 WP가 **자기 수용선을 자기가 정의**한다 — 느슨하게 쓰면 통과가 무의미해진다. 이것은 **수용의 신뢰도를 나쁘게 만든다**. 상쇄는 §5.4b뿐: `CG-*`는 수치 판정선이 아니라 **관측 가능한 산출 증거**를 반드시 명시해야 하며, 증거 아티팩트가 없는 `CG-*`는 CI가 거부한다.

### 2.3b 🔴 `PG-RT-001` 분할 — `PG-RT-001a` / `PG-RT-001b`

**DAG 데드락이었다.** Wave 1 퇴장이 `PG-RT-001` PASS를 요구하는데, `PG-RT-001`의 정본 측정 조건(`15` §2.10 조건4 = **실카메라·실라이터 부하**)은 **Wave 3C 전에는 성립하지 않는다.** 카메라도 기록 라이터도 그때 존재한다. 즉 Wave 1은 **3C를 기다리고 3C는 Wave 1을 기다린다** — 영구 정지.

**게이트를 둘로 쪼갠다. 새 FSM 상태를 만들지 않는다** — 쪼개는 쪽이 싸다.

| | `PG-RT-001a` | `PG-RT-001b` |
|---|---|---|
| 대역 | **Wave 1** (`WP-1-04`) | **Wave 3C** (통합 처리량 — WP 정본 = `02b`) |
| 부하 조건 | **합성 GIL 부하 하네스**(`WP-0C-06`, `15` §2.10 조건4 **대역**) | 🔴 **실카메라·실라이터** = `15` §2.10 조건4 **정본** |
| 판정의 지위 | **잠정** — Wave 1 퇴장을 허가할 뿐 | **최종** |
| 명세 매핑 | `NFR-PRF-054`, `NFR-PRF-055`, `15` §2.10 (**`16` M-8 아님** — CI-10) | 동일 |
| 무엇을 못 하는가 | 🔴 **실부하를 대표하지 못한다.** 합성 부하는 GIL 경합을 흉내낼 뿐 USB 대역·인코더 스레드·디스크 flush를 만들지 않는다 | — |

**a와 b의 관계 — 이것이 분할의 핵심이다:**

```
PG-RT-001a: PASS (잠정)  ──> Wave 1 퇴장 허가 ──> 2A~2D, 3A, 3B 진행
                                                          │
                                                          ▼
                                            PG-RT-001b 측정 (실카메라·실라이터)
                                                          │
                          ┌───────────────────────────────┴──────────────┐
                          ▼                                              ▼
              b가 a를 확인 → a 유지                    🔴 b가 a를 뒤집음 → a = SUPERSEDED
                                                              │
                                        ART-F-MAX-PYTHON stale → 마찰동정 로깅 대역·GMO 임계 재도출
```

> 🔴 **`a`의 PASS는 `b`가 무효화할 수 있다(`SUPERSEDED`).** 이것은 분할의 결함이 아니라 **분할이 정직한 지점**이다 — 합성 부하 위에서 낸 판정을 최종이라 부르면 그게 날조다. `a`를 소비하는 산출(`f_max_python`, 마찰동정 로깅 대역, GMO 임계)은 전부 **잠정 표시**를 달고, `b` 확정 시 §4.4 stale 전파를 받는다(`stale_on: PG-RT-001b:PASS`).
>
> **왜 새 FSM 상태(`PROVISIONAL_PASS` 같은 것)를 만들지 않는가.** 상태를 늘리면 **14개 게이트 전부의 상태머신이 바뀌고**, §3.3의 4종 자동 동작이 새 상태마다 정의되어야 하며, CI-05c의 커버리지 검사가 전 게이트에서 다시 계산된다. 게이트 하나를 둘로 쪼개는 것은 **ID 두 개와 `SUPERSEDED` 간선 하나**로 끝난다. 기존 기계가 이미 표현할 수 있는 것에 새 개념을 만들지 않는다.

> **레지스트리상 귀결**: `PG-RT-001`(접미사 없는 ID)은 **더 이상 유효한 게이트 ID가 아니다** — 단, 그 무효는 **ID가 `값`으로 오는 자리에서만** 성립한다. `gate[]`에 나타나면 `03`에 실재하지 않으므로 CI-11b가 잡는다. 후손 인덱스(§7)도 a/b로 갈린다.
>
> 🔴 **그러나 `PG-RT-001`을 산문에서 쓰는 것은 위반이 아니다.** 이것은 `M-8` 봉인(CI-10)이 이미 배운 함정이며, **같은 처방을 쓴다**: 봉인은 **필드 값**에 걸리지 **텍스트**에 걸리지 않는다. `00`·`04`는 `PG-RT-001`을 **계열(family) 지칭**으로 정당하게 쓴다 — *"제어루프 측정은 반드시 `PG-RT-001`"*(`16` M-8이 아니라는 고지), *"`PG-RT-001` 결과와 무관하게 유효하다"*, 실행클래스 예시, 축약어 표(`RT` → `PG-RT-001`). 이들은 **a/b 중 어느 쪽도 지목하지 않아야 옳은 자리**다: 분할 이전을 서술하거나, a와 b 양쪽에 동시에 해당하거나, 게이트 축 자체를 부르는 것이기 때문이다. 전역 `\bPG-RT-001\b` 금지는 **이 정당한 18여 개 문장을 빌드 실패로 만들고**, 분할을 설명하는 이 문단이 분할 규칙 자신에게 걸리는 재귀를 만든다. 범위와 예외는 CI-11b(§5)가 못 박는다.

### 2.4 워크플로우 형상 (`workflow`)

**정본은 `05` §1의 5종이며, 이 문서는 그것을 인용만 한다.** 형상 이름을 이 파일이 짓지 않는다.

| 값 | 뜻 | 이 문서에서의 제약 |
|---|---|---|
| `SHAPE-CF` | 계약동결 · 단일 소유(폭 = 항상 1) | 같은 심볼을 만지는 WP 집합 전체가 이 형상. 계약 파일 + 그것이 정의하는 심볼 전량을 소유 |
| `SHAPE-IM` | 구현 — 픽스처 대상별 fan-out(폭 = N) | `owns[].mode = EXCLUSIVE`인 glob이 서로 **교집합 0**이어야 함 |
| `SHAPE-IG` | 통합 배리어(폭 = 1) | 동시 실행 WP 수 = 0을 CI가 확인 |
| `SHAPE-MS` | 측정 — 읽고 잰다. 물리 조작 없음(자원 배타) | `owns[]` 비어야 함(쓰기 금지) |
| `SHAPE-HG` | Human 게이트 — 블록 후 증거 수신 시 재개 | SPINE §4의 6항목(에이전트/사람/중단/초기상태/증거/재개) 전부 필수 |

### 2.6 🔴 재시도는 상태이지 WP가 아니다

**게이트가 뒤집혔을 때 "같은 일을 다르게 다시 한다"는 `RETRY_WITH_VARIANT` 상태가 표현한다.** 이 레지스트리는 그것을 위해 `WP-<원본>R` 같은 ID를 **발급하지 않는다.** 발급했다면 그것은 `02a`~`02d` 어디에도 정의가 없는 유령이고, CI-05b가 잡는다.

| 상황 | 무엇이 표현하는가 | 새 WP ID가 생기는가 |
|---|---|---|
| 같은 WP를 조건·변형을 바꿔 재실행 | `RETRY_WITH_VARIANT` (게이트 상태) | ❌ **아니다** |
| 게이트가 `DEGRADED_ACCEPTED` → 축소 경로 채택 | 상태 + `negative_branch[].action` | ❌ 아니다 (기존 WP의 산출이 축소될 뿐) |
| 게이트 `FAIL_BLOCKING` → **다른 접근**이 필요 | `negative_branch[].spawns` | ✅ **그렇다** — 단, `02a`~`02d`에 실재하는 ID여야 한다(CI-05b) |
| 계약 MAJOR bump → 소비자 이관 | §4.3 절차 | ✅ 그렇다 (`WP-<원본>M2`, `02a`~`02d` 등재 후) |

경계는 **"재실행인가, 다른 작업인가"**다. 재실행이면 상태로 족하고, ID를 늘리면 DAG에 존재하지 않는 노드가 생겨 `downstream` 대조(§5.6)가 조용히 새 나간다.

## 3. 파일/모듈 소유권

### 3.1 소유 모드

| 모드 | 동시 편집 | 규칙 |
|---|---|---|
| `EXCLUSIVE` | ❌ | 한 시점에 **정확히 1개 WP**. 교집합 glob이 있으면 빌드 실패 |

전 회차에 이 파일은 `CT-SCHEDULER-MAILBOX`·`CT-RT-BUDGET`·`CT-POLICY-COMPAT`를 **발명했다.** 셋 다 `01` §6.2에 없다. 정정:

### 4.3 변경은 새 웨이브를 연다

**동결된 계약은 편집되지 않는다. 새 버전이 발행되고, 그 발행이 웨이브다.**

**버전 축은 `01` §6.3 `CR-3`가 정본이다 — 변경은 `CTR-x@v(n+1)` 발행이다.** `@v<n>`이 동결 세대이며, 그 아래 semver 자릿수는 없다.

| 변경 종류 | 버전 | 하류에 무슨 일이 | 새 웨이브 |
|---|---|---|---|
| 문서·주석·예시만 (**필드 불변** — 스키마 해시 불변) | `@v1` **유지** | 없음 | ❌ (동일 웨이브 내 처리) |
| 🔴 **필드 추가 — 선택 필드 포함** | **`@v1 → @v2`** | ↓ MAJOR와 동일 | ✅ **새 배치를 연다** |
| 필드 제거·의미 변경·필수화 | `@v1 → @v2` | 전 소비자 WP가 `stale_on: CTR-*:MAJOR_BUMP`로 stale, 미통합 작업 취소, 소비자마다 **명명된 대체 WP** 생성 | ✅ **새 배치를 연다** (`01` `CR-3`) |

> 🔴 **정정 — 전 회차에 이 표는 "선택 필드 추가는 `@v1` 유지 + 해시 갱신"을 허용했다. 삭제한다. `CR-2`가 이긴다.**
>
> **자기모순이었다.** `CR-2`는 *"소비 중인 계약은 변경하지 않는다"*고 규정하는데, 이 예외는 **같은 `@v1`이 서로 다른 스키마를 가리키는 것**을 합법화했다. 동결(§4.2)의 정의가 *"동결 후 스키마 파일의 내용 해시 변경 = 빌드 실패"*인데, 예외는 **그 해시를 갱신하라고 지시**한다 — 같은 문서의 두 규칙이 정반대를 말했다.
>
> **무엇이 실제로 깨지는가:**
>
> | | 예외가 있었을 때 |
> |---|---|
> | `@v1`의 의미 | 🔴 **시점에 따라 다른 스키마.** "`CTR-WS@v1`을 소비한다"가 **무엇을 소비하는지 특정하지 못한다** |
> | CI-08 (버전 고정) | `^`/`~`/`latest`를 금지해 놓고 **`@v1` 자체가 movable target**이 된다 — 금지의 목적이 소멸 |
> | CI-09 (동결 위반) | 해시 대조가 **무엇과 대조하는지 모른다.** 갱신이 합법이면 불일치가 위반이 아니다 |
> | 병렬 fan-out | 계약 = **두 워크플로우 사이의 뮤텍스**(§4.1b). 스폰 시점에 읽은 스키마와 착지 시점의 스키마가 달라도 **CI가 침묵한다** |
>
> **동결 후 어떤 필드 추가도 `@v(n+1)`이다.** 선택이든 필수든 구분하지 않는다 — 구분선이 존재하는 순간 "이건 선택 필드니까"가 동결을 우회하는 상시 경로가 된다.
>
> **동결 전에는 자유롭다.** 이 규칙은 `CONTRACT_FROZEN` 잠금 **이후**에만 적용된다. 필드를 늘릴 것 같으면 **동결을 늦추는 것이 옳고**, 그 대가(초기 병렬도 저하)는 §4.3 말미가 이미 인정한 것이다 — **이제 그 셋 중 어느 것이 나쁜지가 확정됐다.**

MAJOR 발행 절차 (CI가 강제):
1. 새 계약 ID를 `CTR-<이름>@v2`로 등재. **`@v1`은 삭제하지 않고 `SUPERSEDED`로 표시**한다.
2. `contract_index` 역인덱스로 소비자 WP 전량 열거 → 각각 `WP-<원본>M2` 대체 WP 생성 후 **`02a`~`02d`에 등재**(등재 전에는 CI-05b가 유령으로 잡는다).
3. 대체 WP가 전부 착지하기 전에는 `@v2`를 `active` 표시하지 않는다. `@v1` 소비 경로는 그동안 살아 있다.
4. 전량 착지 후 `@v1`을 `RETIRED`로 내린다. `RETIRED` 계약을 소비하면 빌드 실패.

> **대가**: 이 규약은 **계약을 고치는 일을 무겁게 만든다** — 필드 하나 **추가**하는 데 배치가 열린다. 그래서 **동결이 늦어질 유인**이 생기고, 동결이 늦으면 fan-out이 시작을 못 한다. 이것은 **초기 병렬도를 나쁘게 만든다.**
>
> 🔴 **그리고 이제 완화가 없다.** 전 회차의 완화(선택 필드는 `@v` 유지)는 위 정정으로 **삭제됐다** — 그것은 완화가 아니라 동결을 우회하는 구멍이었기 때문이다. 셋 중 하나는 반드시 나쁘고, **우리는 "초기 병렬도가 나쁜 쪽"을 골랐다.** 이유는 비대칭이다: 늦은 동결의 대가는 **기다림**이고, movable `@v1`의 대가는 **두 워크플로우가 서로 다른 스키마를 같은 이름으로 부르며 CI가 침묵하는 것**이다. 기다림은 관측되지만 침묵은 관측되지 않는다.
>
> **올바른 대응은 동결 시점을 옳게 고르는 것**이지 동결의 의미를 무르게 만드는 것이 아니다.

### 4.4 stale 전파

게이트 상태가 `PASS` → 다른 상태로 바뀌거나, 계약이 MAJOR bump 되면:

| 단계 | 동작 | 검증 가능한 산출 |
|---|---|---|
| 1 | `gate_index` / `contract_index` 역인덱스로 후손 전량 열거 | `registry/build/stale_set.json`에 `{trigger, 후손 WP 목록, 후손 아티팩트 목록}` |
| 2 | 후손 아티팩트에 `stale: true` + `stale_reason` 스탬프 | 각 아티팩트 경로에 대응하는 스탬프 레코드 존재 |
| 3 | 미통합 작업 취소 | 해당 WP의 열린 worktree 목록이 빈 집합 |
| 4 | **명명된 대체 WP 생성** (재실행이 아니라 다른 접근일 때만 — §2.6) | `negative_branch[].spawns`의 ID가 `02a`/`02b`/`02c`/`02d` 중 하나에 실재 |
| 5 | 리스크 등록부 갱신 | `04-리스크-및-안전-브링업-게이트.md` 항목 링크 (`R-nn` = 리스크 ID. CI 규칙 `CI-nn`과 다른 축이다) |

> 🔴 **1~5 중 하나라도 빠지면 그건 "리스크 등록부에 한 줄 적고 넘어가기"다**(SPINE §3). CI는 게이트 상태 변경 커밋에서 `stale_set.json`이 갱신되지 않으면 빌드를 실패시킨다.

### 5.6 그래프 대조 검사 (CI-16)

§1의 "두 번째 진실" 대가를 상쇄하는 유일한 검사.

| 항목 | 방법 | 실패 |
|---|---|---|
| `downstream` 누락 | 정적 import/참조 그래프에서 `artifact[].path`의 실제 피참조자 추출 → `downstream`의 아티팩트 집합과 대조 | 실제 참조가 `downstream`에 없음 |
| `downstream` 과잉 | 반대 방향 | 선언은 됐으나 참조 없음 → `justification` 필드 필수(예: 런타임 동적 로드) |

> **대가**: CI-16은 **정적 그래프가 볼 수 없는 의존을 못 본다** — WS 봉투를 문자열 키로 소비하는 프런트엔드, YAML로만 이어진 udev↔프로파일, 데이터셋 스키마↔학습 정규화 통계. 이것은 **CI-16의 커버리지를 구조적으로 불완전하게 만든다**. 그 구멍은 `CONTRACT_FROZEN` + CI-09(해시 동결)로만 부분 상쇄되며, 나머지는 게이트가 잡거나 아무도 못 잡는다.

## 8. 이 레지스트리의 실패 모드

정직하게: 아래 넷은 이 설계가 **막지 못하는 것**이다.

| # | 실패 | 왜 못 막는가 | 무엇이 대신 잡는가 |
|---|---|---|---|
| F-1 | **의미 중복** — 두 `OA-*` 코드가 같은 사건을 다르게 부름 | `SHARED_APPEND`는 문자열만 본다(§3.2) | 사람. CI 아님 |
| F-2 | **느슨한 `CG-*`** — WP가 자기 수용선을 통과하기 쉽게 씀 | 판정선을 CI가 평가할 수 없다 | CI-04b(증거 강제)까지. 그 이상은 수용 리뷰 |
| F-3 | **동적 의존 누락** — 문자열 키·YAML·정규화 통계로 이어진 참조 | 정적 그래프가 못 본다(§5.6) | `CONTRACT_FROZEN` + CI-09. 부분적 |
| F-4 | **레지스트리 드리프트** — 파일 이동이 glob을 깨고, 급해서 glob을 넓힘 | 넓힌 glob은 문법적으로 합법이다 | CI-02b(고아 파일). 넓은 glob 자체는 못 잡는다 |
| F-5 | **ID 공간 드리프트** — 이 파일이 잇는 ID가 `02a`~`02d`·`01`·`03`에서 조용히 이름을 바꿈 | 이 레지스트리는 **두 번째 진실**이고, 정본이 움직이면 여기가 먼저 틀린다 | CI-01b·CI-03c·CI-04c·CI-05b·CI-17(전부 실재 검사). 🔴 **접두사를 이 파일이 발명하는 순간 이 검사들이 전부 무력해진다** — 발명된 ID는 어디에도 실재하지 않으므로 대조할 정본이 없다 |

> **F-4가 가장 위험하다.** `owns: {glob: "backend/**", mode: EXCLUSIVE}`는 CI-02를 통과하면서 §3.2의 단일 소유자 보장을 통째로 무의미하게 만든다. 이것을 막는 유일한 검사는 **glob 폭 상한**(전개 파일 수 상한)인데, 그건 정당하게 넓은 소유(예: `contracts/**` 동결)를 오탐한다 — 즉 **정당한 광역 소유를 나쁘게 만든다**. 그래서 이 문서는 폭 상한을 **두지 않는다.** 대신 `EXCLUSIVE` glob의 전개 파일 수를 `registry/build/glob_index.json`에 기록해 **증가를 관측 가능하게만** 만든다. 관측은 강제가 아니다. 이 구멍은 열려 있다.
