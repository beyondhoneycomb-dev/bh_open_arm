# 02d. 작업 패키지 — GUI (교차 스트림)

`01-의존성-DAG-및-병렬화.md`가 선언·DAG 배치한 **18개 GUI WP**(`WP-G-00`~`WP-G-04` + `WP-G-S01`~`WP-G-S13`)를 {입력·산출물·인터페이스 계약·수용 게이트·실행 클래스·워크플로우 형상}으로 전수 명세한다. **이 문서가 `WP-G-*` 네임스페이스의 유일 카탈로그다.**

GUI는 **웨이브가 아니라 교차 스트림**이다. 종단 대역이 없으므로 02a/02b/02c처럼 대역별로 자르지 않고, **공유 기반(§1)** 과 **화면(§2)** 으로 자른다.

---

## 0. 이 문서를 읽는 법

### 0.1 형식 (02a/02b/02c와 동일)

각 절은 **표 2개**로 기술한다. 산문 서술은 배치 규칙·대가 진술에만 허용한다.

| 표 | 열 |
|---|---|
| **(A) WP 정의** | `WP` · 무엇 · 입력 · 산출 · 실행클래스 · 워크플로우 형상 |
| **(B) 계약·게이트** | `WP` · 동결 인터페이스 계약 · 수용 검사(검증가능) · 음성 분기 · 명세 근거 |

- **WP ID**: `WP-G-nn`(공유 기반) / `WP-G-Snn`(화면). 불변.
- **게이트**: `docs/plan/03-측정-게이트.md` §3의 `PG-*` 전수표가 유일 정본이며 **본 문서는 `PG-*`를 신설하지 않는다.** `PG-*`에 대응하지 않는 검사는 **`CG-*` WP 수용 검사**이지 게이트가 아니다(`00` §8).
- 🔴 **`CG-*`는 상태머신을 갖지 않는다 — `PASS`/`FAIL` 이진이다**(`00` §8.0a 정본). 5상태(`RETRY_WITH_VARIANT`·`DEGRADED_ACCEPTED`·`FAIL_BLOCKING`·`SUPERSEDED`)는 **`PG-*` 게이트만의 것**이다. 특히 **`CG-*`에 `DEGRADED_ACCEPTED`는 없다** — WP가 자기 수용선을 자기가 낮추는 것이므로 심판과 선수가 같다. `06`의 CI-05e가 이것을 빌드 실패로 잡는다.
  - **음성 분기 칸을 읽는 법**: `①실패`처럼 번호로 지시되는 것은 `CG-*`이므로 판정은 `FAIL` 이진이고, 뒤따르는 서술은 **상태 부여가 아니라 재작업 방향**이다. `FAIL`은 "이 WP를 아직 못 끝냈다"는 뜻일 뿐이며 후손 무효화를 발화시키지 않는다. 5상태 이름이 붙는 것은 **`PG-*`가 주어일 때뿐**이다(예: `PG-CAM-001` `DEGRADED_ACCEPTED` → 축소, `PG-VR-001` `RETRY_WITH_VARIANT` → WebXR 폴백). `CG-*`가 `FAIL`인데 원인이 "다른 접근이 필요하다"라면 그것은 상태 문제가 아니라 **WP 설계가 틀린 것**이며, 대체 WP는 `PG-*`의 `negative_branch`나 웨이브 재배치(`01`)가 낳는다.
- **계약 ID**: `CTR-<NAME>@v<n>`(`01` §6.2 레지스트리).
- **워크플로우 형상** (**어휘** = `05` §1.1 정본 · 🔴 **배정** = **이 문서 소유**, `00` §9.2a): `SHAPE-CF` 계약동결(1, 병렬 금지) / `SHAPE-IM` 구현(N = 배타 쓰기 단위 수) / `SHAPE-IG` 통합 배리어(1) / `SHAPE-MS` 측정(1, 자원 배타) / `SHAPE-HG` Human 게이트(0, 블록).
- **실행 클래스**(SPINE §4): `AI-offline` / `AI-on-HW` / `Human-assisted-HW` / `Human-judgment`.
- 구현자는 **Claude dynamic workflow 병렬 fan-out**이다. 워크플로우 형상은 그 fan-out 배치를 뜻한다.
- 표기 규약은 명세를 계승한다: `FR-<영역>-nnn` · 우선 `M/S/C` · `[확정]`/`[미확인]`/`[결정필요]`/`[신규구현]`.

### 0.2 이 문서가 소유하는 것 / 소유하지 않는 것

| | |
|---|---|
| **소유한다** | `WP-G-*` 18개의 정의 · 화면↔백엔드 WP 결선 · GUI 수용 검사 `CG-G-*` |
| **소유하지 않는다** | **화면 인벤토리 정본 = `13` §2.6**(13행). 계획은 그 13행에 웨이브를 붙일 뿐 화면을 추가·삭제하지 않는다 · **도메인 로직 정본**(조그 산식·IK·임계·정책·데이터셋 스키마)은 각 영역 문서가 소유한다 · **카메라 하드웨어 정본 = `06`** · **포트맵 정본 = `01` §2.17 + `14` §2.1** |

> 🔴 **모든 화면 WP의 제1 규율: 화면은 창구이지 소유자가 아니다.** 화면이 백엔드의 검증·클램프·집계를 재구현하면 **두 개의 진실**이 생기고, 그 순간 화면은 조용히 틀린다. 이 규율의 위반은 전부 **`CG-*` `FAIL`(정본 이중화)** 이며, 고쳐서 다시 돌린다 — `CG-*`는 `PASS`/`FAIL` 이진이다(`00` §8.0a).

### 0.3 🔴 배치의 단일 규칙 — Wave 3B 이전에 GUI는 존재하지 않는다

**`CTR-WS@v1`(WS 봉투, 소유 `WP-3A-04`)이 Wave 3A에서 동결되고, GUI는 그 이후에만 소비한다.** 따라서 **어떤 `WP-G-*`도 Wave 3B 이전에 착수될 수 없다.** 예외 없음.

> **이것은 편의가 아니라 안전 판정이다.** 단일 WS 멀티플렉싱은 RFC6455상 프레임이 인터리빙되지 않으므로 **head-of-line 블로킹이 구조적으로 실재**한다(`FR-GUI-040`·`FR-GUI-042`). 즉 **브라우저 소프트 E-Stop은 HOL만큼 늦을 수 있다.** 그러므로 Wave 1·2에서 팔이 처음 움직일 때의 개입 수단은 **하네스 + 전원라인 물리 E-Stop 버튼**이지 GUI가 아니다(`16` §4 `M-2`, `NFR-SAF-009`). GUI를 앞당기는 것은 개입 수단을 앞당기는 것이 아니라, **개입 수단을 HOL 뒤로 숨기는 것**이다.
>
> 두 번째 근거: **미동결 계약을 소비하는 병렬 구현은 재작업으로 귀결된다**(`01` §6.3 `CR-2` 소비 중 변경 금지). WS 스키마 "초안"에 화면을 붙이면 동결본과의 차이가 화면 전량에 번진다.

**GUI = 교차 스트림, 두 갈래 배치:**

| 갈래 | 대상 | 배치 |
|---|---|---|
| **공유 기반** | `WP-G-00`~`WP-G-04`(앱 셸 · WS 클라이언트 · 3D 뷰포트 · 상시 요소 · 모드/제어권) | `CTR-WS@v1` 동결 **직후 = Wave 3B**에 일괄 착지 |
| **개별 화면** | `WP-G-S01`~`WP-G-S13` | **per-backend fan-out.** 각자의 백엔드 WP가 착지할 때 갈라진다 |

**화면 WP의 진입 조건은 단일 식이다:**

```
진입(WP-G-Sxx) = max( BOOT 통과(Wave −2),  CTR-WS@v1 동결(Wave 3A 종료),  해당 백엔드 WP 전량 착지 )
```

이 식이 §2.0 배치표의 **유일한 산출 근거**다. 배치표의 웨이브 칸은 선언이 아니라 이 식의 **결과**다. `BOOT` 항은 §0.3.1이 정의한다 — 실무적으로 `CTR-WS@v1` 동결이 항상 그보다 뒤이므로 `max`를 지배하지 않지만, **식에 없으면 없는 전제가 된다.**

### 0.3.1 🔴 Wave −2 `BOOT` — `WP-G-*` 18개 전부의 절대 선행

`CTR-WS@v1` 동결도, `SHAPE-IM` fan-out도, `CG-G-*`의 `PASS`/`FAIL` 판정도 **그것을 등재·질의·잠글 기계가 있어야** 성립한다. 그 기계를 짓는 것이 신설 대역 **`BOOT`**(`00` §3.5 · `05` §0.1 · `06` §1.1, `WP-BOOT-01`~`WP-BOOT-05`)이며, **Wave −1보다도 앞선다.**

| `BOOT` 산출 | GUI가 그것에 의존하는 지점 |
|---|---|
| `registry/traceability.yaml` 전량 + JSON Schema | §2.0 배치표의 **백엔드 결선**(`06`:654·660 정정(실제 오류 위치 = 레지스트리)을 포함한 화면↔백엔드 간선이 여기서 질의 가능해진다) |
| WP 매니페스트 스키마 + 인덱스 생성기 | `WP-G-*` 18개 등재 |
| **CI-01~CI-18 검사기 실물** | §0.1의 `CG-*` 이진 강제(CI-05e) · `PG-*` 미신설 강제 |
| 워크플로우 상태 저장소 + **취소 메커니즘** | `SHAPE-IG`(`WP-G-S01` 배리어) · per-backend fan-out의 재배치 |
| 계약 해시 등재/**동결 잠금**(`CONTRACT_FROZEN`) | 🔴 **§0.3의 두 번째 근거를 실제로 집행하는 것이 이것이다** — `CR-2`("소비 중 변경 금지")는 규칙이지만 잠금은 기계다. 잠금이 없으면 `CTR-WS@v1`의 "동결"은 **선언일 뿐**이고, 화면 13개가 초안 위에 서게 된다 |

> 🔴 **`BOOT` 수용 = 177 WP 전량 등재 + CI-01~17이 자기 코퍼스에서 green.** 미달 = `FAIL_BLOCKING`이며, 그 상태에서는 **어떤 `WP-G-*`도 착수 불가**다. §0.3이 "Wave 3B 이전에 GUI는 존재하지 않는다"고 말할 때, 그 문장은 **`BOOT` 통과를 이미 가정하고 있다.**
>
> **대가**: GUI는 이 계획에서 가장 늦게 착지하는 대역인데, 그 위에 **가장 이른 배리어**가 하나 더 얹힌다. 그러나 GUI가 `BOOT`으로 얻는 것은 배리어 하나가 아니라 **동결의 실재성**이다 — §1.3이 "화면 재작업 리스크를 계약 동결 품질에 전부 건다"고 적은 그 베팅은, 동결을 잠글 기계가 없으면 **애초에 성립하지 않는 베팅**이었다.

## 1. 공유 기반 — `WP-G-00` ~ `WP-G-04` (Wave 3B, `CTR-WS@v1` 동결 직후)

### 1.0 이 대역의 성격

5개 WP는 **13화면 전부의 전제**이며, 서로 파일 소유권이 분리된다(셸 라우팅 / WS 클라이언트 / 3D 컴포넌트 / 상시 요소 / 모드·제어권). 따라서 **`SHAPE-IM` fan-out(5)** 이 성립한다 — 계약(`CTR-WS@v1`)이 이미 동결되어 있고, 각자 배타 쓰기 단위를 갖기 때문이다.

`WP-G-02`·`WP-G-03`·`WP-G-04`는 **안전 표면**이다. 나머지 화면이 아무리 늦어도 이 셋은 3B에 선다 — 화면이 없어서 못 보는 것과, 화면이 있는데 틀리게 보이는 것은 다른 사고다.

### 2.3 이 대역이 남기는 대가

| 항목 | 대가 |
|---|---|
| **per-backend fan-out** | 화면이 **13번에 나눠 착지한다.** 앱이 "완성된 하나"로 보이는 시점이 마지막까지 없고, 화면 간 일관성(용어·단위 표기·배지 위치)이 착지 순서대로 표류한다. 이것을 잡는 것은 화면 10개를 만드는 일이 아니라 **흩어져 착지한 13화면이 하나의 앱인지 확인하는 일**이며, Wave 5의 완성 감사가 그 몫이다 |
| **늦은 UI 피드백** | 화면을 백엔드에 붙여 늦게 fan-out하면 **폼·조작 설계의 문제가 계약이 굳은 뒤에 발견된다.** 반대로 먼저 만들면 계약이 흔들려 다시 만든다. 계획은 후자를 택했고, 그 대가는 **폼 재작업 리스크를 계약 동결 품질에 건다**는 것이다 |
| **`WP-G-S01` 배리어** | 대시보드가 `SHAPE-IG`라서 **전 백엔드가 착지할 때까지 열리지 않는다.** 즉 "시스템 전체가 지금 어떤가"를 한 화면에서 보는 능력이 **가장 마지막에** 생긴다 — 정확히 그것이 가장 필요한 통합 구간에 |
| **화면 = 창구 (전 화면)** | 백엔드가 값을 안 주면 화면은 **빈칸이고, 빈칸을 채울 권한이 없다.** 임시 계산이라는 저렴한 우회가 전면 금지된다 |
| **`WP-G-S11` 2회 착지** | 추론 화면이 4B에 열리고 성공률 패널이 4C에 붙는다 → 그 사이 구간에 **평가는 도는데 통계가 없다.** 점추정으로 메우는 것은 `CG-G-S11b` 위반이므로, 그 구간에는 **아무 숫자도 표시하지 않는다** |

## 3. 이 문서가 남기는 전역 대가

| 항목 | 대가 |
|---|---|
| **GUI 18 WP가 전부 `AI-offline`** | 화면은 픽스처 위에서만 검증된다(단 `WP-G-S13`의 읽기 계측만 `AI-on-HW`). **픽스처가 실기 분포를 잘못 흉내내면 화면은 그린인데 실기에서 틀린다.** 이것을 막는 것은 화면의 몫이 아니라 각 백엔드 WP의 픽스처 품질이며, GUI는 그 품질에 전부를 건다 |
