# 측정 게이트 — 전수 명세

계획이 소유하는 게이트 15종의 정의·상태머신·음성분기·후손 무효화 규칙. **명세의 깨진 게이트 ID(`M-8`, `M-25`)를 재사용하지 않고 `PG-*`로 대체하는 근거를 포함한다.**

> 🔴 **`PG-RT-001`은 더 이상 게이트 ID가 아니다.** DAG 데드락(§5.1.0)을 해소하기 위해 **`PG-RT-001a`(Wave 1, 잠정) / `PG-RT-001b`(Wave 3C, 최종 정본)** 둘로 쪼개졌다. 계획 어디에서도 맨 `PG-RT-001`을 판정 근거로 인용하지 않는다 — CI가 게이트 선언·참조 위치에서 이를 빌드 실패로 막는다(§3).

---

## 1. 게이트란 무엇인가 — 읽는 법

게이트는 **테스트가 아니다.** 테스트는 "구현이 명세를 만족하는가"를 묻고, 게이트는 **"어느 아키텍처를 지을 것인가"** 를 묻는다. 게이트는 측정 결과가 하류 설계를 **갈라놓을 때에만** 게이트다. 갈라놓지 않으면 그것은 회귀 테스트이지 게이트가 아니다(→ `PG-CAN-001`이 이 이유로 강등된다, §5.2).

| 규약 | 내용 |
|---|---|
| ID | `PG-<영역>-nnn`. **불변.** 계획이 소유하며 명세 ID와 1:1 대응하지 않는다 |
| 상태 | `PASS` / `RETRY_WITH_VARIANT` / `DEGRADED_ACCEPTED` / `FAIL_BLOCKING` / `SUPERSEDED` (§2) |
| 실행 클래스 | `AI-offline` / `AI-on-HW` / `Human-assisted-HW` / `Human-judgment` (스핀 §4) |
| 명세 매핑 | 실재하는 FR/NFR ID + `문서:절`. **깨진 ID는 §4에서 격리한다** |
| 우선 | 명세 규약 계승 — M(필수) / S(권장) / C(선택) |
| 상태 태그 | 명세 규약 계승 — `[확정]` / `[미확인]` / `[결정필요]` / `[신규구현]` |

**측정 선행 규율(스핀 §2-6).** 게이트 통과 전에는 하류 주파수·지연 수치를 못 박지 않는다. 이 문서가 인용하는 수치는 **명세가 이미 판정선으로 명기한 것뿐**이며(`NFR-PRF-054`의 오버런율 ≤ 0.1%, `NFR-CAM-005`의 RTF ≤ 1.0, `NFR-CAM-002`의 q99 ≤ 16.7 ms 등), **`[미확인]` 목표치는 판정선으로 승격하지 않는다**(대표 사례: `NFR-MAN-002`의 20 ms → §5.7).

### 🔴 1.1 원칙 단서 — 측정은 실현가능성을 정하지 안전을 정의하지 않는다

**측정 선행 규율은 *성능 목표*에 적용되며 *안전 한계*에는 적용되지 않는다.** 이 둘을 섞으면 게이트가 조용히 뒤집힌다 — "우리가 잴 수 있는 것"이 "우리가 허용해도 되는 것"으로 승격되기 때문이다.

| 축 | 어디서 나오는가 | 리그가 하는 일 |
|---|---|---|
| **성능 목표** (루프 주파수·IK p99·RTF·fps) | **측정이 정한다.** 측정 전에는 못 박지 않는다 | 값을 **낳는다** |
| 🔴 **안전 한계** (허용 정지시간·운영 속도·워크스페이스 경계) | **측정이 정하지 않는다.** **정지거리·에너지·도달 워크스페이스·수동 구속**에서 **먼저 도출**한다 | 도출된 기준에 **대고 잰다** — 리그는 그 기준을 만족하는지 답할 뿐 기준을 고르지 않는다 |

**따라서 안전 축의 게이트는 두 단계다: ① 관측과 무관하게 한계를 도출한다 ② 리그가 그 한계 안에 있는지 검증한다.** 게이트가 ①을 건너뛰고 "실측 최대치"나 "우리가 관측 가능한 최소 지연"을 한계로 채택하면, 그것은 **관측 능력을 안전 근거로 위장한 것**이다.

- `PG-VEL-001`(§5.6)은 이 규율을 따른다 — 리미터를 **먼저 도출**하고 스윕은 그 아래에서 돈다. **실측 최대치를 한계로 채택하지 않는다.**
- `PG-STOP-001`(§5.7)의 허용 정지시간도 정지거리·낙하영역 기하에서 도출한다 — **측정된 P99가 판정선을 정의하지 않는다.** 측정은 "도출된 한계 안에 드는가"에만 답한다.
- `PG-SAFE-001`(§5.5)은 애초에 소프트웨어를 재지 않는다 — 물리적 형상이 기준이다.

> **반례로 읽어야 하는 문장:** *"측정해 보니 P99가 35 ms였으므로 판정선을 35 ms로 한다"* — 이것은 게이트가 아니라 **측정치의 자기 승인**이다. CI가 안전 축 게이트의 판정선 필드에 **도출 근거 URI**를 요구하며, 그 URI가 자기 결과 레코드를 가리키면 거부한다.

## 2. 게이트 상태머신 (15종 공통)

### 2.1 상태 정의

| 상태 | 뜻 | 후손 처리 | 등록부 |
|---|---|---|---|
| `PASS` | 판정선 충족. 측정 조건이 **정본 조건**과 일치 | 후손 착수 허용 | 결과 레코드 필수 |
| `RETRY_WITH_VARIANT` | 미달이나 **명명된 변형(variant)이 남아 있다.** 변형은 게이트별로 사전 열거되며 발명 금지 | 후손 **보류**(stale 아님 — 아직 산출물이 없다) | 변형 ID + 재실행 조건 기록 |
| `DEGRADED_ACCEPTED` | 미달이나 **축소된 능력으로 수용.** 반드시 **무엇이 꺼지는가**를 명명 | 후손 중 **축소 범위에 걸린 것만** stale + 대체 WP 생성 | 꺼진 능력 목록 + 표시 의무(UI/리포트) |
| `FAIL_BLOCKING` | 후손 착수 불가. 리그·자산·계약을 고쳐 재실행하는 것 외의 경로 없음 | 후손 **전량 stale + 미통합 작업 취소** | 차단 사유 + 해제 조건 |
| `SUPERSEDED` | 결과는 유효했으나 **입력이 바뀌어 무효**(SW SHA / 리그 프로파일 / 스키마 버전 / 상류 게이트 상태 변경) | 후손 **재검증 대기**로 표시(취소 아님) | 대체 레코드 URI |

### 2.2 전이 규칙

| 전이 | 트리거 | 강제 |
|---|---|---|
| `RETRY_WITH_VARIANT` → `PASS`/`DEGRADED_ACCEPTED`/`FAIL_BLOCKING` | 열거된 변형을 **전부** 소진 | 변형이 남았는데 `FAIL_BLOCKING`으로 가면 CI가 거부 |
| `*` → `SUPERSEDED` | 결과 레코드의 4대 입력 중 하나라도 변경(§3) | 자동. 사람·에이전트 판단 개입 없음 |
| `PASS` → `SUPERSEDED` → `PASS` | 재실행이 동일 판정 | 후손 재검증 해제 |
| `DEGRADED_ACCEPTED` → `PASS` | 변형·수정 후 재실행 | 꺼졌던 능력의 **복구 WP**가 자동 생성 |
| `*` → `FAIL_BLOCKING` | 정본 조건에서 판정선 미달 + 변형 소진 | 후손 자동 stale |

> 🔴 **"리스크 등록부에 한 줄 적고 넘어가기" 금지(스핀 §3).** 상태 변경은 **반드시** 3개를 동시에 만든다: ① 후손 아티팩트 stale 표시 ② 미통합 작업 취소 ③ **명명된 대체 WP 생성.** 셋 중 하나라도 없으면 그 상태 변경은 무효이며 CI가 거부한다.

### 2.3 후손 무효화 (자동 stale 전파)

```
게이트 상태 변경
  → 결과 레코드 URI를 입력으로 선언한 아티팩트 전수 조회 (06-추적성-레지스트리)
  → 직접 후손 stale
  → 그 후손을 입력으로 선언한 아티팩트 재귀 stale (전이적 폐포)
  → 미통합(unmerged) 워크플로우 산출물 취소
  → 대체 WP 생성 + 원 게이트 레코드 URI 역참조
```

**전파는 전이적이며 정지하지 않는다.** "이 후손은 영향이 적어 보인다"는 판단으로 전파를 끊는 것은 금지한다 — 영향 판정은 stale 해제 시점에 하는 것이지 전파 시점에 하는 것이 아니다.

**대가:** 이것은 **처리량을 나쁘게 만든다.** 상류 게이트가 흔들리면 무관해 보이는 하류까지 재검증 대기에 들어가 fan-out 폭이 일시적으로 줄어든다. 그 대가를 지불하는 이유는, 조용히 stale된 아티팩트 위에 쌓은 후속 작업이 **훨씬 나중에·훨씬 비싸게** 무너지기 때문이다.

## 3. 결과 레코드 — 무엇을 저장하는가

**모든 게이트 결과는 아래 4대 입력을 반드시 포함한다.** 하나라도 빠진 레코드는 CI가 거부하고, 그 게이트는 `PASS`가 될 수 없다.

| 필드 | 뜻 | 왜 필수인가 |
|---|---|---|
| `schemaVersion` | 결과 레코드 스키마 버전 | 스키마가 바뀌면 과거 레코드의 필드 의미가 달라진다 → 자동 `SUPERSEDED` 판정 근거 |
| `rigProfile` | 리그 프로파일(팔 구성·모터 ID 맵·CAN 비트레이트·카메라 세트·컴퓨트 타깃) | **이기종 플릿**(Jetson Nano/Orin/RTX 5090/A6000, 스핀 §7)에서 측정치는 리그에 종속. 리그 없는 수치는 무의미 |
| `swSha` | LeRobot 핀 commit SHA + 우리 코드 SHA + 자산(MJCF/URDF) SHA | LeRobot **v0.6.0** 정본, `pyproject.toml:28`의 `0.6.1`은 **유령 버전**(`16` §3.1) → 자칭 버전 문자열은 신뢰 불가, SHA만 신뢰 |
| `rawArtifactUri` | 원본 아티팩트 URI(`candump` 로그·히스토그램 원자료·`ip -s -d link show` 덤프·프레임 인덱스) | 요약 통계는 재판정 불가. 판정선이 바뀌면 **원본으로 재판정**한다 |

**부가 필수 필드:** `gateId`, `state`, `variantId`(변형 실행 시), `executionClass`, `conditionId`(정본 조건 식별자), `upstreamGateRecords[]`(상류 게이트 레코드 URI), `degradedCapabilities[]`(`DEGRADED_ACCEPTED` 시 필수).

### CI 강제

| 검사 | 실패 시 |
|---|---|
| **게이트 ID 유일성** — `PG-*`가 계획 전체에서 정확히 1회 정의 | 빌드 실패 |
| 🔴 **맨 `PG-RT-001` 사용 금지** — 게이트 선언 위치(게이트 표 ID 칸, WP의 게이트 필드)에서 `PG-RT-001a`/`PG-RT-001b` 접미사 없는 인용 | 빌드 실패 (§5.1.0 — 데드락 서술·원장 항목은 명시 예외) |
| 🔴 **`PG-RT-001a` 산출 `f_max_python`의 `provisional` 플래그 보존** — 플래그를 벗겨 인용하는 아티팩트 | 레코드 거부 (§5.1a) |
| 🔴 **안전 축 게이트의 판정선 도출 근거 URI 실재**(`PG-VEL-001`·`PG-STOP-001`) — URI가 **자기 결과 레코드**를 가리키면 자기 승인 | 레코드 거부 → `PASS` 불가 (§1.1) |
| 🔴 **`PG-STOP-001`의 `clockProvenance` 완비** — 방식(`KERNEL_TIMESTAMPING`/`INDEPENDENT_MARKER`) + 오프셋 추정 + 불확실성 구간. 서로 다른 두 시계의 스탬프를 정렬 근거 없이 뺀 흔적 | 레코드 거부 → **P99 발행 금지** (§5.7.0) |
| 🔴 **`PG-VEL-001` 스윕의 3구속 어서트** — 단일 관절 · 기계 구속 · 부트스트랩 리미터 아래 | 아티팩트 발행 거부 (§5.6.0) |
| 게이트 ID 참조 무결성 — 참조된 `PG-*`가 §5에 정의 존재 | 빌드 실패 (**`M-25` 유형의 dangling 참조 재발 방지, §4.2**) |
| 명세 매핑 실재성 — 인용된 FR/NFR ID가 `docs/spec/`에 실재 | 빌드 실패 |
| 4대 입력 완비 | 레코드 거부 → `PASS` 불가 |
| `DEGRADED_ACCEPTED`의 `degradedCapabilities[]` 비어있지 않음 | 레코드 거부 |
| 상태 변경 시 후손 stale + 대체 WP 동반 | 상태 변경 거부 |
| 금지 ID 사용 — `M-8` / `M-25`를 판정 근거로 인용 | 빌드 실패 (§4) |

## 4. 깨진 명세 ID 격리 — 계획이 `M-*`를 쓰지 않는 이유

### 4.1 🔴 `M-8` — 같은 ID, 두 개의 상반된 뜻

| 출처 | `M-8`의 뜻 | 실제 텍스트 |
|---|---|---|
| `16-미해결-이슈.md:103` | **"무조사 종결 — Quest 추적 정확도는 알려진 대로, 신경 쓰지 않는다"** | 근거: `DECISIONS-v3 M-8` |
| `15-비기능-요구사항.md:319, 116, 279` | **"LeRobot Python 제어 루프의 실측 사이클 타임 — 최우선. 다른 모든 주파수 NFR의 선행 조건"** | `NFR-PRF-054` / `NFR-PRF-055` / `NFR-PRF-004` / `NFR-PRF-040` / `NFR-PRF-053`이 전부 `M-8`을 지목 |

**왜 이것이 치명적인가.** `16`은 **결정 등록부**다 — 워크플로우가 "이 항목이 조사 대상인가"를 물을 때 찾아가는 문서다. 그 등록부에서 `M-8`을 조회한 워크플로우는 **"무조사 종결, 신경 쓰지 않음"** 을 읽고 종료한다. 그런데 같은 `M-8`이 `15`에서는 **프로젝트 최우선 측정**이자 *"모든 주파수 NFR의 뿌리"*(`15:279`)다. 즉 **등록부를 신뢰한 워크플로우는 프로젝트에서 가장 중요한 측정을 건너뛴다.** 이것은 문서 오타가 아니라 **자동 실행 경로의 함정**이다.

> **따라서: 계획은 `M-8`을 쓰지 않는다.** 제어루프 측정은 **오직 `PG-RT-001a`/`PG-RT-001b`**(§5.1)이며, 명세 매핑은 **`NFR-PRF-054` + `15` §2.10**이다. `16` §8의 `M-8`(Quest 추적 정확도)은 **계획의 게이트가 아니며 인용도 하지 않는다.** CI가 `M-8` 인용을 빌드 실패로 막는다(§3).

### 4.2 🔴 `M-25` — 정의 없는 유령 ID (dangling reference)

**`M-25`는 `16-미해결-이슈.md`에 정의가 없다.** `16`의 `M-*` 전수는 `M-1`~`M-4`, `M-8`, `M-10`~`M-13`, `M-16`, `M-17`, `M-20`~`M-24`(16개)이며 **`M-25`는 없다.** 그럼에도 4개 문서가 이를 정본으로 참조한다:

| 참조 위치 | 무엇을 `M-25`라 부르는가 | 어느 문서 소유라 주장하는가 |
|---|---|---|
| `01-시스템-아키텍처.md:213` | 16 vs 32 프레임 candump 실측 | **`15` M-25** |
| `01-시스템-아키텍처.md:561` (`NFR-SYS-002`) | 동일 | **`15` M-25** |
| `14-시스템-운영.md:317, 319, 908, 926` | 동일 | **`16` M-25** |
| `15-비기능-요구사항.md:89, 344, 420, 588` | 동일 (`§2.10` 조건 7) | **`16` M-25** |

### 5.1 🔴 `PG-RT-001a` / `PG-RT-001b` — LeRobot 제어루프 사이클타임 (**두 게이트**)

> **이 측정이 계획 전체의 뿌리다.** `15:279`: *"LeRobot Python 루프의 실측 사이클 타임이 모든 주파수 NFR의 뿌리다. 확정 전까지 어떤 목표도 못 박지 않는다."*

#### 🔴 5.1.0 왜 하나가 아니라 둘인가 — 기록된 데드락과 그 해소

**이전 판(단일 `PG-RT-001`)은 DAG를 영구 정지시켰다.** 순환은 정확히 이랬다:

```
Wave 1 퇴장  ──요구──→  PG-RT-001 PASS
     ↑                        │
     │                        │ PASS 조건 = 조건 4(실카메라·실라이터)
     │                        ↓
     │                   Wave 3C 도달 필요
     │                        │
     └────요구(선행)───────────┘   ← Wave 3C는 Wave 1 퇴장 없이는 도달 불가
```

이전 판이 **동시에** 주장한 두 문장이 서로를 잡아먹었다:
1. §5.0/§6.1 — **`PG-RT-001`은 Wave 1④의 퇴장 게이트**다(`WP-1-04` 소유).
2. 옛 §5.1 — *"**RT-canonical(3C) 없이는 `PG-RT-001`이 `PASS`가 될 수 없다**"* + *"RT-approx의 `PASS`는 존재하지 않는다"*.

즉 **Wave 1은 3C를 기다리고, 3C는 Wave 1을 기다린다.** 어떤 상태도 이 순환을 깨지 못한다 — `RETRY_WITH_VARIANT`는 변형을 소진하면 끝나고, `DEGRADED_ACCEPTED`는 조건 4 결과 없이는 무엇이 꺼지는지 명명할 수 없다.

> **해소: 게이트를 둘로 쪼갠다. 새 FSM 상태(`PROVISIONAL` 등)를 만들지 않는다.** 상태를 늘리면 15개 게이트 전부의 상태머신·CI 검사기·후손 전파 규칙이 바뀌지만, 게이트를 쪼개면 바뀌는 것은 **이 항목 하나**다. **싼 쪽을 택한다.**
>
> - **`PG-RT-001a`** — 기준 = **합성 GIL 부하 하네스**(`WP-0C-06` 산출). **잠정.** Wave 1 퇴장을 **허가한다.**
> - **`PG-RT-001b`** — 기준 = **실카메라 5스트림 + 무손실 PNG 쓰기 + 데이터셋 쓰기 + GUI 전송이 같은 프로세스**(`15` §2.10 조건 4 = 진짜 조건). **최종 정본.**
> - 🔴 **`a`의 `PASS`는 `b`가 `SUPERSEDED`로 무효화할 수 있다.** `a`는 거짓말할 권리를 가진 채로 통과하는 게이트이고, `b`는 그 거짓말을 회수하는 게이트다. 순환이 끊기는 지점이 정확히 여기다 — **`a`가 최종 정본을 자칭하지 않기 때문에** Wave 1이 퇴장할 수 있다.
>
> **대가: 이것은 Wave 2·3의 재작업 위험을 나쁘게 만든다.** `a`가 `PASS`한 뒤 `b`가 뒤집으면 Wave 2·3에서 단일프로세스 전제로 지은 아티팩트가 stale이 된다(§5.1b 하류 소비자). 그 대가를 지불하는 이유는 대안이 **아무것도 짓지 못하는 것**이기 때문이다 — 재작업 위험은 유한하고 데드락은 무한하다. 위험은 두 방향으로 관리한다: ① `a`의 합성 부하 4파라미터를 산출물에 못 박아 `b`와 **대비 가능**하게 만든다 ② `a`가 미달이면 그 신호만으로 워커 분리 설계 분기를 **앞당겨** 시작한다(근사가 이미 미달이면 실기는 반드시 미달).

| 항목 | 두 게이트 공통 |
|---|---|
| **무엇을 재는가** | `robot.get_observation()` + `safety_filter()` + `robot.send_action()` (+텔레옵이면 `teleop.get_action()`) 각 단계의 사이클 타임 분포(p50/p95/p99/max)와 **오버런율**. 단완/양완 각각, 각 10,000 사이클, `time.perf_counter_ns()` 기준 |

`15` §2.10(`15:344` 주변)의 조건표는 7개 조건을 정의하며, **조건 4에만 "🔴 이것이 진짜 조건이다"라는 표시가 붙어 있다**:

### 5.2 `PG-CAN-001` — 사이클당 CAN 프레임 16 vs 32 (**검증용 — BLOCKER 아님**)

| 항목 | 내용 |
|---|---|
| **무엇을 재는가** | `candump -t d can0`로 제어 루프 **한 사이클의 CAN 프레임 수**를 센다. 8+8+8+8=32(패턴 B)인가, 8+8=16(패턴 A)인가 |
| **명세 매핑** | `15` §2.10 조건 7(`15:344`) · `NFR-PRF-003`(`15:420`, M, [확정](원리)/[미확인](프레임 수)) · `NFR-PRF-047`(`15:515`, 패턴 A vs B 실험) · `NFR-SYS-002`(`01:561`) · `NFR-SAF-001`(`12:824`) · **`M-25` 아님 (§4.2)** |
| **WP 바인딩** | **소유 = `WP-1-04`**(읽기전용 측정 — `PG-RT-001a` 조건 7과 동일 세션·동일 WP). **하류 소비자 WP:** `WP-2B-05`(패턴 A 로거 — 16프레임/사이클 전제가 이 게이트 산출에 종속) |

#### 🔴 왜 BLOCKER가 아닌가 — U-1이 이 게이트에서 판정력을 제거했다

이 게이트는 명세에서 **625 Hz vs 1250 Hz 상한**을 가르는 것으로 기술된다(`15:89`: *"이것이 32 vs 16 프레임, 625 vs 1250 Hz를 가른다"*). 그러나 **그 두 상한 사이에서 갈리는 설계가 계획에 남아 있지 않다**:

| 대역 | 32프레임(패턴 B, 상한 ≈625 Hz)에서 | 16프레임(패턴 A, 상한 ≈1250 Hz)에서 | 갈리는가? |
|---|---|---|---|
| **주경로 30–250 Hz** (record·추론·평가) | 통과 (250 Hz = 부하 40%) | 통과 (250 Hz = 부하 20%) | **아니오** |
| **바이래터럴 500 Hz** | 부하 80% — 빠듯 | 부하 40% — 여유 | **U-1로 종결(OUT)** |
| **1 kHz 안전 루프** | 물리적 불가 (`01:561`: ≈160%) | 성립 안 함 (`01:561`: *"어느 경로에서도 1 kHz 안전 루프는 성립하지 않는다"*) | **아니오 — 양쪽 다 불가로 이미 확정** |

- **주경로는 어느 쪽이든 통과한다.** 30–250 Hz는 32프레임 상한 625 Hz의 40% 이하다.
- **1 kHz 안전 루프는 어느 패턴에서도 불가로 이미 확정**되어 있다 — `NFR-SYS-002`(`01:561`)가 **패턴 A(1250 Hz 상한)에서조차** 성립하지 않는다고 명기했다. 즉 이 측정 결과가 1 kHz 여부를 가르지 않는다.
- **500 Hz 바이래터럴이 유일하게 갈리던 대역이었으나 U-1로 종결**됐다 — 동력 리더암이 없다(`01:264`/`16:145` KER 무모터, Quest 진동만). 힘반사가 없으면 500 Hz 요구가 없고, 500 Hz 요구가 없으면 32 vs 16이 예산에서 갈리지 않는다.

> **따라서 `PG-CAN-001`은 검증용으로 강등된다.** 재는 이유는 아키텍처를 가르기 위해서가 아니라, **`15` §2.1의 프레임 예산 정본이 실제 코드경로와 일치하는지 확인**하기 위해서다(예산표가 틀렸다면 그 사실 자체가 다른 곳에서 문제를 낸다).
>
> **대가:** 이 강등은 **CAN 예산표에 대한 조기 경보를 나쁘게 만든다.** BLOCKER였다면 Wave 1에서 예산표 오류가 즉시 드러났을 것이나, 검증용이면 미달이 나와도 하류가 멈추지 않아 발견이 늦어질 수 있다. 그럼에도 강등하는 이유는, **아무 설계도 가르지 않는 측정을 BLOCKER로 두면 fan-out이 근거 없이 직렬화**되기 때문이다.

### 5.6 `PG-VEL-001` — 관절 속도 한계 **도출값 대비 검증** (한계 발견 아님)

> 🔴 **이 게이트는 한계를 발견하지 않는다.** §1.1 원칙 단서가 여기에 그대로 적용된다 — **속도는 안전 한계이지 성능 목표가 아니다.** 이전 판은 이 게이트를 "무부하 최대 속도를 실측해 한계를 찾는다"로 기술했고, 그것이 두 개의 결함을 낳았다:
>
> 1. **순환 의존**: 스윕이 한계를 찾는데(`WP-1-06`), 그 스윕을 안전하게 돌려면 리미터가 있어야 하고, 리미터(`WP-2A-04`)는 스윕 결과를 기다린다.
> 2. **관측의 자기 승인**: "실측 최대치"를 한계로 채택하면, 리그가 낼 수 있는 속도가 곧 허용 속도가 된다 — **안전 근거가 아니라 하드웨어 능력의 자백**이다.
>
> **해소: 리미터가 선행한다.** 보수 리미터를 **도출**해 먼저 세우고, 스윕은 **그 리미터 아래에서** 돈다. 게이트가 답하는 질문은 *"이 관절은 얼마나 빠른가"*가 아니라 🔴 **"도출한 보수값이 실제 리그에서 유효한가 — 그 아래에서 관절이 지령을 따르는가"**다.

#### 🔴 5.7.0 시작 이벤트는 `candump`로 찍을 수 없다 — 두 시계 문제

**이전 판은 측정 불가능한 것을 측정한다고 주장했다.** *"조그 버튼 릴리즈 시각과 CAN 정지 프레임을 `candump` 하드웨어 타임스탬프로 **동시** 측정"* — 이것은 성립하지 않는다:

> 🔴 **`candump`는 CAN 프레임에만 타임스탬프를 찍는다. 버튼 릴리즈는 CAN 프레임이 아니다.** 릴리즈는 CAN 버스에 아무것도 내보내지 않는다 — 그것이 SocketCAN 하드웨어 타임스탬프 영역에 들어오는 유일한 경로는 **이미 지연 경로를 통과한 뒤 정지 프레임으로 나타나는 것**뿐이다. 즉 옛 절차대로 재면 **Δt의 시작점이 곧 끝점**이 되어 상수 0에 가까운 수치가 나오거나, 릴리즈 시각을 **다른 시계**(Python `time.time()`, 브라우저 이벤트 시각)에서 몰래 가져와 CAN HW 타임스탬프와 빼게 된다. 🔴 **두 시계의 오프셋과 드리프트를 모르는 채 뺀 값은 지연이 아니다 — 그것은 날조다.**

**허용되는 시작 이벤트 출처는 둘뿐이다:**

| 방법 | 무엇을 | 왜 성립하는가 | 시계 비교 가능성 증명 |
|---|---|---|---|
| **(A) 커널 시계 계측** | 입력 이벤트를 **커널이 스탬프한 시각**으로 취한다(`evdev`/`input_event`의 커널 타임스탬프). SocketCAN HW 타임스탬프를 **같은 커널 시계 도메인으로 정렬**한다(`SO_TIMESTAMPING`의 HW↔시스템 시계 상관 삼중자 = `(sw_pre, hw, sw_post)`) | 두 스탬프가 **같은 커널이 소유하는 시계 도메인**에 놓인다 — 오프셋이 커널이 보고하는 양이 된다 | 🔴 **필수 산출물**: HW↔시스템 시계 오프셋 추정치 + 그 **불확실성 구간**. 이 구간이 관측 Δt와 같은 자릿수면 **P99 발행 금지** |
| **(B) 독립 전기/GPIO 마커** | 버튼 접점을 **분기**해 ① 시스템에 GPIO 인터럽트로 넣거나 ② 로직 애널라이저/오실로스코프로 **버튼 접점과 CAN 물리선(CAN_H/CAN_L)을 같은 장비의 같은 시간축에서** 동시 캡처 | 🔴 **시계가 하나다.** 비교 가능성 문제가 애초에 발생하지 않는다 — 가장 강한 증거이며 **(A)가 증명에 실패하면 유일한 경로** | 장비 채널 간 스큐 사양 + 트리거 정렬 캡처 원본 |

> 🔴 **수용 게이트에 박는다: 두 시계가 비교 가능함을 증명하지 못하면 P99를 발행하지 않는다.**
>
> - 결과 레코드는 **`clockProvenance` 필드 필수** — `KERNEL_TIMESTAMPING` | `INDEPENDENT_MARKER` 중 하나 + 시계 오프셋 추정치 + 불확실성 구간(또는 단일 시간축 증거).
> - 🔴 **이 필드가 없거나, 오프셋 불확실성이 관측 Δt와 같은 자릿수이거나, 서로 다른 두 시계의 스탬프를 정렬 근거 없이 뺀 흔적이 있으면 — 레코드 거부. `PASS` 불가. P99 미발행.**
> - **`P99 = 없음`은 유효한 산출물이다.** 이 게이트에서 발행 금지는 실패가 아니라 **정직**이다. 비교 불가능한 시계로 만든 수치를 발행하면 하류 전체(`NFR-MAN-002`·`NFR-GUI-005`의 [미확인]/[결정필요] 해제)가 **날조 위에** 얹힌다.
>
> **대가: 이것은 계측 리그를 나쁘게 만든다.** (A)는 `SO_TIMESTAMPING` 상관 삼중자를 다루는 하네스를 요구하고, (B)는 접점 분기와 캡처 장비를 요구한다 — 둘 다 `candump` 한 줄보다 무겁다. 그 대가를 지불하는 이유는, **정지 시간은 안전 축이고**(§1.1) 안전 축의 수치는 틀리면 늦게·비싸게 드러나는 것이 아니라 **팔이 떨어질 때 드러나기** 때문이다.
