# 05. 워크플로우 오케스트레이션

이 문서는 `00`~`04`·`06`이 **선언한** WP·게이트·웨이브를 Claude dynamic workflow 병렬 fan-out이 **실제로 어떻게 돌리는가**의 절차 규격이다 — fan-out 형상, worktree 격리의 한계, verify 루프, 블록/재개, 무효화 전파, 계약 위반 탐지, 무하드웨어 회귀, 인용 규율.

### 0.1 🔴 이 문서의 절차는 **Wave −2 `BOOT` 대역이 지은 기계 위에서 돈다**

## 1. WP 유형별 fan-out 형상

구현자는 사람이 아니라 **동시에 뜨는 Claude 워크플로우 인스턴스**다. 따라서 "누가 한다"가 아니라 **"몇 개가 동시에 뜨고, 각자 무엇을 배타로 쓰고, 무엇이 배리어인가"**만이 실행 변수다. 형상은 5종이며, 모든 WP는 정확히 하나에 속한다.

### 1.1 형상 **어휘** 정의 — **이 5종이 어휘 정본이다 (배정 정본이 아니다)**

> 🔴 **이 문서는 어휘를 소유하고 배정을 소유하지 않는다**(`00` §9.2a). *"`SHAPE-IM`이 무엇인가"*는 여기서 답하고, *"`WP-0C-02`가 `SHAPE-IM`인가"*는 **`02a`~`02d`**가 답한다. `01`의 형상 칸은 **인용**이며, 갈리면 카탈로그가 이긴다 — 대조는 `06` **CI-14b**가 기계로 한다. 이 구분이 없던 동안 `01`↔카탈로그가 **39개 중 14개** 어긋나 있었다(2026-07-16 리뷰).

> 🔴 **워크플로우 형상의 유일 어휘는 `SHAPE-CF` / `SHAPE-IM` / `SHAPE-IG` / `SHAPE-MS` / `SHAPE-HG` 5종이며, 이 문서가 소유한다.** `00`이 형상 정의를 이 문서에 위임했고, `02a`·`02b`·`02c`·`02d`·`06`은 전부 이 5종을 **그대로** 쓴다. 별칭도 매핑표도 없다 — 형상 칸에 5종 밖의 코드가 있으면 그것은 번역 대상이 아니라 **오류**다. 새 형상 코드를 만들지 마라.

| 형상 | 뜻 | 동시성 | 쓰기 배타 단위 | 종료 조건 | 실행클래스(`00` §4) |
|---|---|---|---|---|---|
| **`SHAPE-CF`** 계약동결 | 심볼·스키마·파일소유권을 **최초로 확정**한다 | **1 (병렬 금지)** | 계약 파일 + 그것이 정의하는 심볼 전량 | 계약 해시 발행 + 소비자 스텁 컴파일 통과 | `AI-offline` |
| **`SHAPE-IM`** 구현 | 동결된 계약을 **픽스처 대상별로** 채운다 | **N** (= 픽스처 대상 수) | 대상별 배타 파일 집합 1개 | 대상별 픽스처 그린 + 계약 해시 일치 | `AI-offline` |
| **`SHAPE-IG`** 통합 게이트 | N개 산출물을 **한자리에 모아** 교차 검증 | **1 (배리어)** | 통합 지점 파일 + 락파일 | 교차검증 전항 통과, 하나라도 실패 시 배리어 유지 | `AI-offline` |
| **`SHAPE-MS`** 측정 | 실물에서 **읽고 잰다.** 물리 조작 없음 | **1 (자원 배타)** | CAN writer 락 / 카메라 노드 / 세션 `connect()` | 게이트 산출 아티팩트 + 재현 메타데이터 | `AI-on-HW` |
| **`SHAPE-HG`** Human 게이트 | 워크플로우가 **블록**하고, 사람 행위 후 **재개** | **1 (블록)** | 물리 리그 전체 | 증거 패킷 수신 + 판정 기록 | `Human-assisted-HW` / `Human-judgment` |

### 1.1.1 형상 칸이 틀리기 쉬운 네 지점

5종 정의(§1.1)만 읽으면 형상 칸을 네 가지로 잘못 채운다. 각 오류는 **정보를 지운다** — 무엇이 지워지는지가 규칙의 근거다.

| # | 규칙 | 무엇이 지워지는가 |
|---|---|---|
| ① | **`SHAPE-MS`의 fan-out 폭은 1로 강제된다 — 선택이 아니라 물리다.** 측정은 CAN writer 락 / 카메라 노드 / `connect()` 세션이라는 **물리 자원 배타** 위에 서고, **리그는 하나**다. 측정 WP(`WP-0B-06`·`WP-0B-07`·`WP-1-04`)는 예외 없이 `SHAPE-MS`다 | 측정 WP를 `SHAPE-IM`으로 적으면 **리그가 하나라는 사실이 지워진다.** 스케줄러는 폭 N을 띄우려 하고, 조건 스윕이 순차라는 §1.2의 처리량 상한이 계획에서 사라진다 |
| ② | **verify 루프는 형상이 아니다.** §3의 verify 루프는 5종 **전부에 붙는 공통 절차**이며, 형상 축과 직교한다 | verify를 형상 칸에 적으면 **그 WP의 실제 형상이 비어 버린다.** 형상 칸은 여전히 `SHAPE-CF`/`IM`/`IG`/`MS`/`HG` 중 하나여야 하고, verify는 그 위에 얹히는 의무다 |
| ③ | **순차 서브-DAG는 형상이 아니라 DAG 배치**(`01` 소관)다. 순차 체인의 **각 노드가 자기 형상을 따로 갖는다** — 대개 `SHAPE-CF` 또는 `SHAPE-IM`(n=1) | "순차"를 형상 칸에 적으면 **노드별 형상이 지워진다.** 체인이라는 사실은 `01`의 DAG가 이미 인코딩하고 있으므로, 형상 칸에서 그것을 반복하면 진짜 정보만 잃는다 |
| ④ | **`SHAPE-CF` vs `SHAPE-IM`(n=1) 판정**: **새 심볼/스키마/파일소유권을 정의하면 `SHAPE-CF`**(예: `WP-0A-02` 스키마 소유자). **동결된 계약을 채우기만 하면 `SHAPE-IM`(n=1)**(예: `WP-1-02`→`WP-1-03` 순차 인계). 동시성이 1이라는 사실은 둘을 구별하지 못한다 | `SHAPE-CF`를 `SHAPE-IM`으로 오독하면 **계약동결 WP에 병렬 빌더가 붙어** 계약이 둘이 된다(§1.2). 반대로 오독하면 채우기만 하는 WP가 근거 없이 배리어가 된다 |

> **역사 기록 한 줄.** 병렬 작성 과정에서 세 문서가 각자 형상 코드(`02a`의 `S0`~`S5`, `02b`의 `solo-barrier`/`fan-out(n)`/`seq`/`human-gated`, `06`의 `WF-*`)를 유도했으나, 수리 라운드에서 **전면 치환**됐다 — 세 어휘 모두 **형상 선언 위치**(WP 표의 형상 칸)에서 0 히트이며 `SHAPE-CF`/`IM`/`IG`/`MS`/`HG` 5종이 유일 어휘다. 번역표는 존재하지 않고, 다시 만들지 않는다. **이 줄은 역사 서술이므로 그 0 히트 규칙의 명시 예외다** — 폐기 어휘를 산문에서 *언급*하는 것과 형상 칸에서 *사용*하는 것은 다르다. 폐기 사실은 `06` §1(접두사 표)이 함께 기록한다.

### 1.2 형상별 fan-out 사양

| 형상 | fan-out 폭을 결정하는 것 | 공유 상태 | 실패 시 격리 반경 | 이 형상이 나쁘게 만드는 것 |
|---|---|---|---|---|
| `SHAPE-CF` | **없다 — 항상 1.** 폭을 늘리는 순간 계약이 둘이 된다 | 없음(유일 라이터) | WP 1개 | **지연 병렬성.** 계약이 끝나기 전에는 하류 `SHAPE-IM`이 0개다. 척추 폭이 좁아지는 대가를 병렬화 총량에서 지불한다 |
| `SHAPE-IM` | **소유권이 분리 가능한 파일 집합의 수.** 대상 수가 아니라 *배타 쓰기 단위*의 수다 | 계약 해시(읽기전용) + 픽스처(읽기전용) | 대상 1개 | **통합 리스크의 후불화.** 각자 그린인데 합치면 빨간 상황이 `SHAPE-IG`까지 숨는다. 그래서 `SHAPE-IG`가 선택이 아니라 필수다 |
| `SHAPE-IG` | 없다 — 배리어 | N개 산출물 전량 | 웨이브 전체 | **직렬 구간의 실재.** 배리어 앞에서 N−1개 워크플로우가 놀고, 배리어가 깨지면 N개가 함께 stale이 된다 |
| `SHAPE-MS` | 없다 — **물리 자원이 하나다** | CAN 버스(`FR-OPS-029` flock writer 배타), `connect()` 세션 | 세션 전체 | **측정 처리량.** 조건 스윕이 순차가 된다. `NFR-PRF-055`가 "세션당 `connect()` 1회로 조건 1~7 전부"를 요구하는 이유가 이것이며, 이는 우회 불가다(반복 `connect()` = 영점 파괴, `FR-OPS-083`) |
| `SHAPE-HG` | 없다 — 블록 | 리그 전체 | 하류 전량 | **워크플로우 진행의 사람 종속.** 블록 중 다른 WP는 돌 수 있으나, 이 게이트의 후손은 **한 개도** 못 돈다 |

## 2. worktree 격리 — 쓸 수 있는 곳과 못 쓰는 곳

**worktree는 텍스트 충돌을 없앤다. 의미 충돌은 손도 못 댄다.** 이 구분이 Wave 0-Env를 단독 배리어로 만든다.

### 2.1 격리 가능성 판정표

| 공유 자원 | worktree로 격리되는가 | 왜 | 처리 |
|---|---|---|---|
| 대상별 소스 파일 (`lerobot_teleoperator_vr/`, `lerobot_camera_*/`, 화면별 SPA 컴포넌트) | ✅ | 파일 경로가 겹치지 않는다. 병합 = 파일 추가 | `SHAPE-IM` fan-out |
| 대상별 픽스처·골든 파일 | ✅ | 같음 | `SHAPE-IM` |
| 문서 (`docs/spec/` 인용, `docs/plan/`) | ✅ | 라인 단위 병합 가능 | 자유 fan-out |
| **`pyproject.toml`** | ❌ **의미 충돌** | 두 worktree가 각각 `[project.dependencies]`에 다른 버전 제약을 적으면 **텍스트는 깔끔히 병합되고 의존성 해가 달라진다.** git은 "둘 다 유효한 TOML"만 본다 | **Wave 0-Env 단독 배리어** |
| **락파일 (`uv.lock`/`poetry.lock`)** | ❌ | 락은 **해 전체의 스냅샷**이다. 두 worktree의 락을 병합하면 어느 쪽에서도 검증되지 않은 제3의 해가 나온다 | 동일 |
| **LeRobot commit SHA 핀** | ❌ | 핀이 갈리면 `docs/spec` 전체의 **라인 번호 인용이 두 기준을 갖는다**(`14` §5: "이 문서의 모든 라인 번호는 이 스냅샷 기준") | 동일. v0.6.0 SHA 핀, `0.6.1`은 유령(`14` §2.4) |
| **`register_third_party_plugins()` 엔트리포인트 네임스페이스** | ❌ | 두 worktree가 같은 배포명(`lerobot_robot_*`)을 등록하면 병합 후 **런타임에 late-import가 하나를 조용히 이긴다** | `SHAPE-CF`로 배포명 사전 동결 (`01` FR-SYS-014) |
| **CAN 인터페이스 / 물리 리그** | ❌ **물리적으로 하나** | flock은 협조 프로세스 간 합의일 뿐이다(`14` FR-OPS-029) | `SHAPE-MS`/`SHAPE-HG` 배타 |
| **모터 NV 영점 (`0xFE`)** | ❌ | worktree 개념 자체가 없는 상태 — **모터 안**에 산다 | `03` FR-MOT-045: 명시 플로우 외 전 경로 하드 차단 |
| **디스크 캘리브레이션 SoT** | ❌ | 파일은 하나이고 원자적 교체가 전제다(`02` FR-CON-064 `checksum`/`schema_version`) | 단일 라이터 |
| **CI 설정 · 계약회귀 검사기** | ❌ | 검사기가 둘이면 무엇이 게이트인지 정의되지 않는다 | Wave 0-Env |

### 2.2 그래서 Wave 0-Env는 **먼저, 혼자** 돈다

```
Wave −2 BOOT (실행 부트스트랩, 블로킹) ← §0.1. 여기 이전엔 아무것도 착수 못 한다.
   · registry/traceability.yaml 전량 + JSON Schema
   · WP 매니페스트 스키마 + 인덱스 생성기
   · CI 검사기 실물 (CI-01~CI-17)
   · 워크플로우 상태 저장소 + 취소 메커니즘 (latch-to-hold 포함)
   · 계약 해시 등재/동결 잠금 (CONTRACT_FROZEN)
        ↓  부트스트랩 수용 = 177 WP 등재 + CI-01~17 자기 코퍼스 green
Wave −1 (명세 정규화, 블로킹)
        ↓  정규화 해시
Wave 0-Env  ← 단독. 동시 WP 0개.
   · LeRobot v0.6.0 commit SHA 핀
   · 락파일 1개 확정
   · CI 파이프라인 + 계약회귀 검사기(§3.4) + 불변식 정적검사(§6.3)
        ↓  환경 해시 = {SHA, 락 해시, 검사기 버전}
Wave 0-A / 0-B / 0-C / 0-Ops  ← 여기서부터 fan-out
```

> `BOOT`이 CI **검사기 실물**을 낳고, Wave 0-Env가 그것을 **환경 해시에 묶는다.** 순서가 뒤집히면 0-Env는 존재하지 않는 검사기의 버전을 해시에 적게 된다.

| 규칙 | 내용 |
|---|---|
| **착수 금지** | 부트스트랩 수용(Wave −2 `BOOT`, §0.1) + 정규화 해시(Wave −1) + 환경 해시(Wave 0-Env)를 **셋 다 참조하지 못하는 WP는 착수하지 않는다.** 참조 없는 산출물은 병합 대상이 아니다 |
| **환경 해시 변경** | Wave 0-Env를 **다시 단독으로** 연다. 진행 중 `SHAPE-IM`은 전부 일시정지, 재개 시 새 해시로 픽스처 재실행 |
| **worktree 사용 조건** | ① 쓰기 대상이 §2.1에서 ✅인 것만, ② 그 목록을 WP 착수 시 **선언**, ③ 선언 밖 경로 수정은 CI가 거부(§6.1) |

**대가**: 이것은 **초기 병렬성을 0으로 만든다.** Wave 0-Env가 도는 동안 fan-out 폭은 정확히 1이고, 다른 모든 워크플로우는 대기한다. 그 대신 이후 모든 병합에서 "환경이 달라서 안 된다" 부류의 진단 불가능한 실패가 구조적으로 제거된다. **worktree를 더 공격적으로 쓰면 이 배리어를 앞당길 수 있다는 주장은 틀렸다** — 병합되는 것은 텍스트이지 의존성 해가 아니다.

## 3. verify 루프

### 3.1 3단 구조

```
① 생성        SHAPE-IM 워크플로우가 산출물 + 자기검증 결과를 낸다
       ↓  (생성기의 추론 컨텍스트를 넘기지 않는다)
② 적대적 검증  독립 워크플로우가 "이것을 어떻게 깨뜨리는가"만 수행
       ↓
③ 판정        게이트 상태머신 5상태 중 하나로 기록 (§5.1)
```

### 3.2 적대적 검증기의 계약

| 규칙 | 내용 | 이유 |
|---|---|---|
| **컨텍스트 차단** | 검증기는 생성기의 추론·자기검증 서술을 **받지 않는다.** 받는 것은 {산출물, 계약 해시, 인용된 명세 라인} | 생성기의 논거를 읽으면 검증기가 그 논거를 재확인하게 된다 |
| **반증 지향** | 산출물이 "맞다"를 확인하지 않는다. **실패 시나리오를 구성**한다: 구체 입력/상태 → 잘못된 출력/크래시 | "동작한다"는 판정이 아니다 |
| **명세 재인용 의무** | 생성기가 인용한 FR/NFR ID를 **파일을 열어 다시 확인**한다(§8). 요약을 신뢰하지 않는다 | 존재하지 않는 ID 인용이 실제 발생하는 실패 모드다 |
| **불변식 재검** | `00` §2 전역 불변식 7항을 산출물에 대해 개별 재검 | 영역마다 재구현 금지 = 영역마다 재검사 의무 |

### 3.3 다수결 — 쓰는 곳과 **금지되는 곳**

| 대상 | 다수결 | 근거 |
|---|---|---|
| 코드 구조·명명·리팩터 판단 | ✅ 허용 | 되돌릴 수 있다 |
| 명세 해석이 갈릴 때(어느 FR이 이기는가) | ✅ 허용하되 **Wave −1 정규화 원장에 등재** — 다수결 결과가 새 정본이 되므로 반드시 기록 | `01` §4.2 vs `02` §2.0.4 같은 실재 모순 |
| 픽스처 통과/실패 | ❌ **금지** — 결정적이다. 투표할 것이 없다 | 픽스처는 재현 가능해야 한다 |
| **모든 `PG-*` 게이트 판정** | 🔴 **금지** | 측정치는 투표 대상이 아니다. 값이 애매하면 판정이 아니라 **측정 조건**이 틀린 것이다 |
| **안전 판정** (`PG-SAFE-001`, `PG-RID-001`, 토크-ON 가부) | 🔴 **절대 금지** | 3개 워크플로우 중 1개가 "낙하 위험"이라 했으면 **소수의견이 이긴다.** 다수결은 그 1표를 지우는 기계다 |

**대가**: 안전 경로에서 소수의견이 이기게 하면 **거짓 양성 블록이 늘어난다** — 근거 없는 우려 1건이 웨이브 전체를 세울 수 있다. 이 비용은 지불한다. 반대 방향의 오류(다수결로 진짜 위험을 지움)는 `00` §2-5(홀딩 브레이크 없음 = 영구 물리 위험)에서 **되돌릴 수 없기** 때문이다.

**절차**:

| 단계 | 행위 |
|---|---|
| 1 | 업그레이드 후보 SHA로 **격리 환경**에서 `CR-a`~`CR-i` 전량 실행 |
| 2 | 하나라도 FAIL_BLOCKING → **환경 해시 갱신 차단.** 업그레이드는 여기서 끝난다 |

| 4 | 명세 갱신 착지 후 검사기 기준값 갱신 → 1로 복귀 |
| 5 | 전항 통과 시에만 환경 해시 갱신 → **모든 진행 중 `SHAPE-IM` 일시정지 + 픽스처 재실행**(§2.2) |

## 4. Human 게이트 — 블록과 재개

### 4.1 블록 프로토콜

`SHAPE-HG`는 워크플로우가 **사람에게 묻는 것이 아니라, 사람이 물리 세계에서 무언가를 한 뒤 그 증거를 받아 재개하는 것**이다. `00` §4가 요구하는 6항목 — {에이전트가 하는 일 / 사람이 하는 일 / 중단 조건 / 안전 초기상태 / 산출 증거 / 재개 지점} — 은 게이트 정의에 **전부** 있어야 하며, 하나라도 없으면 그 게이트는 열 수 없다.

| 단계 | 워크플로우 | 산출 |
|---|---|---|
| **HG-1 프리플라이트** | 안전 초기상태를 **스스로 검증**한다. 사람에게 확인시키지 않는다 | 프리플라이트 리포트 (`07` FR-REC-018 형태: 관절 리밋 실측·CAN 링크(`14` FR-OPS-084)·카메라 협상 포맷·토크 스위치 상태·`push_to_hub` 상태) |
| **HG-2 하네스 기동** | 사람이 개입할 창을 여는 하네스를 띄운다. **토크-ON은 `PG-SAFE-001` PASS 이후에만** | 하네스 PID + 세션 ID + `connect()` 1회 기록 |
| **HG-3 블록** | 실행을 멈추고 **요청 패킷**을 발행한다 (§4.2) | 요청 패킷 |
| **HG-4 사람 행위** | 워크플로우는 관측만 한다. 유도하지 않는다 | — |
| **HG-5 증거 수집·재개** | 증거 패킷 검증 → 게이트 상태 기록 → **재개 지점**에서 계속 | 증거 패킷 + 게이트 상태 |

**중단 조건은 블록 전에 정의된다.** 블록 중 조건 충족 시 워크플로우는 사람의 응답을 기다리지 않고 **하네스를 안전 상태로 되돌린다** — 단, `00` §2-3(CAN 명령 스트림 SW가 끊지 않음)에 따라 "안전 상태"는 **홀드**이지 스트림 중단이 아니다.

## 5. 게이트 실패 시 후손 무효화 전파

### 5.1 상태머신 (전 게이트 공통)

| 상태 | 뜻 | 후손 처리 | 대체 WP |
|---|---|---|---|
| `PASS` | 판정선 충족 | 해제 | — |
| `RETRY_WITH_VARIANT` | 조건/파라미터를 바꿔 재측정 | **동결 유지** (stale 아님 — 아직 무효가 아니다) | 변종 측정 WP |
| `DEGRADED_ACCEPTED` | 미달이나 축소 스코프로 수용 | **후손 재검증 필수** + 축소된 계약 재발행 | 스코프 축소 WP + 재검증 WP |
| `FAIL_BLOCKING` | 후손 진행 불가 | **전량 stale + 미통합 작업 취소** | 명명된 대체 WP |
| `SUPERSEDED` | 다른 게이트/결정이 이 게이트를 대체 | 후손을 **새 게이트로 재부모화** | 재부모화 WP |

### 5.2 전파 절차 (`FAIL_BLOCKING` / `DEGRADED_ACCEPTED`)

| 단계 | 행위 | 산출 |
|---|---|---|
| **P-0 🔴 하드웨어 래치 (실행클래스 `AI-on-HW`·`Human-assisted-HW`만)** | 폐포 계산도 목록화도 기다리지 않는다. **즉시 `ActuationScheduler`에 안전래치 홀드를 건다**(latch-to-hold) — 게이트가 무효화된 순간 팔이 **움직이고 있을 수 있다**. 진행 중 `SHAPE-MS`/`SHAPE-HG` 하네스는 그 즉시 홀드 하위로 떨어진다 | 래치 시각 + 래치 사유 `{gateId, 이전상태, 새상태}` |
| **P-1 정지** | 해당 게이트의 **후손 전량**에서 새 fan-out 발행 중단. 진행 중 워크플로우 정지 규율은 **실행클래스가 가른다**(아래 §5.2.1) | 정지 목록 |
| **P-2 후손 폐포 산출** | `06-추적성-레지스트리.md`의 `게이트 → 하류 소비자` 간선으로 **이행적 폐포**를 계산. 직접 후손이 아니라 폐포다 | 후손 집합 |
| **P-3 아티팩트 stale 표시** | 폐포 내 전 아티팩트에 `staleReason = {gateId, 이전상태, 새상태, 전파 시각}` 부착. **삭제하지 않는다** — 대체 WP가 무엇이 왜 죽었는지 읽어야 한다 | stale 마킹 |
| **P-4 미통합 작업 취소** | 병합되지 않은 worktree 브랜치 취소. **병합된 것은 취소가 아니라 되돌림 WP** — 취소와 되돌림은 다른 절차다 | 취소 목록 |
| **P-5 데이터 오염 판정** | 게이트가 **데이터 정체성**에 걸리면(`CR-b`/`CR-e`/`CR-f`/`CR-g`, `PG-J7-001`, 영점 파괴) 그 게이트 이후 **기록된 에피소드까지** 폐포에 포함 | 오염 에피소드 목록 |
| **P-6 대체 WP 명명 생성** | 게이트의 **음성 분기**(`03-측정-게이트.md` 소유)가 지정한 대체 WP를 실제로 만든다. **"리스크 등록부에 한 줄 적고 넘어가기 금지"** | 대체 WP |
| **P-7 상태 기록** | 게이트 상태 + 전파 결과를 추적성 레지스트리에 기록. CI가 **stale 아티팩트에 의존하는 WP 착수를 거부**(§6) | — |

### 5.2.1 🔴 취소 규율은 **실행클래스별로 갈린다** — 전역 단일 절차가 아니다

"진행 중 워크플로우는 현 스텝을 끝내고 정지한다"는 **컴파일에는 옳고 모션에는 위험하다.** 안전·RID·정지 게이트가 무효화된 그 순간, 그 게이트를 근거로 토크-ON된 팔이 스텝 중간에 **움직이고 있다.** "현 스텝"을 끝내게 허용하는 것은 **무효화된 근거 위에서 모션을 계속 시키는 것**이다. 그래서 취소는 실행클래스로 가른다.

| 실행클래스(`00` §4) | 취소 규율 | 근거 |
|---|---|---|
| **`AI-offline`** | ✅ **현재 스텝 완주 허용.** 그 뒤 정지 | 중간 중단은 **부분 산출물**을 남긴다 — 반쯤 쓴 파일, 반쯤 생성된 픽스처. 물리 위험이 0이므로 부분 산출물 쪽이 더 비싼 실패다 |
| **`AI-on-HW`** | 🔴 **즉시 스케줄러 latch-to-hold**(P-0). 스텝 완주 허용 **없음**. 우아한 브랜치 취소(P-4)는 **래치가 걸린 뒤에** | 읽기·측정이라도 **세션은 토크-ON 상태일 수 있다**(`PG-RT-001a/b` 측정은 명령 스트림 위에서 돈다). 무효화된 게이트 위에서 스트림이 계속 도는 것을 허용하지 않는다 |
| **`Human-assisted-HW`** | 🔴 **즉시 스케줄러 latch-to-hold**(P-0). 그 뒤 §4.1 HG 블록 프로토콜의 중단 경로로 넘긴다 | 사람이 **팔 근처에 있다.** 이 클래스에서 "스텝 완주"는 사람이 잡고 있는 동안 무효화된 근거로 모션을 계속한다는 뜻이다 |
| **`Human-judgment`** | ✅ 완주 허용 (라벨링·판정 — 물리 조작 없음) | `AI-offline`과 동일 |

| 규칙 | 내용 |
|---|---|
| **순서가 규칙이다** | **래치 먼저, 취소 나중.** P-0 → P-1 → … → P-4. 폐포 계산(P-2)이 오래 걸린다는 사실이 래치를 늦추는 근거가 되어서는 안 된다 — P-0은 폐포를 **입력으로 받지 않는다**(실행클래스만 본다) |
| **래치 ≠ 스트림 중단** | latch-to-hold는 `00` §2-1의 **안전래치 홀드 방출**이다. 스케줄러는 계속 매 틱 방출한다. `00` §2-3(CAN 명령 스트림 SW가 끊지 않음)은 여기서도 유효하다 — **취소가 낙하를 만들면 안 된다** |
| **래치 해제는 자동이 아니다** | 홀드 래치 해제 = **명시적 재무장 핸드셰이크**(새 세대 ID + 오퍼레이터 의도 확인). 게이트가 나중에 `PASS`로 바뀌었다는 사실만으로 모션이 재개되지 않는다 |
| **실행체** | P-0의 래치 경로는 `BOOT`의 **워크플로우 상태 저장소 + 취소 메커니즘**에 산다(§0.1). 워크플로우별 지역 구현 금지 — 지역 구현은 래치를 못 건 워크플로우가 하나 생기는 순간 무의미하다 |

**대가**: `AI-on-HW`를 즉시 래치로 만들면 **측정 세션이 스텝 중간에 끊기고, 그 세션의 부분 측정치는 버려진다** — `SHAPE-MS`는 `connect()` 세션당 1회라는 §1.2의 제약 위에 있으므로, 래치 1회가 **조건 스윕 전체의 재실행**을 뜻할 수 있다. 이 비용은 지불한다. 반대 방향의 오류(무효화된 안전 근거 위에서 스텝을 마저 돌림)는 `00` §2-5(홀딩 브레이크 없음 = 영구 물리 위험)에서 **되돌릴 수 없다.**

## 6. 계약 위반 탐지 — 두 워크플로우가 같은 것을 다르게 만들 때

병렬 fan-out의 고유 실패는 "둘 다 그린인데 합치면 정의가 둘"이다. worktree는 이것을 **감추는 쪽으로** 작동한다(§2). 따라서 탐지는 CI에만 있다.

### 6.1 소유권 레지스트리 검사 (`XV-OWN`)

### 6.3.1 🔴 직접-버스 예외 등록부 — **닫힌 목록. 토크-ON 항목은 없고, 추가할 수 없다**

`00` §2-1은 토크-ON부터 토크-OFF까지 `ActuationScheduler`가 **유일 CAN 라이터**라고 못 박는다. 이 등록부는 그 불변식의 **유일한 구멍**이므로, 구멍의 모양이 곧 안전 경계다.

| 등재 예외 | 실행클래스 | 토크 상태 | 송신 | 근거 |
|---|---|---|---|---|
| `PG-RT-001a` / `PG-RT-001b` 조건 2 (패턴 A 실험) | `AI-on-HW` | 🔴 **토크-OFF 전용** | 송신 有 (패턴 A가 `robot.bus`를 직접 다뤄야 한다) | `15` §2.10 |
| 하드 E-Stop 경로 | — | — | — (전원+CAN 동시 차단) | `16` M-2 · 불변식 3 예외 |

> 🔴 **등재 규칙: 토크-ON 항목을 이 등록부에 추가하지 마라.** 유일하게 허용되는 송신 예외는 **토크-OFF 상태인 `PG-RT-001a`/`PG-RT-001b` 실험뿐**이다. 토크-ON 상태에서 스케줄러 밖의 송신자가 하나라도 생기면 **CAN 라이터가 둘이 되고**, 스케줄러가 정지하는 순간 그 팔은 **떨어진다**(홀딩 브레이크 없음, `00` §2-5). 이것은 "예외를 신중히 심사한다"가 아니라 **심사 대상이 아니다** — 등록부에 토크-ON 항목을 추가하는 PR은 근거의 질과 무관하게 거부된다.

**연동: 1kHz 마찰 로깅은 예외가 아니다.** `WP-2B-05`의 마찰 동정 로깅이 `robot.bus`를 직접 잡으면 그것이 곧 **토크-ON 상태의 두 번째 라이터**다. 허용 경로는 둘뿐이고, **둘 다 송신하지 않는다**:

| 경로 | 형태 | 로깅 레이트 |
|---|---|---|
| **(a) 스케줄러 내부 탭** | 스케줄러가 **자기 틱에서** pos/vel/tau를 로그로 흘린다. 새 소켓도 새 송신자도 없다 | = 스케줄러 틱 레이트 (`PG-RT-001a` 산출 `f_max_python` 종속) |
| **(b) 완전 수동 RX 탭** | 별도 **read-only SocketCAN 소켓**. RX만 한다 — **송신 0** | 버스 트래픽 종속 |

**`WP-2B-05`의 `robot.bus` 직접 접근은 금지된다.** `XV-INV` 불변식 1(스케줄러 외 모듈의 버스 write 심볼 참조 → 거부)이 이것을 기계로 잡으며, 마찰 로깅은 §6.3.1 등록부에 **등재되지 않는다** — 등재할 필요가 없기 때문이다. 두 허용 경로 모두 write 심볼을 건드리지 않는다.

**대가**: `XV-INV`는 **정당한 예외를 거부한다** — 패턴 A 측정 하네스와 E-Stop 경로가 즉시 그 예다. 예외는 코드 주석이 아니라 **명시 등록부**로만 인정하며, 등록 없는 예외는 위반이다. 이것은 **하네스 작성을 번거롭게 만든다.** 대신 "이 파일만 특별히"라는 주장이 리뷰 없이 통과하는 경로가 없어진다. 마찰 로깅에 지불하는 구체적 대가는 **로깅 레이트가 스케줄러 틱에 묶인다는 것**(경로 a) 또는 **송신 타이밍과 로그가 같은 시계를 공유하지 않는다는 것**(경로 b)이며, 식별 대역이 그만큼 좁아진다. 그 대가로 낙하 경로 하나가 **구조적으로 존재하지 않게** 된다.

## 8. 명세 인용 규율

**요약은 가설이다. 파일을 열어 라인을 읽은 것만이 근거다.** 이 규율은 취향이 아니라, 이 명세가 **LeRobot 내부 구현에 라인 번호까지 의존**하고(`14` FR-OPS-089) **공식 문서가 소스와 어긋난다**는 확인된 사실(`07` §2.8: `openarm.mdx`의 record 예제는 `--repo-id`를 쓰나 실제 `RecordConfig`는 `--dataset.repo_id`이고, Configuration Options 표에 `use_velocity_and_torque`가 **grep 0회**) 위에 서 있다. 명세 자신이 결론을 이미 적었다 — **"이 문서는 문서가 아니라 소스를 인용한다."**

### 8.1 인용 등급

| 등급 | 형태 | 허용 |
|---|---|---|
| **A. 라인 인용** | `파일:라인` + 그 라인이 무엇을 하는지 | ✅ 근거로 사용 가능 |
| **B. ID 인용** | `FR-<영역>-nnn` + `문서:섹션` — **해당 문서를 열어 그 행을 읽은 뒤에만** | ✅ 근거로 사용 가능 |
| **C. 요약 인용** | 다른 워크플로우/서브에이전트의 요약, grep 히트 수 | ❌ **근거 아님.** 어디를 열어볼지 알려줄 뿐 |
| **D. 기억 인용** | 컨텍스트에 없는 ID를 그럴듯하게 적음 | 🔴 **금지.** 이것이 이 계획의 최악 실패 모드다 |

### 8.2 절차

| 단계 | 행위 |
|---|---|
| 1 | grep으로 **열거**한다 → 등급 C. 여기서 멈추면 근거가 없다 |
| 2 | 각 히트 주변을 **읽는다** → 등급 A/B |
| 3 | 인용 시 **무엇을 읽었는지 진술**한다: "`14`:726 FR-OPS-089를 읽었고 (e)항이 `send_action`의 tau 하드코딩 재검사를 요구한다" — "grep에서 20건 나왔으니 쓰인다"는 진술이 아니다 |
| 4 | 적대적 검증기가 **같은 파일을 다시 연다**(§3.2). 검증기는 생성기의 인용을 신뢰하지 않는다 |
| 5 | `CR-i`(§3.4)가 라인 인용의 유효성을 환경 해시 갱신 시마다 재검사 |

### 8.4 명세가 스스로 모순일 때
